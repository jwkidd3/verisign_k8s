📜 The Kubernetes API — Your Contract Interface

The API Server is the single point of interaction

Every interaction flows through the API server — kubectl, CI/CD pipelines, controllers, and the scheduler all speak the same API.

# Every developer action is an API call
kubectl apply -f deployment.yaml    # POST/PUT to /apis/apps/v1/deployments
kubectl get pods                    # GET to /api/v1/pods
kubectl logs my-pod                 # GET to /api/v1/pods/my-pod/log
kubectl port-forward svc/myapp 8080 # Upgrade to SPDY/WebSocket
Why this matters: The API is versioned, documented, and consistent. Your manifests from today will work on future clusters because Kubernetes guarantees API compatibility.
The API server is the front door to everything in Kubernetes. Understanding that kubectl is just an HTTP client helps developers reason about what's happening under the hood. The versioned API is a stability guarantee — you're writing against a contract, not an implementation.

metadata:
  name: myapp
  labels:

🔗 Services — Stable Networking

apiVersion: v1
kind: Service
metadata:
  name: myapp
spec:
  selector:
    app: myapp
  ports:
  - port: 80
    targetPort: 8080
    protocol: TCP
  type: ClusterIP
DNS auto-registered:
myapp.default.svc.cluster.local

Why Services exist:

  • Pods have dynamic IPs — they change on restart
  • Services provide a stable virtual IP
  • Built-in load balancing across pod replicas
  • DNS-based discovery — no hardcoded IPs
  • Selector connects Service to Pods via labels
Think of it as: A stable address that always routes to healthy pods.
The Service is how pods find and talk to each other. The selector is the glue — it matches labels on pods. Kubernetes automatically updates the endpoints as pods come and go. This is the foundation of service discovery, which we'll cover in depth in Module 4.

🔄 Two Ways to Interact with Kubernetes

🚨 Imperative (Avoid in Production)

# Step-by-step commands
kubectl create deployment myapp \
  --image=myapp:v1
kubectl scale deployment myapp \
  --replicas=3
kubectl expose deployment myapp \
  --port=80 --target-port=8080
kubectl set image deployment/myapp \
  myapp=myapp:v2

Tells Kubernetes what to do step by step.

✅ Declarative (Best Practice)

# Declare desired state
kubectl apply -f deployment.yaml
kubectl apply -f service.yaml

# Update? Change the YAML, re-apply
kubectl apply -f deployment.yaml

# Everything is in version control
git diff deployment.yaml

Tells Kubernetes what you want — it figures out how.

The declarative approach is fundamental to Kubernetes' power. With imperative commands, you lose the ability to audit, diff, review, and version control your infrastructure. Declarative manifests are code — they go in Git, they get reviewed in PRs, and they serve as documentation of your system's desired state.

🏆 Why Declarative Wins

🔎 Auditability

  • Git history shows every change
  • PR reviews for infra changes
  • Blame shows who changed what

🔄 Reproducibility

  • Same YAML = same result
  • Recreate environments from Git
  • Disaster recovery is a kubectl apply

🤖 Automation

  • GitOps workflows (ArgoCD, Flux)
  • CI/CD pipelines apply manifests
  • Drift detection built-in
The reconciliation loop: Kubernetes continuously compares desired state (your YAML) to actual state (what's running) and takes action to converge. This is the superpower.
The reconciliation loop is the heart of Kubernetes. It's not a one-time "run this command" — it's a continuous process. If a pod crashes, the controller sees the discrepancy and creates a new one. This is self-healing infrastructure, driven by your declarative manifests.

📋 Environment Parity with Kubernetes

The Promise

The same container image runs identically in every environment. Configuration is injected, not baked in.

What stays the same

  • Container image (immutable artifact)
  • Application code and dependencies
  • Runtime (JVM, Node, Python version)
  • OS-level packages

What changes per environment

  • ConfigMaps (feature flags, URLs)
  • Secrets (credentials, API keys)
  • Resource requests/limits
  • Replica counts
  • Ingress rules / hostnames
This is a direct implementation of the 12-factor app principle of dev/prod parity. The image is your immutable artifact — it's the same bytes in every environment. Only the configuration changes. Kubernetes makes this natural with ConfigMaps and Secrets.

🛠️ Environment Parity in Practice

# Directory structure for multi-environment manifests
k8s/
├── base/
│   ├── deployment.yaml        # Common Deployment spec
│   ├── service.yaml           # Common Service spec
│   └── kustomization.yaml     # Base kustomization
├── overlays/
│   ├── dev/
│   │   ├── kustomization.yaml # Dev-specific patches
│   │   ├── replicas-patch.yaml
│   │   └── configmap.yaml
│   ├── staging/
│   │   ├── kustomization.yaml
│   │   ├── replicas-patch.yaml
│   │   └── configmap.yaml
│   └── prod/
│       ├── kustomization.yaml
│       ├── replicas-patch.yaml
│       └── configmap.yaml
# Deploy to each environment
kubectl apply -k k8s/overlays/dev/
kubectl apply -k k8s/overlays/staging/
kubectl apply -k k8s/overlays/prod/
Kustomize is built into kubectl and is the standard approach for managing environment-specific configurations. The base contains the common definition, and overlays patch only what differs. This ensures environments stay in sync while allowing necessary differences like replica counts and resource limits.

🎯 12-Factor App Principles

How Kubernetes embodies modern application design

The 12-factor app methodology predates Kubernetes, but Kubernetes was essentially designed to run 12-factor apps. Let's map the principles to Kubernetes features.

📚 12-Factor Principles & Kubernetes

Factor Principle Kubernetes Implementation
I Codebase One repo, one container image
II Dependencies Bundled in container image
III Config ConfigMaps, Secrets, env vars
IV Backing Services Services, ExternalName, endpoints
V Build, Release, Run Image build → tag → Deployment
VI Processes Stateless pods, external state
This is the first half of the 12 factors. Notice how each one maps cleanly to a Kubernetes concept. This isn't a coincidence — Kubernetes was designed by people who deeply understood these principles. If your application follows 12-factor design, it will thrive on Kubernetes.

📚 12-Factor Principles & Kubernetes (cont.)

Factor Principle Kubernetes Implementation
VII Port Binding containerPort + Service
VIII Concurrency Horizontal scaling (replicas, HPA)
IX Disposability Fast startup, graceful shutdown (preStop)
X Dev/Prod Parity Same image + Kustomize overlays
XI Logs Write to stdout/stderr, collected by platform
XII Admin Processes Jobs, CronJobs, kubectl exec
Design implication: Applications that follow these principles are "Kubernetes-native" — they leverage the platform fully and avoid fighting it.
Factor IX — Disposability — is particularly important. Kubernetes will kill and reschedule your pods. If your application can't handle graceful shutdown (SIGTERM), you'll have problems. Factor XI — Logs — means your app should never write to log files inside the container. Write to stdout/stderr and let the platform handle collection.

🚫 Factor IX Deep Dive — Disposability

spec:
  terminationGracePeriodSeconds: 30
  containers:
  - name: myapp
    image: myapp:v1.2.3
    lifecycle:
      preStop:
        exec:
          command: ["/bin/sh", "-c", "sleep 5"]  # Allow in-flight requests to drain
    # When Kubernetes stops a pod:
    # 1. Pod marked as Terminating
    # 2. Removed from Service endpoints
    # 3. preStop hook runs
    # 4. SIGTERM sent to PID 1
    # 5. Wait terminationGracePeriodSeconds
    # 6. SIGKILL if still running
Common mistake: Not handling SIGTERM in your application. If your process doesn't gracefully shut down on SIGTERM, in-flight requests will be dropped.
The pod termination lifecycle is critical for zero-downtime deployments. The preStop sleep of 5 seconds gives kube-proxy and ingress controllers time to update their routing tables before the pod stops accepting traffic. This prevents dropped connections during rolling updates.

🎯 Environment Parity

The Twelve-Factor Principle: Dev/Prod Parity

Keep development, staging, and production as similar as possible. Kubernetes makes this achievable by standardizing the runtime environment.

💻 Dev

  • Same container images
  • Same K8s manifests (different values)
  • Namespace isolation

🛠️ Staging

  • Mirrors production topology
  • Same network policies
  • Realistic resource quotas

🌍 Production

  • Proven images from pipeline
  • Full observability stack
  • Autoscaling enabled
The key insight is that Kubernetes manifests plus Kustomize or Helm overlays let you keep a single source of truth with environment-specific overrides. We'll cover Helm and Kustomize in detail in a later module.

🔧 Self-Service Infrastructure

Shift Left: Developers can provision what they need without filing infrastructure tickets — within guardrails set by the platform team.
  • Namespaces — Isolated environments on shared clusters
  • Resource Quotas — Guaranteed resource allocation per team
  • RBAC — Scoped permissions for deploy, debug, and observe
  • PVCs — Request storage dynamically through StorageClasses
  • Ingress rules — Expose services without touching load balancers
The result: Faster iteration cycles. Developers deploy in minutes, not days.
This is particularly relevant in enterprise environments like Verisign, where the platform team sets the guardrails (quotas, network policies, PSA) and developers operate freely within them.

🔁 Reproducible Deployments

What Makes It Reproducible

  • Immutable container images — same bytes in every environment
  • Declarative manifests — desired state is version-controlled
  • Helm/Kustomize — parameterized templating
  • GitOps — Git is the single source of truth

💡 Why It Matters

  • Rollback is trivial: git revert + sync
  • Audit trail for every change
  • Disaster recovery = reapply manifests
  • New team members can deploy day one
Reproducibility is the foundation of trust in your deployment pipeline. When a deployment is fully described in code and driven by GitOps, you eliminate the "what changed?" question during incidents.

📊 Resource Management: The Developer's View

You Define the Contract, K8s Enforces It

Resource requests guarantee capacity. Resource limits cap consumption. Getting this right is a developer responsibility.

Requests

resources:
  requests:
    cpu: "250m"
    memory: "256Mi"

Scheduler uses this to place your Pod. Underestimate = potential eviction. Overestimate = wasted cluster capacity.

Limits

resources:
  limits:
    cpu: "1"
    memory: "512Mi"

Hard ceiling. Exceeding memory limit = OOMKilled. CPU is throttled, not killed.

Resource management is often treated as an ops concern, but it directly affects developer experience. OOMKilled pods and CPU throttling are developer problems. We'll dive deeper into VPA and HPA in Module 2.

📐 Quotas and LimitRanges

Platform guardrails that protect everyone — including you from noisy neighbors.

ResourceQuota

Per-namespace caps on total CPU, memory, object counts

apiVersion: v1
kind: ResourceQuota
metadata:
  name: team-quota
spec:
  hard:
    requests.cpu: "10"
    requests.memory: 20Gi
    pods: "50"

LimitRange

Per-pod/container defaults and bounds

apiVersion: v1
kind: LimitRange
metadata:
  name: defaults
spec:
  limits:
  - default:
      cpu: "500m"
      memory: "256Mi"
    type: Container
Developers need to understand quotas because they'll encounter them as deployment errors — "exceeded quota" messages. Knowing what quotas exist in your namespace and how to right-size your workloads is essential.

📸 Lab 1

Exploring Your Kubernetes Environment

Objective: Verify your cluster access, explore the environment, and observe the Pod lifecycle firsthand.

  • Verify cluster access with kubectl cluster-info and kubectl get nodes
  • Explore existing namespaces and their resource quotas
  • Deploy a sample application using a Deployment manifest
  • Expose it with a Service and verify connectivity
  • Examine the Pod lifecycle: Pending → ContainerCreating → Running
  • Inspect events, logs, and resource consumption with kubectl describe and kubectl logs
This lab ensures everyone has a working environment and is comfortable with kubectl. It also sets the baseline for the more complex labs to come. Allow about 30-40 minutes. Circulate and help anyone with access issues early.
