CI/CD Fundamentals for Kubernetes

From commit to cluster

Before diving into specific tools, we need to establish a shared mental model for what a deployment pipeline looks like when the target is Kubernetes rather than a traditional VM or bare-metal server.

🛠 The Deployment Pipeline

A deployment pipeline is the automated path code travels from a developer's commit to a production workload.

Build

  • Compile / bundle code
  • Build container image
  • Push to registry

Test

  • Unit tests
  • Integration tests
  • Security scanning

Deploy

  • Apply manifests to cluster
  • Validate rollout health
  • Promote across environments
The three major phases are build, test, and deploy. In the Kubernetes world, the build phase produces a container image rather than a JAR or binary artifact. Testing includes not just unit and integration tests but also image vulnerability scanning. Deployment means applying manifests -- Deployments, Services, ConfigMaps -- and confirming the rollout succeeds.

⚡ Why Automation Matters

Manual Deployments

  • Error-prone kubectl commands
  • No audit trail
  • Inconsistent between environments
  • "It worked on my machine"

Automated Pipelines

  • Repeatable and deterministic
  • Full audit log of every change
  • Enforced quality gates
  • Fast feedback to developers
Automation is not optional at scale -- it is a prerequisite for reliability.
Teams running Kubernetes without automated pipelines eventually hit a wall. Manual kubectl apply commands do not scale, they create drift between environments, and there is no record of who deployed what. Automation gives you speed, consistency, and traceability.

🔄 Push-Based vs Pull-Based Deployment

Push-Based (CI/CD) Pull-Based (GitOps)
Trigger Pipeline pushes to cluster Agent in cluster pulls from Git
Credentials CI needs cluster access Agent already has cluster access
Drift Detection Not built-in Continuous reconciliation
Example Tools Jenkins, GitHub Actions, GitLab CI FluxCD, ArgoCD
Rollback Re-run previous pipeline git revert
Key insight: Many teams combine both models -- CI builds and tests, then GitOps deploys.
This is the most important architectural distinction in this module. Push-based pipelines like Jenkins actively apply changes to the cluster. Pull-based GitOps agents live inside the cluster and continuously reconcile desired state from a Git repository. The hybrid approach is increasingly common: CI handles build and test, then commits the updated manifest to a Git repo that a GitOps agent watches.

Jenkins CI/CD Pipeline Integration

Building and deploying with Jenkins

Jenkins remains one of the most widely deployed CI/CD servers in the enterprise. Understanding how to integrate Jenkins with Kubernetes is valuable even if your organization is also evaluating GitOps tools, because Jenkins often handles the build and test phases in hybrid architectures.

🧑‍🛠 Jenkins in the Kubernetes Ecosystem

  • Jenkins can run inside Kubernetes as a Deployment
  • Dynamic agents spin up as Pods, then terminate
  • Kubernetes plugin manages agent lifecycle
  • Helm chart available for standardized installation

Benefits of Jenkins on K8s

  • Elastic build capacity
  • Isolated build environments
  • No permanent agent VMs to manage
  • Resource limits per build
Running Jenkins itself on Kubernetes is a natural fit. The controller runs as a Deployment with persistent storage for configuration. Build agents are ephemeral Pods that the Kubernetes plugin creates on demand. Each build gets a clean, isolated environment and the cluster handles scheduling and resource allocation automatically.

📝 Jenkinsfile Structure for K8s

pipeline {
    agent {
        kubernetes {
            yaml '''
            spec:
              containers:
              - name: docker
                image: docker:24-dind
                securityContext:
                  privileged: true
              - name: kubectl
                image: bitnami/kubectl:1.28
                command: ['sleep', 'infinity']
            '''
        }
    }
    stages {
        stage('Build')   { steps { container('docker')  { /* ... */ } } }
        stage('Test')    { steps { container('docker')  { /* ... */ } } }
        stage('Deploy')  { steps { container('kubectl') { /* ... */ } } }
    }
}
Pod template: Each stage can target a different container within the same agent Pod, giving you specialized tooling per phase.
The declarative Jenkinsfile defines an agent block with a Kubernetes pod template. This template can contain multiple containers -- for example, one with Docker for building images and another with kubectl for deploying. Each stage references the appropriate container. This approach keeps the pipeline self-contained and version-controlled alongside the application code.

📦 Building Container Images in CI

Docker-in-Docker (DinD)

  • Run Docker daemon inside agent Pod
  • Requires privileged mode
  • Full Docker CLI compatibility

Rootless Alternatives

  • Kaniko -- builds in userspace, no daemon
  • Buildah -- OCI-native, daemonless
  • ko -- Go-specific, fast
Security note: Privileged DinD is a security risk. Prefer Kaniko or Buildah in production pipelines where pod security standards are enforced.
Building container images inside Kubernetes requires careful consideration. Docker-in-Docker is the most compatible option but requires privileged containers, which breaks PodSecurity standards. Kaniko from Google is the most popular alternative -- it builds images entirely in userspace without needing a Docker daemon. Buildah is another strong option, especially in Red Hat ecosystems.

🚀 Deploying to K8s from Jenkins

stage('Deploy') {
    steps {
        container('kubectl') {
            withCredentials([file(credentialsId: 'kubeconfig',
                                  variable: 'KUBECONFIG')]) {
                sh '''
                    kubectl set image deployment/myapp \
                        myapp=registry.example.com/myapp:${BUILD_TAG} \
                        --namespace production
                    kubectl rollout status deployment/myapp \
                        --namespace production --timeout=300s
                '''
            }
        }
    }
}
  • Store kubeconfig as a Jenkins credential -- never in source control
  • Use rollout status to block until healthy or timeout
  • Consider kubectl apply -f with versioned manifests for full declarative deploys
The deploy stage uses kubectl to update the image tag on the target Deployment. The kubeconfig is injected securely from Jenkins credentials. The rollout status command acts as a gate -- the pipeline will not proceed until the Deployment reports all replicas healthy, or it will fail on timeout. For more sophisticated deployments, teams often use Helm upgrade or kustomize build piped into kubectl apply.

Quality gates: Each stage acts as a gate. A vulnerability above the severity threshold or a failing test halts the pipeline before any deployment occurs.
This is the canonical five-stage pipeline for Kubernetes. Lint catches YAML syntax and policy violations early. Build produces the container image. Scan runs a vulnerability scanner like Trivy or Grype against the image layers. Test executes the application test suite -- often inside a container built from the same image. Deploy is the final stage and only runs if all previous gates pass. Each stage is a quality gate that protects production.

🏁 Environment Promotion Strategies

Branch-Based Promotion

  • develop ➔ dev cluster
  • staging ➔ staging cluster
  • main ➔ production
  • Merge triggers promotion

Tag-Based Promotion

  • Every commit builds an image
  • Promote by tagging the image
  • v1.2.3 tag triggers prod deploy
  • Same artifact across all envs
Promote artifacts, not code rebuilds -- the image deployed to production should be the same image tested in staging.
There are two primary promotion strategies. Branch-based promotion maps branches to environments: merging to main triggers a production deploy. Tag-based promotion builds once and promotes the same image artifact through environments by applying tags. The critical principle is that you should never rebuild for production -- the same binary artifact should flow through every environment to avoid "it built differently" issues.

⚙ Jenkins Agents on Kubernetes

apiVersion: v1
kind: Pod
metadata:
  labels:
    jenkins: agent
spec:
  serviceAccountName: jenkins-agent
  containers:
  - name: jnlp
    image: jenkins/inbound-agent:latest
    resources:
      requests: { cpu: "500m", memory: "512Mi" }
      limits:   { cpu: "2",    memory: "2Gi"   }
  - name: kaniko
    image: gcr.io/kaniko-project/executor:debug
    command: ['sleep', 'infinity']
    volumeMounts:
    - name: docker-config
      mountPath: /kaniko/.docker
  volumes:
  - name: docker-config
    secret:
      secretName: registry-credentials
  • Pod templates define multi-container agent environments
  • The jnlp container connects back to the Jenkins controller
  • Sidecar containers provide build tools (Kaniko, Helm, kubectl)
This is an example Pod template for a Jenkins agent. The jnlp container is the Jenkins agent process that communicates with the controller. Alongside it, you add sidecar containers with the tools you need -- Kaniko for building images, kubectl for deploying, Helm for chart operations. The Pod is created when a build starts and deleted when it finishes. Resource requests and limits ensure builds do not starve the cluster.

GitOps with FluxCD & ArgoCD

Git as the single source of truth

Now we transition from push-based CI/CD to the pull-based GitOps model. GitOps has gained enormous traction in the Kubernetes ecosystem because it leverages Git -- a tool every developer already knows -- as the single source of truth for cluster state. Let us explore how this works in practice.

📚 What Is GitOps?

GitOps is an operational framework where the desired state of infrastructure and applications is stored in Git, and automated agents ensure the live system matches that desired state.

  • Git repository is the single source of truth
  • All changes go through pull requests -- code review for infrastructure
  • No manual kubectl apply against production
  • Cluster state is always auditable via git log
"If it is not in Git, it does not exist."
GitOps was coined by Weaveworks in 2017. The core idea is simple: your Git repository defines what should be running. An agent inside the cluster watches that repository and reconciles any differences. This means every change is versioned, reviewed, and auditable. It also means rollback is just a git revert. No one needs direct kubectl access to production.

📌 Core GitOps Principles

📜 Declarative

System state described declaratively, not imperatively

🔒 Versioned

Desired state stored in Git with full history

⚡ Automated

Approved changes applied automatically

🔄 Reconciled

Agents continuously correct drift

Reconciliation is what separates GitOps from just "storing YAML in Git." The agent does not run once -- it runs continuously, detecting and correcting drift.
These four principles come from the OpenGitOps specification. Declarative means you describe the desired end state, not the steps to get there -- which is exactly how Kubernetes works natively. Versioned means Git tracks every change. Automated means approved changes apply without manual intervention. Reconciled means an agent continuously compares actual state to desired state and corrects drift. That last one is the key differentiator.

🔧 FluxCD Architecture

Core Controllers

  • Source Controller -- fetches manifests from Git/Helm/OCI
  • Kustomize Controller -- applies Kustomize overlays
  • Helm Controller -- manages HelmRelease resources
  • Notification Controller -- sends alerts and receives webhooks

How It Works

  1. Source Controller polls Git repo
  2. Detects new commit on target branch
  3. Pulls and validates manifests
  4. Kustomize/Helm Controller applies to cluster
  5. Reports status back to Git/Slack
FluxCD v2 is composed of specialized controllers that each handle a specific concern. The Source Controller is responsible for fetching desired state from various sources -- Git repositories, Helm chart repositories, or OCI registries. The Kustomize Controller renders and applies Kustomize overlays. The Helm Controller manages Helm charts as custom resources. The Notification Controller handles bidirectional communication -- sending alerts to Slack or Teams and receiving webhooks from Git providers to trigger immediate reconciliation.

📡 ArgoCD Overview

  • Pull-based GitOps agent, same model as FluxCD
  • Rich web UI for visualizing sync state
  • Application CRD defines source repo, path, and target cluster
  • Supports Kustomize, Helm, Jsonnet, and plain YAML
  • Multi-cluster management from a single control plane

FluxCD vs ArgoCD

UIFluxCD: CLI-first
ArgoCD: Web UI
ArchitectureFluxCD: multiple CRDs
ArgoCD: Application CRD
Multi-tenancyBoth support it, different approaches
CNCF StatusBoth graduated projects
ArgoCD is the other major GitOps tool in the CNCF ecosystem. Its biggest differentiator is the web UI, which provides a real-time visualization of application sync state, resource health, and diff views. Both FluxCD and ArgoCD are CNCF graduated projects. The choice between them often comes down to whether your team prefers a CLI-first workflow with FluxCD or a UI-driven workflow with ArgoCD. Functionally, they achieve the same goal.

🔄 The Reconciliation Loop

Git Repository (desired state)
↓
GitOps Agent compares desired vs actual
↓
Kubernetes Cluster (actual state)
↓
Drift detected? ➔ Auto-remediate or Alert

  • Default polling interval: ~1 minute (configurable)
  • Webhooks enable near-instant reconciliation on push
  • Manual edits via kubectl are reverted automatically
  • Failed applies are retried with exponential backoff
Warning: The reconciliation loop will overwrite manual changes. Communicate this to all team members before enabling GitOps.
The reconciliation loop is the heart of GitOps. The agent continuously compares what Git says should exist with what actually exists in the cluster. Any drift is corrected. This means if someone manually edits a Deployment via kubectl, the GitOps agent will revert it. This is by design -- it enforces the principle that Git is the sole source of truth. Teams need to understand this before adoption, because the first time someone's manual fix gets reverted, there will be questions.

Repository structure matters a great deal in GitOps. This is a common pattern: the apps directory holds base manifests, the clusters directory holds per-environment kustomize overlays, and the infrastructure directory holds shared platform components like monitoring and ingress controllers. The GitOps agent for each cluster points to its respective clusters directory, which uses Kustomize to compose the final set of resources from bases and overlays.

🔐 Handling Secrets in GitOps

Problem: You cannot commit plaintext Secrets to Git. But GitOps requires everything to be in Git.

Sealed Secrets

  • Bitnami Sealed Secrets controller
  • Encrypt with public key, decrypt only in-cluster
  • SealedSecret CRD committed to Git
  • Controller decrypts to native Secret

SOPS + Age/KMS

  • Mozilla SOPS encrypts YAML values in-place
  • Keys managed by Age, AWS KMS, or GCP KMS
  • FluxCD has native SOPS decryption support
  • Diff-friendly -- structure visible, values encrypted
Alternative: Use External Secrets Operator to sync secrets from Vault, AWS Secrets Manager, or Azure Key Vault -- only the reference lives in Git.
Secrets are the thorniest problem in GitOps. You must never commit plaintext secrets to Git, but GitOps requires all desired state to be in Git. There are three main solutions. Sealed Secrets uses asymmetric encryption: you encrypt with a public key, and only the in-cluster controller can decrypt. SOPS encrypts individual values within YAML files, keeping the structure readable while protecting sensitive data. External Secrets Operator takes a different approach entirely -- it stores only a reference in Git and fetches the actual secret value from an external secrets manager at runtime.

Benefits for Developers

Why CI/CD and GitOps improve the developer experience

Let us take a step back from the tooling and talk about why this matters to the people who write code every day. Automated deployment pipelines and GitOps are not just operational improvements -- they fundamentally change how developers interact with Kubernetes.

🎯 Simplified Deployment Workflow

Developer's workflow:
git commit ➔ git push ➔ open PR ➔ merge ➔ deployed

  • Developers interact with Git, not kubectl
  • No need to understand cluster internals to ship code
  • Pull request reviews serve as deployment approvals
  • Same workflow regardless of target environment
The developer experience simplifies to what developers already know: Git. They commit, push, open a PR, get it reviewed and merged, and the automation handles everything else. They do not need kubeconfig files, they do not need to remember kubectl flags, and they do not need to worry about which context they are pointing at. This removes an entire category of human error.

📖 Audit Trail and Rollback

Audit Trail

  • git log shows every change, who made it, and when
  • PR comments capture review discussion
  • CI logs record build and test results
  • Compliance teams can trace any deployment to its approval

Rollback

  • git revert <sha> to undo any change
  • GitOps agent auto-reconciles the revert
  • No need to remember previous image tags
  • Rollback is just another reviewed commit
"Who deployed what, when, and why?" is always answerable with git log.
The audit trail is a massive benefit, especially for regulated industries. Every change to every environment is captured in the Git history with author, timestamp, reviewer, and the exact diff. Rollback becomes trivial -- git revert the offending commit and the GitOps agent takes care of the rest. No more scrambling to figure out which image tag was running before an incident.

💫 Self-Service & Reduced Cognitive Load

Self-Service Deployments

  • Teams own their deployment manifests
  • No ticket to the platform team to deploy
  • Guardrails enforced by policy, not process
  • Platform team provides templates and tooling

Reduced Cognitive Load

  • One workflow for all environments
  • No context-switching between Git and kubectl
  • Pipeline handles complexity behind the scenes
  • Focus on code, not deployment mechanics
Self-service does not mean ungoverned. The platform team establishes guardrails through admission controllers, OPA policies, and pipeline quality gates. Application teams then operate independently within those guardrails. This reduces bottlenecks and allows teams to ship at their own pace. The reduced cognitive load is real -- developers do not need to become Kubernetes experts to deploy their applications safely.

🕑 Faster Feedback Loops

Automated pipelines shrink the time between writing code and seeing it run in a real environment.

  • Lint on commit -- catch misconfigurations in seconds
  • Test on PR -- validate changes before merge
  • Deploy on merge -- see results in staging within minutes
  • Reconcile continuously -- production matches Git in under a minute
Shorter feedback loops lead to smaller changes, fewer defects, and faster recovery.
Research from the DORA team consistently shows that high-performing teams deploy more frequently with shorter lead times and recover faster from incidents. CI/CD and GitOps are the mechanisms that enable this. The faster a developer learns their change has a problem, the cheaper it is to fix. Catching an issue in lint costs seconds; catching it in production costs hours or days.

Key Takeaways

  • CI/CD pipelines automate the path from commit to cluster through build, test, scan, and deploy stages -- eliminating manual, error-prone deployment steps.
  • Jenkins on Kubernetes uses ephemeral Pod agents for elastic, isolated build environments with multi-container pod templates for specialized tooling.
  • Push-based vs pull-based deployment models each have strengths -- many teams combine CI for build/test with GitOps for deployment.
  • GitOps uses Git as the single source of truth with continuous reconciliation to detect and correct drift automatically.
  • FluxCD and ArgoCD are both CNCF graduated GitOps tools -- FluxCD is CLI-first while ArgoCD offers a rich web UI.
  • Secrets in GitOps require encryption (Sealed Secrets, SOPS) or external references (External Secrets Operator) to keep sensitive data out of plaintext Git.
These are the core concepts from this module. The overarching theme is that automating deployments through CI/CD and GitOps improves reliability, security, auditability, and developer experience. Jenkins handles the build and test lifecycle well, while GitOps tools handle the deployment and reconciliation loop. Together they form a robust, end-to-end automation strategy for Kubernetes workloads.
