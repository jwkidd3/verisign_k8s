The Problem with Raw Manifests

Why plain YAML falls short at scale

Before diving into tools, let us establish the problem they solve. This section frames the challenges that Helm and Kustomize were designed to address.
📁 Managing Many YAML Files

A Typical Microservice Stack

  • Deployment + Service per microservice
  • ConfigMaps, Secrets, Ingress rules
  • RBAC, NetworkPolicies, PDBs
  • HPA, VPA, ServiceMonitors

Multiply Across Environments

  • dev, staging, production
  • 10 services × 8 manifests × 3 envs
  • = 240 YAML files
  • Each slightly different
Even a modest microservices deployment generates a large number of manifests. When multiplied across environments, the file count becomes difficult to manage manually. This is where most teams begin to feel pain.
⚠️ Duplication and Drift
Copy-paste YAML is the root cause of configuration drift
  • Duplication: Same resource definitions repeated across environments with minor differences (replica count, image tag, resource limits)
  • Drift: Over time, copies diverge -- a fix applied in staging is missed in production
  • Inconsistency: No single source of truth for what "the application" looks like
  • Onboarding friction: New team members cannot quickly understand the deployment model
Duplication is not just an inconvenience -- it is a reliability risk. Configuration drift between environments leads to the classic "works in staging, fails in production" scenario. These problems motivate the need for templating and overlay tools.
🔧 Why Templating & Overlay Tools Exist

Reduce Duplication

Define a resource once, parameterize the differences

Enforce Consistency

Single source of truth with environment-specific overrides

Enable Reuse

Package applications for sharing across teams and organizations

Two approaches emerged: Helm uses templating (fill in the blanks), while Kustomize uses overlays (patch the base). Both are valid; they solve the problem differently.
This is the key conceptual distinction for the entire module. Helm takes a templating approach where you define placeholders and fill them in with values. Kustomize takes an overlay approach where you start with valid YAML and layer patches on top. Understanding this difference helps students choose the right tool.

⎈ Helm

The Package Manager for Kubernetes

Helm is the most widely adopted packaging tool in the Kubernetes ecosystem. We will cover its architecture, workflow, and best practices in depth.
📦 What is Helm?

Helm is a package manager for Kubernetes

Think of it as apt/yum/brew for your cluster. Helm packages Kubernetes manifests into charts, which can be versioned, shared, and installed with a single command.

Helm provides:
  • Templated manifests with configurable values
  • Release management (install, upgrade, rollback)
  • Dependency management between charts
  • A rich ecosystem of community charts
Current version: Helm 3
  • No more Tiller (server-side component)
  • Uses Kubernetes Secrets for release state
  • Improved security model
  • Three-way strategic merge for upgrades
Helm 3 removed the controversial Tiller component that had elevated cluster privileges. Now Helm operates purely client-side, using the user's kubeconfig credentials. Release state is stored as Secrets in the target namespace.
🗂️ Charts, Releases, Repositories

Chart

A package of templated Kubernetes manifests. Versioned and distributable.

Example: nginx-ingress, prometheus, cert-manager

Release

A running instance of a chart in a cluster. Each install creates a uniquely named release.

Example: my-app-prod (release) from my-app (chart)

Repository

An HTTP server hosting packaged charts. Can be public or private.

Example: ArtifactHub, Bitnami, your internal registry

OCI Support: Helm 3 also supports storing charts in OCI-compliant registries (Docker Hub, ECR, GCR, ACR) -- no separate chart repository needed.
The chart/release/repository model mirrors traditional package management. A chart is like an RPM or Deb package. A release is an installed instance. A repository is like a yum repo or apt source. The OCI support is increasingly popular because it lets teams reuse existing container registry infrastructure.
📂 Chart Structure
my-chart/
├── Chart.yaml          # Chart metadata (name, version, dependencies)
├── values.yaml         # Default configuration values
├── charts/             # Dependency charts (subcharts)
├── templates/          # Kubernetes manifest templates
│   ├── deployment.yaml
│   ├── service.yaml
│   ├── ingress.yaml
│   ├── _helpers.tpl    # Reusable template functions
│   ├── NOTES.txt       # Post-install usage instructions
│   └── tests/
│       └── test-connection.yaml
├── .helmignore         # Files to exclude from packaging
└── README.md           # Chart documentation
Chart.yaml defines the chart identity. values.yaml provides defaults that users override at install time. templates/ contains Go-templated Kubernetes manifests.
Walk through each file and its purpose. Emphasize that _helpers.tpl is where you define reusable named templates, and NOTES.txt is rendered after install to show the user how to access the deployed application. The tests directory holds test pods that can be run with helm test.
📝 Chart.yaml & values.yaml

Chart.yaml

apiVersion: v2
name: my-web-app
description: A web application chart
type: application
version: 1.2.0        # Chart version
appVersion: "3.5.1"   # App version
dependencies:
  - name: postgresql
    version: "12.x"
    repository: "https://charts.bitnami.com"
    condition: postgresql.enabled

values.yaml

replicaCount: 2

image:
  repository: myregistry/web-app
  tag: "3.5.1"
  pullPolicy: IfNotPresent

service:
  type: ClusterIP
  port: 80

resources:
  limits:
    cpu: 500m
    memory: 256Mi
  requests:
    cpu: 100m
    memory: 128Mi

postgresql:
  enabled: true
Chart.yaml uses apiVersion v2 for Helm 3. The version field tracks chart changes while appVersion tracks the packaged application version. Dependencies can be conditionally enabled via values, as shown with the postgresql condition.
🔤 Template Syntax (Go Templates)
apiVersion: apps/v1
kind: Deployment
metadata:
  name: {{ include "my-chart.fullname" . }}
  labels:
    {{- include "my-chart.labels" . | nindent 4 }}
spec:
  replicas: {{ .Values.replicaCount }}
  selector:
    matchLabels:
      {{- include "my-chart.selectorLabels" . | nindent 6 }}
  template:
    spec:
      containers:
        - name: {{ .Chart.Name }}
          image: "{{ .Values.image.repository }}:{{ .Values.image.tag }}"
          ports:
            - containerPort: {{ .Values.service.port }}
          {{- if .Values.resources }}
          resources:
            {{- toYaml .Values.resources | nindent 12 }}
          {{- end }}
Go template syntax uses double curly braces for expressions. The dot represents the current scope. .Values accesses values.yaml, .Chart accesses Chart.yaml, and .Release accesses release metadata. The dash inside braces trims whitespace. nindent handles YAML indentation, which is one of the trickiest parts of Helm templates.
🧩 Key Template Constructs

Built-in Objects

  • .Values -- values.yaml + overrides
  • .Chart -- Chart.yaml metadata
  • .Release -- release name, namespace, revision
  • .Template -- current template info
  • .Capabilities -- cluster API versions

Control Flow

{{- if .Values.ingress.enabled }}
  # render ingress resource
{{- end }}

{{- range .Values.env }}
  - name: {{ .name }}
    value: {{ .value | quote }}
{{- end }}

{{- with .Values.nodeSelector }}
nodeSelector:
  {{- toYaml . | nindent 2 }}
{{- end }}
Cover if/else, range for iteration, and with for scope narrowing. Mention the pipe operator for chaining functions like quote, toYaml, nindent, default, and required. These are the building blocks students will use when creating their own charts.
🚀 Managing Releases
# Add a repository
helm repo add bitnami https://charts.bitnami.com/bitnami
helm repo update

# Install a chart (creates a release)
helm install my-release bitnami/nginx -f custom-values.yaml -n web

# List releases
helm list -n web

# Upgrade with new values or chart version
helm upgrade my-release bitnami/nginx --set replicaCount=3 -n web

# Check release history
helm history my-release -n web

# Roll back to a previous revision
helm rollback my-release 1 -n web

# Uninstall a release
helm uninstall my-release -n web
Atomic installs: Use --atomic to automatically roll back if the install/upgrade fails. Combine with --timeout to set a deadline.
Walk through the full lifecycle: add repo, install, upgrade, rollback, uninstall. Emphasize that helm upgrade is idempotent and can be used for both value changes and chart version bumps. The --atomic flag is a production best practice -- it prevents partial deployments.
🪝 Helm Hooks

Hooks let you intervene at specific points in the release lifecycle

HookPurpose
pre-installRun before resources are created
post-installRun after all resources are created
pre-upgradeRun before an upgrade
post-upgradeRun after an upgrade
pre-deleteRun before deletion
testRun via helm test
 
apiVersion: batch/v1
kind: Job
metadata:
  name: db-migrate
  annotations:
    "helm.sh/hook": pre-upgrade
    "helm.sh/hook-weight": "1"
    "helm.sh/hook-delete-policy":
      hook-succeeded
spec:
  template:
    spec:
      containers:
        - name: migrate
          image: myapp/migrate:v2
          command: ["./migrate"]
      restartPolicy: Never
Hooks are commonly used for database migrations, cache warming, or notification triggers. The hook-weight controls ordering when multiple hooks exist. hook-delete-policy determines whether the hook resource is cleaned up after execution. Common values are hook-succeeded, hook-failed, and before-hook-creation.
🛠️ Creating Your Own Chart
# Scaffold a new chart
helm create my-web-app

# Review and customize generated templates
# Edit values.yaml with your defaults
# Modify templates/ to match your application

# Validate your chart
helm lint my-web-app

# Render templates locally (dry-run)
helm template my-release my-web-app -f values-prod.yaml

# Install from local directory
helm install my-release ./my-web-app -f values-prod.yaml

# Package for distribution
helm package my-web-app
# Creates my-web-app-1.0.0.tgz
Development workflow: helm create → customize → helm lint → helm template → helm install --dry-run → helm install
The helm create scaffold is a great starting point, but students should understand every file it generates. Encourage removing unused templates rather than leaving them in. The lint and template commands are essential for catching errors before deployment.
✅ Helm Best Practices

Chart Design

  • Use _helpers.tpl for reusable labels and names
  • Provide sensible defaults in values.yaml
  • Document every value with comments
  • Pin dependency versions in Chart.yaml
  • Include helm test pods for validation

Operations

  • Always use --atomic in CI/CD pipelines
  • Store custom values files in version control
  • Use helm diff plugin before upgrading
  • Limit template complexity -- if logic is deep, consider a subchart
  • Use OCI registries for chart distribution
Avoid: Putting too much logic in templates. If your _helpers.tpl exceeds 200 lines, your chart may be trying to do too much.
Emphasize that Helm charts should be treated as code -- versioned, reviewed, tested, and documented. The helm diff plugin is especially valuable in production as it shows exactly what will change before you apply an upgrade.

🔀 Kustomize

Template-Free Configuration Management

Now we shift to Kustomize, which takes a fundamentally different approach. Instead of templates with placeholders, Kustomize works with plain, valid YAML and layers modifications on top.
🎯 What is Kustomize?

Kustomize: template-free configuration customization

Kustomize lets you customize Kubernetes resources without modifying the original YAML. It uses a declarative overlay approach -- you define a base and then apply patches per environment.

Key principles:
  • All manifests are valid Kubernetes YAML -- no template syntax
  • Customization is additive (overlays on top of bases)
  • Built directly into kubectl since v1.14
  • No learning a new templating language
Kustomize was developed at Google and eventually merged into kubectl. Its philosophy is that Kubernetes manifests should always be valid YAML that you can apply directly. This makes them easier to read, lint, and understand compared to heavily templated Helm charts.
📋 kustomization.yaml Structure
apiVersion: kustomize.config.k8s.io/v1beta1
kind: Kustomization

# Resources to include
resources:
  - deployment.yaml
  - service.yaml
  - ingress.yaml

# Common transformations
namespace: my-app-prod
namePrefix: prod-
commonLabels:
  app: my-web-app
  environment: production
commonAnnotations:
  team: platform-engineering

# Config and secret generators
configMapGenerator:
  - name: app-config
    literals:
      - LOG_LEVEL=info
      - DB_HOST=db.prod.internal

# Image overrides
images:
  - name: myregistry/web-app
    newTag: "3.5.1"
Walk through each section of the kustomization.yaml. The resources field lists the manifests to include. Common transformations like namespace, namePrefix, and commonLabels are applied to all resources. configMapGenerator creates ConfigMaps from literals or files and appends a hash suffix for automatic rollout on changes.
📐 Bases and Overlays
my-app/
├── base/
│   ├── kustomization.yaml     # Lists resources
│   ├── deployment.yaml        # Standard deployment
│   ├── service.yaml           # Standard service
│   └── ingress.yaml           # Standard ingress
├── overlays/
│   ├── dev/
│   │   ├── kustomization.yaml # References ../base, applies dev patches
│   │   └── replica-patch.yaml
│   ├── staging/
│   │   ├── kustomization.yaml
│   │   └── resource-patch.yaml
│   └── prod/
│       ├── kustomization.yaml # References ../base, applies prod patches
│       ├── replica-patch.yaml
│       └── hpa.yaml           # Additional prod-only resource
Base: The shared, common resource definitions. Valid YAML you can apply directly.
Overlay: Environment-specific modifications layered on top of the base.
The base/overlay pattern is Kustomize's core concept. The base contains the canonical resource definitions. Each overlay references the base and adds environment-specific customizations. Overlays can also add entirely new resources, as shown with the HPA in the production overlay.
🔧 Overlay in Practice

overlays/dev/kustomization.yaml

apiVersion: kustomize.config.k8s.io/v1beta1
kind: Kustomization

resources:
  - ../../base

namespace: my-app-dev
namePrefix: dev-

commonLabels:
  environment: dev

images:
  - name: myregistry/web-app
    newTag: "latest"

patches:
  - path: replica-patch.yaml

overlays/prod/kustomization.yaml

apiVersion: kustomize.config.k8s.io/v1beta1
kind: Kustomization

resources:
  - ../../base
  - hpa.yaml        # Prod-only resource

namespace: my-app-prod
namePrefix: prod-

commonLabels:
  environment: production

images:
  - name: myregistry/web-app
    newTag: "3.5.1"

patches:
  - path: replica-patch.yaml
  - path: resource-patch.yaml
Notice how the dev overlay uses the latest tag while prod pins a specific version. The prod overlay also adds an HPA resource that does not exist in dev. This is the power of overlays -- you can add, modify, or even remove resources per environment.
🔄 Common Transformations
TransformerEffectExample
namespace Sets namespace on all resources namespace: production
namePrefix Prepends string to all resource names namePrefix: prod-
nameSuffix Appends string to all resource names nameSuffix: -v2
commonLabels Adds labels to all resources and selectors commonLabels: {env: prod}
commonAnnotations Adds annotations to all resources commonAnnotations: {team: sre}
images Overrides image names and tags images: [{name: app, newTag: v2}]
These transformations are applied globally across all resources in the kustomization. Note that commonLabels also updates selector fields, which can be problematic for immutable selectors on existing Deployments. In those cases, use the labels transformer with a more targeted configuration.
🩹 Patching Strategies

Strategic Merge Patch

Merges with existing resource structure

# replica-patch.yaml
apiVersion: apps/v1
kind: Deployment
metadata:
  name: web-app
spec:
  replicas: 5
  template:
    spec:
      containers:
        - name: web-app
          resources:
            limits:
              cpu: "2"
              memory: 1Gi

Only specify fields you want to change. Kubernetes merges the rest.

JSON Patch (RFC 6902)

Precise operations: add, remove, replace, move, copy, test

# In kustomization.yaml
patches:
  - target:
      kind: Deployment
      name: web-app
    patch: |-
      - op: replace
        path: /spec/replicas
        value: 5
      - op: add
        path: /spec/template/metadata
              /annotations/prometheus.io~1scrape
        value: "true"
      - op: remove
        path: /spec/template/spec
              /containers/0/env/2
Strategic merge patches are more intuitive -- you write partial YAML that gets merged. JSON patches give you surgical precision for operations like removing fields or modifying array elements. Use strategic merge patches for most cases and JSON patches when you need to remove something or modify a specific array index.
⌨️ Built-in with kubectl
# Preview rendered output
kubectl kustomize overlays/prod

# Apply directly (equivalent to kustomize build | kubectl apply -f -)
kubectl apply -k overlays/prod

# Diff before applying
kubectl diff -k overlays/prod

# Delete resources managed by kustomization
kubectl delete -k overlays/prod

# Use standalone kustomize binary for latest features
kustomize build overlays/prod | kubectl apply -f -
Version note: The kubectl-embedded Kustomize may lag behind the standalone binary. For production, consider using the standalone kustomize CLI to get the latest features and bug fixes.
The -k flag in kubectl is the simplest way to use Kustomize. However, the embedded version may be several releases behind the standalone binary. In CI/CD pipelines, it is common to use the standalone kustomize build piped to kubectl apply for consistency and access to the latest features.

🤔 Choosing the Right Tool

Helm, Kustomize, or both?

This is the practical decision-making section. Teams often debate Helm vs Kustomize, but in reality they serve different purposes and can be used together effectively.
⚖️ Helm vs Kustomize
AspectHelmKustomize
Approach Templating (Go templates) Overlays (patch-based)
Learning curve Steeper (template syntax) Gentler (plain YAML)
Release management Built-in (install, upgrade, rollback) None (relies on kubectl apply)
Packaging & sharing First-class (charts, repos, OCI) Limited (directory references, remote bases)
Ecosystem Thousands of community charts Smaller, growing
Readability Templates can be hard to read Always valid, readable YAML
Tooling Separate CLI (helm) Built into kubectl (-k flag)
This table captures the fundamental trade-offs. Neither tool is universally superior. Walk through each row and discuss how these differences influence the choice in real-world scenarios.
📦 When to Use Helm
Helm excels when you need:
  • Third-party applications: Install Prometheus, NGINX Ingress, cert-manager, or databases from community charts
  • Complex parameterization: Applications with many configurable options that vary between consumers
  • Release lifecycle management: Built-in install, upgrade, rollback, and history tracking
  • Application distribution: Sharing packaged applications across teams or organizations
  • Dependency management: Applications with subcharts (e.g., app + database + cache)
Rule of thumb: If you are consuming someone else's application or distributing your own to others, Helm is the natural choice.
Helm's strength is in its ecosystem and lifecycle management. When you need to deploy a complex third-party application like a monitoring stack, Helm charts save enormous effort. The parameterization model also works well when different teams need to install the same application with different configurations.
🔀 When to Use Kustomize
Kustomize excels when you need:
  • Environment variations: Same application deployed to dev, staging, and production with minor differences
  • Simplicity: No templating language to learn -- just valid Kubernetes YAML
  • GitOps workflows: Kustomize integrates naturally with Argo CD and Flux
  • Patching upstream manifests: Customizing third-party YAML without forking it
  • Lightweight customization: Adding labels, changing namespaces, updating image tags
Rule of thumb: If your team owns the application and the primary need is environment-specific configuration, Kustomize keeps things simple and transparent.
Kustomize shines in internal platform scenarios where the same team writes the manifests and deploys them. It is the default kustomization engine for both Argo CD and Flux, making it the natural choice for GitOps pipelines.
🤝 Using Both Together

Helm + Kustomize: A common production pattern

Pattern: Helm Template + Kustomize Overlay

  1. Use helm template to render a chart to plain YAML
  2. Store the rendered output as a Kustomize base
  3. Apply Kustomize overlays for environment-specific patches

Gives you Helm's ecosystem with Kustomize's overlay model.

Pattern: Kustomize helmCharts

# kustomization.yaml
helmCharts:
  - name: nginx-ingress
    repo: https://helm.nginx.com
          /stable
    version: 0.17.1
    releaseName: ingress
    namespace: ingress-system
    valuesFile: values.yaml

Kustomize natively renders Helm charts (standalone binary only).

Argo CD pattern: Use Helm charts for third-party components and Kustomize overlays for your own applications, all managed through GitOps.
The combination of Helm and Kustomize is increasingly common in mature Kubernetes environments. Helm handles packaging and third-party software, while Kustomize handles environment-specific customization. Argo CD supports both natively, making this pattern straightforward to implement.

Key Takeaways

  • Raw YAML does not scale -- duplication and drift across environments create reliability risks that templating and overlay tools are designed to solve.
  • Helm is a full package manager with templating, release lifecycle management, dependency handling, and a vast ecosystem of community charts.
  • Kustomize provides template-free customization through bases and overlays, keeping all manifests as valid Kubernetes YAML.
  • Helm excels at distributing complex applications and consuming third-party software; Kustomize excels at environment-specific configuration for internally managed workloads.
  • Using both together is a common and powerful pattern -- Helm for packaging, Kustomize for environment overlays, unified through GitOps.
Recap the main takeaways. The most important message is that Helm and Kustomize are complementary, not competing. Mature teams typically use both, choosing the right tool based on the specific use case. Encourage students to practice with both in the lab.
