🔭 Observability Fundamentals

Understanding the signals your applications emit

This section introduces the conceptual foundation. Before we dive into specific tools, we need to establish what observability means and why it matters for developers — not just operations teams. The goal is to shift thinking from "monitoring as someone else's job" to "observability as a first-class design concern."

🏛️ The Three Pillars of Observability

📋 Logs

Discrete, timestamped records of events. The narrative of what happened.

  • Structured (JSON) or unstructured
  • Request-scoped context
  • High volume, high cardinality

📊 Metrics

Numeric measurements aggregated over time. The vital signs of your system.

  • Counters, gauges, histograms
  • Low cost to store and query
  • Ideal for dashboards and alerts

🔗 Traces

End-to-end request paths across services. The map of distributed work.

  • Spans and trace IDs
  • Latency breakdown per service
  • Critical for microservices
Each pillar answers a different question. Logs tell you "what happened." Metrics tell you "how is the system performing right now." Traces tell you "where did this specific request spend its time." No single pillar is sufficient — you need all three working together. A metric alert tells you something is wrong, logs help you understand what broke, and traces help you find where in the call chain the problem occurred. Emphasize that this module focuses primarily on logs and metrics, since those are the most immediately actionable for application developers.

👩‍💻 Why Observability Matters for Developers

Without Observability

  • User reports the problem first
  • "Works on my machine" debugging
  • SSH into pods, tail log files
  • Guessing at root cause
  • Hours to mean-time-to-detection

With Observability

  • Alerts fire before users notice
  • Dashboards show exact state
  • Structured queries in Splunk
  • Correlate metrics with deploys
  • Minutes to mean-time-to-detection

"You build it, you run it" — observability is how you run it well.

This is the developer motivation slide. In a Kubernetes environment, you cannot SSH into containers like you could with traditional VMs. Containers are ephemeral — they restart, they get rescheduled. If your only debugging strategy is "look at the logs on the container," you will lose those logs. Observability must be designed into the application from the start. It is a feature, not an afterthought. Ask participants: "Have you ever had an incident where better logging or metrics would have cut your resolution time in half?" Most will say yes.

📝 Structured Logging Best Practices

Unstructured (Avoid) 
ERROR: Failed to process
order for user john.
Connection timed out after
30s to payment-svc
Structured (Prefer) 
{
  "level": "error",
  "msg": "order processing failed",
  "user_id": "john",
  "order_id": "ORD-4521",
  "downstream": "payment-svc",
  "error": "connection timeout",
  "duration_ms": 30000,
  "trace_id": "abc123"
}
Key fields to include: timestamp, level, message, request/correlation ID, service name, error details, duration
Structured logging is the single most impactful practice for operational readiness. JSON logs are machine-parseable, which means Splunk can index individual fields and you can write precise queries like "show me all errors from payment-svc where duration_ms > 5000." Unstructured logs require regex parsing, which is fragile and slow. Encourage teams to standardize on a logging library that outputs JSON — most modern frameworks support this out of the box. The trace_id field is critical for correlating logs across microservices.

🔍 Application vs Infrastructure Observability

Aspect Infrastructure Level Application Level
Who owns it Platform / SRE team Application developers
Metrics CPU, memory, disk, network Request rate, error rate, latency
Logs kubelet, kube-apiserver, etcd Application events, business logic
Alerts Node down, disk full, OOMKilled SLO breach, error spike, latency P99
Tools Node exporter, kube-state-metrics Custom metrics, application logs
Remember: Infrastructure metrics tell you something is wrong. Application metrics tell you what is wrong and who is affected.
This distinction is crucial. Many teams rely solely on infrastructure metrics — they know a pod restarted or CPU spiked, but they cannot answer "are users experiencing errors?" or "is checkout latency degraded?" Application-level observability fills that gap. As a developer, you are responsible for the application-level signals. The platform team provides the infrastructure signals. Both are necessary, and they need to be correlated — for example, high application latency might correlate with a node-level CPU saturation event.

📋 Logging to Splunk

Getting your application logs where they need to go

Now we move from concepts to practical implementation. This section covers how Kubernetes handles logs natively, the different patterns for aggregating logs, and how those logs flow into Splunk for querying and analysis. The key message: your application just needs to write to stdout/stderr — the platform handles the rest, but you need to understand the pipeline to troubleshoot effectively.

🏗️ Kubernetes Logging Architecture

Application
writes to stdout/stderr
→
Container Runtime
captures stream
→
Node Log File
/var/log/containers/
→
Log Agent
ships to backend
→
Splunk
index & search
  • Container runtime redirects stdout/stderr to JSON-wrapped log files on the node
  • Log files are stored at /var/log/containers/<pod>_<namespace>_<container>-<id>.log
  • Log rotation is handled by the kubelet (10MB default, 5 files)
  • When a pod is deleted, its log files on the node are eventually cleaned up
This is the fundamental flow. In Kubernetes, the universally agreed-upon contract is: your application writes to stdout and stderr, and the platform takes it from there. The container runtime (containerd) wraps each log line with a timestamp and stream identifier (stdout or stderr) and writes it to a file on the node. These files are what kubectl logs reads, and they are what log agents scrape. Emphasize that log files are ephemeral — they exist only as long as the node has them. This is why log aggregation is essential, not optional.

📂 Container stdout/stderr to Node Logs

The Golden Rule of Kubernetes Logging

Write logs to stdout and stderr — never to files inside the container.

What kubectl logs actually reads:

{"log":"order processed\n",
 "stream":"stdout",
 "time":"2026-03-11T14:22:01.3Z"}

{"log":"connection refused\n",
 "stream":"stderr",
 "time":"2026-03-11T14:22:01.5Z"}

Why stdout/stderr?

  • Works with every log collector
  • No volume mounts needed
  • Kubelet manages rotation
  • 12-factor app compliant
  • Container-native approach
This pattern may seem obvious, but many legacy applications write to log files inside the container (e.g., /var/log/app.log). In Kubernetes, those files disappear when the container restarts. If your application currently writes to files, you have two options: refactor it to write to stdout, or use a sidecar container to tail the file and re-emit to stdout. The CRI log format shown here is what containerd produces — each line is a JSON object wrapping the original log message. This is important to know when configuring Splunk parsing.

🔄 Log Aggregation Patterns

🖥️ DaemonSet Agent

Most common pattern.

  • One agent per node
  • Reads node log files
  • Low overhead per pod
  • e.g., Splunk UF, Fluentd
Recommended for most workloads

🚗 Sidecar Container

For special cases.

  • Per-pod log agent
  • Reads from shared volume
  • App-specific parsing
  • Higher resource cost
Use when app writes to files, not stdout

📡 Direct Push

Application sends directly.

  • App uses logging SDK
  • No agent needed
  • Tight coupling to backend
  • Harder to change later
Avoid unless required by design
The DaemonSet pattern is the standard in most Kubernetes environments, including at Verisign. A single agent per node (running as a DaemonSet) reads the /var/log/containers/ directory and ships all logs to the backend. This is efficient because you only run one agent per node regardless of how many pods are on that node. The sidecar pattern is used when an application cannot write to stdout — the sidecar tails the file and either re-emits to stdout or ships directly. The direct push pattern tightly couples the app to the log backend, making it hard to change later. In practice, DaemonSet should be your default choice.

🔎 Splunk Integration for Kubernetes

Splunk Connect for Kubernetes

  • Helm chart deploys DaemonSet log collector
  • Collects container logs, Kubernetes metadata, and cluster metrics
  • Enriches logs with: pod name, namespace, labels, annotations
  • Ships to Splunk HTTP Event Collector (HEC)

Kubernetes Metadata Enrichment

Every log line automatically includes:

  • pod, namespace
  • container_name
  • node_name
  • labels.* (app, version)
  • annotations.*
Key benefit: You can search Splunk by namespace, deployment, or label — not just by text pattern.
Splunk Connect for Kubernetes is the official way to get Kubernetes logs into Splunk. It deploys as a Helm chart with three components: a DaemonSet for log collection, a Deployment for cluster-level metrics, and a Deployment for Kubernetes objects. The metadata enrichment is extremely valuable — it means you can filter logs in Splunk by namespace, by specific deployment, or by custom labels without any additional configuration. This is why labeling your Kubernetes resources consistently is so important — those labels become searchable dimensions in Splunk.

✅ Log Formatting for Searchability

  • Use JSON format — Splunk auto-extracts fields from JSON logs
  • Include a correlation ID — trace requests across services with a shared request_id or trace_id
  • Use consistent field names — agree on level, msg, error across all services
  • Add business context — user_id, order_id, account make logs actionable
  • Avoid logging sensitive data — no passwords, tokens, PII in log fields
  • Use appropriate log levels — reserve ERROR for things that need attention, not routine events
Anti-pattern: Logging at DEBUG level in production generates massive volumes, increases Splunk costs, and buries important signals in noise.
These practices directly impact how efficiently you can troubleshoot in production. JSON formatting is the most impactful — when Splunk can parse your log fields automatically, you go from regex-based searching to field-based querying, which is orders of magnitude faster. Consistent field naming across services is a team agreement — publish a logging standard and enforce it in code reviews. The correlation ID point is critical for microservices: without a shared request ID propagated through HTTP headers, you cannot trace a request across service boundaries in Splunk.

🔍 Common Log Queries for Troubleshooting

Essential Splunk Queries

# All errors in a namespace in the last hour
index=k8s namespace="production" level="error"
| stats count by container_name, msg

# Latency outliers for a specific service
index=k8s app="payment-svc" duration_ms>5000
| timechart avg(duration_ms), p99(duration_ms)

# Trace a specific request across services
index=k8s trace_id="abc-123-def"
| sort _time
| table _time, app, level, msg

# Error rate trend for a deployment
index=k8s app="order-svc"
| timechart span=5m count(eval(level="error"))
  AS errors, count AS total
| eval error_rate = errors/total*100
Tip: Save these as Splunk dashboards and share them with your team — they become your first stop during incidents.
Walk through each query and explain the use case. The first query is your starting point during any incident — "show me all errors in production." The second helps with latency investigations — P99 is the key metric because it represents the worst experience your users have. The third query demonstrates the power of correlation IDs — you can reconstruct the entire journey of a request. The fourth computes error rate as a percentage over time, which is what SLOs measure. Encourage participants to build a library of saved queries that their team can share. These queries become much more powerful when your logs are structured JSON with consistent field names.

🚨 Alerting to Netcool, Email, and WebEx

Turning signals into action — without the noise

Alerting is where observability becomes operational. The worst outcome is an alerting system that cries wolf — teams start ignoring alerts, and real problems get missed. This section focuses on designing alerts that are actionable and routing them through enterprise channels: Netcool for incident management, email for notifications, and WebEx for team collaboration. The goal: every alert should require a human action, and the right human should receive it.

🎯 Alert Design Principles

✅ Good Alerts Are

  • Actionable — someone can do something about it
  • Symptomatic — alert on user impact, not causes
  • Contextual — include runbook links and dashboards
  • Tiered — severity matches required response time
  • Owned — clear routing to a responsible team

❌ Bad Alerts Are

  • Noisy — fire frequently with no action needed
  • Causal — "CPU at 80%" without impact context
  • Vague — "something is wrong with prod"
  • Undifferentiated — everything is "critical"
  • Orphaned — no one knows who should respond

"The best alert is one that wakes someone up and they say 'I'm glad you woke me.'"

This slide sets the philosophy. The number one mistake teams make is alerting on causes rather than symptoms. "CPU at 80%" is not an alert — it is a metric. "Error rate exceeds 1% for 5 minutes" is an alert because it describes user impact. Walk through each principle and ask the group to identify which anti-patterns they have experienced. Most teams have been through alert fatigue — where so many alerts fire that they all get ignored. The cure is fewer, better alerts with clear ownership and runbook documentation.

⚙️ Prometheus Alertmanager Concepts

Prometheus
evaluates rules
→
Alertmanager
routes & groups
→
Netcool / Email / WebEx
notification channels

Alertmanager Responsibilities

  • Grouping — combine related alerts into one notification
  • Inhibition — suppress downstream alerts when parent fires
  • Silencing — mute known/expected alerts during maintenance
  • Routing — direct alerts to the right receiver

Example Alert Rule

- alert: HighErrorRate
  expr: |
    rate(http_errors_total[5m])
    / rate(http_requests_total[5m])
    > 0.01
  for: 5m
  labels:
    severity: critical
    team: payments
  annotations:
    summary: "Error rate > 1%"
    runbook: "https://wiki/..."
Prometheus evaluates alerting rules at a regular interval (typically every 15-30 seconds). When an expression is true for the specified "for" duration, the alert transitions to "firing" and is sent to Alertmanager. Alertmanager then applies its configuration: it groups related alerts (so you get one notification for 50 similar alerts, not 50 notifications), inhibits child alerts (if the entire cluster is down, you don't also need "service A is down" alerts), and routes each alert to the correct receiver based on labels. The "for" clause is crucial — it prevents flapping alerts by requiring the condition to be true for a sustained period.

🏢 Routing Alerts to Netcool

  • Netcool/OMNIbus — enterprise event management platform
  • Alertmanager sends via webhook receiver to Netcool probe
  • Netcool provides: deduplication, enrichment, escalation workflows
  • Integrates with on-call schedules and incident management
  • Maps Prometheus labels to Netcool event fields

Alertmanager Webhook Config

receivers:
  - name: 'netcool'
    webhook_configs:
      - url: 'http://netcool-probe:80/...'
        send_resolved: true

route:
  receiver: 'netcool'
  group_by: ['alertname','namespace']
  group_wait: 30s
  group_interval: 5m
  routes:
    - match:
        severity: critical
      receiver: 'netcool'
Netcool is the enterprise incident management system. The integration works through Alertmanager's webhook receiver — when an alert fires, Alertmanager sends a JSON payload to the Netcool HTTP probe, which creates an event in the ObjectServer. The key configuration points: send_resolved should be true so Netcool automatically clears events when the alert resolves; group_by should match your organizational structure (typically alertname and namespace); and group_wait introduces a brief delay to batch related alerts together. The label-to-field mapping is configured in the Netcool probe rules file.

📬 Email and WebEx Notifications

📧 Email Notifications

receivers:
  - name: 'email-team'
    email_configs:
      - to: 'team@example.com'
        from: 'alerts@k8s.local'
        smarthost: 'smtp:587'
        require_tls: true
        html: '{{ template "email" . }}'

Best for: Low-urgency warnings, daily summaries, stakeholder notifications

💬 WebEx Notifications

receivers:
  - name: 'webex-ops'
    webhook_configs:
      - url: 'https://webex.api/...'
        http_config:
          bearer_token: '...'

Best for: Team awareness, real-time collaboration during incidents, non-paging alerts

Routing strategy: Critical → Netcool (page on-call) → WebEx (notify team). Warning → Email (next business day). Info → Dashboard only.
Different notification channels serve different purposes. Netcool is for critical, page-worthy alerts that require immediate response. WebEx is for team awareness — the whole team sees the alert in their ops channel and can collaborate on resolution. Email is for low-urgency items that can wait until the next business day. The routing strategy shown at the bottom is a good starting template: severity critical goes to Netcool AND WebEx, severity warning goes to email, and informational alerts are visible only on dashboards. This tiered approach prevents alert fatigue while ensuring critical issues get immediate attention.

🚦 Alert Severity Levels and Escalation

Severity Response Time Channel Example
Critical Immediate (page) Netcool + WebEx SLO breach, service down
Warning Next business hour Email + WebEx Error rate elevated, disk 80%
Info Best effort Dashboard only Deployment completed, scale event
Escalation policy: If a critical alert is unacknowledged for 15 minutes, escalate to the secondary on-call. After 30 minutes, escalate to the engineering manager.
Health check: If more than 50% of alerts result in "no action taken," your alert thresholds need tuning. Review monthly.
This table should be posted on your team's wiki and agreed upon by everyone. The most common mistake is making everything critical — when everything is critical, nothing is. The escalation policy ensures that alerts don't get lost: if the primary on-call doesn't acknowledge within 15 minutes, the alert escalates automatically. The health check metric at the bottom is something to track in retrospectives: if more than half of your alerts require no action, you are generating noise and need to raise thresholds or eliminate those alerts entirely. Aim for a target where every page-worthy alert results in a meaningful human response.

📏 SLIs, SLOs, and Error Budgets

📐 SLI — Service Level Indicator

A measurement of service behavior.

  • Request latency P99
  • Error rate percentage
  • Availability (uptime)

🎯 SLO — Service Level Objective

A target for your SLI.

  • P99 latency < 300ms
  • Error rate < 0.1%
  • 99.9% availability

💰 Error Budget

The allowable amount of failure.

  • 99.9% = 43 min/month
  • Budget consumed → slow down
  • Budget remaining → ship faster
The connection to alerting: Alert when your error budget burn rate suggests you will exhaust the budget before the end of the window. This is a symptom-based alert by definition.
This is a brief but important concept. SLIs, SLOs, and error budgets come from Google's SRE practices and represent the gold standard for symptom-based alerting. The key insight: instead of alerting on every individual error, you alert when the rate of errors threatens to breach your SLO for the measurement window. For example, if your SLO is 99.9% availability per month (43 minutes of downtime allowed), and you've consumed 30 minutes in the first week, your burn rate is too high — that's when you alert. This approach dramatically reduces noise while catching real problems. Even if you don't implement full SLO-based alerting today, understanding the concept helps you design better alerts.

📊 Monitoring Your Application

Prometheus metrics and Grafana dashboards

This final technical section brings everything together visually. Prometheus collects the metrics, Alertmanager fires the alerts, and Grafana gives you the dashboards to understand what is happening right now and what happened during an incident. For developers, Grafana dashboards are your primary operational interface — they should answer the question "is my application healthy?" at a glance.

📡 Prometheus Metrics Collection

Application
/metrics endpoint
←
Prometheus
scrapes targets
→
Grafana
queries & visualizes

Pull-Based Model

  • Prometheus scrapes your app's /metrics endpoint
  • ServiceMonitor CRDs define scrape targets
  • Default scrape interval: 15-30 seconds
  • Metrics stored in Prometheus TSDB

ServiceMonitor Example

apiVersion: monitoring.coreos.com/v1
kind: ServiceMonitor
metadata:
  name: my-app
spec:
  selector:
    matchLabels:
      app: my-app
  endpoints:
    - port: http-metrics
      interval: 15s
      path: /metrics
Prometheus uses a pull-based model — it reaches out to your application and scrapes the /metrics endpoint at a regular interval. This is different from push-based systems where the application sends metrics to a collector. The pull model has advantages: Prometheus knows immediately if a target is down (it fails to scrape), and applications don't need to know where Prometheus is. In Kubernetes, the Prometheus Operator uses ServiceMonitor custom resources to define what to scrape. Your application just needs to expose a /metrics endpoint in Prometheus exposition format — most language frameworks have libraries for this (micrometer for Java, prometheus_client for Python, promhttp for Go).

🔴 The RED Method: Application Metrics

The RED method — three signals every service should expose for request-driven workloads.

📈 Rate

Requests per second your service is handling.

rate(
  http_requests_total[5m]
)

Tells you: Is traffic normal? Are we being overloaded?

❌ Errors

Requests per second that are failing.

rate(
  http_requests_total
  {status=~"5.."}[5m]
)

Tells you: Are users experiencing failures?

⏱️ Duration

Time each request takes (latency distribution).

histogram_quantile(0.99,
  rate(http_request_duration
  _seconds_bucket[5m])
)

Tells you: How fast is the user experience?

The RED method was popularized by Tom Wilkie and is the standard framework for monitoring request-driven microservices. Rate tells you how much work the system is doing — deviations from normal traffic patterns are an early warning. Errors tell you how much of that work is failing — this directly maps to user experience. Duration tells you how fast the successful work is completing — slow responses are often as bad as errors from the user's perspective. These three signals, combined, give you a comprehensive view of your service's health. Every service you deploy should expose these three metrics at minimum. They directly feed your SLIs and SLOs.

📊 Grafana Dashboards for Application Monitoring

Essential Dashboard Panels

  • Request rate — line chart, compare to baseline
  • Error rate % — with SLO threshold line
  • Latency P50/P95/P99 — histogram heatmap
  • Active connections — gauge panel
  • Pod resource usage — CPU and memory vs limits
  • Deployment events — annotations on timeline

Dashboard Organization

  • Top row: SLO status — green/red at a glance
  • Second row: RED metrics — rate, errors, duration
  • Third row: Resource utilization — CPU, memory, replicas
  • Bottom row: Downstream dependencies — database, external APIs
Pro tip: Use Grafana template variables for namespace, deployment, and pod — one dashboard serves all environments.
A well-designed dashboard tells a story from top to bottom. Start with the most important question — "is my SLO being met?" — and then drill down into details. The SLO status row should be immediately visible as green or red, answering the "is everything OK?" question in under one second. Template variables are extremely powerful — by parameterizing namespace and deployment name, a single dashboard definition works for dev, staging, and production. Deployment annotations on the timeline are critical for correlating changes with problems — "latency spiked exactly when version 2.3.1 was deployed" is immediately visible.

🛠️ Custom Application Metrics

Metric Types

  • Counter — monotonically increasing (requests, errors, jobs completed)
  • Gauge — can go up or down (queue depth, active connections, cache size)
  • Histogram — distribution of values (request duration, response size)
  • Summary — similar to histogram, pre-computed quantiles

Example: Go Application

var ordersProcessed = prometheus.
  NewCounterVec(
    prometheus.CounterOpts{
      Name: "orders_processed_total",
      Help: "Total orders processed",
    },
    []string{"status", "payment_type"},
  )

// In your handler:
ordersProcessed.
  WithLabelValues("success","credit").
  Inc()
Cardinality warning: Avoid labels with unbounded values (user IDs, IP addresses). High cardinality = high memory usage in Prometheus.
Custom metrics go beyond the RED method to capture business-specific signals. For example, "orders processed" with a label for payment type tells you not just that your service is handling requests, but what kind of business transactions are flowing through. Counters are the most common type — always use counters for things that only go up, and apply rate() in your queries. Gauges are for point-in-time measurements. Histograms are best for latency because they let you compute arbitrary percentiles at query time. The cardinality warning is critical: if you create a metric with a user_id label and you have 10 million users, you create 10 million time series, which can crash Prometheus. Keep label values bounded — status codes, payment types, regions, not user IDs.

✨ Dashboard Best Practices

Do

  • Design for the on-call engineer at 3am
  • Include SLO threshold lines on charts
  • Add deployment annotations to correlate changes
  • Use template variables for reuse across environments
  • Link to runbooks from dashboard panel descriptions
  • Version dashboards in Git (dashboard-as-code)

Avoid

  • Dashboards with 50+ panels — split by concern
  • Vanity metrics that don't drive decisions
  • Raw counters without rate() — always show rate
  • Average-only latency — always include P99
  • No context — panels without titles or descriptions
  • Static time ranges — let users adjust the window
The "3am test" is the ultimate design principle: if an on-call engineer is woken up by an alert, they should be able to open the dashboard and understand the situation within 30 seconds. That means clear panel titles, SLO threshold lines that show "good" vs "bad" at a glance, and links to runbooks for next steps. Dashboard-as-code (exporting Grafana dashboards as JSON and storing them in Git) ensures dashboards are reviewed, versioned, and recoverable. The average-latency antipattern deserves emphasis: if your average latency is 100ms but P99 is 5 seconds, 1% of your users are having a terrible experience and the average hides it completely.
