🌐 Network Security in Kubernetes

Understanding the default behavior and why it matters

Let's start by understanding what Kubernetes networking looks like out of the box, and why that default behavior creates real security risks in production environments.

⚠️ Default Behavior: The Open Network

By default, Kubernetes has zero network isolation.

Every pod can communicate with every other pod across all namespaces — no restrictions whatsoever.

❌ What This Means

  • A compromised frontend pod can reach the database directly
  • Any pod can scan and discover services across namespaces
  • Lateral movement is trivial for an attacker
  • No boundary between dev, staging, and prod namespaces

✅ What We Want

  • Frontend can only reach backend API
  • Backend can only reach its database
  • Database accepts connections only from backend
  • Namespaces are isolated by default
This is the fundamental problem. Kubernetes networking was designed for ease of use, not security. The flat network model makes service discovery simple, but it also means that if an attacker compromises one pod, they have network access to every other pod in the cluster. This is a massive blast radius.

💥 Blast Radius & Compliance

Blast Radius

Without network policies, a single compromised pod gives an attacker access to:

  • All services in the same namespace
  • All services in other namespaces
  • Kubernetes DNS for service discovery
  • Metadata APIs (cloud provider)
  • External endpoints (data exfiltration)

Compliance Requirements

  • PCI-DSS: Requires network segmentation for cardholder data
  • HIPAA: Mandates access controls on PHI
  • SOC 2: Requires logical access controls
  • CIS Benchmarks: Recommend network policies on all namespaces
  • NIST 800-190: Container-specific network segmentation
This slide drives home two key motivations for network policies. First, the blast radius problem: without network segmentation, one compromised pod means full cluster compromise. Second, compliance: virtually every security framework requires network segmentation. If you're going through a SOC 2 audit or need PCI-DSS compliance, network policies are not optional.

🔒 Least Privilege for Networking

"Every pod should have only the network access it needs to perform its function — nothing more."
Default: Allow All
➡️
Deny All
➡️
Allow Only What's Needed

💡 The Pattern

  1. Start with deny-all — block all ingress and egress traffic
  2. Identify required flows — map your application's communication paths
  3. Create targeted allow rules — open only the ports and paths needed
  4. Test and verify — confirm both allowed and blocked traffic
This is the fundamental approach. Just like RBAC follows least privilege for API access, network policies follow least privilege for network access. The pattern is deny-all first, then progressively allow what's needed. This is the opposite of the default Kubernetes behavior, where everything is allowed and you have to explicitly deny.

🔍 Network Policies Deep Dive

Selectors, rules, and YAML in practice

Now let's dive into the mechanics. We'll look at how network policies actually work, how selectors target pods, and how ingress and egress rules control traffic flow.

⚙️ How Network Policies Work

🎯 Pod Selector

Selects which pods the policy applies to using label matching

spec:
  podSelector:
    matchLabels:
      app: backend
      tier: api

📦 Namespace Selector

Restricts traffic to/from pods in specific namespaces

namespaceSelector:
  matchLabels:
    environment: production

Key Concepts

  • Additive only: Policies can only allow, never explicitly deny
  • Namespace-scoped: Each policy lives in a namespace
  • Label-driven: Targets pods via label selectors
  • Directional: Separate ingress and egress rules
  • Implicit deny: Selecting a pod type activates deny-by-default for that direction
  • Union of policies: Multiple policies on same pod are OR'd together
Let's break down the key building blocks. A network policy has two main parts: the pod selector that determines which pods the policy applies to, and the rules that define what traffic is allowed. The podSelector uses the same label matching as services and deployments. A critical concept is that policies are additive — you can't write a policy that says "deny traffic from X." Instead, selecting a pod activates implicit deny, and then your rules define exceptions.

📥 Ingress Rules: Controlling Incoming Traffic

Ingress rules define which sources can send traffic to the selected pods.

apiVersion: networking.k8s.io/v1
kind: NetworkPolicy
metadata:
  name: backend-ingress
  namespace: production
spec:
  podSelector:
    matchLabels:
      app: backend
  policyTypes:
    - Ingress
  ingress:
    - from:
        - podSelector:
            matchLabels:
              app: frontend
        - namespaceSelector:
            matchLabels:
              name: monitoring
      ports:
        - protocol: TCP
          port: 8080

What this policy does: Allows incoming TCP traffic on port 8080 to app: backend pods from either app: frontend pods in the same namespace OR any pod in the monitoring namespace.

Let's walk through this ingress policy line by line. The podSelector targets pods with the label app=backend. The policyTypes array includes Ingress, which means we're controlling incoming traffic. The ingress rules have a "from" block with two entries — these are OR'd together. Either the traffic comes from frontend pods OR from the monitoring namespace. The ports block restricts this to TCP port 8080 only.

📤 Egress Rules: Controlling Outgoing Traffic

Egress rules define which destinations the selected pods can send traffic to.

apiVersion: networking.k8s.io/v1
kind: NetworkPolicy
metadata:
  name: backend-egress
  namespace: production
spec:
  podSelector:
    matchLabels:
      app: backend
  policyTypes:
    - Egress
  egress:
    - to:
        - podSelector:
            matchLabels:
              app: database
      ports:
        - protocol: TCP
          port: 5432
    - to:                        # DNS access (critical!)
        - namespaceSelector: {}
      ports:
        - protocol: UDP
          port: 53
        - protocol: TCP
          port: 53

⚠️ Don't forget DNS! If you restrict egress without allowing UDP/TCP port 53, your pods will lose DNS resolution and most things will break.

Egress rules are just as important as ingress. Here the backend pod can only talk to the database on port 5432 and to DNS on port 53. The DNS rule is the most common mistake when implementing egress policies — if you forget it, pods can't resolve service names and everything appears to break. The namespaceSelector with empty braces means "any namespace," which is necessary because kube-dns runs in the kube-system namespace.

📋 Policy Types & Default Behaviors

policyTypes Ingress Effect Egress Effect
[Ingress] Only listed ingress rules allowed All egress allowed (unchanged)
[Egress] All ingress allowed (unchanged) Only listed egress rules allowed
[Ingress, Egress] Only listed ingress rules allowed Only listed egress rules allowed
Omitted (with ingress rules) Only listed ingress rules allowed All egress allowed (unchanged)

💡 Key insight: Simply selecting a pod with a NetworkPolicy and including a direction in policyTypes activates default deny for that direction. If the rules array is empty, all traffic in that direction is blocked.

This table is critical for understanding how network policies work. The policyTypes field determines which directions the policy controls. If you only specify Ingress, egress traffic is completely unaffected — the default allow-all remains. The most common production pattern is to specify both Ingress and Egress and then write explicit rules for both directions.

🚫 Deny-All Policy

The foundation of network security — start by blocking everything.

Deny All Ingress & Egress

apiVersion: networking.k8s.io/v1
kind: NetworkPolicy
metadata:
  name: deny-all
  namespace: production
spec:
  podSelector: {}
  policyTypes:
    - Ingress
    - Egress

Empty podSelector {} = selects ALL pods in the namespace

No ingress/egress rules = nothing is allowed

What Happens

  • ❌ Pod-to-pod traffic blocked
  • ❌ External traffic blocked
  • ❌ DNS resolution blocked
  • ❌ Internet access blocked
  • ✅ Traffic to the Kubernetes API server still works (not pod traffic)

This is your baseline. Apply this first, then layer on allow rules for each required communication path.

This is the single most important network policy you'll write. The empty podSelector matches all pods. Including both Ingress and Egress in policyTypes with no rules means nothing is allowed. Apply this to every namespace as your baseline, then start adding allow policies on top. Note that the Kubernetes API server communication via the service account still works because that's not pod-to-pod traffic handled by the CNI.

✅ Allow Specific Traffic

Layer targeted allow rules on top of your deny-all baseline.

Allow Frontend to Backend

apiVersion: networking.k8s.io/v1
kind: NetworkPolicy
metadata:
  name: allow-frontend-to-backend
  namespace: production
spec:
  podSelector:
    matchLabels:
      app: backend
  policyTypes:
    - Ingress
  ingress:
    - from:
        - podSelector:
            matchLabels:
              app: frontend
      ports:
        - protocol: TCP
          port: 8080

Allow DNS for All Pods

apiVersion: networking.k8s.io/v1
kind: NetworkPolicy
metadata:
  name: allow-dns
  namespace: production
spec:
  podSelector: {}
  policyTypes:
    - Egress
  egress:
    - to:
        - namespaceSelector: {}
      ports:
        - protocol: UDP
          port: 53
        - protocol: TCP
          port: 53

💡 Apply the DNS policy immediately after deny-all to restore name resolution.

Here we see the practical pattern. After the deny-all policy, we layer on specific allows. The left example allows frontend pods to reach backend pods on port 8080. The right example is the essential DNS policy — without this, no pod in the namespace can resolve service names. Notice the empty namespaceSelector on the DNS policy, which means "allow DNS queries to any namespace" since kube-dns runs in kube-system.

📦 Namespace-Based Policies

Control traffic between namespaces using namespace label selectors.

apiVersion: networking.k8s.io/v1
kind: NetworkPolicy
metadata:
  name: allow-monitoring-ingress
  namespace: production
spec:
  podSelector: {}           # All pods in production
  policyTypes:
    - Ingress
  ingress:
    - from:
        - namespaceSelector:
            matchLabels:
              purpose: monitoring
          podSelector:       # AND — both must match
            matchLabels:
              app: prometheus
      ports:
        - protocol: TCP
          port: 9090

🏷️ Important: You must label your namespaces for namespace selectors to work.

kubectl label namespace monitoring purpose=monitoring
kubectl label namespace production purpose=production
Namespace-based policies are critical for multi-tenant clusters and for allowing cross-cutting concerns like monitoring and logging. In this example, we allow Prometheus pods from the monitoring namespace to scrape metrics on port 9090 from any pod in the production namespace. Note the important prerequisite: namespaces must be labeled for selectors to match. This is a common gotcha.

🌍 IPBlock Rules for External CIDR Ranges

Control traffic to and from external IP addresses outside the cluster.

apiVersion: networking.k8s.io/v1
kind: NetworkPolicy
metadata:
  name: allow-external-api
  namespace: production
spec:
  podSelector:
    matchLabels:
      app: payment-service
  policyTypes:
    - Egress
  egress:
    - to:
        - ipBlock:
            cidr: 203.0.113.0/24     # Payment gateway IP range
            except:
              - 203.0.113.128/25     # Exclude test subnet
      ports:
        - protocol: TCP
          port: 443

✅ Common Use Cases

  • Allow egress to specific external APIs
  • Restrict access to corporate subnets
  • Block traffic to cloud metadata endpoints

⚠️ Limitations

  • Cannot use DNS names, only CIDR ranges
  • Does not apply to pod-to-pod traffic within the cluster
  • IP ranges may change for cloud services
IPBlock rules handle a different use case than pod and namespace selectors. They control traffic to and from external IP addresses. This is useful for allowing access to specific external APIs — like a payment gateway — while blocking all other external traffic. The "except" field lets you carve out exclusions from a CIDR range. A key limitation is that you can't use DNS names; you must specify IP ranges, which can be challenging for cloud services with dynamic IPs.

🧠 AND vs OR Logic in Policy Rules

Understanding how multiple selectors combine is critical for correct policies.

🟢 OR Logic (Separate Array Items)

ingress:
  - from:
      - podSelector:         # Rule 1
          matchLabels:
            app: frontend
      - namespaceSelector:   # Rule 2
          matchLabels:
            name: monitoring

Match pods with app: frontend OR any pod in the monitoring namespace.

Two list items = OR

🟡 AND Logic (Same Array Item)

ingress:
  - from:
      - podSelector:         # Combined
          matchLabels:
            app: prometheus
        namespaceSelector:   # Single rule
          matchLabels:
            name: monitoring

Match pods with app: prometheus AND in the monitoring namespace.

One list item = AND

🚨 This is the #1 source of network policy misconfiguration. A misplaced dash (-) in YAML changes AND to OR, potentially opening traffic to an entire namespace instead of specific pods within it.

This is arguably the most important slide in the module. The difference between AND and OR logic in network policies comes down to a single YAML dash. On the left, the podSelector and namespaceSelector are separate list items (two dashes), so they're OR'd. On the right, they're part of the same list item (one dash), so they're AND'd. Getting this wrong is the number one cause of network policy misconfigurations. I strongly recommend reviewing every policy carefully for this exact distinction.

🐅 Calico & CNI Plugins

Why your CNI choice determines your network policy capabilities

Now let's talk about CNI plugins and why they matter for network policies. Not all CNI plugins support network policies, and those that do offer very different levels of functionality.

🔌 Why Your CNI Plugin Matters

Network policies are only enforced if your CNI plugin supports them. The Kubernetes API will accept NetworkPolicy resources regardless, but without a compatible CNI, they are silently ignored.

CNI Plugin NetworkPolicy Extended Features Notes
Calico ✅ Full GlobalNetworkPolicy, tiers, DNS policies Most widely deployed
Cilium ✅ Full L7 policies, identity-based, eBPF High performance
Weave Net ✅ Full Limited Simpler setup
Flannel ❌ None None L3 only, no policy
AWS VPC CNI ⚠️ Partial Requires Calico add-on EKS default
This is a critical point that catches many teams. If you're using Flannel, your network policies do absolutely nothing — Kubernetes will accept the YAML, but no enforcement happens. This is a false sense of security. Calico and Cilium are the two most popular choices for production clusters that need policy enforcement. On EKS, you need to install the Calico add-on alongside the AWS VPC CNI to get policy support.

🐅 Calico: Beyond Standard NetworkPolicy

🌐 GlobalNetworkPolicy

  • Cluster-wide scope (not namespaced)
  • Apply to all namespaces at once
  • Ideal for baseline security
  • Can target host endpoints
apiVersion: projectcalico.org/v3
kind: GlobalNetworkPolicy
metadata:
  name: deny-external
spec:
  selector: all()
  types:
    - Egress
  egress:
    - action: Deny
      destination:
        nets:
          - 0.0.0.0/0

📚 Policy Tiers

  • Ordered evaluation layers
  • Security team controls top tier
  • Dev teams control lower tiers
  • Higher tier overrides lower
apiVersion: projectcalico.org/v3
kind: Tier
metadata:
  name: security
spec:
  order: 100
---
apiVersion: projectcalico.org/v3
kind: Tier
metadata:
  name: platform
spec:
  order: 200

🔧 Additional Features

  • DNS policies: Allow/deny by FQDN
  • Application layer: HTTP method/path rules
  • Explicit deny: Action: Deny (not just implicit)
  • Logging: Per-rule packet logging
  • Network sets: Reusable IP ranges
  • Host protection: Policies on node interfaces
Calico extends the standard Kubernetes NetworkPolicy significantly. GlobalNetworkPolicy is cluster-scoped, so you can write a single policy that applies across all namespaces. Policy tiers let you separate security concerns: the security team manages the top tier that can override everything, while developers manage policies in lower tiers. Calico also supports explicit deny actions, DNS-based policies, and per-rule logging, none of which are available in the standard Kubernetes NetworkPolicy API.

📊 Network Policy Ordering & Precedence

Standard Kubernetes

  • No ordering — all policies are equal
  • Policies selecting the same pod are unioned (OR'd)
  • Traffic is allowed if any policy permits it
  • Cannot write an explicit deny rule
Standard NetworkPolicy Logic allowed = policy1.allows OR policy2.allows OR policy3.allows

Calico with Tiers

  • Ordered evaluation by tier priority
  • Higher tier (lower number) evaluated first
  • Explicit Allow, Deny, or Pass actions
  • Pass defers to the next tier
Security Tier
➡️
Platform Tier
➡️
App Tier
Understanding precedence is crucial. In standard Kubernetes, all policies are equal — there's no way to say "this policy overrides that one." If any policy allows traffic, it's allowed. This means you can't write an explicit deny that overrides an allow. Calico solves this with tiers. The security tier evaluates first and can explicitly deny traffic before the application tier even sees it. The Pass action in Calico lets a tier defer to the next lower tier for a decision.

🛡️ Security Principles

Defense in depth, zero trust, and microsegmentation

Now let's zoom out and look at network policies in the context of broader security principles. Network policies are one piece of a larger security strategy.

🏰 Defense in Depth

Network policies are one layer in a multi-layered security strategy.

🌐 Cloud Firewall / Security Groups
🛡️ Kubernetes Network Policies
🔒 RBAC & Pod Security Standards
🔍 Service Mesh (mTLS)
📦 Container Image Security
✅ Application-Level Auth

💡 Principle: No single layer is sufficient. Each layer reduces the blast radius if another layer is breached. Network policies are the network segmentation layer that limits lateral movement even if pod-level security is compromised.

Defense in depth means having multiple overlapping security controls so that if one fails, others still protect you. Network policies sit at the network layer, controlling which pods can talk to which. But they work alongside cloud firewalls at the infrastructure level, RBAC at the API level, service meshes for encrypted communication, image scanning for supply chain security, and application-level authentication. Each layer reduces the blast radius of a breach.

🚫 Zero Trust Networking

"Never trust, always verify. Assume the network is hostile, even inside the cluster."

❌ Traditional Model

  • Trust everything inside the perimeter
  • Focus on keeping attackers out
  • Flat internal network
  • One breach = full access

✅ Zero Trust Model

  • Trust nothing by default
  • Verify every connection
  • Segment all communication paths
  • One breach = limited blast radius

Implementing Zero Trust in Kubernetes

  • Network policies: Deny all, allow explicitly
  • mTLS: Encrypt and authenticate all pod-to-pod traffic (service mesh)
  • RBAC: Least privilege for API access
  • Pod Security: Restrict container capabilities
Zero trust is the guiding philosophy behind network policies. The traditional castle-and-moat model says "if you're inside the cluster, you're trusted." Zero trust says "nothing is trusted, ever." In Kubernetes terms, this means deny-all network policies as baseline, mTLS for encryption (via a service mesh like Istio or Linkerd), strict RBAC, and pod security standards. Network policies are the foundational layer that implements the network segmentation aspect of zero trust.

🧩 Microsegmentation

Isolating individual workloads rather than just network segments.

Traditional Segmentation

Broad network zones (DMZ, internal, database)

DMZ Zone
App Zone
DB Zone

All apps in same zone can see each other

Microsegmentation

Per-workload isolation using labels

frontend-a
frontend-b
api-a
api-b
db-a
db-b

Each workload has its own policy

Kubernetes advantage: Label-based selectors make microsegmentation natural. Each application can have its own network policy without managing IP addresses or subnets.

Microsegmentation takes network isolation down to the individual workload level. Instead of broad zones where all apps in the same zone can see each other, each application has its own fine-grained policy. Kubernetes is actually very well suited for this because network policies use label selectors rather than IP addresses. You can write a policy that says "this specific application can only talk to that specific database" without ever knowing the IPs involved.

🔗 Combining Network Policies with Other Controls

Security Control What It Protects Complements Network Policies By
RBAC API access Preventing unauthorized policy modification
Pod Security Standards Container runtime Preventing privilege escalation that bypasses policies
Service Mesh (Istio/Linkerd) Transport layer Adding mTLS encryption + L7 authorization
OPA/Gatekeeper Policy enforcement Requiring network policies on every namespace
Secrets Management Credentials Ensuring compromised network access doesn't leak secrets

💡 Pro tip: Use OPA/Gatekeeper to enforce that every namespace has a deny-all network policy. This prevents teams from accidentally deploying without network isolation.

Network policies don't exist in a vacuum. They're most effective when combined with other security controls. RBAC prevents unauthorized users from modifying network policies. Pod security standards prevent containers from gaining privileges that could bypass network controls. A service mesh adds encryption and L7 authorization on top of the L3/L4 controls that network policies provide. And OPA/Gatekeeper can enforce organizational requirements like "every namespace must have a deny-all policy."

🏗️ Common Pattern: Three-Tier Isolation

The classic frontend → backend → database isolation pattern.

🌐 Frontend
port 80
➡️
⚙️ Backend API
port 8080
➡️
💾 Database
port 5432

Frontend Policy

  • Ingress: Allow from ingress controller
  • Egress: Allow to backend:8080 + DNS

Backend Policy

  • Ingress: Allow from frontend:8080
  • Egress: Allow to database:5432 + DNS

Database Policy

  • Ingress: Allow from backend:5432
  • Egress: DNS only

Result: Frontend cannot reach the database. Database cannot reach the internet. Each tier can only communicate with its immediate neighbor.

This is the most common network policy pattern you'll implement. The three-tier app with frontend, backend, and database. Each tier has its own ingress and egress policies. The key outcomes are: the frontend can never directly access the database, the database has no outbound internet access, and each tier can only talk to its immediate neighbor. This dramatically reduces the blast radius of a compromise at any tier.

🌐 Default Behavior: Kubernetes allows all pod-to-pod traffic by default — a significant security risk in production environments.
🔒 Least Privilege: Start with deny-all policies, then progressively allow only the traffic each workload actually needs.
🔍 Policy Mechanics: Pod selectors, namespace selectors, and IPBlock rules control traffic. AND vs OR logic depends on YAML structure.
🔌 CNI Requirement: Network policies require a compatible CNI plugin (Calico, Cilium). Without one, policies are silently ignored.
🐅 Calico Extensions: GlobalNetworkPolicy, policy tiers, explicit deny actions, and DNS-based rules extend the standard API.
🛡️ Security Principles: Network policies implement zero trust and microsegmentation as part of a defense-in-depth strategy.
