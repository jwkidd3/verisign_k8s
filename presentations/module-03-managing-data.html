The Challenge of Stateful Applications

Why data persistence matters in Kubernetes

Before we dive into PVs and PVCs, we need to understand the fundamental tension between container ephemerality and stateful workloads. This context will make the storage APIs much more intuitive.

💨 The Ephemeral Container

Core principle: Every container starts from a clean filesystem image. Any data written inside the container is lost when the container stops or restarts.

  • Container filesystem is a copy-on-write layer on top of the image
  • Pod restarts, rescheduling, and scaling all destroy container-local data
  • Even a simple kubectl rollout restart wipes all in-container state
  • This is by design -- immutability enables reproducibility and scale
Emphasize that this is a feature, not a bug. The immutability of containers is what makes them reliable and scalable. The challenge is that many real-world applications need to persist data beyond the lifecycle of any single container.

💾 Why Persistence Matters

Applications That Need State

  • Databases (PostgreSQL, MySQL, MongoDB)
  • Message queues (RabbitMQ, Kafka)
  • File uploads and media storage
  • Machine learning model artifacts
  • Session data and caches

What Happens Without Persistence

  • Database tables vanish on pod restart
  • Uploaded files disappear
  • Queue messages are lost
  • Application state resets to zero
  • Users experience data loss
Walk through concrete examples participants will recognize. Most production workloads have at least one stateful component. Even microservices architectures typically require databases, caches, or file storage somewhere in the stack.

⚖️ Stateful vs Stateless Patterns

Characteristic Stateless Stateful
Data storage External (DB, cache, object store) Local volumes or attached storage
Scaling Horizontal, any replica is identical Requires careful coordination
Pod identity Interchangeable Unique, stable network identity
Kubernetes resource Deployment StatefulSet
Recovery Replace and go Must reattach data volume

Best practice: Design stateless whenever possible. When state is unavoidable, use Kubernetes storage primitives to manage it explicitly.

This comparison helps participants decide which pattern fits their workload. Stateless is always simpler to operate, but databases and similar workloads are inherently stateful. Kubernetes provides robust tools for both patterns.

Persistent Volumes & Claims

PV, PVC, and StorageClass

Now we enter the core of Kubernetes storage. The PV/PVC model separates storage provisioning from storage consumption, following the same separation-of-concerns philosophy that runs through all of Kubernetes.

🏗️ Storage Architecture

PersistentVolume (PV)

  • A piece of storage in the cluster
  • Provisioned by an admin or dynamically
  • Cluster-scoped resource
  • Independent of any single pod

PersistentVolumeClaim (PVC)

  • A request for storage by a user
  • Namespace-scoped resource
  • Binds to a matching PV
  • Referenced by pods

StorageClass

  • Defines a "class" of storage
  • Specifies the provisioner
  • Enables dynamic provisioning
  • Parameters vary by provider
Think of PVs as the supply side and PVCs as the demand side. StorageClass is the blueprint that connects them. This three-part model decouples storage management from application deployment, allowing platform teams and developers to work independently.

🔗 How PV Binding Works

Lifecycle: Provisioning → Binding → Using → Releasing → Reclaiming

  1. Provisioning -- Admin creates a PV (static) or StorageClass triggers creation (dynamic)
  2. Binding -- Kubernetes matches a PVC to a PV based on size, access mode, and StorageClass
  3. Using -- Pod mounts the PVC as a volume; reads and writes flow to the underlying storage
  4. Releasing -- Pod is deleted; PVC is deleted; PV enters "Released" state
  5. Reclaiming -- PV is retained, deleted, or recycled based on reclaim policy
Walk through each step carefully. The binding step is automatic -- Kubernetes finds the best match. If no PV satisfies the PVC, the claim stays in Pending state until a suitable PV becomes available (or dynamic provisioning creates one).

⚙️ Static vs Dynamic Provisioning

Static Provisioning

  • Admin pre-creates PV objects
  • PV defines capacity, access mode, and backend
  • PVC binds to an existing PV
  • Good for pre-allocated infrastructure
  • Requires manual lifecycle management

Dynamic Provisioning

  • PVC references a StorageClass
  • StorageClass triggers the provisioner
  • PV is created automatically on demand
  • Preferred for cloud-native workflows
  • Self-service for developers

Production recommendation: Use dynamic provisioning with StorageClasses. It reduces operational overhead and gives developers autonomy within guardrails set by the platform team.

Dynamic provisioning is the standard in most production clusters. Static provisioning is still useful for pre-existing storage (e.g., NFS shares, SAN LUNs) or for highly regulated environments where storage must be pre-approved.

🔒 Access Modes

Mode Abbreviation Description Use Case
ReadWriteOnce RWO Read-write by a single node Databases, single-instance apps
ReadOnlyMany ROX Read-only by many nodes Shared config, static assets
ReadWriteMany RWX Read-write by many nodes Shared uploads, CMS content

Important: Not all storage backends support all access modes. Cloud block storage (EBS, Azure Disk) typically supports only RWO. For RWX, you need a file-based backend such as NFS, EFS, or CephFS.

Access modes are a common source of confusion. RWO means the volume can be mounted as read-write by pods on a single node, not by a single pod. Multiple pods on the same node can share an RWO volume. Also mention ReadWriteOncePod (RWOP) which was introduced in Kubernetes 1.22 for single-pod access.

♻️ Reclaim Policies

Retain

  • PV is kept after PVC deletion
  • Data is preserved
  • Admin must manually clean up
  • Safest for production data

Delete

  • PV and underlying storage are deleted
  • Default for dynamic provisioning
  • Automatic cleanup
  • Risk of accidental data loss

Recycle (Deprecated)

  • Runs rm -rf on the volume
  • Makes PV available again
  • Deprecated in favor of dynamic provisioning
  • Avoid in new deployments
Retain is the safest policy for production databases. Delete is convenient for ephemeral workloads and CI/CD pipelines. Recycle is deprecated and should not be used -- dynamic provisioning with Delete achieves the same goal more reliably. Remind participants to check their StorageClass default reclaim policy.

📝 PersistentVolume YAML

apiVersion: v1
kind: PersistentVolume
metadata:
  name: my-pv
  labels:
    type: local
spec:
  capacity:
    storage: 10Gi
  accessModes:
    - ReadWriteOnce
  persistentVolumeReclaimPolicy: Retain
  storageClassName: standard
  hostPath:
    path: /mnt/data

Key fields: capacity defines size, accessModes controls who can mount, persistentVolumeReclaimPolicy determines what happens after release, and the backend section (hostPath, nfs, csi) specifies the actual storage.

This is a static PV using hostPath, which is suitable for single-node development clusters. In production, you would use a CSI driver or cloud-native volume type instead of hostPath. Point out that PVs are cluster-scoped -- there is no namespace field.

📝 PersistentVolumeClaim YAML

apiVersion: v1
kind: PersistentVolumeClaim
metadata:
  name: my-pvc
  namespace: default
spec:
  accessModes:
    - ReadWriteOnce
  resources:
    requests:
      storage: 5Gi
  storageClassName: standard

Binding: Kubernetes will find a PV that matches the requested accessModes, has at least the requested storage capacity, and shares the same storageClassName. If dynamic provisioning is enabled, a new PV is created automatically.

Note that the PVC requests 5Gi but the PV we defined earlier has 10Gi. Kubernetes will bind them because the PV meets the minimum requirement. The PVC is namespace-scoped, so different teams can create PVCs independently. Emphasize that storageClassName must match between PV and PVC for static binding.

📦 Mounting Volumes in Pods

apiVersion: apps/v1
kind: Deployment
metadata:
  name: my-app
spec:
  replicas: 1
  selector:
    matchLabels:
      app: my-app
  template:
    metadata:
      labels:
        app: my-app
    spec:
      containers:
        - name: app
          image: nginx:1.25
          volumeMounts:
            - name: data-volume
              mountPath: /usr/share/nginx/html
      volumes:
        - name: data-volume
          persistentVolumeClaim:
            claimName: my-pvc
Two pieces connect the PVC to the container: the volumes section declares the PVC by name, and the volumeMounts section mounts it at a specific path inside the container. The volume name (data-volume) ties the two together. Multiple containers in the same pod can mount the same volume at different paths.

🏭 StorageClass Deep Dive

apiVersion: storage.k8s.io/v1
kind: StorageClass
metadata:
  name: fast-ssd
  annotations:
    storageclass.kubernetes.io/is-default-class: "false"
provisioner: ebs.csi.aws.com
parameters:
  type: gp3
  iops: "5000"
  throughput: "250"
reclaimPolicy: Retain
allowVolumeExpansion: true
volumeBindingMode: WaitForFirstConsumer

provisioner: The CSI driver or in-tree plugin responsible for creating volumes

volumeBindingMode: WaitForFirstConsumer delays binding until a pod is scheduled, enabling topology-aware placement

StorageClasses are the bridge between Kubernetes and the underlying storage infrastructure. The provisioner field determines which driver handles volume creation. Parameters are provider-specific. WaitForFirstConsumer is recommended for cloud environments to ensure the volume is created in the same availability zone as the pod.

☁️ Common Storage Providers

Provider CSI Driver Volume Type Access Modes
AWS EBS ebs.csi.aws.com Block (gp3, io2) RWO
AWS EFS efs.csi.aws.com File (NFS) RWX
Azure Disk disk.csi.azure.com Block (Premium SSD) RWO
GCE PD pd.csi.storage.gke.io Block (SSD, Standard) RWO
Longhorn driver.longhorn.io Block (replicated) RWO, RWX
This table gives participants a quick reference for the most common storage backends. Emphasize that the CSI (Container Storage Interface) standard means Kubernetes storage is pluggable -- any vendor can write a CSI driver. Longhorn is a popular open-source option for bare-metal clusters.

When to Use Ephemeral Volumes

Temporary storage for non-persistent needs

Not every volume needs to survive a pod restart. Kubernetes provides several ephemeral volume types for scratch space, configuration injection, and metadata exposure. Knowing when to use these instead of PVCs simplifies your architecture.

📂 emptyDir Volumes

emptyDir creates a temporary directory that exists for the lifetime of the pod. It starts empty and is deleted when the pod is removed.

Use Cases

  • Scratch space for sorting or processing
  • Sharing files between containers in a pod
  • Cache that can be rebuilt
  • Checkpointing for long computations
volumes:
  - name: scratch
    emptyDir: {}
  - name: cache
    emptyDir:
      medium: Memory
      sizeLimit: 256Mi

medium: Memory backs the volume with tmpfs (RAM). Fast, but counts against the container memory limit.

emptyDir is the most common ephemeral volume type. A typical pattern is a sidecar container that processes files written by the main container -- they communicate through a shared emptyDir. The Memory medium is excellent for sensitive data that should never touch disk.

🗂️ configMap and secret Volumes

configMap Volume

volumes:
  - name: config
    configMap:
      name: app-config
      items:
        - key: app.properties
          path: application.properties
  • Projects ConfigMap keys as files
  • Updates propagate automatically
  • Read-only by default

secret Volume

volumes:
  - name: certs
    secret:
      secretName: tls-certs
      defaultMode: 0400
  • Projects Secret keys as files
  • Stored in tmpfs (never written to disk)
  • Set restrictive file permissions
configMap and secret volumes are the standard way to inject configuration and credentials into containers as files. They are ephemeral -- the data comes from the Kubernetes API, not from persistent storage. Mention that secret volumes use tmpfs, so the sensitive data only lives in memory on the node.

📡 downwardAPI Volumes

volumes:
  - name: pod-info
    downwardAPI:
      items:
        - path: "labels"
          fieldRef:
            fieldPath: metadata.labels
        - path: "cpu-limit"
          resourceFieldRef:
            containerName: app
            resource: limits.cpu

The Downward API exposes pod and container metadata as files. Useful for self-discovery -- applications can read their own labels, annotations, resource limits, and namespace without calling the Kubernetes API.

The downwardAPI volume is less commonly used but valuable in automation scenarios. For example, a monitoring sidecar can read pod labels to tag metrics, or an application can read its own resource limits to configure thread pools appropriately.

🤔 Persistent vs Ephemeral Decision Guide

Question If Yes If No
Must data survive pod restarts? PV/PVC emptyDir
Shared across multiple pods? PV with RWX / RWO emptyDir or configMap
Configuration or secrets? configMap / secret volume --
Pod metadata needed at runtime? downwardAPI --
Temporary processing space? emptyDir PV/PVC

Use the simplest volume type that meets your requirements.

This decision guide helps participants quickly identify the right volume type. The key question is always: does this data need to survive a pod restart? If not, an ephemeral volume is simpler and more efficient. If yes, PVCs are the answer.

Basic Backup Concepts

Protecting application data in Kubernetes

Now that we understand how to provision and mount storage, we need to consider how to protect the data that lives on those volumes. This section covers fundamental backup strategies and the division of responsibility between developers and platform teams.

🛡️ Data Is Externalized

Key insight: Kubernetes storage is backed by external systems (cloud disks, NFS, SAN). Backups happen at the storage layer, not inside the container.

  • PVs are references to external storage resources
  • The storage backend (EBS, Azure Disk, Ceph) handles durability and replication
  • Backups operate on the underlying volume, not on the pod
  • Kubernetes orchestrates attachment -- the storage system manages data integrity
This is an important mental model shift. When participants think about backing up a database in Kubernetes, they should think about backing up the PersistentVolume (via snapshots or external tools) rather than running backup commands inside the container -- though application-level backups are also valuable.

📊 Backup Strategies

Application-Level Backups

  • pg_dump, mysqldump, mongodump
  • Logical backups (SQL, JSON)
  • Application-consistent snapshots
  • Can run as CronJobs in Kubernetes
  • Portable across storage backends

Infrastructure-Level Backups

  • Volume snapshots (CSI VolumeSnapshot)
  • Cloud-native snapshots (EBS, Azure)
  • Block-level, fast and efficient
  • Tools: Velero, Kasten K10
  • Can capture entire cluster state

Best practice: Combine both approaches. Application-level backups ensure logical consistency. Infrastructure-level snapshots provide fast recovery and disaster protection.

Neither approach alone is sufficient for production. A volume snapshot may capture a database mid-transaction, leading to corruption on restore. An application-level dump ensures consistency but is slower and uses more storage. The combination provides both speed and reliability.

📸 Volume Snapshot Concepts

apiVersion: snapshot.storage.k8s.io/v1
kind: VolumeSnapshot
metadata:
  name: my-data-snapshot
spec:
  volumeSnapshotClassName: csi-snapclass
  source:
    persistentVolumeClaimName: my-pvc
  • VolumeSnapshot -- captures the state of a PVC at a point in time
  • VolumeSnapshotClass -- defines the snapshot driver and parameters
  • VolumeSnapshotContent -- the actual snapshot resource (like a PV for snapshots)
  • Restore by creating a new PVC with dataSource referencing the snapshot
Volume snapshots are a Kubernetes-native API that works through CSI drivers. Not all storage backends support snapshots. The three-object model (Snapshot, SnapshotClass, SnapshotContent) mirrors the PV/PVC/StorageClass pattern. Snapshots are incremental on most backends, making them storage-efficient.

👥 Shared Responsibility Model

Developer Responsibilities

  • Design applications for data durability
  • Define PVC requirements (size, access mode)
  • Implement application-level backups
  • Test restore procedures
  • Handle graceful shutdown and data flushing

Platform Team Responsibilities

  • Provision and maintain StorageClasses
  • Configure volume snapshot infrastructure
  • Set up automated backup schedules
  • Monitor storage capacity and performance
  • Manage disaster recovery runbooks

Reliable data management requires collaboration between developers and platform teams.

This slide is important for setting expectations. Developers own the application-level concerns: what data matters, how to export it, and how to restore it. Platform teams own the infrastructure: StorageClasses, snapshot policies, monitoring, and disaster recovery. Neither can do it alone effectively.

Key Takeaways

  • Containers are ephemeral: Any data stored inside a container is lost on restart -- external storage is required for persistence
  • PV/PVC separation: PersistentVolumes represent storage supply; PersistentVolumeClaims represent developer demand -- Kubernetes binds them automatically
  • Dynamic provisioning: StorageClasses eliminate manual PV creation and enable self-service storage for development teams
  • Access modes matter: Choose RWO for single-node workloads, ROX for shared reads, and RWX (with compatible storage) for shared writes
  • Ephemeral volumes: Use emptyDir for scratch space, configMap/secret for configuration injection, and downwardAPI for pod metadata
  • Backup both layers: Combine application-level dumps with infrastructure-level volume snapshots for comprehensive data protection
These six takeaways summarize the module. The most critical concepts for day-to-day work are the PV/PVC model, dynamic provisioning via StorageClasses, and the distinction between ephemeral and persistent volume types. Backup strategies become critical as workloads move toward production.
