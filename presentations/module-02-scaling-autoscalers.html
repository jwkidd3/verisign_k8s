💰 Resource Requests & Limits

The foundation everything else depends on

Before we can talk about autoscaling, we need to understand resource requests and limits. These are the foundation that the scheduler and autoscaler both depend on.

💰 Requests vs. Limits

Requests (Guaranteed)

  • What the scheduler uses for placement
  • Guaranteed minimum resources
  • Used by HPA for percentage calculations
  • Must be set for HPA to work

Limits (Maximum)

  • Enforced ceiling the container cannot exceed
  • CPU: throttled when exceeded
  • Memory: OOMKilled when exceeded
  • Protects neighbors on the same node
resources:
  requests:
    cpu: "250m"       # 0.25 CPU cores guaranteed
    memory: "256Mi"   # 256 MiB guaranteed
  limits:
    cpu: "500m"       # Can burst to 0.5 cores, then throttled
    memory: "512Mi"   # Killed if exceeds 512 MiB
The distinction between requests and limits is critical. Requests are a scheduling guarantee — the scheduler will only place your pod on a node that has at least that much capacity available. Limits are enforcement — the kubelet enforces these at runtime. CPU is compressible (throttled), memory is not (OOMKilled).

🔢 Understanding Resource Units

CPU Units

ValueMeaning
11 vCPU / 1 core
500mHalf a core (500 millicores)
250mQuarter core
100m1/10th of a core

1000m = 1 CPU. Always compressible.

Memory Units

ValueMeaning
128Mi128 mebibytes (2^20)
256Mi256 MiB
1Gi1 gibibyte (2^30)
512M512 megabytes (10^6)

Use Mi/Gi (binary). Not compressible.

Common pitfall: 128M (megabytes) ≠ 128Mi (mebibytes). Always use Mi/Gi for consistency with how Kubernetes actually allocates memory.
Millicores are the standard CPU unit — think of them as thousandths of a CPU core. For memory, always use Mi and Gi (binary units) because that's what Kubernetes uses internally. Using M or G (decimal) can cause subtle allocation mismatches.

🎯 Quality of Service (QoS) Classes

QoS Class Criteria Eviction Priority
Guaranteed requests == limits for all containers Last to be evicted
Burstable requests < limits (or only requests set) Evicted after BestEffort
BestEffort No requests or limits set First to be evicted 
# Guaranteed QoS — best for production critical workloads
resources:
  requests:
    cpu: "500m"
    memory: "512Mi"
  limits:
    cpu: "500m"      # Same as request
    memory: "512Mi"  # Same as request
Recommendation: Production workloads should be Guaranteed or Burstable. Never deploy BestEffort to production.
QoS class determines what happens under memory pressure. When a node runs low on memory, the kubelet evicts BestEffort pods first, then Burstable, and Guaranteed last. For critical services, Guaranteed QoS gives you the most protection. However, Burstable is often a good trade-off that allows efficient resource utilization while still providing baseline guarantees.

📊 Metrics Server

The data pipeline for autoscaling

Before the HPA can make scaling decisions, it needs data. The Metrics Server is the component that provides this data. Let's understand how it works.

📊 Metrics Server Architecture

How metrics flow through the system

  1. kubelet collects resource usage from the container runtime (containerd)
  2. Metrics Server scrapes kubelet's /metrics/resource endpoint every 15 seconds
  3. Metrics Server aggregates and serves data via the metrics.k8s.io API
  4. HPA controller queries metrics.k8s.io every 15 seconds (default)
  5. HPA controller calculates desired replicas and patches the Deployment
# Verify Metrics Server is running on your cluster
kubectl get deployment metrics-server -n kube-system

# Check the metrics API is available
kubectl get apiservice v1beta1.metrics.k8s.io

# View node-level metrics
kubectl top nodes

# View pod-level metrics
kubectl top pods -n my-namespace
The Metrics Server is a prerequisite for HPA. On EKS, it's typically installed as an add-on. It's an in-memory store — it only keeps the latest metrics, not historical data. For historical metrics and dashboards, you need Prometheus. The 15-second scrape interval means there's always some lag in autoscaling decisions.

⚡ Horizontal Pod Autoscaler

The core scaling mechanism

Now we get to the main event — the Horizontal Pod Autoscaler. This is the primary autoscaling mechanism most developers will use. We'll cover the algorithm, the API, and the tuning options.

🧮 The HPA Algorithm

Core Formula

desiredReplicas = ⌈ currentReplicas × ( currentMetric / desiredMetric ) ⌉

Example Calculation

ParameterValue
Current replicas3
CPU request per pod250m
Current average CPU usage200m per pod (80%)
Target CPU utilization50%
Calculation⌈ 3 × (80 / 50) ⌉ = ⌈ 4.8 ⌉ = 5 replicas
Key detail: The HPA uses the ratio of current to desired metric values. It's percentage-based against requests, not limits.
The algorithm is straightforward once you understand it. The ceiling function means Kubernetes always rounds up — it prefers to have slightly more capacity than slightly less. The key insight is that HPA computes utilization as a percentage of the REQUEST, not the limit. This is why setting appropriate requests is critical for HPA to work correctly.

🔢 HPA Algorithm — Edge Cases

  • Tolerance zone (±10%): HPA won't scale if the ratio is within 0.9–1.1 of the target. This prevents flapping.
  • Missing metrics: Pods with missing metrics are treated conservatively — scale-down assumes 100% utilization, scale-up assumes 0%.
  • Unready pods: Excluded from the metric average during scale-up (included for scale-down).
  • Multiple metrics: HPA evaluates each metric independently and takes the maximum desired replica count.
The 10% tolerance band is critical for stability. If your target is 50% CPU and current usage is between 45%–55%, the HPA takes no action. This prevents constant scaling oscillation.
These edge cases are what make HPA production-ready. Without the tolerance zone, you'd get constant scaling up and down by 1 replica. Without the conservative treatment of missing metrics, a pod that's temporarily unreachable could cause inappropriate scale-down. These details matter when troubleshooting why HPA isn't behaving as expected.

Autoscaling depends on proper resource specifications

Requests

  • Guaranteed resources for the container
  • Used by the scheduler for placement

  • Maximum resources the container can use

Stress that without resource requests, HPA cannot calculate utilization percentages. This is the most common misconfiguration. CPU is measured in millicores (1000m = 1 core). Memory in MiB or GiB.

🔢 How HPA Uses Resource Requests

HPA Formula:

desiredReplicas = ceil[ currentReplicas × ( currentMetric / targetMetric ) ]

Example Calculation

  • 3 Pods, each requesting 200m CPU
  • Target utilization: 50% (100m per Pod)
  • Actual usage: 160m per Pod (80%)
  • Desired = ceil[ 3 × (160 / 100) ] = ceil[ 4.8 ] = 5 Pods
Important: Without resource requests, HPA cannot compute utilization-based targets. Always set requests on containers you want to autoscale.
Walk through the math. This formula is the core of HPA's control loop. The ceil function ensures we always round up so we don't under-provision. Show that higher actual usage drives replica count up.

↔️ Horizontal Pod Autoscaler (HPA)

Automatically adjusting replica counts based on metrics

This is the core section of the module. HPA is the most widely used autoscaler in Kubernetes. We'll go deep on configuration, behavior, and best practices.

🔄 How HPA Works

┌──────────────┐     queries      ┌────────────────┐
│  HPA Control │ ───────────────▶ │ Metrics Server  │
│    Loop      │                  │ (or custom API) │
│  (every 15s) │ ◀─────────────── │                 │
└──────┬───────┘   metric values  └────────────────┘

┌──────────────┐

  • HPA controller runs a control loop every 15 seconds (default)
  • Queries the Metrics API for current resource utilization
  • Compares actual vs target, then adjusts spec.replicas
  • Updates the Scale subresource of the target workload
The 15-second interval is configurable via the --horizontal-pod-autoscaler-sync-period flag on the controller manager. In practice the default is fine for most workloads.

📊 HPA with Custom Metrics

apiVersion: autoscaling/v2
kind: HorizontalPodAutoscaler
metadata:
  name: myapp-hpa
spec:
  scaleTargetRef:
    apiVersion: apps/v1
    kind: Deployment
    name: myapp
  minReplicas: 3
  maxReplicas: 50
  metrics:
  # Standard CPU metric
  - type: Resource
    resource:
      name: cpu
      target:
        type: Utilization
        averageUtilization: 60
  # Custom metric: requests per second per pod
  - type: Pods
    pods:
      metric:
        name: http_requests_per_second
      target:
        type: AverageValue
        averageValue: "1000"
  # External metric: SQS queue depth
  - type: External
    external:
      metric:
        name: sqs_queue_length
        selector:
          matchLabels:
            queue: processing-queue
      target:
        type: Value
        value: "100"
Custom metrics are where HPA gets really powerful. The Pods type targets a metric exposed by the pods themselves (via Prometheus adapter). The External type targets metrics from outside the cluster — like an SQS queue depth. This requires a custom metrics adapter (like prometheus-adapter or KEDA) to be installed on the cluster. The highest desired replica count across all metrics wins.

⚙️ HPA Behavior Policies

apiVersion: autoscaling/v2
kind: HorizontalPodAutoscaler
metadata:
  name: myapp-hpa
spec:
  scaleTargetRef:
    apiVersion: apps/v1
    kind: Deployment
    name: myapp
  minReplicas: 3
  maxReplicas: 100
  metrics:
  - type: Resource
    resource:
      name: cpu
      target:
        type: Utilization
        averageUtilization: 60
  behavior:
    scaleUp:
      stabilizationWindowSeconds: 0     # Scale up immediately
      policies:
      - type: Percent
        value: 100                       # Double current replicas
        periodSeconds: 60
      - type: Pods
        value: 10                        # Or add 10 pods
        periodSeconds: 60
      selectPolicy: Max                  # Use whichever adds more
    scaleDown:
      stabilizationWindowSeconds: 300    # Wait 5 min before scaling down
      policies:
      - type: Percent
        value: 10                        # Remove 10% of replicas
        periodSeconds: 60
      selectPolicy: Min                  # Use whichever removes fewer
Behavior policies are the fine-tuning knobs for HPA. The key pattern: scale UP aggressively (respond to spikes quickly) and scale DOWN conservatively (prevent premature scale-down). The stabilizationWindowSeconds for scale-down means HPA looks at the highest recommended replica count over the last 5 minutes — this prevents flapping on bursty workloads.

🕒 Stabilization Windows Explained

How Stabilization Works

The stabilization window defines a time period during which the HPA examines all computed replica counts and selects the most appropriate one.

Scale-Up Window

  • Default: 0 seconds
  • HPA selects the highest recommendation from the window
  • 0 = immediate scale-up
  • Set higher to smooth out transient spikes

Scale-Down Window

  • Default: 300 seconds (5 min)
  • HPA selects the highest recommendation from the window
  • Prevents premature scale-down
  • Increase for workloads with slow ramp-down
Real-world tip: For DNS-related services, consider a longer scale-down window (600s+) because DNS caching can redirect traffic back to recently removed endpoints.
The stabilization window is a sliding window. For scale-down with a 300-second window, the HPA looks at all the recommendations it computed in the last 5 minutes and picks the highest one (most conservative). This means even if utilization drops suddenly, the HPA won't immediately scale down — it waits to see if the drop is sustained. This is especially important for Verisign's DNS-intensive workloads.

📈 Vertical Pod Autoscaler

Right-sizing pod resources

While HPA scales the number of pods, VPA scales the resources allocated to each pod. Let's explore this complementary approach.

📈 VPA — Vertical Pod Autoscaler

What VPA does

VPA automatically adjusts CPU and memory requests and limits for pods based on observed usage patterns.

Recommender

Watches resource usage and computes optimal requests/limits

Updater

Evicts pods that need resource updates (triggers restart with new values)

Admission Controller

Mutates new pods to use recommended resources on creation

VPA has three components that work together. The Recommender watches metrics and computes suggestions. The Updater evicts pods that are significantly over or under-resourced. The Admission Controller intercepts pod creation and injects the recommended values. Note that VPA requires pod restarts to apply changes — it cannot change resources on a running pod.

⚙️ VPA Operating Modes

Mode Recommends Applies on Create Evicts to Update Use Case
Off ✅ ❌ ❌ Dry run — observe recommendations
Initial ✅ ✅ ❌ Apply on new pods only
Auto ✅ ✅ ✅ Fully automatic (restarts pods) 
apiVersion: autoscaling.k8s.io/v1
kind: VerticalPodAutoscaler
metadata:
  name: myapp-vpa
spec:
  targetRef:
    apiVersion: apps/v1
    kind: Deployment
    name: myapp
  updatePolicy:
    updateMode: "Off"   # Start with Off to observe
  resourcePolicy:
    containerPolicies:
    - containerName: myapp
      minAllowed:
        cpu: "100m"
        memory: "128Mi"
      maxAllowed:
        cpu: "2"
        memory: "4Gi"
Start with Off mode to observe VPA's recommendations before letting it make changes. This is especially important in production. The minAllowed and maxAllowed provide guardrails to prevent VPA from setting unreasonable values. Important caveat: do NOT use VPA in Auto mode with HPA on the same CPU/memory metrics — they will conflict. You can combine them if HPA uses custom metrics and VPA manages CPU/memory.

⚠️ VPA + HPA — Compatibility

Do NOT use VPA (Auto mode) and HPA on the same metrics (CPU/memory) simultaneously. They will fight each other.

❌ Conflicting Setup

  • HPA scales on CPU utilization
  • VPA adjusts CPU requests
  • VPA changes requests → HPA sees different utilization → scales differently → VPA readjusts...

✅ Compatible Setup

  • HPA scales on custom metrics (RPS, queue depth)
  • VPA manages CPU/memory requests
  • Each autoscaler owns different dimensions
  • Or: VPA in Off mode for recommendations only
This is one of the most common mistakes teams make. If HPA targets CPU and VPA adjusts CPU requests, they create a feedback loop. The safe patterns are: use VPA in Off mode alongside HPA (just for recommendations), or have HPA scale on custom metrics while VPA handles resource sizing.

⚡ KEDA

Kubernetes Event-Driven Autoscaling

KEDA extends Kubernetes autoscaling to handle event-driven workloads and scale-to-zero scenarios. Let's understand where it fits.

⚡ KEDA — Event-Driven Autoscaling

What KEDA adds beyond HPA

KEDA is a lightweight component that can scale Kubernetes workloads based on external event sources — and can scale to zero.

KEDA Capabilities

  • Scale to/from zero replicas
  • 60+ built-in scalers (SQS, Kafka, Prometheus, Cron, etc.)
  • Works alongside native HPA
  • ScaledObject and ScaledJob resources

When to use KEDA

  • Queue-based workers (SQS, Kafka)
  • Schedule-based scaling (cron patterns)
  • Scale-to-zero for cost savings
  • Complex multi-metric scaling logic
KEDA is not a replacement for HPA — it extends it. Under the hood, KEDA creates and manages HPA objects. Its superpower is the ability to scale to zero and the rich library of event source scalers. For workloads like queue consumers that should only exist when there's work to do, KEDA is the right choice.

📝 KEDA ScaledObject Example

apiVersion: keda.sh/v1alpha1
kind: ScaledObject
metadata:
  name: sqs-consumer-scaler
spec:
  scaleTargetRef:
    name: sqs-consumer          # Deployment to scale
  pollingInterval: 15            # Check every 15s
  cooldownPeriod: 60             # Wait 60s before scale-to-zero
  minReplicaCount: 0             # Can scale to zero!
  maxReplicaCount: 50

      queueURL: https://sqs.us-east-1.amazonaws.com/123456789/processing-queue

      name: aws-credentials      # TriggerAuthentication reference
Scale-to-zero: When the SQS queue is empty, KEDA scales the Deployment to 0 replicas. When messages arrive, it scales back up. This can significantly reduce costs for bursty workloads.
This example shows a common pattern: a worker that processes messages from an SQS queue. When the queue is empty, why pay for running pods? KEDA scales to zero. When messages arrive, KEDA scales up based on queue depth. The queueLength of 5 means KEDA targets 5 messages per pod — so 50 messages in the queue would result in 10 pods.

🎯 Scaling Best Practices

✅ Do

  • Always set resource requests
  • Start with CPU-based HPA at 60-70% target
  • Use conservative scale-down (300s+ window)
  • Set meaningful min/max replicas
  • Monitor HPA events and conditions
  • Load test to validate scaling behavior
  • Use VPA in Off mode to get right-sizing data

❌ Don't

  • Skip resource requests (HPA won't work)
  • Set target too high (90%+, no headroom)
  • Set target too low (20%, over-provision)
  • Use VPA Auto + HPA on same metric
  • Set maxReplicas too low for actual demand
  • Ignore readiness probes (affects scaling lag)
  • Forget about cluster-level capacity
These best practices come from real-world production experience. The 60-70% CPU target is a common starting point — it leaves enough headroom for the scaling lag we discussed. Load testing is essential to validate that your scaling configuration can actually handle your traffic patterns. And always remember: pod autoscaling is useless if the cluster doesn't have node capacity — that's where the Cluster Autoscaler comes in.

Scale-Up Strategy

  • React quickly to load spikes

  • Prevents thrashing from brief metric dips
  • Looks at the highest recommendation in the window

↕️ Vertical Pod Autoscaler (VPA)

Right-sizing your containers automatically

VPA is covered conceptually here. It's not as widely used as HPA in practice, but understanding it helps participants make informed architecture decisions.

📐 What VPA Does

Vertical Pod Autoscaler

Automatically adjusts CPU and memory requests/limits on containers based on observed usage patterns.

When to Use VPA

  • You don't know the right resource values
  • Workloads have variable resource needs
  • Stateful applications that can't horizontally scale
  • Right-sizing after initial deployment

When to Avoid VPA

  • Stateless workloads better served by HPA
  • When Pod restarts are unacceptable
  • When running alongside HPA on same CPU/memory metrics
VPA is a separate project (not built into core Kubernetes). It must be installed as an add-on. It's particularly valuable for discovery — even in "Off" mode it tells you what your resources should be.

🎛️ VPA Modes

Mode Behavior Pod Restarts? Use Case
Off Recommendations only — no changes applied No Discovery and analysis
Initial Sets resources on Pod creation only No (existing Pods unchanged) Safe production usage
Auto Applies changes, evicts Pods if needed Yes Full automation
Recommendation: Start with Off mode to gather data, then graduate to Initial or Auto once you trust the recommendations.
"Off" mode is the safest starting point. You get recommendations without any risk. "Auto" mode will evict pods to apply new resource values, which can cause brief disruptions. Use PodDisruptionBudgets to mitigate this.

🤼 VPA vs HPA — When to Choose Which

Aspect HPA VPA
What it scales Number of Pods (replicas) Resource requests/limits per Pod
Best for Stateless web services, APIs Databases, batch jobs, JVM apps
Pod disruption None (adds/removes Pods) May evict Pods to resize
Built-in? Yes (core Kubernetes) No (separate add-on)
Can combine? Yes, but not on the same metric. Use VPA for memory + HPA for custom metrics.
The key takeaway: don't run VPA and HPA on the same metric (e.g., both targeting CPU). They will conflict. You can safely combine VPA on memory with HPA on custom metrics or CPU.

⚠️ VPA Limitations & Considerations

  • Not a core component — must be installed separately
  • Pod eviction in Auto mode — disrupts running workloads to apply changes
  • Cannot run with HPA on same metrics — creates conflicting scaling signals
  • Resource ceilings — limited by node capacity; cannot request more than a node offers
  • In-place resize (alpha) — Kubernetes 1.27+ has in-place Pod resource resize, reducing the need for eviction
Production Tip: Use VPA in Off mode alongside HPA to get resource recommendations without interfering with horizontal autoscaling.
In-place resource resize is a game-changer but still alpha/beta. Once it's GA, VPA will become much more practical for production. For now, recommend using VPA in Off mode for insights.

📡 Metrics Server and Its Role

What Is Metrics Server?

A lightweight, in-memory aggregator of resource metrics from kubelets. It exposes the Metrics API (metrics.k8s.io) consumed by HPA, VPA, and kubectl top.

Metrics Server is a cluster add-on, not part of the core API server. It scrapes kubelets every 15 seconds. It stores only the latest data point — it is not a monitoring solution. For historical data, use Prometheus.

📊 Resource Metrics vs Custom Metrics

Resource Metrics

metrics.k8s.io

  • CPU and Memory only
  • Provided by Metrics Server
  • Available out of the box
  • Real-time, no history

Custom / External Metrics

custom.metrics.k8s.io

  • Application-defined metrics
  • Requires adapter (e.g., Prometheus)
  • Requests/sec, queue depth, latency
  • More meaningful for application scaling
Evolution: Start with resource metrics (CPU-based HPA), then graduate to custom metrics as your observability stack matures.
Resource metrics get you started quickly but are a proxy for load. Custom metrics let you scale on what actually matters — like request rate or queue depth. The Prometheus Adapter bridges Prometheus queries to the custom metrics API.

🔗 Connecting Metrics to Scaling Decisions

The Scaling Pipeline:
  1. Instrument — Set resource requests on containers; expose application metrics
  2. Collect — Metrics Server gathers CPU/memory; Prometheus scrapes application metrics
  3. Expose — Metrics API makes data available to HPA/VPA controllers
  4. Evaluate — HPA compares current vs target and computes desired replicas
  5. Act — Controller updates the Deployment's replica count
  6. Observe — Use kubectl get hpa and kubectl top to verify behavior
Golden Rule: Scaling is only as good as your metrics. Invest in proper resource requests and meaningful metrics collection.
This pipeline view helps participants see the full picture. Each step is a potential failure point — missing requests, unhealthy metrics server, misconfigured HPA targets. Troubleshooting means checking each link in the chain.
