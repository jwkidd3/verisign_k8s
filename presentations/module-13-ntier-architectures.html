📐 N-Tier Application Patterns

Foundational architecture models for Kubernetes workloads

We'll start by reviewing the foundational N-tier patterns and how they map to Kubernetes resources. Even if participants are familiar with these patterns from traditional infrastructure, the Kubernetes implementation has important differences.

🏛️ Classic 3-Tier Architecture

🌐 Presentation Tier
Ingress + Frontend Pods
⬇️ ⬆️
⚙️ Business Logic Tier
API / Service Pods
⬇️ ⬆️
💾 Data Tier
StatefulSet + PVCs
  • Presentation: Handles user requests, serves static assets, routes traffic
  • Business Logic: Processes data, enforces rules, orchestrates workflows
  • Data: Persists state, manages transactions, handles queries
  • Each tier scales independently based on its specific resource demands
The 3-tier model is the most common pattern for production applications. In Kubernetes, each tier maps to different resource types: Deployments for stateless tiers, StatefulSets for the data tier, Services for internal routing, and Ingress for external access. The key advantage is independent scaling — your frontend might need 10 replicas during peak traffic while the API needs only 3 and the database remains at a fixed replica count.

🔀 Microservices vs Monolith in Kubernetes

Monolith in K8s

  • Single Deployment, multiple replicas
  • Simpler service mesh requirements
  • All-or-nothing scaling
  • Single CI/CD pipeline
  • Shared database connection pool
Best when: Small teams, tightly coupled logic, limited blast radius

Microservices in K8s

  • Multiple Deployments, each independently versioned
  • Per-service scaling and resource allocation
  • Isolated failure domains
  • Team ownership boundaries
  • Service-to-service communication overhead
Best when: Multiple teams, independent release cadences, varied scaling needs

"Kubernetes excels at both — the choice depends on organizational structure, not technology."

This is a crucial discussion point. Kubernetes does not force a microservices architecture. Many successful production systems run monoliths on Kubernetes, benefiting from self-healing, rolling updates, and resource management without the complexity of service-to-service communication. The decision should be driven by Conway's Law — your architecture will mirror your organization's communication structure.

🔧 Sidecar and Ambassador Patterns

🛡️ Sidecar

Extends the main container with auxiliary functionality within the same Pod.

  • Log forwarding
  • TLS termination
  • Service mesh proxy (Envoy)
  • Config file watchers

🌐 Ambassador

Acts as a proxy between the main container and external services.

  • Connection pooling
  • Rate limiting
  • Circuit breaking
  • Protocol translation

🔄 Adapter

Transforms the output of the main container to a standard format.

  • Metrics export (Prometheus)
  • Log normalization
  • Health check translation
  • API version bridging
All three patterns use multi-container Pods — containers share network namespace, can share volumes, and have the same lifecycle.
These patterns are fundamental to production Kubernetes. The sidecar pattern is especially common — service meshes like Istio inject an Envoy proxy as a sidecar into every Pod. The ambassador pattern is useful when you need to abstract away the complexity of connecting to external services. The adapter pattern is key for observability — exporting metrics in Prometheus format from an application that doesn't natively support it. Emphasize that all three share the same Pod, so they share localhost networking and can mount the same volumes.

📡 Service-to-Service Communication

Synchronous

REST / gRPC

  • Direct request-response
  • ClusterIP Services for discovery
  • DNS-based: svc.namespace.svc.cluster.local
  • Requires retry logic and timeouts
  • Service mesh can add mTLS, retries, circuit-breaking

Asynchronous

Message Queues / Events

  • Decoupled producers and consumers
  • Kafka, RabbitMQ, NATS as intermediaries
  • Better fault isolation between tiers
  • Natural backpressure handling
  • Event sourcing and CQRS patterns
Key Decision: Synchronous is simpler but creates tight coupling. Asynchronous adds resilience but increases operational complexity (broker management, message schema evolution).
In multi-tier Kubernetes applications, the choice between synchronous and asynchronous communication has deep implications. Synchronous works well for simple request-response flows where you need immediate results. Asynchronous is better for workflows where eventual consistency is acceptable and you want isolation between failures. Many production systems use a hybrid: synchronous for the user-facing request path and asynchronous for background processing, notifications, and data pipeline work.

🚀 Deploying a Multi-Tier Application

From architecture design through production deployment

Now we move from theory to practice. This section walks through every aspect of deploying a complete multi-tier application on Kubernetes. We'll cover the full stack: frontend, API, and database, including all the Kubernetes resources needed for each tier.

📋 Architecture Design: Frontend, API, Database

🌐 Frontend

⚙️ API Service
Deployment + HPA
→
💾 Database
StatefulSet + PVC

Design Principles

  • Each tier in its own Deployment (or StatefulSet)
  • Each tier exposed via a ClusterIP Service
  • Only the frontend tier exposed externally via Ingress
  • Namespace isolation for environment separation

Naming Conventions

  • app: myapp-frontend
  • app: myapp-api
  • app: myapp-db
  • Consistent tier: labels for policy targeting
Start with a clean architecture diagram so everyone has the same mental model. Labels are critical — they serve as the targeting mechanism for Services, NetworkPolicies, and monitoring selectors. Consistent naming conventions across all tiers make everything from debugging to policy enforcement much simpler. The tier label is especially important for NetworkPolicies, which we'll cover shortly.

This table is a useful reference. Walk through each row and discuss the rationale for each choice. For example, the database uses a StatefulSet because it needs stable network identities and persistent storage. The frontend and API use Deployments because they're stateless. HPAs are applied to the stateless tiers but usually not to databases, where scaling is more nuanced. The headless Service for the database gives each Pod a stable DNS name for replication.

🔗 Inter-Service Communication & DNS

Kubernetes DNS Resolution

# Short form (same namespace)
http://myapp-api:8080/v1/resources

# Fully qualified domain name
http://myapp-api.production.svc.cluster.local:8080/v1/resources

# StatefulSet Pod DNS (headless service)
myapp-db-0.myapp-db.production.svc.cluster.local

Frontend → API

  • Environment variable injection via ConfigMap
  • API_URL=http://myapp-api:8080
  • Connection timeout and retry policies

API → Database

  • Connection string from Secret
  • DB_HOST=myapp-db-0.myapp-db
  • Connection pooling in application code
DNS is the primary service discovery mechanism in Kubernetes. The short form works within the same namespace and is the most common pattern. Cross-namespace calls require the fully qualified name. For StatefulSets, each Pod gets a predictable DNS entry, which is essential for database replication where Pods need to know about each other. Stress that hardcoding IP addresses is an anti-pattern — always use DNS names through Services.

🔐 ConfigMaps & Secrets Across Tiers

Principle: Each tier owns its configuration. Cross-tier references use service DNS names, never direct Pod IPs.

Configuration Strategy

  • Per-tier ConfigMaps: Application-specific settings
  • Shared ConfigMap: Common environment (log level, tracing endpoint)
  • Volume mounts for config files (nginx.conf, app.yaml)
  • Environment variables for simple key-value pairs

Secrets Management

  • RBAC-scoped Secrets: Only the tier that needs credentials can read them
  • External Secrets Operator: Sync from Vault or AWS Secrets Manager
  • Sealed Secrets: Safe to store in Git
  • Rotation: Rolling restart strategy for credential rotation
Configuration management across tiers is a common source of deployment issues. The key principle is that each tier should own its configuration and only reference other tiers via DNS names. Shared ConfigMaps are useful for global settings like log levels or tracing endpoints, but be careful not to over-share. For Secrets, RBAC should ensure that the frontend Pods cannot read database credentials — only the API tier should have access to those.

🛡️ Network Policies for Tier Isolation

Deny-All Default + Explicit Allow

# Default deny all ingress in the namespace
apiVersion: networking.k8s.io/v1
kind: NetworkPolicy
metadata:
  name: default-deny-ingress
spec:
  podSelector: {}
  policyTypes: [Ingress]
---
# Allow frontend → API only
apiVersion: networking.k8s.io/v1
kind: NetworkPolicy
metadata:
  name: allow-frontend-to-api
spec:
  podSelector:
    matchLabels:
      tier: api
  ingress:
  - from:
    - podSelector:
        matchLabels:
          tier: frontend
    ports:
    - port: 8080
Rule of thumb: Frontend → API → Database. Never allow frontend → database directly.
Network policies are the firewall rules of Kubernetes. The pattern we recommend is "default deny, explicit allow." Start by denying all ingress traffic in the namespace, then create specific policies that allow only the communication paths your architecture requires. This means the frontend can talk to the API, the API can talk to the database, but the frontend can never reach the database directly. This is defense-in-depth for your application architecture.

💓 Health Checks & Readiness Gates

Frontend

  • Liveness: HTTP GET /healthz
  • Readiness: HTTP GET / + upstream API check
  • Startup: 30s for static asset loading

API Service

  • Liveness: HTTP GET /health
  • Readiness: DB connection pool + dependency checks
  • Startup: 60s for migrations

Database

  • Liveness: TCP check on port
  • Readiness: Query execution (SELECT 1)
  • Startup: 120s for recovery
Readiness Gates: Use readinessGates for external conditions — e.g., waiting for a load balancer to register the Pod, or for a sidecar proxy to initialize.
Each tier has different health check requirements. The frontend is the simplest — an HTTP check on the root path. The API tier needs to verify it can reach its dependencies, especially the database. The database tier should use a TCP check for liveness (is the process running?) and a real query for readiness (can it actually serve data?). Startup probes are essential for containers with slow initialization — without them, the liveness probe might kill the container before it finishes starting up. Readiness gates extend this concept to external conditions.

📊 Resource Sizing per Tier

Tier CPU Request CPU Limit Memory Request Memory Limit
Frontend (Nginx) 50m 200m 64Mi 128Mi
API (Node/Go/Java) 250m 1000m 256Mi 512Mi
Database (PostgreSQL) 500m 2000m 512Mi 2Gi
Requests = scheduling guarantee. Set based on steady-state usage observed in production.
Limits = hard ceiling. Set based on peak usage plus headroom. OOMKill if exceeded for memory.

"Start with VPA recommendations, then tune based on observed metrics."

These numbers are illustrative starting points. The actual values depend heavily on your application's characteristics. The key principles: requests should match observed steady-state usage so the scheduler makes good placement decisions; limits should allow for bursts but prevent runaway resource consumption. Note the database tier has the highest resource allocation because it's doing the most intensive work. Use the Vertical Pod Autoscaler in recommendation mode to get data-driven sizing suggestions, then validate with load testing.

🗂️ Complete Deployment Walkthrough

Deployment order matters: Dependencies must be ready before dependents.
  1. Namespace & RBAC: Create namespace, ServiceAccounts, Roles
  2. Secrets & ConfigMaps: Database credentials, TLS certs, app configuration
  3. Network Policies: Default-deny + tier-specific allow rules
  4. Database tier: StatefulSet, Headless Service, PVC
  5. API tier: Deployment, Service, HPA
  6. Frontend tier: Deployment, Service, HPA
  7. Ingress: External routing to frontend
  8. Observability: ServiceMonitors, dashboards, alerts
With Helm: All of this can be packaged into a single helm install command with dependency ordering handled by Helm hooks.
Walk through the deployment order and explain why each step depends on the previous one. The database must exist before the API starts because the API's readiness probe checks the database connection. Network policies should be in place before workloads to ensure security from the start. With Helm, you can codify this entire sequence into a chart with pre-install hooks for secrets and post-install hooks for smoke tests. This is where Module 5's Helm content pays off.

🏗️ Verisign Architecture Patterns

Infrastructure Conventions

  • Standardized namespace taxonomy per team and environment
  • Shared platform services (logging, monitoring, secret management)
  • Consistent labeling schema for cost tracking and ownership
  • GitOps-driven deployment pipelines

Architectural Standards

  • Multi-cluster topology for blast radius isolation
  • Standardized Helm chart templates for team adoption
  • Mandatory resource quotas and limit ranges
  • Centralized Ingress controllers with team-scoped routes
Verisign's approach: Platform teams provide guardrails and shared infrastructure; product teams own their application lifecycle within those boundaries.
Verisign, like many large enterprises, has established conventions for how teams use Kubernetes. These conventions help maintain consistency across teams and reduce operational risk. The key concept is the separation between platform engineering (which provides the cluster, shared services, and guardrails) and product engineering (which builds and deploys applications). Encourage participants to discuss how their specific team interacts with the platform layer.

🌐 DNS-Centric Workloads

Verisign's core mission — operating critical internet infrastructure — makes DNS performance and reliability a first-class concern in every architecture decision.

DNS Workload Characteristics

  • Ultra-low latency requirements (sub-millisecond)
  • Extremely high query-per-second rates
  • Strict availability SLAs (100% uptime targets)
  • Geographically distributed query patterns

Kubernetes Implications

  • Pod anti-affinity for zone distribution
  • Host networking for reduced latency
  • Dedicated node pools with guaranteed resources
  • Custom CoreDNS configuration and tuning
Verisign operates the .com and .net TLD infrastructure among other critical DNS services. This means their workloads have exceptionally stringent availability and performance requirements. When deploying DNS-centric workloads on Kubernetes, special considerations include using hostNetwork for performance-sensitive Pods, node affinity to dedicated high-performance nodes, and Pod disruption budgets that ensure enough capacity is always running. Discuss how participants' specific teams handle these requirements.

🔄 High-Availability Considerations

Pod Level

  • Multiple replicas per tier
  • Pod Disruption Budgets
  • Anti-affinity rules (zone/node)
  • Graceful shutdown handlers

Cluster Level

  • Multi-AZ node pools
  • Control plane HA (3+ masters)
  • etcd cluster health monitoring
  • Node auto-repair policies

Multi-Cluster

  • Active-active across regions
  • Global load balancing (DNS-based)
  • Cross-cluster service discovery
  • Data replication strategies
Verisign standard: No single point of failure at any layer. Every tier must survive the loss of an entire availability zone without service interruption.
High availability at Verisign goes beyond basic Kubernetes HA. Walk through the three levels: Pod-level HA ensures individual container failures don't cause service disruption; cluster-level HA ensures the platform itself is resilient; multi-cluster HA protects against entire region failures. PodDisruptionBudgets are mandatory for any production workload — they prevent cluster upgrades or node drains from taking down too many replicas simultaneously. Discuss topology spread constraints as a more flexible alternative to anti-affinity.

CI/CD Pipeline Integration

Observability Stack

  • PagerDuty / OpsGenie alert routing

Integration pattern: Applications expose standard interfaces (Prometheus metrics, structured JSON logs), and the platform layer collects, aggregates, and routes them.
Internal tooling integration is where organizational efficiency comes from. Rather than each team building their own monitoring and deployment pipelines, Verisign's platform team provides standardized tooling. Applications integrate by following conventions: expose metrics on a known endpoint, emit structured logs, and use the standard Helm chart templates. 👥 Team Workflows & Deployment Procedures

GitOps Workflow: All changes flow through version-controlled repositories. The cluster state matches what is declared in Git.
  1. Develop: Feature branch → code review → merge to main
  2. Build: CI pipeline builds container image, runs tests, scans for vulnerabilities
  3. Package: Helm chart version bumped, values updated for target environment
  4. Deploy to Staging: ArgoCD / Flux syncs from Git, automated integration tests run
  5. Promote to Production: Approval gate, canary or blue-green rollout
  6. Monitor: Automated health checks, alert thresholds, SLO tracking
  7. Rollback: Revert Git commit → ArgoCD automatically syncs previous state
The GitOps workflow ensures auditability and reproducibility. Every change to the cluster can be traced back to a specific Git commit, which is essential for compliance and incident investigation. The key advantage of this approach is that rollback is simply a Git revert — no manual kubectl commands needed. Discuss how this workflow interacts with the participants' existing processes and what adaptations might be needed.
