🏗️ The Twelve-Factor App

Factor III: Store Config in the Environment

The 12-factor app methodology, originally from Heroku, provides foundational guidance for building cloud-native applications. Factor III specifically addresses configuration management.

📚 Why Externalize Configuration?

❌ Anti-Pattern: Embedded Config

  • Config hardcoded in application code
  • Requires rebuilding images for each environment
  • Secrets committed to source control
  • No separation of concerns

✅ Best Practice: External Config

  • Config injected at runtime
  • Same image across dev, staging, prod
  • Secrets managed independently
  • Config changes without redeployment
The core principle is strict separation of config from code. Configuration varies between deploys — code does not. At Verisign, this means the same container image runs in dev, staging, and production EKS clusters with different ConfigMaps and Secrets.

🚀 Kubernetes Config Primitives

Kubernetes provides two native resources for external configuration:

  • ConfigMaps — Non-sensitive configuration data (key-value pairs, files)
  • Secrets — Sensitive data (credentials, tokens, certificates)
Both can be consumed as: Environment variables, command-line arguments, or files mounted into volumes.
ConfigMaps and Secrets are the two primary mechanisms Kubernetes provides. They share a similar API but have important behavioral differences, especially around security. We'll cover both in depth.

📃 ConfigMaps

Externalizing Non-Sensitive Configuration

Let's start with ConfigMaps — the primary mechanism for injecting non-sensitive configuration into your pods.

🔧 Creating ConfigMaps — From Literals

Use --from-literal for individual key-value pairs:

# Create ConfigMap from literal values
kubectl create configmap app-config \
  --from-literal=DATABASE_HOST=postgres.db.svc.cluster.local \
  --from-literal=DATABASE_PORT=5432 \
  --from-literal=LOG_LEVEL=info \
  --from-literal=CACHE_TTL=300
# Verify the ConfigMap
kubectl get configmap app-config -o yaml
Tip: Literal creation is great for simple key-value pairs, but becomes unwieldy for more than a handful of values.
The simplest way to create a ConfigMap is from literal values on the command line. Each --from-literal flag adds one key-value pair. This is useful for quick tests but for production, declarative YAML is preferred.

📄 Creating ConfigMaps — From Files

Load entire configuration files into a ConfigMap:

# Create from a single file (key = filename)
kubectl create configmap nginx-conf \
  --from-file=nginx.conf

# Create from a file with a custom key name
kubectl create configmap nginx-conf \
  --from-file=main-config=nginx.conf

# Create from a directory (each file becomes a key)
kubectl create configmap app-configs \
  --from-file=./config-dir/
Key Insight: When using --from-file, the filename becomes the key and the file contents become the value. With a directory, each file in the directory becomes a separate key.
File-based ConfigMaps are extremely useful for injecting entire configuration files — things like nginx.conf, application.properties, or JSON config files. The entire file content is stored as the value.

📝 Creating ConfigMaps — Declarative YAML

The preferred approach for production workloads:

apiVersion: v1
kind: ConfigMap
metadata:
  name: app-config
  namespace: verisign-apps
  labels:
    app: dns-service
    environment: production

  DATABASE_PORT: "5432"
  LOG_LEVEL: "info"
  CACHE_TTL: "300"
  # Multi-line configuration file
  application.yaml: |
    server:
      port: 8080
      shutdown: graceful
    logging:
      level:
        root: INFO
        com.verisign: DEBUG
    cache:
      type: redis
      ttl: 300
Declarative YAML is the recommended approach for Verisign. It can be version-controlled, reviewed in PRs, and applied through GitOps pipelines. Notice how you can mix simple key-value pairs with multi-line file content in the same ConfigMap.

⚙️ Externalizing Configuration

                
Separating config from code

                

                    We start with the foundational principle: why configuration should live outside your application code and container images. This is not just a Kubernetes concept -- it is a fundamental software engineering practice.
                

            


            
            

                
📖 The Twelve-Factor App: Config

                

                    "An app's config is everything that is likely to vary between deploys (staging, production, developer environments)."

                    -- twelve-factor.net, Factor III

⚠️ Why Hardcoding Config Is Problematic

                

                    

                        
🔒 Security Risk

                        
Credentials embedded in images or code end up in version control, container registries, and layer caches

                    

                    

                        
🔄 Inflexibility

                        
Changing a single environment variable requires rebuilding and redeploying the entire image

                    

                    

                        
📈 Environment Drift

                        
Maintaining separate images or branches per environment leads to divergence and deployment errors

                    

                

                

                    
Real-world impact: GitHub reports that thousands of secrets are leaked to public repositories daily. Once a secret is in a container layer, it persists even if removed in a subsequent layer.

                

                

                    Walk through each risk. The security risk is most urgent -- emphasize that Docker image layers are additive. Even if you delete a secret in a later layer, it is still accessible in the earlier layer. The inflexibility issue impacts velocity: every config change requiring a rebuild slows down operations significantly.

📜 ConfigMaps

                
Decoupling configuration from container images

                

                    ConfigMaps are the primary Kubernetes primitive for non-sensitive configuration data. They are flexible, support multiple creation and consumption patterns, and are essential knowledge for any Kubernetes practitioner.
                

            


            
            

                
📄 What Is a ConfigMap?

                

                    
A ConfigMap is a Kubernetes API object that stores non-confidential configuration data as key-value pairs. Pods can consume ConfigMaps as environment variables, command-line arguments, or configuration files in a volume.

                

                

                    

                        
Key Characteristics

                        
    
                            
  • Namespace-scoped resource
    • 
                            
  • Max size: 1 MiB
    • 
                            
  • Stored in etcd
    • 
                            
  • Not designed for sensitive data
    • 
                        

                    

                    

                        
Common Use Cases

                        
    
                            
  • Application properties files
    • 
                            
  • nginx / HAProxy configurations
    • 
                            
  • Feature flags
    • 
                            
  • Connection strings (non-secret)
    • 
                        

                    

                

                

                    Emphasize the 1 MiB size limit. If students need to store larger configuration, they should consider mounting from a persistent volume or using an init container. ConfigMaps are namespace-scoped, so they must exist in the same namespace as the pods that consume them.
                

            


            
            

                
🔨 Creating ConfigMaps

                
From Literals

                

# From literal key-value pairs
kubectl create configmap app-config \
  --from-literal=DATABASE_HOST=postgres.default.svc \
  --from-literal=DATABASE_PORT=5432 \
  --from-literal=LOG_LEVEL=info
                

                
From Files

                

# From a single file (key = filename, value = file contents)
kubectl create configmap nginx-config \
  --from-file=nginx.conf

# From a file with a custom key name
kubectl create configmap nginx-config \
  --from-file=my-nginx-config=nginx.conf
                

                
From Directories

                

# Each file in the directory becomes a key-value pair
kubectl create configmap app-configs \
  --from-file=./config-dir/
                

                

                    Demonstrate each method. The from-literal approach is ideal for simple key-value pairs. The from-file approach is powerful for injecting entire config files like nginx.conf. The from-directory approach creates one key per file, where the filename is the key and the file content is the value.

Hot Reload: ConfigMap volume mounts are updated automatically by the kubelet — typically within ~60 seconds (synced on the kubelet sync period + cache propagation delay). Your application must watch for file changes to pick up updates.
Volume mounts are the most powerful consumption pattern. Each key becomes a file in the mount directory. The huge advantage is hot reload — the kubelet automatically updates the mounted files when the ConfigMap changes, typically within about 60 seconds. But your application must be designed to detect and reload configuration from disk.

🔄 Hot Reload Behavior Deep Dive

How Volume-Mounted ConfigMap Updates Work

  • Kubelet periodically syncs ConfigMap data (default: every 60s)
  • Updates use atomic symlink swap: ..data → ..data_tmp → rename
  • All files update atomically together
  • Total propagation delay: kubelet sync + API server cache TTL
Important Exceptions:
  • subPath mounts do NOT receive updates — they are a one-time copy
  • Environment variables from ConfigMaps are never updated — require pod restart
Understanding the update mechanism is crucial. The kubelet uses symlinks for atomic updates, which means all files in a ConfigMap mount update together. But there are two critical exceptions: subPath mounts and environment variables never receive updates. If you need live config updates, use volume mounts without subPath.

🔐 Kubernetes Secrets

Managing Sensitive Configuration Data

Now let's move to Secrets. While similar to ConfigMaps in API structure, Secrets have fundamentally different security characteristics that you must understand.

🔧 Creating Secrets

# Opaque secret from literals
kubectl create secret generic db-credentials \
  --from-literal=username=admin \
  --from-literal=password='S3cur3P@ssw0rd!'

# TLS secret from certificate files
kubectl create secret tls app-tls-cert \
  --cert=tls.crt \
  --key=tls.key

# Docker registry secret
kubectl create secret docker-registry ecr-creds \
  --docker-server=123456789.dkr.ecr.us-east-1.amazonaws.com \
  --docker-username=AWS \
  --docker-password=$(aws ecr get-login-password)
Shell History Risk: Creating secrets from the command line leaves credentials in your shell history. Use --from-file or declarative YAML with sealed/external secrets for production.
Here are the three most common imperative creation patterns. Note the shell history warning — this is a real security concern. For Verisign production workloads, secrets should be managed through External Secrets Operator syncing from Vault, not created imperatively.

🔄 ConfigMap Updates and Pod Behavior

Consumption Method Auto-Update? Delay
Environment variables ❌ No Requires pod restart
Volume mount ✅ Yes kubelet sync period (~60s default)
Volume mount with subPath ❌ No Requires pod restart
Command arguments ❌ No Requires pod restart

Key insight: Only standard volume mounts (without subPath) receive automatic updates. Your application must still detect and reload the changed files -- Kubernetes only updates the filesystem.

This is a critical operational detail. Even with volume mounts, the application must be designed to detect file changes and reload. Many applications require a SIGHUP signal or file-watcher pattern. The kubelet sync period is configurable but defaults to roughly 60 seconds. Note that cached ConfigMaps (using the watch or TTL-based cache) may introduce additional delay.

🔒 Immutable ConfigMaps


apiVersion: v1
kind: ConfigMap
metadata:
  name: app-config-v3
data:
  DATABASE_HOST: "postgres.default.svc"
  LOG_LEVEL: "warn"
immutable: true

✅ Benefits

  • Protects against accidental changes
  • Significant performance gain at scale (no watch overhead)
  • Reduces load on kube-apiserver

💡 Usage Pattern

  • Version names: app-config-v3
  • Delete and recreate to "update"
  • Update pod spec to reference new version
  • Enables rollback by switching versions
Immutable ConfigMaps are a GA feature since Kubernetes 1.21. At scale, they dramatically reduce apiserver load because the kubelet does not need to maintain watches on them. The versioned naming pattern works well with deployment strategies -- you update the Deployment to reference a new ConfigMap version, which triggers a rolling update.

🔐 Secrets

Managing sensitive data in Kubernetes

Now we move to Secrets -- the Kubernetes primitive designed for sensitive data. While structurally similar to ConfigMaps, Secrets have important behavioral differences and security implications that every operator must understand.

🔓 Secrets vs ConfigMaps

Aspect ConfigMap Secret
Purpose Non-sensitive config Sensitive data
Encoding Plain text Base64-encoded
Size limit 1 MiB 1 MiB
etcd storage Plain text Base64 (encryption at rest optional)
tmpfs mount No Yes (never written to disk on node)
Typed No Yes (Opaque, TLS, docker-registry, etc.)
kubectl output Shown by default Hidden by default (requires -o yaml)
Walk through each row. The tmpfs behavior is important: when Secrets are mounted as volumes, they are stored in a tmpfs filesystem on the node, meaning they are held in memory and never written to the node's physical disk. This is a meaningful security difference. However, emphasize that base64 is NOT encryption -- it is merely encoding and provides zero confidentiality.

📚 Secret Types

Type Description Required Keys
Opaque Generic / arbitrary data (default) None (user-defined)
kubernetes.io/tls TLS certificate + private key tls.crt, tls.key
kubernetes.io/dockerconfigjson Docker registry credentials .dockerconfigjson
kubernetes.io/basic-auth Basic authentication username, password
kubernetes.io/ssh-auth SSH private key ssh-privatekey
kubernetes.io/service-account-token ServiceAccount token (legacy) Auto-populated

Validation: Typed Secrets enforce that the required keys are present. Using the correct type enables better tooling integration and RBAC policies.

Secret types provide schema validation. For example, a TLS Secret must contain tls.crt and tls.key, and the API server validates this at creation time. The docker-registry type is used with imagePullSecrets for pulling from private registries. Emphasize that Opaque is the default and most commonly used type.

      mountPath: /etc/tls
      readOnly: true
  volumes:
  - name: tls-certs
    secret:
      secretName: app-tls-cert
      defaultMode: 0400  # Read-only for owner
Consuming secrets follows the same patterns as ConfigMaps — env vars or volumes. Notice the defaultMode of 0400 for the TLS cert volume — restricting file permissions is an important security practice. Secret files should only be readable by the application process.

📊 Environment Variable Precedence

Understanding Override Order

When you have multiple sources setting the same environment variable, understanding the precedence rules is essential to avoid surprises.

📋 Environment Variable Precedence Rules

Order of Precedence (highest to lowest):

  1. env[].value — Inline value defined directly in the Pod spec
  2. env[].valueFrom — Single key reference (configMapKeyRef / secretKeyRef)
  3. envFrom — Bulk import from ConfigMap or Secret
spec:
  containers:
  - name: app
    envFrom:
    - configMapRef:
        name: app-defaults     # LOG_LEVEL=info (lowest priority)
    env:
    - name: LOG_LEVEL
      value: "debug"           # This WINS (highest priority)
    - name: DB_HOST
      valueFrom:
        configMapKeyRef:
          name: app-overrides
          key: DATABASE_HOST   # Overrides envFrom value
The precedence rule is simple: explicit env entries override envFrom entries. Among envFrom entries, later entries in the list override earlier ones. This pattern lets you set defaults in a shared ConfigMap via envFrom, then override specific values with targeted env entries.

📁 Advanced Volume Patterns

Projected Volumes and subPath Mounts

Let's look at two advanced volume patterns that solve real-world configuration challenges.

📂 Projected Volumes

Combine multiple sources into a single volume mount:

apiVersion: v1
kind: Pod
metadata:
  name: dns-service
spec:
  containers:
  - name: app
    volumeMounts:
    - name: all-config
      mountPath: /etc/app
      readOnly: true
  volumes:
  - name: all-config
    projected:
      sources:
      - configMap:
          name: app-config
          items:
          - key: application.yaml
            path: application.yaml
      - secret:
          name: db-credentials
          items:
          - key: password
            path: secrets/db-password
      - downwardAPI:
          items:
          - path: labels
            fieldRef:
              fieldPath: metadata.labels
      - serviceAccountToken:
          path: token
          expirationSeconds: 3600
          audience: vault
Projected volumes are incredibly powerful. You can combine ConfigMaps, Secrets, the Downward API, and service account tokens into a single mount point. This is especially useful for applications that expect all their configuration in one directory. The service account token projection with audience and expiration is the modern approach, replacing the legacy auto-mounted tokens.

📄 subPath Mounts

Mount a single key without overwriting the entire directory:

❌ Without subPath

# This REPLACES everything
# in /etc/nginx/
volumeMounts:
- name: config
  mountPath: /etc/nginx/
  # All existing files in
  # /etc/nginx/ are hidden!

✅ With subPath

# Only adds nginx.conf,
# preserving other files
volumeMounts:
- name: config
  mountPath: /etc/nginx/nginx.conf
  subPath: nginx.conf
  # Other files in /etc/nginx/
  # remain visible!
⚠️ Trade-off: subPath mounts do NOT receive automatic updates when the ConfigMap changes. If you need hot reload, use a regular volume mount to a different directory and symlink to it.
subPath is essential when you need to inject a single config file into a directory that already has other files — like adding a custom nginx.conf without hiding the rest of /etc/nginx. But remember the critical trade-off: subPath mounts never receive updates. This is a common source of confusion in production.

  • Create new ConfigMap with version suffix
  • Update Deployment to reference new name

Immutable ConfigMaps and Secrets are a production best practice, especially at scale. Once marked immutable, the resource cannot be modified — only deleted and recreated. 📐 Secrets: Env Vars vs Volumes

📄 Environment Variables

  • Simple to consume in code
  • Visible in kubectl describe pod
  • Exposed in /proc/<pid>/environ
  • Cannot be updated without pod restart
  • May appear in application crash dumps
  • Risk of accidental logging

📂 Volume Mounts

  • Stored in tmpfs (memory only)
  • File permissions enforced
  • Auto-updated when Secret changes
  • Not visible in pod describe
  • Application reads from file path
  • Preferred for production workloads

For production workloads, prefer volume mounts over environment variables for Secrets.

This comparison is important for security-conscious organizations. The environment variable approach is convenient for development but creates multiple exposure vectors in production. Volume mounts on tmpfs ensure secrets never touch the node's persistent storage, and file permissions add an additional layer of protection.

🛡️ Best Practices for Secrets

Production-grade secret management

This section covers what it takes to move from the default Kubernetes secret handling to a production-hardened configuration. Default Kubernetes installations provide minimal secret protection, and additional measures are essential for regulated environments.

🔒 Encryption at Rest

By default, Secrets are stored unencrypted in etcd. Anyone with etcd access can read all secrets. Encryption at rest encrypts Secret data before it is written to etcd.


# /etc/kubernetes/manifests/encryption-config.yaml
apiVersion: apiserver.config.k8s.io/v1
kind: EncryptionConfiguration
resources:
  - resources:
    - secrets
    providers:
    - aescbc:
        keys:
        - name: key1
          secret: 
    - identity: {}   # Fallback: read unencrypted secrets

Providers (strongest to weakest): kms (recommended) > aescbc > aesgcm > secretbox > identity (no encryption)

Walk through the encryption configuration. The kms provider integrates with external KMS systems like AWS KMS, Azure Key Vault, or GCP Cloud KMS and is the recommended approach for production. The identity provider is a passthrough that provides no encryption -- it should only appear as a fallback for reading existing unencrypted secrets during migration. After enabling encryption, run kubectl get secrets --all-namespaces -o json | kubectl replace -f - to re-encrypt all existing secrets.

  • Encrypts Secret data before writing to etcd
  • Decrypts transparently on read via API server
  • Protects against etcd backup exposure
  • Does NOT protect against API access
Remember: Encryption at rest protects etcd data. It does not prevent users with kubectl get secret permissions from reading secrets through the API.
Encryption at rest is essential for compliance. On EKS, Verisign uses AWS KMS envelope encryption. The API server encrypts each Secret with a data encryption key (DEK), and that DEK is encrypted with your KMS key. This protects against etcd snapshot exposure but not against API-level access.

🔄 External Secrets Operator: How It Works


apiVersion: external-secrets.io/v1beta1
kind: ExternalSecret
metadata:
  name: db-credentials
  namespace: production
spec:
  refreshInterval: 1h
  secretStoreRef:
    name: aws-secrets-manager
    kind: ClusterSecretStore
  target:
    name: db-credentials        # Kubernetes Secret to create
    creationPolicy: Owner
  data:
  - secretKey: username         # Key in the K8s Secret
    remoteRef:
      key: prod/database        # Path in AWS Secrets Manager
      property: username
  - secretKey: password
    remoteRef:
      key: prod/database
      property: password

Flow: ExternalSecret CRD → ESO controller reads from AWS Secrets Manager → Creates/updates native Kubernetes Secret → Pods consume normally

The External Secrets Operator follows a declarative pattern. You define what secrets you need and where they live externally, and the operator handles synchronization. The refreshInterval controls how often the operator checks for updates. This approach means the actual secret values never appear in your YAML manifests or Git repositories.

🚧 Avoiding Secrets in Container Images

❌ Anti-Patterns

  • Hardcoded credentials in source code
  • COPY .env in Dockerfile
  • ARG / ENV for secrets in build
  • Secrets in Docker build context
  • Embedding keys in application config files

✅ Correct Patterns

  • Inject at runtime via Secrets
  • Use .dockerignore rigorously
  • Multi-stage builds (secrets in builder only)
  • Docker BuildKit --secret flag
  • Init containers for secret bootstrapping

Remember: Docker image layers are immutable and cumulative. A secret added in layer 3 and deleted in layer 4 still exists in layer 3 and can be extracted.

This is a common area of mistakes. Even experienced engineers sometimes use ARG to pass secrets during builds, not realizing that ARG values are stored in the image metadata and visible via docker history. The BuildKit --secret flag was introduced specifically to address this -- it mounts secrets during the build without persisting them in any layer.

🔄 Secret Rotation Strategies

Secret rotation is the practice of periodically changing credentials to limit the blast radius of a compromise. Effective rotation requires coordination between the secret store, Kubernetes, and the application.

🕐 Manual Rotation

  1. Create new credentials at source
  2. Update Kubernetes Secret
  3. Perform rolling restart
  4. Verify application health
  5. Revoke old credentials

⚙️ Automated Rotation

  1. Vault dynamic secrets (TTL-based)
  2. External Secrets Operator (refreshInterval)
  3. Cloud provider auto-rotation
  4. Application-level secret reloading
  5. Dual-credential (blue/green) rotation
Rotation is often the most operationally challenging aspect of secret management. The manual process is error-prone, so automation is strongly recommended. Vault's dynamic secrets are the gold standard -- each pod gets unique, short-lived credentials that automatically expire. The dual-credential pattern maintains two active credentials simultaneously during rotation, ensuring zero downtime.

🏢 Enterprise Secret Management Checklist

Production-ready secret management requires a layered approach:

  • 🔒 Encryption at rest: Enable etcd encryption using KMS provider integration
  • 👥 RBAC: Restrict secret access by namespace, name, and verb -- apply least privilege
  • 📜 Audit logging: Enable Kubernetes audit logging for all secret access events
  • 🔄 Rotation: Implement automated rotation with defined schedules (e.g., 90-day cycle)
  • 🌐 External stores: Use Vault, cloud KMS, or external-secrets for centralized management
  • 🔍 Scanning: Integrate secret scanning in CI/CD pipelines (gitleaks, truffleHog)
  • 📦 Network policies: Restrict which pods can reach secret management infrastructure
This checklist represents the full spectrum of secret management maturity. For organizations in regulated industries (financial services, DNS infrastructure, etc.), all of these items should be implemented. The audit logging point is especially important for compliance -- you need to demonstrate who accessed what secrets and when. Secret scanning in CI/CD catches accidental commits before they reach production.
