🔒 Why External Secrets Management?

The problem with Kubernetes Secrets alone

Before we jump into Vault, we need to establish why native K8s Secrets are insufficient for production workloads. This section sets the motivation for everything that follows.

The Kubernetes Secrets Problem

⚠️ base64 is encoding, NOT encryption
Anyone with RBAC access to read Secrets can decode them instantly.
  • No encryption at rest by default — etcd stores Secrets in plaintext
  • Secrets in Git — even with SealedSecrets or SOPS, the secret lifecycle is tied to Git
  • No audit trail — who read which secret, and when?
  • No dynamic secrets — every credential is static and long-lived
  • No automatic rotation — credentials persist until manually changed
  • No centralized policy — access control is scattered across RBAC rules
Walk through each limitation. Emphasize that base64 is trivially decodable — it is not a security measure. etcd encryption at rest helps but does not address the other concerns. The key takeaway is that K8s Secrets are a delivery mechanism, not a secret management platform.

The Evolution of Secret Management

① Kubernetes Secrets

  • Simple, built-in
  • base64 encoded
  • No versioning
  • No audit

② Sealed Secrets / SOPS

  • Encrypted in Git
  • Decrypted in-cluster
  • Still static secrets
  • No centralized management

③ External Secret Stores (Vault, AWS Secrets Manager, etc.)

  • Centralized, encrypted, audited, policy-driven
  • Dynamic secrets, automatic rotation, versioning
  • Secrets never stored in Git — only references
Show the progression from simplest to most robust. SealedSecrets and SOPS are a step up because they allow encrypted secrets in Git, but they still produce static K8s Secrets. External stores like Vault represent the production-grade approach: secrets live outside the cluster, are managed centrally, and are synced on-demand.

🛡️ HashiCorp Vault Overview

Centralized secret management platform

Now we introduce Vault itself — what it is, how it is architected, and the core concepts that underpin everything.

What is HashiCorp Vault?

Vault is a centralized secret management platform

It provides a unified interface to manage secrets, enforce access control, and maintain a detailed audit log — across any infrastructure.

Core Capabilities

  • Secret storage and retrieval
  • Dynamic secret generation
  • Data encryption (transit)
  • Identity-based access control

Why Vault for Kubernetes?

  • Native K8s auth method
  • Pod identity → Vault policy
  • Helm chart for easy deployment
  • Multiple integration patterns
Vault is the industry-standard open-source secret management tool. It is not Kubernetes-specific — it works across cloud providers, on-prem, and hybrid environments. For Kubernetes, the key value is the native auth method that maps pod service accounts to Vault policies.

Vault Architecture

┌──────────────┐     ┌─────────────────────────────────┐
│   Client     │     │         Vault Server             │
│  (CLI/API/   │────▶│                                  │
│   K8s Pod)   │     │  ┌─────────┐  ┌──────────────┐  │
│              │◀────│  │  Auth   │  │   Secrets     │  │
└──────────────┘     │  │ Methods │  │   Engines     │  │
                     │  └─────────┘  └──────────────┘  │
                     │  ┌─────────┐  ┌──────────────┐  │
                     │  │ Policies│  │ Audit Logging │  │
                     │  └─────────┘  └──────────────┘  │
                     │         │                        │
                     │  ┌──────▼───────────────────┐   │
                     │  │    Storage Backend        │   │
                     │  │  (Consul/Raft/File/etc.)  │   │
                     │  └──────────────────────────┘   │
                     └─────────────────────────────────┘
Key flow: Client authenticates → Vault validates identity → Returns a Vault token → Client uses token to access secrets (governed by policies)
Walk through the architecture diagram. The client can be the Vault CLI, an API call, or a Kubernetes pod. Auth methods validate the client's identity. Policies determine what the client can access. Secrets engines manage the actual secrets. The storage backend persists everything. In the Verisign platform, Vault runs in dev mode with file storage.

Vault Core Concepts

🔑 Secrets Engines

Components that store, generate, or encrypt data. Each is mounted at a path (e.g., secret/).

👤 Auth Methods

Verify client identity and map to policies. Examples: Kubernetes, AppRole, AWS IAM.

📜 Policies

HCL rules that define path-based access control. Determine who can read, write, or list which secrets.

📑 Audit Logging: Every Vault operation is logged with identity, timestamp, and request/response details. This is the audit trail that K8s Secrets lack.
These four concepts are the pillars of Vault. Secrets engines handle data, auth methods handle identity, policies handle authorization, and audit logs handle accountability. Every interaction with Vault touches all four.

Vault Secrets Engines

Engine Purpose Use Case
KV v2 Versioned key-value store Application secrets, config values
Database Dynamic database credentials Short-lived DB usernames/passwords
PKI Certificate generation TLS certs, mTLS between services
AWS Dynamic IAM credentials Temporary AWS access keys
Transit Encryption as a service Encrypt/decrypt without storing keys
Verisign uses KV v2 for application secrets — this is the most common starting point and what we will focus on.
Briefly describe each engine. KV v2 is the workhorse — it is what most teams start with and what the Verisign platform provisions. Database and PKI engines provide dynamic secrets which are more advanced. Transit is interesting because it provides encryption without ever exposing keys. Focus the remaining discussion on KV v2.

KV v2 Secrets Engine Deep Dive

Writing Secrets

# Store a secret
vault kv put secret/myapp/db \
  username=admin \
  password=s3cureP@ss

# Update with check-and-set (CAS)
# Prevents overwriting if version changed
vault kv put -cas=1 secret/myapp/db \
  username=admin \
  password=newP@ss123

Reading Secrets

# Get current version
vault kv get secret/myapp/db

# Get specific version
vault kv get -version=1 secret/myapp/db

# Get as JSON (useful for scripts)
vault kv get -format=json \
  secret/myapp/db

# View metadata (all versions)
vault kv metadata get secret/myapp/db
💡 KV v2 API path note: The API path includes /data/ — so secret/myapp/db becomes secret/data/myapp/db in policies and API calls. This is a common source of confusion.
Walk through the CLI commands. Emphasize the CAS flag for concurrent access safety. The version flag lets you retrieve old values — KV v2 keeps a configurable number of versions. Highlight the API path distinction because it trips up many people when writing policies.

KV v2 Versioning

Version Lifecycle

# Soft delete a version (recoverable)
vault kv delete -versions=2 secret/myapp/db

# Recover a soft-deleted version
vault kv undelete -versions=2 secret/myapp/db

# Permanently destroy a version
vault kv destroy -versions=2 secret/myapp/db

# Delete all versions and metadata
vault kv metadata delete secret/myapp/db
Why versioning matters: Roll back to a previous secret version without redeploying. Audit who changed what and when. Soft delete gives you a safety net before permanent destruction.
Versioning is one of the key advantages over K8s Secrets. If someone updates a password and breaks something, you can roll back to the previous version instantly. Soft delete means accidental deletions are recoverable. This is a significant operational improvement over managing Secrets in YAML files.

👤 Vault Authentication Methods

How workloads prove their identity to Vault

Authentication is how Vault knows who is asking for secrets. For Kubernetes workloads, the Kubernetes auth method is the primary mechanism. Let us understand how it works.

Vault Authentication Methods

✅ Kubernetes Auth

Primary for K8s workloads. Uses the pod's ServiceAccount token to authenticate with Vault.

🔑 Token Auth

Direct token-based access. Used for initial setup and root operations. Not recommended for workloads.

🔄 AppRole Auth

Machine-to-machine auth with RoleID + SecretID. Good for CI/CD pipelines and automation.

☁️ AWS IAM Auth

Uses AWS IAM identity to authenticate. Useful for EC2/Lambda workloads outside K8s.

Kubernetes auth is the focus for this course because it provides seamless identity for pods. Token auth is how we bootstrap Vault but should never be used for application workloads. AppRole is useful for CI/CD. AWS IAM auth is relevant for workloads outside the cluster that need Vault access.

Kubernetes Auth Method: How It Works

┌──────────────┐    1. Present SA Token    ┌──────────────┐
│   K8s Pod    │──────────────────────────▶│    Vault     │
│  (with SA)   │                           │   Server     │
│              │    4. Return Vault Token   │              │
│              │◀──────────────────────────│              │
└──────────────┘                           └──────┬───────┘
                                                  │
                                           2. Validate token
                                            with K8s API
                                                  │
                                           ┌──────▼───────┐
                                           │  Kubernetes  │
                                           │  API Server  │
                                           │              │
                                           │ 3. Confirm   │
                                           │   identity   │
                                           └──────────────┘
The flow: Pod presents its ServiceAccount JWT → Vault calls the K8s TokenReview API → K8s confirms the identity → Vault maps SA to a role → Role maps to policies → Vault returns a Vault token with those policies.
This is the core flow for K8s auth. The pod does not need a pre-configured Vault token. Its identity comes from the ServiceAccount assigned to the pod. Vault validates that token with the Kubernetes API server itself, so there is no way to forge the identity. The returned Vault token has a TTL and is scoped to the policies defined in the Vault role.

Configuring Kubernetes Auth

# Enable the Kubernetes auth method
vault auth enable kubernetes

# Configure it to talk to the K8s API
vault write auth/kubernetes/config \
  kubernetes_host="https://kubernetes.default.svc" \
  kubernetes_ca_cert=@/var/run/secrets/kubernetes.io/serviceaccount/ca.crt

# Create a role mapping SA to policies
vault write auth/kubernetes/role/my-app-role \
  bound_service_account_names=my-app-sa \
  bound_service_account_namespaces=my-app-ns \
  policies=my-app-policy \
  ttl=1h
💡 Role binding: A Vault role ties together three things: which ServiceAccount, in which namespace, and which Vault policies. This is the bridge between K8s identity and Vault authorization.
In the Verisign platform, this configuration is done by the bootstrap Job that runs after Vault is deployed. The kubernetes_host points to the in-cluster API server. The role definition is the key piece — it says 'if a pod running as my-app-sa in my-app-ns authenticates, give it the my-app-policy policies with a 1-hour TTL.'

Vault Policies

Policy Syntax (HCL)

# Allow reading secrets under myapp/
path "secret/data/myapp/*" {
  capabilities = ["read", "list"]
}

# Allow managing secrets for admins
path "secret/data/myapp/*" {
  capabilities = ["create", "read",
                   "update", "delete",
                   "list"]
}

# Deny access to other paths
# (implicit — deny by default)

Capabilities

createCreate new secrets
readRead secret values
updateModify existing secrets
deleteDelete secrets
listList secret paths
denyExplicitly deny (overrides)
Vault is deny-by-default. If no policy grants access, the request is denied.
Policies use HCL syntax with path-based rules. The path matches the API path — remember that KV v2 paths include /data/. Capabilities are similar to CRUD operations. Vault is deny-by-default, which is the secure default. Emphasize least privilege: give only the capabilities needed, scoped to the narrowest path.

Policy Best Practices

✅ Least Privilege: Grant the minimum access needed. A web app typically only needs read on its specific secret paths.

❌ Too Broad

# Don't do this!
path "secret/*" {
  capabilities = ["read", "list"]
}

Grants access to ALL secrets

✅ Properly Scoped

# Correct — scoped to app
path "secret/data/myapp/*" {
  capabilities = ["read"]
}

Grants access only to myapp secrets

Show the contrast between a broad policy and a properly scoped one. In production, each application should have its own policy that only grants access to its own secret paths. Never use wildcard paths at the top level. The Verisign bootstrap creates per-application policies that follow this pattern.

🔄 External Secrets Operator (ESO)

Bridging Vault and Kubernetes

Now we shift from Vault itself to how secrets get into Kubernetes. The External Secrets Operator is the bridge between Vault and K8s Secrets — it is the integration layer that the Verisign platform uses.

What is the External Secrets Operator?

ESO is a Kubernetes operator that synchronizes secrets from external providers into Kubernetes Secrets.

It watches ExternalSecret CRDs, fetches values from the configured provider (Vault, AWS SM, etc.), and creates/updates native K8s Secrets.

Why ESO over alternatives?

Approach How it works Trade-off
Vault Agent Sidecar Sidecar container fetches and renders secrets Extra container per pod, more resource usage
Vault CSI Driver Mounts secrets as files via CSI volume Does not create K8s Secrets (unless configured)
ESO Controller syncs to native K8s Secrets Clean separation, works with existing workloads
ESO is the cleanest integration for teams that want to keep their pod specs simple. The Vault Agent sidecar approach requires injecting a container into every pod. CSI driver mounts secrets as files but does not natively produce K8s Secrets (though it can be configured to). ESO operates at the cluster level — no changes to pod specs are needed. The application consumes a standard K8s Secret.

ESO Architecture

┌─────────────────────────────────────────────────────────┐
│                   Kubernetes Cluster                     │
│                                                          │
│  ┌──────────────┐    watches    ┌───────────────────┐   │
│  │     ESO      │◀────────────│  ExternalSecret    │   │
│  │  Controller  │              │      CRD           │   │
│  └──────┬───────┘              └───────────────────┘   │
│         │                                               │
│         │ fetches secrets                                │
│         │ via ClusterSecretStore                         │
│         ▼                                               │
│  ┌──────────────┐    creates    ┌───────────────────┐   │
│  │    Vault     │──────────────▶│  Kubernetes       │   │
│  │   (external) │              │  Secret            │   │
│  └──────────────┘              └───────────────────┘   │
│                                         │               │
│                                         ▼               │
│                                  ┌──────────────┐       │
│                                  │     Pod      │       │
│                                  │ (env/volume) │       │
│                                  └──────────────┘       │
└─────────────────────────────────────────────────────────┘
Key CRDs: ClusterSecretStore (connection to Vault) + ExternalSecret (what to sync) = Kubernetes Secret (result)
The ESO controller runs as a Deployment in the cluster. It watches for ExternalSecret resources. When it finds one, it reads the referenced ClusterSecretStore to know how to connect to Vault, fetches the specified secrets, and creates or updates a native K8s Secret. The pod consumes the K8s Secret through normal volume mounts or environment variables — it has no idea Vault exists.

ClusterSecretStore CRD

Configures the connection from ESO to Vault (cluster-scoped)

apiVersion: external-secrets.io/v1beta1
kind: ClusterSecretStore
metadata:
  name: vault-backend
spec:
  provider:
    vault:
      server: "http://vault.vault.svc.cluster.local:8200"
      path: "secret"
      version: "v2"
      auth:
        kubernetes:
          mountPath: "kubernetes"
          role: "eso-role"
          serviceAccountRef:
            name: "external-secrets"
            namespace: "external-secrets"
💡 Cluster-scoped vs namespace-scoped: ClusterSecretStore is available to all namespaces. SecretStore is namespace-scoped — use it to limit which namespaces can access which secret providers.
Walk through each field. The server URL uses the in-cluster DNS name for Vault. The path is the mount path of the KV v2 engine. The auth block tells ESO how to authenticate to Vault — here it uses the Kubernetes auth method with a specific role and service account. The Verisign platform creates this ClusterSecretStore as part of the FluxCD bootstrap.

ExternalSecret CRD

Defines which secrets to sync from Vault into a K8s Secret

apiVersion: external-secrets.io/v1beta1
kind: ExternalSecret
metadata:
  name: db-credentials
  namespace: my-app
spec:
  refreshInterval: "5m"
  secretStoreRef:
    name: vault-backend
    kind: ClusterSecretStore
  target:
    name: db-credentials       # K8s Secret name
    creationPolicy: Owner      # ESO owns the Secret lifecycle
  data:
    - secretKey: username       # Key in K8s Secret
      remoteRef:
        key: secret/myapp/db   # Path in Vault
        property: username      # Field within the Vault secret
    - secretKey: password
      remoteRef:
        key: secret/myapp/db
        property: password
This is the primary CRD that developers interact with. The refreshInterval controls how often ESO polls Vault for updates. secretStoreRef points to the ClusterSecretStore. The target defines the K8s Secret that will be created. The data array maps individual fields from Vault to keys in the K8s Secret. Walk through each field carefully.

ExternalSecret: Sync Entire Paths

Individual Keys (spec.data)

spec:
  data:
    - secretKey: username
      remoteRef:
        key: secret/myapp/db
        property: username
    - secretKey: password
      remoteRef:
        key: secret/myapp/db
        property: password

Explicit mapping, full control over K8s Secret keys

Entire Path (spec.dataFrom)

spec:
  dataFrom:
    - extract:
        key: secret/myapp/db

Syncs ALL key-value pairs from the Vault path. K8s Secret keys match Vault keys.

⚠️ Trade-off: dataFrom is simpler but gives less control. If someone adds an unexpected key in Vault, it automatically appears in the K8s Secret. Use data for explicit control.
Show both patterns. dataFrom is convenient when you want to sync everything from a Vault path. data is more explicit and safer because you control exactly which keys end up in the K8s Secret. For production, data is generally preferred because it makes the contract between Vault and the application explicit.

The Full End-to-End Flow

  ① Developer stores secret in Vault
     $ vault kv put secret/myapp/db username=admin password=s3cure

  ② Developer creates ExternalSecret CR in their namespace
     $ kubectl apply -f external-secret.yaml -n my-app

  ③ ESO controller detects the ExternalSecret
     → Reads ClusterSecretStore for Vault connection info
     → Authenticates to Vault via K8s auth

  ④ ESO fetches the secret from Vault
     → GET /v1/secret/data/myapp/db

  ⑤ ESO creates/updates the K8s Secret
     → K8s Secret "db-credentials" in namespace "my-app"

  ⑥ Pod mounts the Secret (env vars or volume)
     → Application reads DB_USERNAME, DB_PASSWORD
💡 The developer never handles raw secrets in YAML or Git. Only references (ExternalSecret CRDs) are stored in version control. Actual secret values live exclusively in Vault.
Walk through each numbered step. This is the complete lifecycle. The key insight is that the ExternalSecret YAML is safe to commit to Git — it contains no secret values, only references. The actual secret value lives in Vault and is fetched at runtime by ESO. This is the pattern the Verisign platform uses end-to-end.

🔄 Secret Rotation

How ESO Handles Rotation

  • spec.refreshInterval polls Vault periodically
  • When the value changes in Vault, ESO updates the K8s Secret
  • Pods using volume mounts see updates automatically (~60s kubelet sync)
  • Pods using env vars need a restart to pick up changes

Handling Env Var Restarts

Reloader pattern: Use stakater/Reloader to automatically trigger rolling restarts when a Secret changes. 
metadata:
  annotations:
    reloader.stakater.com/auto: "true"

Add this annotation to your Deployment to enable automatic restarts on Secret changes.

Secret rotation is a critical operational concern. ESO polls Vault at the configured refreshInterval. Volume-mounted secrets are updated by the kubelet automatically (roughly every 60 seconds). But environment variables are set at pod start and never updated — you need to restart the pod. Stakater Reloader watches for Secret changes and triggers rolling restarts. This is an important pattern to know for production.

Vault vs AWS Secrets Manager

Feature HashiCorp Vault AWS Secrets Manager
Hosting Self-hosted / HCP Fully managed by AWS
Dynamic Secrets Yes (DB, AWS, PKI, etc.) Lambda rotation only
Auth Methods K8s, AppRole, AWS, OIDC, LDAP, ... IAM only
Multi-cloud Yes — cloud agnostic AWS only
ESO Support Yes Yes
Operational Burden Higher (self-managed) Lower (managed service)
💡 ESO abstracts the provider. Your ExternalSecret CRDs look the same regardless of whether the backend is Vault or AWS Secrets Manager. Only the SecretStore changes.
This comparison is relevant because Verisign runs on AWS EKS and could use either. Vault provides more features (dynamic secrets, multi-cloud, more auth methods) but requires operational investment. AWS Secrets Manager is simpler but limited to AWS and IAM auth. The key point is that ESO abstracts the difference — if you migrate from one to the other, only the ClusterSecretStore changes. ExternalSecrets remain the same.

✅ Best Practices

  • Never store Vault tokens in Git — use Kubernetes auth, not static tokens
  • Scope policies narrowly — each app gets its own policy with minimum capabilities
  • Enable audit logging — know who accessed what and when
  • Use ESO refreshInterval wisely — 5m is a good default; too frequent wastes resources
  • Use secretStoreRef to limit blast radius — namespace-scoped SecretStores where possible
  • Version your secrets — KV v2 gives you rollback capability
  • Never put secret values in ExternalSecret CRDs — only references
  • Monitor ExternalSecret status — set up alerts for sync failures
Walk through each best practice. These are informed by real-world production experience. The most common mistakes are overly broad policies, static tokens checked into Git, and no monitoring of ESO sync status. Emphasize that ExternalSecret CRDs are safe to commit to Git because they contain only references, not values.

🔧 Troubleshooting

Common Issues and Diagnostics

ClusterSecretStore Not Ready

# Check status
kubectl get clustersecretstore
kubectl describe clustersecretstore vault-backend

Common causes: wrong Vault address, Vault not running, K8s auth misconfigured, ESO service account missing permissions

ExternalSecret SecretSyncedError

# Check the ExternalSecret status
kubectl get externalsecret -n my-app
kubectl describe externalsecret db-credentials -n my-app

Common causes: Vault policy denies access, wrong path (missing /data/ prefix), wrong property name, secret does not exist in Vault

Troubleshooting is essential. The most common issue is a ClusterSecretStore that is not Ready — usually due to a wrong Vault address or auth misconfiguration. The second most common is SecretSyncedError on ExternalSecrets — usually a policy denial or an incorrect path. The describe command is your best friend here because it shows the exact error message from the ESO controller.

🔧 Troubleshooting (continued)

Stale Secrets

# Check refreshInterval in the ExternalSecret
kubectl get externalsecret db-credentials -n my-app -o yaml | grep refreshInterval

# Force a reconciliation by annotating
kubectl annotate externalsecret db-credentials \
  force-sync=$(date +%s) -n my-app

If the refresh interval is too long, secrets may appear stale after a Vault update

Useful Diagnostic Commands

# ESO controller logs
kubectl logs -l app.kubernetes.io/name=external-secrets -n external-secrets

# Verify Vault is accessible from within the cluster
kubectl exec -it vault-0 -n vault -- vault status

# Test K8s auth manually
kubectl exec -it vault-0 -n vault -- vault login -method=kubernetes \
  role=eso-role jwt=@/var/run/secrets/kubernetes.io/serviceaccount/token
Additional troubleshooting steps. If secrets are stale, check the refreshInterval and consider force-reconciling. The ESO controller logs contain detailed error messages. You can also test Vault connectivity and authentication from within the cluster using kubectl exec. These commands will be useful in the upcoming lab.
