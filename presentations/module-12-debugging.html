🚨 Common Application Failure States

Recognizing the symptoms is half the diagnosis

This section covers the failure states you'll encounter most frequently. For each one, we'll look at what the status means, what causes it, and the first commands you should run to investigate. The key insight is that Kubernetes communicates problems through a well-defined set of pod statuses and event messages — learning to read them is your most important debugging skill.

💥 CrashLoopBackOff

The container starts, crashes, and Kubernetes keeps restarting it with exponential backoff.

Common Causes

  • Application error on startup
  • Missing environment variables or config
  • Incorrect command or entrypoint
  • Dependency not available (database, API)
  • File permission issues in the container

Diagnosis Steps

  • kubectl logs <pod> — check current crash
  • kubectl logs <pod> -p — check previous crash
  • kubectl describe pod <pod> — check events
  • Look at Exit Code: 1 = app error, 137 = OOM killed, 139 = segfault
CrashLoopBackOff is probably the single most common error you'll encounter. The name describes the behavior: the container crashes, kubelet restarts it, it crashes again, and the backoff delay increases (10s, 20s, 40s, up to 5 minutes). The key diagnostic tool here is 'kubectl logs' — use the -p flag to see logs from the previous container instance, because the current one may have already crashed with no output. Exit codes are critical: code 1 is a generic application error, 137 means the process received SIGKILL (usually OOM), and 139 is a segmentation fault.

📦 ImagePullBackOff

Kubernetes cannot pull the container image from the registry.

Common Causes

  • Image name or tag misspelled
  • Image tag does not exist (e.g., :latest removed)
  • Private registry — missing imagePullSecrets
  • Registry authentication expired or rotated
  • Network policy blocking egress to registry

Diagnosis Steps

  • kubectl describe pod — read the Events section
  • Verify the image exists: docker pull <image>
  • Check imagePullSecrets on pod spec
  • Verify Secret exists and has valid .dockerconfigjson
"Always pin image tags in production. Relying on :latest is a debugging nightmare."
ImagePullBackOff and ErrImagePull are closely related. ErrImagePull is the initial failure; ImagePullBackOff kicks in after repeated failures. The most common cause in enterprise environments is authentication — someone rotated the registry credentials but didn't update the Kubernetes secret. The describe command will show you the exact error message from the registry, which usually tells you whether it's a 401 (auth), 404 (image not found), or a network error. Remind participants to always check whether the imagePullSecrets reference is correct and that the referenced Secret actually exists in the same namespace.

⏳ Pending Pods

The pod has been accepted but cannot be scheduled to a node.

Scheduling Failures

  • Insufficient resources — no node has enough CPU or memory
  • Node affinity/anti-affinity — no matching nodes
  • Taints and tolerations — pod doesn't tolerate node taints
  • PVC binding — PersistentVolumeClaim cannot find a matching PV

Investigation

  • kubectl describe pod — check scheduler Events
  • kubectl get nodes — verify node status
  • kubectl describe node — check allocatable vs. requested
  • kubectl get pvc — check volume claim status
Pending pods are frustrating because the pod just sits there without any container-level logs to examine. The scheduler tells you why it couldn't place the pod through events on the pod itself. Run 'kubectl describe pod' and look at the Events section — you'll see messages like "0/5 nodes are available: 3 Insufficient cpu, 2 node(s) had taint that the pod didn't tolerate." This tells you exactly what's wrong. For PVC issues, check whether the StorageClass exists and whether the PV provisioner is working correctly.

💾 OOMKilled

The container exceeded its memory limit and was terminated by the kernel OOM killer.

Understanding OOMKilled (Exit Code 137)

  • Memory limit exceeded — the container tried to use more memory than its resources.limits.memory
  • No graceful shutdown — the process receives SIGKILL, not SIGTERM
  • Often intermittent — may happen only under load or with specific data
Quick Check:
kubectl get pod <pod> -o jsonpath='{.status.containerStatuses[0].lastState}'
Fix Options:
Increase memory limit, fix memory leak, tune JVM/GC settings, add memory profiling
OOMKilled is exit code 137 (128 + 9, where 9 = SIGKILL). This is one of the trickiest issues because there are often no logs — the process is killed instantly. Check the pod's lastState to confirm OOMKilled as the reason. Common causes include Java applications with heap sizes exceeding the container limit, memory leaks that accumulate over time, or simply setting limits too low. Remind participants that requests and limits serve different purposes: requests affect scheduling, limits affect runtime enforcement. A best practice is to set memory requests equal to limits to avoid the node-level OOM killer making unpredictable choices.

💓 Readiness & Liveness Probe Failures

Liveness Probe Failure

  • Container is killed and restarted
  • Causes: deadlocked process, hung thread, unresponsive endpoint
  • Too-aggressive thresholds cause unnecessary restarts
Symptom: High restart count with "Liveness probe failed" events

Readiness Probe Failure

  • Container is removed from Service endpoints
  • Causes: slow startup, dependency unavailable, high load
  • Pod stays Running but receives no traffic
Symptom: Pod is Running but Endpoints list shows 0 ready
"Liveness probes answer: Is it alive? Readiness probes answer: Can it serve traffic?"
Probe failures are a very common source of confusion. A failing liveness probe causes restarts — which can look like CrashLoopBackOff but is actually the kubelet killing the container. Check the restart count and events to distinguish between application crashes and liveness kills. A failing readiness probe is more subtle: the pod stays Running, but it's removed from service endpoints, so no traffic reaches it. Use 'kubectl get endpoints' to check. Common mistakes include setting initialDelaySeconds too low for applications that need time to start up, or pointing the probe at an endpoint that depends on external services.

🔧 Init Container Failures

Init containers must complete successfully before the main container starts.
  • Pod stuck in Init:0/2 — the first init container hasn't completed
  • Common causes: waiting for a dependency, incorrect script, missing Secret/ConfigMap
  • Diagnosis: kubectl logs <pod> -c <init-container-name>
  • Check status: kubectl get pod <pod> -o jsonpath='{.status.initContainerStatuses}'

Pro Tip

Init containers run sequentially. If the first one hangs, none of the others — or the main containers — will ever start. Always add timeouts or exit conditions to init containers that wait for external dependencies.

Init containers are often used for database migrations, waiting for services to become available, or populating shared volumes. When they fail or hang, the pod status shows "Init:0/N" and the main container never starts. The most important thing is to check the logs of the specific init container that's failing using the -c flag. A common pattern is an init container that does "until nc -z my-db 5432; do sleep 1; done" — if the database never comes up, the pod hangs forever. Always add a timeout or maximum retry count.

📋 Configuration Errors

Missing or incorrect ConfigMaps, Secrets, and volume mounts prevent pod startup.

Missing ConfigMap or Secret

  • Pod stays in ContainerCreating
  • Event: "configmap 'my-config' not found"
  • Fix: Create the resource or mark it optional: true

Key Not Found in ConfigMap

  • Pod fails to start — missing env var
  • Event: "key 'DB_HOST' not found in configmap"
  • Fix: Add the key or use optional: true on the reference
# Mark a ConfigMap reference as optional
env:
  - name: DB_HOST
    valueFrom:
      configMapKeyRef:
        name: app-config
        key: DB_HOST
        optional: true  # Pod will start even if key is missing
Configuration errors are extremely common during initial deployments and environment promotions. When you deploy to a new namespace and forget to create the ConfigMap or Secret that the deployment references, pods get stuck in ContainerCreating. The describe command will show exactly which resource is missing. Best practice: use a deployment tool like Helm or Kustomize that packages all dependent resources together. The 'optional: true' flag is useful for development but should be used carefully in production — you probably want the pod to fail fast if critical config is missing rather than starting with empty values.

🛠️ Essential Troubleshooting Tools

Your kubectl debugging toolkit

Now that we've covered what can go wrong, let's look at the tools you'll use every day to investigate. All of these are kubectl subcommands — no additional tooling required. Mastering these commands is what separates someone who can deploy applications from someone who can keep them running in production.

📝 kubectl describe

Your first command for any debugging investigation. Shows detailed state, conditions, and events. 
# Describe a pod — focus on Events section at the bottom
kubectl describe pod my-app-7d9f5b6c4-x2k9m

# Describe a node — check resource pressure and conditions
kubectl describe node worker-01

# Describe a service — check endpoints and selector
kubectl describe svc my-app-service

What to Look For

  • Events — the timeline of what happened (scheduling, pulling, starting, failing)
  • Conditions — Ready, ContainersReady, Initialized, PodScheduled
  • State — Waiting (with reason), Running (with start time), Terminated (with exit code)
The describe command is the Swiss Army knife of Kubernetes debugging. Always start here. The Events section at the bottom of the output is the most valuable part — it shows you the sequence of things that happened, including scheduler decisions, image pulls, container starts, probe failures, and OOM kills. Events are time-limited (default 1 hour), so if you're debugging something that happened overnight, you may need to look at logs or monitoring tools instead. The Conditions section gives you a boolean summary of the pod's health dimensions.

📜 kubectl logs

Read stdout/stderr from containers — your window into what the application is doing. 
# Basic log output
kubectl logs my-app-7d9f5b6c4-x2k9m

# Logs from the PREVIOUS container instance (essential for CrashLoopBackOff)
kubectl logs my-app-7d9f5b6c4-x2k9m -p

# Follow logs in real time
kubectl logs my-app-7d9f5b6c4-x2k9m -f

# Logs from a specific container in a multi-container pod
kubectl logs my-app-7d9f5b6c4-x2k9m -c sidecar

# Last 100 lines from the past hour
kubectl logs my-app-7d9f5b6c4-x2k9m --tail=100 --since=1h

# Logs from all pods matching a label
kubectl logs -l app=my-app --all-containers=true
Key tip: Use -p (previous) when a container is in CrashLoopBackOff — the current container may have no logs yet because it just restarted.
kubectl logs is your second essential command. The -p flag is critically important — when a container crashes and restarts, you need to see the logs from the previous instance that actually crashed, not the new instance that just started. The --since and --tail flags help you narrow down large log volumes. The label selector (-l) is powerful for viewing logs across all replicas of a deployment at once. Remember that logs only show stdout/stderr — if the application writes logs to files inside the container, you'll need to use 'kubectl exec' or 'kubectl cp' to access them.

🖥️ kubectl exec

Execute commands inside a running container for interactive investigation. 
# Open an interactive shell
kubectl exec -it my-app-7d9f5b6c4-x2k9m -- /bin/sh

# Run a single command
kubectl exec my-app-7d9f5b6c4-x2k9m -- env

# Check if the app is listening on the expected port
kubectl exec my-app-7d9f5b6c4-x2k9m -- netstat -tlnp

# Test DNS resolution from inside the pod
kubectl exec my-app-7d9f5b6c4-x2k9m -- nslookup my-service.default.svc.cluster.local

# Exec into a specific container in a multi-container pod
kubectl exec -it my-app-7d9f5b6c4-x2k9m -c sidecar -- /bin/bash
Limitation: Many production images are minimal (distroless, scratch) and have no shell or debugging tools. Use kubectl debug for those cases.
kubectl exec gives you a shell inside the running container, which is invaluable for checking environment variables, testing network connectivity, inspecting files, and verifying the application state. However, modern best practices favor minimal container images that don't include shells or debugging tools. Distroless images and scratch-based Go binaries have no shell at all, so exec won't work. That's where kubectl debug comes in, which we'll cover next. Also note that exec requires the container to be running — you can't exec into a CrashLoopBackOff container that keeps dying immediately.

🐛 kubectl debug

Attach ephemeral debug containers to running pods — even those with no shell. 
# Add a debug container to a running pod
kubectl debug -it my-app-7d9f5b6c4-x2k9m --image=busybox --target=my-app

# Create a copy of the pod with a debug container (for CrashLoopBackOff)
kubectl debug my-app-7d9f5b6c4-x2k9m -it --copy-to=debug-pod \
  --container=debug --image=ubuntu -- /bin/bash

# Debug with a full networking toolkit
kubectl debug -it my-app-7d9f5b6c4-x2k9m \
  --image=nicolaka/netshoot --target=my-app

# Debug a node directly
kubectl debug node/worker-01 -it --image=ubuntu

Why Ephemeral Containers?

They share the pod's network namespace and process namespace, giving you full visibility into the target container's environment without modifying the pod spec or restarting it.

kubectl debug was promoted to stable in Kubernetes 1.25 and is a game-changer for production debugging. Ephemeral containers are added to a running pod without restarting it. The --target flag shares the process namespace with the specified container, letting you see its processes with 'ps' and inspect files in its filesystem via /proc. The --copy-to flag creates a copy of the pod, which is useful when the original pod keeps crashing. The nicolaka/netshoot image is excellent for network debugging — it includes curl, dig, nslookup, tcpdump, and many more tools. Node debugging creates a privileged pod on the specified node with the host filesystem mounted at /host.

📡 kubectl get events

View the cluster-wide event stream — see what's happening across all resources. 
# All events in the current namespace, sorted by time
kubectl get events --sort-by='.lastTimestamp'

# Watch events in real time
kubectl get events -w

# Events for a specific resource
kubectl get events --field-selector involvedObject.name=my-app-7d9f5b6c4-x2k9m

# Only warning events
kubectl get events --field-selector type=Warning

# Events across all namespaces
kubectl get events -A --sort-by='.lastTimestamp'
Note: Events have a default TTL of 1 hour. For historical event data, use a monitoring solution (Prometheus, Datadog) or a log aggregator.
Events are the cluster's activity log. While 'describe' shows events for a specific resource, 'get events' gives you the big picture. This is especially useful when you're not sure which resource is causing the problem. The -w (watch) flag lets you see events as they happen in real time, which is invaluable when you're actively making changes and want to see the results. The --field-selector flag lets you filter by type (Normal vs Warning) or by the specific resource involved. Remember the 1-hour TTL — if you need to investigate an incident from hours ago, you'll need to rely on your logging and monitoring infrastructure.

🔌 port-forward & cp

kubectl port-forward

Create a local tunnel to a pod or service for testing.

# Forward local port 8080 to pod port 80
kubectl port-forward pod/my-app 8080:80

# Forward to a service
kubectl port-forward svc/my-app 8080:80

# Bind to all interfaces (for remote access)
kubectl port-forward --address 0.0.0.0 \
  pod/my-app 8080:80

kubectl cp

Copy files to and from containers.

# Copy file from pod to local
kubectl cp my-app:/var/log/app.log \
  ./app.log

# Copy file to pod
kubectl cp ./config.json \
  my-app:/app/config.json

# Copy from specific container
kubectl cp my-app:/data ./data \
  -c sidecar
Use case: port-forward is perfect for testing an endpoint that isn't exposed via Ingress, or for accessing a database admin UI during debugging.
Port-forward creates a TCP tunnel from your local machine to a pod or service, which is invaluable for testing endpoints without exposing them externally. It's also useful for connecting local database clients to in-cluster databases during debugging. kubectl cp requires tar to be installed in the container, which is a limitation with minimal images. Use it to pull log files, heap dumps, or configuration files out of the container for local analysis. Both of these commands are for debugging only — they should not be used as production access patterns.

🗺️ Systematic Debugging Approach

A repeatable methodology for any Kubernetes issue

Having the right tools is necessary but not sufficient. You also need a systematic approach so you don't waste time chasing red herrings. This section introduces a debugging flowchart that works for the vast majority of Kubernetes application issues. The key principle is to work from the outside in: start with the pod status, then check the container, then the application, and finally the network.

🔄 The Debugging Flowchart

1. Pod Status
Is it Running?
→
2. Container
Is it started?
→
3. Application
Is it healthy?
→
4. Network
Is it reachable?

At Each Level, Ask:

  • What is the current state?
  • What do the events say?
  • What do the logs show?
  • What changed recently?

Principle: Outside In

  • Don't jump to the application layer
  • Verify infrastructure first
  • Check the simple things before the complex
  • Most issues are configuration, not code
This four-step flowchart should become second nature. When someone reports an issue, resist the urge to immediately SSH into a container or read application logs. Start at step 1: Is the pod even running? If it's Pending, the issue is scheduling — no amount of log reading will help. If it's Running but not Ready, the issue is likely probes or the application. Only after confirming the pod and container are running do you move to application-level debugging. And networking issues — DNS resolution, service discovery, network policies — should be checked last because they're the hardest to diagnose and least likely to be the problem.

📊 Checking Resource Status & Events

Step 1: Establish the current state before diving into details. 
# Quick status overview — what's the pod doing?
kubectl get pods -o wide

# Detailed pod information
kubectl describe pod my-app-7d9f5b6c4-x2k9m

# Check the deployment rollout status
kubectl rollout status deployment/my-app

# View the deployment's replica sets
kubectl get rs -l app=my-app

# Check if the service has endpoints
kubectl get endpoints my-app-service

Read the Status Column Carefully

  • Pending → Scheduling issue (check nodes, resources, affinity)
  • ContainerCreating → Image pull or volume mount issue
  • Running but READY 0/1 → Probe failure or slow startup
  • CrashLoopBackOff → Application crash (check logs -p)
  • Terminating stuck → Finalizer issue or unresponsive graceful shutdown
The 'get pods -o wide' command gives you the quick picture: status, restarts, age, node, and IP. The -o wide output is important because it shows which node the pod is on and its IP address, both of which are useful for further debugging. The READY column is key — 0/1 means the readiness probe is failing. A high restart count suggests CrashLoopBackOff or liveness probe failures. The deployment rollout status command is useful for ongoing rollouts — it tells you if the rollout is stuck and why.

📖 Reading Logs Effectively

Structured Approach

  • Start with the end — use --tail=50 to see recent output first
  • Check previous instance — -p for crashed containers
  • Time-bound your search — --since=10m to limit noise
  • All containers — --all-containers for sidecar output
  • Aggregate across replicas — -l app=my-app with labels

Common Patterns to Grep For

  • ERROR, FATAL, panic
  • connection refused
  • timeout, deadline exceeded
  • permission denied
  • out of memory, OOM
  • no such host (DNS failure)
# Combine tail + follow for real-time debugging
kubectl logs -f --tail=20 -l app=my-app --all-containers | grep -i error
Reading logs effectively is a skill. Don't start from the top of a 10,000-line log file. Start from the end with --tail, or use --since to scope the time window. When debugging across multiple replicas, use the label selector to see all instances at once. Pipe through grep to filter for errors. For JSON-structured logs, consider using jq to parse the output. If the application writes logs to files rather than stdout, you'll need to use 'kubectl exec' or 'kubectl cp' to access those files — this is a common issue with legacy applications running in Kubernetes.

🌐 Network Debugging

DNS resolution and service connectivity are the most common network issues.

DNS Troubleshooting

# Test DNS from inside a pod
kubectl exec -it my-app -- \
  nslookup my-service

# Full FQDN resolution
kubectl exec -it my-app -- \
  nslookup my-service.default.\
svc.cluster.local

# Check CoreDNS is running
kubectl get pods -n kube-system \
  -l k8s-app=kube-dns

# Check CoreDNS logs
kubectl logs -n kube-system \
  -l k8s-app=kube-dns

Connectivity Testing

# Test service connectivity
kubectl exec -it my-app -- \
  wget -qO- http://my-service:8080/health

# Test pod-to-pod connectivity
kubectl exec -it my-app -- \
  nc -zv 10.244.1.5 8080

# Check if a port is open
kubectl exec -it my-app -- \
  curl -v telnet://my-service:8080

# Verify service endpoints exist
kubectl get endpoints my-service
Network issues are the hardest to debug because the symptoms — timeouts, connection refused, no such host — can have many causes. Always start with DNS: can the pod resolve the service name? If DNS works, test connectivity with wget or curl. If the service resolves but no endpoints are returned, check that the service selector matches the pod labels. If endpoints exist but connectivity fails, check NetworkPolicies. Remember that services in different namespaces need the full FQDN: service-name.namespace.svc.cluster.local.

🧰 Using Debug Containers for Network Diagnosis

When your application container lacks networking tools, spin up a debug pod. 
# Launch a temporary debug pod with full networking toolkit
kubectl run debug-net --rm -it --image=nicolaka/netshoot -- /bin/bash

# Inside the debug pod:
# DNS lookup
dig my-service.default.svc.cluster.local

# TCP connectivity test
nmap -p 8080 my-service.default.svc.cluster.local

# HTTP request test
curl -v http://my-service:8080/health

# Trace the network path
traceroute my-service

# Capture packets (requires NET_ADMIN capability)
tcpdump -i eth0 port 8080 -n

Popular Debug Images

  • nicolaka/netshoot — full networking toolkit (curl, dig, tcpdump, nmap, iperf)
  • busybox — lightweight, includes basic utilities (nslookup, wget, ping)
  • curlimages/curl — minimal image with curl for HTTP testing
The netshoot image by nicolaka is the gold standard for network debugging in Kubernetes. It includes every networking tool you could want. The key is to run the debug pod in the same namespace as the service you're testing — this ensures it uses the same DNS search domain and is subject to the same NetworkPolicies. Use 'kubectl run' with --rm and -it for a temporary pod that cleans itself up when you exit. For debugging within the same network namespace as an existing pod, use 'kubectl debug' with the --target flag instead.

🔬 Advanced Debugging Techniques

Beyond the fundamentals — strategies for complex issues

Once you've mastered the fundamental tools and methodology, you'll encounter situations that require more advanced techniques. This section covers resource monitoring during active incidents, comparing working versus broken deployments, rollback strategies, and knowing when to escalate. These techniques are particularly valuable in production environments where the pressure to resolve issues quickly is high.

📈 Resource Monitoring During Issues

Real-time resource monitoring helps identify memory leaks, CPU throttling, and capacity issues. 
# Real-time resource usage per pod (requires metrics-server)
kubectl top pods

# Resource usage per node
kubectl top nodes

# Sort by memory usage to find the heaviest consumers
kubectl top pods --sort-by=memory

# Watch resource usage over time
watch -n 5 kubectl top pods -l app=my-app

# Compare requested vs actual usage
kubectl describe node worker-01 | grep -A 10 "Allocated resources"
CPU throttling: Pod uses 100% of its CPU limit. Application becomes slow but doesn't crash.
Memory pressure: Pod approaches its memory limit. Risk of OOMKilled if usage spikes.
kubectl top requires the metrics-server to be installed in the cluster. It gives you a point-in-time snapshot of CPU and memory usage. Use 'watch' to poll repeatedly and observe trends. The key comparison is between what a pod is using versus what it requested and what its limits are. If actual usage is consistently much higher than requests, the scheduler is making placement decisions based on incorrect data. If actual usage approaches the limit, you're at risk of OOMKilled (memory) or throttling (CPU). For longer-term monitoring, you need Prometheus + Grafana or a similar monitoring stack.

🔎 Comparing Working vs. Broken Deployments

When something breaks after a change, diff the working and broken states systematically. 
# Export the current (broken) deployment spec
kubectl get deployment my-app -o yaml > broken.yaml

# Compare with the previous revision
kubectl rollout history deployment/my-app
kubectl rollout history deployment/my-app --revision=3

# Diff two deployment revisions
diff <(kubectl rollout history deployment/my-app --revision=2) \
     <(kubectl rollout history deployment/my-app --revision=3)

# Check what changed in ConfigMaps or Secrets
kubectl get configmap my-config -o yaml
kubectl get secret my-secret -o jsonpath='{.data}' | base64 -d

Checklist: What Changed?

  • Container image tag?
  • Environment variables or ConfigMap/Secret values?
  • Resource requests/limits?
  • Probe configuration?
  • Volume mounts or permissions?
  • Node selectors, affinity, or tolerations?
The most efficient debugging approach when something was working and then stopped is to identify exactly what changed. Kubernetes tracks deployment revisions, so you can compare the current spec against previous revisions. The diff approach works well because the change is usually small and obvious once you see it side by side. Common culprits are image tag changes, environment variable modifications, and resource limit adjustments. If you're using GitOps, you can also diff the Git commits to see what changed in the manifests.

⏪ Rollback Strategies

When in doubt, rollback. Restore service first, then debug at leisure. 
# Rollback to the previous revision
kubectl rollout undo deployment/my-app

# Rollback to a specific revision
kubectl rollout undo deployment/my-app --to-revision=3

# Check rollout history to find the right revision
kubectl rollout history deployment/my-app

# Verify the rollback succeeded
kubectl rollout status deployment/my-app

# Pause a rollout that's going badly
kubectl rollout pause deployment/my-app

When to Rollback Immediately

  • Production traffic is impacted
  • Error rate is spiking
  • All new pods are crashing
  • The fix is not immediately obvious

When to Debug First

  • Issue is in staging/dev
  • Only some pods are affected
  • The cause is likely obvious
  • Rolling back has its own risks
The instinct to debug and fix forward is strong, but in production, the priority is always restoring service. A rollback takes seconds and immediately restores a known-good state. You can then debug the broken version in a non-production environment. Kubernetes keeps a revision history by default (revisionHistoryLimit, default 10), so you can roll back to any previous version. The 'rollout pause' command is useful when you see a bad rollout in progress — it stops the rollout from progressing while leaving the currently running pods untouched. After pausing, you can investigate and then either resume or undo.

🚀 When to Escalate to the Platform Team

Not every issue is an application problem. Know when the issue is beyond your scope.

Escalate When You See:

  • Node NotReady — node-level failure
  • All pods on a node failing — likely node issue
  • PV provisioning failures — storage infrastructure
  • DNS resolution failing cluster-wide — CoreDNS issue
  • Certificate errors across services — PKI/cert-manager issue
  • etcd or API server errors — control plane issue

What to Include When Escalating

  • Affected namespace, pods, and nodes
  • Timeline: when it started, what changed
  • Output of kubectl describe
  • Relevant events and logs
  • Steps you've already tried
  • Impact: how many users/services affected
There's no shame in escalating — in fact, escalating quickly and with good information is a sign of maturity. The key is knowing which issues are within your control as an application team and which require platform team intervention. Node failures, storage provisioner issues, DNS infrastructure, and control plane problems all require cluster-level access to diagnose and fix. When you do escalate, provide a clear summary: what's affected, when it started, what you've checked, and what the impact is. This saves the platform team from re-doing your investigation and accelerates resolution.

Lab 12: Debugging Applications in Kubernetes

Diagnose and fix a series of intentionally broken deployments using the systematic debugging approach covered in this module.

  • Scenario A: Fix a CrashLoopBackOff deployment — identify the application error, correct the configuration, and verify the fix
  • Scenario B: Resolve an ImagePullBackOff issue — investigate registry authentication, correct the image reference, and deploy successfully
  • Scenario C: Diagnose a resource constraint problem — identify why pods are Pending or OOMKilled, adjust resource specifications
  • Scenario D: Troubleshoot probe failures — find and fix misconfigured liveness and readiness probes causing restarts and traffic loss

Duration: 45 minutes  |  Format: Individual or pairs

This lab provides four independent debugging scenarios. Each scenario deploys a broken application that exhibits one of the failure states covered in the module. Participants should use the systematic debugging flowchart: check pod status, check events, check logs, identify the root cause, apply the fix, and verify. Encourage participants to take notes on their debugging process — which commands they ran and in what order. This helps reinforce the methodology. For the CrashLoopBackOff scenario, the fix involves correcting an environment variable. For ImagePullBackOff, they need to create an imagePullSecret. For resource constraints, they need to adjust requests and limits. For probes, they need to fix the probe endpoint path and timing. Allow about 10 minutes per scenario with 5 minutes for wrap-up discussion.

📌 Key Takeaways

Module 12 — Debugging Your Application in Kubernetes
  • Know the failure states — CrashLoopBackOff, ImagePullBackOff, Pending, OOMKilled, and probe failures each have distinct causes and diagnostic paths
  • Start with describe and events — they provide the most context with the least effort and should always be your first investigation step
  • Use -p for previous container logs — this single flag is the difference between seeing the crash output and seeing nothing
  • Follow the flowchart: Pod → Container → App → Network — work outside-in to avoid chasing the wrong layer
  • Ephemeral debug containers are powerful — kubectl debug gives you full diagnostic capability even with minimal production images
  • Rollback first, debug second — in production, restore service before root-causing the issue
Review each takeaway. The most important ones to emphasize are the systematic flowchart approach and the rollback-first mindset. Participants should leave this module with the confidence that they can diagnose any common Kubernetes application issue using a repeatable methodology. The lab will reinforce this. In the final module (Module 13), we'll look at broader operational practices that build on the debugging skills covered here.
