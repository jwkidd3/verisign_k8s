⚙️ NGINX Ingress Controller

Architecture and IngressClass

NGINX Ingress Controller is the most widely deployed ingress controller. Let's understand its architecture before creating Ingress resources.

🏗️ NGINX Ingress Architecture

How It Works

  1. NGINX Ingress Controller pod runs NGINX reverse proxy
  2. Exposed via a LoadBalancer Service (single AWS NLB/ALB)
  3. Watches for Ingress resources across the cluster
  4. Dynamically generates NGINX configuration from Ingress rules
  5. Reloads NGINX when Ingress resources change
IngressClass: Specifies which controller handles an Ingress resource. Multiple controllers can coexist (e.g., internal vs external).
The NGINX Ingress Controller runs as a Deployment in your cluster, typically in an ingress-nginx namespace. It watches the Kubernetes API for Ingress resources, translates them into NGINX config, and reloads. The IngressClass mechanism allows multiple ingress controllers to coexist — critical for Verisign where you might have separate internal and external ingress controllers.

🔧 IngressClass Configuration

apiVersion: networking.k8s.io/v1
kind: IngressClass
metadata:
  name: nginx-external
  annotations:
    ingressclass.kubernetes.io/is-default-class: "true"
spec:
  controller: k8s.io/ingress-nginx
  parameters:
    apiGroup: k8s.io/ingress-nginx
    kind: IngressNginxConfig
    name: external-config
---
apiVersion: networking.k8s.io/v1
kind: IngressClass
metadata:
  name: nginx-internal
spec:
  controller: k8s.io/ingress-nginx-internal
Production Pattern: Verisign runs separate IngressClasses for external (internet-facing) and internal (VPC-only) traffic, each backed by its own controller deployment and load balancer.
IngressClass is mandatory in modern Kubernetes. It decouples the Ingress resource from the controller. By setting is-default-class, any Ingress without an explicit ingressClassName will be picked up by that controller. Running separate internal and external controllers is a common production pattern.

📐 Ingress Resources

Path-Based and Host-Based Routing

Now let's create Ingress resources to define routing rules for incoming HTTP traffic.

📐 Path-Based Routing

apiVersion: networking.k8s.io/v1
kind: Ingress
metadata:
  name: app-ingress
  namespace: verisign-apps
  annotations:
    nginx.ingress.kubernetes.io/rewrite-target: /$2
spec:
  ingressClassName: nginx-external
  rules:
  - host: app.verisign.internal
    http:
      paths:
      - path: /api(/|$)(.*)
        pathType: ImplementationSpecific
        backend:
          service:
            name: api-service
            port:
              number: 8080
      - path: /admin(/|$)(.*)
        pathType: ImplementationSpecific
        backend:
          service:
            name: admin-service
            port:
              number: 8080
      - path: /
        pathType: Prefix
        backend:
          service:
            name: frontend-service
            port:
              number: 80
Path-based routing directs traffic to different services based on the URL path. The rewrite-target annotation strips the path prefix before forwarding to the backend. pathType can be Exact, Prefix, or ImplementationSpecific. Rules are evaluated in order of specificity — more specific paths match first.

🌐 Host-Based Routing

apiVersion: networking.k8s.io/v1
kind: Ingress
metadata:
  name: multi-host-ingress
  namespace: verisign-apps
spec:
  ingressClassName: nginx-external
  rules:
  - host: api.verisign.internal
    http:
      paths:
      - path: /
        pathType: Prefix
        backend:
          service:
            name: api-service
            port:
              number: 8080
  - host: dashboard.verisign.internal
    http:
      paths:
      - path: /
        pathType: Prefix
        backend:
          service:
            name: dashboard-service
            port:
              number: 3000
  # Default backend for unmatched hosts
  defaultBackend:
    service:
      name: default-404-service
      port:
        number: 80
Host-based routing uses the Host header to direct traffic. Each hostname maps to a different backend service. The defaultBackend handles requests that don't match any host rule — useful for returning a custom 404 page. In production, combine this with External DNS to automatically create DNS records.

📌 Path Types Explained

pathTypeMatching BehaviorExample
Exact Matches URL path exactly /api matches /api only, not /api/v1
Prefix Matches path prefix by / segments /api matches /api, /api/v1, /api/v1/users
ImplementationSpecific Depends on IngressClass controller NGINX supports regex: /api(/|$)(.*)
Gotcha: Prefix matching splits on / boundaries. Path /api does NOT match /apiversion — it only matches paths that have /api as a complete segment prefix.
Understanding pathType is critical for correct routing. Prefix matching works on slash-separated segments, not arbitrary string prefixes. This prevents accidental matching of /apiversion when you only meant /api. ImplementationSpecific lets you use NGINX-specific features like regex paths.

🔧 Ingress Annotations

Customizing NGINX Behavior

Annotations are the primary extension mechanism for NGINX Ingress. They control everything from redirects to rate limiting.

🔧 Essential NGINX Ingress Annotations

apiVersion: networking.k8s.io/v1
kind: Ingress
metadata:
  name: production-ingress
  annotations:
    # SSL Redirect
    nginx.ingress.kubernetes.io/ssl-redirect: "true"
    nginx.ingress.kubernetes.io/force-ssl-redirect: "true"

    # Request Body Size (file uploads)
    nginx.ingress.kubernetes.io/proxy-body-size: "50m"

    # Timeouts
    nginx.ingress.kubernetes.io/proxy-connect-timeout: "10"
    nginx.ingress.kubernetes.io/proxy-read-timeout: "120"
    nginx.ingress.kubernetes.io/proxy-send-timeout: "120"

    # Rate Limiting
    nginx.ingress.kubernetes.io/limit-rps: "50"
    nginx.ingress.kubernetes.io/limit-burst-multiplier: "5"

    # CORS
    nginx.ingress.kubernetes.io/enable-cors: "true"
    nginx.ingress.kubernetes.io/cors-allow-origin: "https://app.verisign.com"
    nginx.ingress.kubernetes.io/cors-allow-methods: "GET, POST, PUT, DELETE"
    nginx.ingress.kubernetes.io/cors-allow-headers: "Authorization, Content-Type"
These are the most commonly used annotations in production. SSL redirect ensures all HTTP traffic is upgraded to HTTPS. proxy-body-size controls upload limits. Rate limiting protects your backends from overload. CORS headers are essential for APIs consumed by web frontends. Each annotation maps to an NGINX directive in the generated configuration.

🔧 Advanced Annotations

Session Affinity

annotations:
  nginx.ingress.kubernetes.io/
    affinity: "cookie"
  nginx.ingress.kubernetes.io/
    session-cookie-name: "VSRVSID"
  nginx.ingress.kubernetes.io/
    session-cookie-max-age: "172800"

Custom Headers & Snippets

annotations:
  nginx.ingress.kubernetes.io/
    configuration-snippet: |
      more_set_headers
        "X-Request-ID: $req_id";
      more_set_headers
        "Strict-Transport-Security:
         max-age=31536000";
⚠️ Security Warning: configuration-snippet and server-snippet annotations can inject arbitrary NGINX config. In multi-tenant clusters, disable them via controller flags to prevent abuse.
Session affinity via cookies is useful for stateful applications. Custom snippets allow injecting raw NGINX configuration — powerful but dangerous in multi-tenant environments. An attacker could inject malicious NGINX config to steal traffic. Always consider disabling snippet annotations in shared clusters.

🔒 TLS Termination

Securing Ingress with HTTPS

TLS termination at the Ingress controller is the standard pattern for HTTPS. Let's see how to configure it with both manual certificates and automated cert-manager.

🔒 TLS with Manual Certificates

# Create TLS secret from certificate files
kubectl create secret tls app-tls \
  --cert=fullchain.pem \
  --key=privkey.pem \
  -n verisign-apps
apiVersion: networking.k8s.io/v1
kind: Ingress
metadata:
  name: secure-ingress
  namespace: verisign-apps
  annotations:
    nginx.ingress.kubernetes.io/ssl-redirect: "true"
spec:
  ingressClassName: nginx-external
  tls:
  - hosts:
    - api.verisign.internal
    - dashboard.verisign.internal
    secretName: app-tls    # References the TLS secret
  rules:
  - host: api.verisign.internal
    http:
      paths:
      - path: /
        pathType: Prefix
        backend:
          service:
            name: api-service
            port:
              number: 8080
The tls block references a kubernetes.io/tls type Secret containing the certificate chain and private key. The hosts list must match the certificate's Subject Alternative Names. Multiple hosts can share the same certificate if it's a wildcard cert or has multiple SANs.

External Access Patterns

Getting traffic into (and out of) the cluster

We start with the fundamental challenge: Kubernetes pods live in an internal network. External clients cannot reach them directly. Over the years, the Kubernetes ecosystem has developed increasingly sophisticated solutions to this problem.

The Challenge: Traffic Into the Cluster

Pods are ephemeral and live on an internal network unreachable from outside

Inbound (Ingress) Traffic

  • Clients need stable endpoints
  • HTTP/HTTPS routing by host and path
  • TLS termination at the edge
  • Load balancing across pods

Outbound (Egress) Traffic

  • Pods calling external APIs or databases
  • NAT and IP masquerading
  • Network policies for egress control
  • Compliance and audit requirements
Clarify the distinction between ingress and egress early. Most of this module focuses on ingress (inbound), since that is where the Kubernetes resource model has evolved the most. Egress control is typically handled via NetworkPolicies (covered in Module 5) and CNI-specific features. Both directions matter for security and compliance.

Evolution of External Access

Each generation solved limitations of the one before it

NodePort

  • Expose a port on every node
  • Range: 30000-32767
  • Simple but inflexible
  • No HTTP-aware routing

LoadBalancer

  • Cloud-provisioned L4 LB
  • One LB per Service
  • Expensive at scale
  • No path/host routing

Ingress

  • L7 HTTP/HTTPS routing
  • Single entry point
  • Host and path rules
  • Now being superseded

Gateway API

  • Next-generation L4-L7
  • Role-oriented design
  • Highly expressive
  • GA since Kubernetes 1.29
Walk through the timeline. NodePort was the original approach, still useful for development. LoadBalancer works well in cloud environments but creates cost sprawl when you have many services. Ingress consolidated L7 routing behind a single load balancer but had expressiveness limitations. Gateway API is the current standard, bringing role-oriented design and a richer feature set. All four still exist and you will encounter each in production.

NodePort and LoadBalancer Services

NodePort

apiVersion: v1
kind: Service
metadata:
  name: my-app
spec:
  type: NodePort
  selector:
    app: my-app
  ports:
    - port: 80
      targetPort: 8080
      nodePort: 31000

Clients must know a node IP and the high port number. Not suitable for production HTTP traffic.

LoadBalancer

apiVersion: v1
kind: Service
metadata:
  name: my-app
spec:
  type: LoadBalancer
  selector:
    app: my-app
  ports:
    - port: 80
      targetPort: 8080

Cloud controller provisions an external LB. One LB per service can become costly at scale.

Show these side by side so students can see the simplicity and the limitations. NodePort is often combined with an external load balancer in on-prem setups. LoadBalancer type relies on a cloud controller integration. In enterprise environments like Verisign's, MetalLB or similar solutions may provide LoadBalancer functionality without a cloud provider.

Ingress Legacy

L7 HTTP/HTTPS routing into the cluster

Ingress is marked as legacy because the Kubernetes project now recommends Gateway API for new deployments. However, Ingress remains extremely common in existing clusters and will be around for years. Students need to understand both.

What Ingress Does

A declarative API for HTTP/HTTPS routing rules at Layer 7

Ingress Resource

  • Defines routing rules (host, path)
  • References backend Services
  • Specifies TLS certificates
  • Purely declarative -- does nothing alone

Ingress Controller

  • Watches Ingress resources
  • Configures a reverse proxy
  • Handles actual traffic routing
  • Must be installed separately

Key distinction: An Ingress resource without an Ingress controller has no effect. The controller is the data plane; the resource is the configuration.

This is a common point of confusion. Creating an Ingress resource does not magically expose your service. You need a controller running in the cluster that watches for these resources and configures the underlying proxy. This is analogous to writing a ConfigMap that nothing reads.

Ingress Controllers

Multiple implementations -- each with different capabilities

Controller Proxy Key Strengths Common Use Case
NGINX Ingress NGINX Mature, widely adopted, rich annotations General-purpose clusters
Traefik Traefik Auto-discovery, middleware chains Dynamic environments, GitOps
HAProxy HAProxy High performance, TCP support High-throughput workloads
Contour Envoy HTTPProxy CRD, multi-team Platform teams, delegation
AWS ALB Controller AWS ALB Native AWS integration EKS clusters
Emphasize that the choice of controller matters because each brings different features, performance characteristics, and annotation semantics. NGINX Ingress Controller is the most common starting point. In enterprise environments you may see multiple controllers running in the same cluster, each handling a different IngressClass.

apiVersion: networking.k8s.io/v1
kind: Ingress
metadata:
  name: multi-host

    - host: api.example.com

                name: web-svc
                port:

These are the two fundamental routing modes. Path-based sends different URL prefixes to different services behind a single hostname. Host-based routes by domain name. You can combine both. Note the ingressClassName field, which selects which controller handles this resource. The pathType field (Prefix, Exact, or ImplementationSpecific) controls how paths are matched.

TLS Termination with Ingress

Terminate HTTPS at the edge so backend pods handle plain HTTP

apiVersion: networking.k8s.io/v1
kind: Ingress
metadata:
  name: tls-ingress
  annotations:
    nginx.ingress.kubernetes.io/ssl-redirect: "true"
spec:
  ingressClassName: nginx
  tls:
    - hosts:
        - app.example.com
      secretName: app-tls-secret
  rules:
    - host: app.example.com
      http:
        paths:
          - path: /
            pathType: Prefix
            backend:
              service:
                name: app-svc
                port:
                  number: 80

The TLS secret must be a kubernetes.io/tls type Secret containing tls.crt and tls.key. cert-manager can automate this entirely.

TLS termination is one of the most valuable features of Ingress. The controller handles the TLS handshake and forwards decrypted traffic to backend services. This simplifies application code and centralizes certificate management. The ssl-redirect annotation ensures HTTP requests are redirected to HTTPS. The Secret must exist in the same namespace as the Ingress resource.

Ingress Annotations

Annotations extend Ingress with controller-specific behavior

Annotation (NGINX) Purpose
nginx.ingress.kubernetes.io/rewrite-target Rewrite the URL path before forwarding
nginx.ingress.kubernetes.io/ssl-redirect Force HTTPS redirect
nginx.ingress.kubernetes.io/proxy-body-size Max request body size
nginx.ingress.kubernetes.io/rate-limit Requests per second limit
nginx.ingress.kubernetes.io/auth-type Basic or external auth
nginx.ingress.kubernetes.io/cors-* CORS header configuration

The annotation problem: These are controller-specific, unvalidated strings. A typo is silently ignored. This is a key reason Gateway API was created.

Annotations are both the power and the weakness of Ingress. They let you configure sophisticated behaviors, but they are not validated by the API server, differ across controllers, and cannot be composed or delegated. This is the core architectural limitation that motivated the Gateway API. Make sure students understand that switching controllers may require rewriting all annotations.

Ingress: Current Status

The Kubernetes project considers Ingress effectively frozen. No new features will be added. Gateway API is the recommended path forward for new deployments.

Why Ingress Falls Short

  • Limited to HTTP/HTTPS only
  • No standard for TCP/UDP/gRPC
  • Annotations are unvalidated, non-portable
  • Single flat resource -- no role separation
  • No traffic splitting or header matching

When You Still Use Ingress

  • Existing clusters with established Ingress rules
  • Controllers not yet supporting Gateway API
  • Simple HTTP routing that works today
  • Environments where Gateway CRDs are not installed
  • Migration is in progress but not complete
Be clear that Ingress is not deprecated in the traditional sense -- it will continue to work. But it is frozen: no new features, no enhancements. Teams starting new projects should use Gateway API. Teams with existing Ingress deployments can migrate at their own pace. Many controllers now support both simultaneously.

Gateway API

The next generation of Kubernetes service networking

Gateway API reached GA (v1.0) in October 2023 and v1.2 in 2024. It is now the officially recommended approach for managing traffic into Kubernetes clusters. Let us look at why it was created and how it differs from Ingress.

Why Gateway API Was Created

Solving the structural limitations of Ingress with a purpose-built API

Expressiveness

  • Header-based routing
  • Traffic splitting and weighting
  • Request/response manipulation
  • TCP, UDP, gRPC, and TLS support

Role-Oriented Design

  • Infrastructure provider manages GatewayClass
  • Platform team configures Gateway
  • App developers define Routes
  • Clean separation of concerns

Portability

  • Typed, validated fields replace annotations
  • Core features are standard across implementations
  • Conformance tests ensure consistency
  • Extended features are clearly labeled
The three pillars of Gateway API are expressiveness, role orientation, and portability. Expressiveness means you can express routing rules that were impossible or non-portable with Ingress. Role orientation means different teams manage different resources with appropriate RBAC. Portability means switching implementations does not require rewriting your configuration.

Key Resources: GatewayClass, Gateway, HTTPRoute

GatewayClass

Managed by: Infrastructure Provider

  • Defines the controller implementation
  • Cluster-scoped resource
  • Similar to IngressClass or StorageClass

Gateway

Managed by: Platform Team

  • Binds to a GatewayClass
  • Defines listeners (ports, protocols, TLS)
  • Namespace-scoped

HTTPRoute

Managed by: App Developer

  • Attaches to a Gateway
  • Defines routing rules
  • Can live in app namespace
This layered model is the key design insight of Gateway API. Infrastructure teams provision the controller (GatewayClass). Platform teams configure shared infrastructure like listeners and TLS (Gateway). Application teams define their own routing rules (HTTPRoute) without needing access to infrastructure resources. This maps well to how enterprise organizations like Verisign actually operate.

Gateway API in Practice

Gateway

apiVersion: gateway.networking.k8s.io/v1
kind: Gateway
metadata:
  name: shared-gateway
  namespace: infra
spec:
  gatewayClassName: envoy
  listeners:
    - name: https
      protocol: HTTPS
      port: 443
      tls:
        mode: Terminate
        certificateRefs:
          - name: wildcard-tls
      allowedRoutes:
        namespaces:
          from: Selector
          selector:
            matchLabels:
              gateway: shared

HTTPRoute

apiVersion: gateway.networking.k8s.io/v1
kind: HTTPRoute
metadata:
  name: app-route
  namespace: my-app
spec:
  parentRefs:
    - name: shared-gateway
      namespace: infra
  hostnames:
    - app.example.com
  rules:
    - matches:
        - path:
            type: PathPrefix
            value: /api
      backendRefs:
        - name: api-svc
          port: 80
          weight: 90
        - name: api-svc-canary
          port: 80
          weight: 10

Notice the traffic splitting: 90/10 weighted routing to enable canary deployments -- impossible with standard Ingress.

Walk through both resources. The Gateway lives in the infra namespace, managed by the platform team. It defines an HTTPS listener with TLS termination and restricts which namespaces can attach routes using label selectors. The HTTPRoute lives in the app namespace and references the Gateway. The weighted backendRefs enable canary deployments natively. This is a feature that required service mesh or custom annotations with Ingress.

Gateway API vs. Ingress

Capability Ingress Gateway API
Protocol support HTTP/HTTPS only HTTP, HTTPS, TCP, UDP, gRPC, TLS
Traffic splitting Annotation-dependent Native weighted backendRefs
Header matching Not supported First-class support
Role separation Single resource, single owner GatewayClass / Gateway / Route
Cross-namespace routing Not standard Built-in with ReferenceGrants
Feature extension Unvalidated annotations Typed Policy Attachments
Conformance testing None Official conformance suite
This comparison makes the case clearly. Gateway API is not just a replacement -- it is a generational leap. The conformance testing is particularly important for enterprises: it means you can switch implementations with confidence that core behavior is preserved. Highlight that Gateway API is an official Kubernetes SIG-Network project, not a third-party initiative.

Gateway API: Adoption Status

GA Implementations

  • Envoy Gateway
  • Istio
  • Cilium
  • Contour
  • NGINX Gateway Fabric
  • Traefik (v3+)

What is GA Today

  • GatewayClass, Gateway, HTTPRoute
  • ReferenceGrant for cross-namespace
  • TLS termination and passthrough
  • GRPCRoute (v1.1+)
  • TCPRoute and UDPRoute are still experimental
  • BackendTLSPolicy in progress

Migration path: Most controllers that support Gateway API also continue to support Ingress. You can run both side-by-side and migrate incrementally.

The ecosystem has moved quickly. All major proxy implementations now have Gateway API support. The incremental migration path is important to emphasize -- no one needs to do a big-bang migration. In fact, some organizations run both for years, migrating route by route. CRDs must be installed separately since Gateway API is not bundled with Kubernetes itself.

Verisign Integration

External access in an enterprise DNS environment

Now let us consider how these concepts apply specifically to Verisign's environment. As a company operating critical DNS infrastructure, Verisign has unique requirements around external access, DNS integration, and traffic patterns.

External Access in Enterprise Environments

Production clusters rarely expose services directly to the public internet

Typical Enterprise Pattern

  • External load balancer (F5, HAProxy, etc.) in front of cluster
  • Ingress controller handles internal L7 routing
  • WAF and DDoS protection at the edge
  • Separate ingress for internal vs. external traffic
  • Private clusters with no public node IPs

On-Premises Considerations

  • No cloud LoadBalancer provisioner -- use MetalLB or similar
  • BGP or L2 advertisement for VIPs
  • Integration with existing network infrastructure
  • Firewall rules and network segmentation
  • Multiple availability zones / data centers
Enterprise environments add layers of infrastructure between the internet and Kubernetes pods. Verisign likely has external load balancers, firewalls, and network segmentation in front of any Kubernetes cluster. The Ingress controller or Gateway acts as the last routing hop before traffic reaches pods. Understanding this full path is critical for troubleshooting and architecture decisions.

DNS Integration Considerations

Connecting Kubernetes Services to DNS

ExternalDNS

  • Watches Ingress/Gateway/Service resources
  • Automatically creates DNS records
  • Supports many providers (Route53, CoreDNS, RFC2136, etc.)
  • Keeps records synchronized with cluster state

Enterprise DNS Workflow

  • Change management for DNS updates
  • Split-horizon DNS (internal vs. external)
  • DNSSEC signing requirements
  • Integration with IPAM systems

For Verisign: As a DNS infrastructure company, integrating Kubernetes service discovery with authoritative DNS systems requires careful coordination between cluster operators and DNS engineering teams.

ExternalDNS is the standard tool for automating DNS record creation from Kubernetes resources. In enterprise settings, you may need to integrate with existing DNS infrastructure rather than directly managing records. Split-horizon DNS is common: internal services resolve to cluster IPs, while external services resolve to load balancer IPs. DNSSEC adds another layer of complexity that Verisign teams will be well familiar with.

Internal vs. External Traffic Patterns

External Ingress

  • Public-facing APIs and UIs
  • TLS required
  • Rate limiting and WAF
  • Authentication at the edge
  • DDoS protection

Internal Ingress

  • Service-to-service within the org
  • mTLS between services
  • Private DNS names
  • No public exposure
  • Often separate controller instance

Egress Control

  • Pods calling external APIs
  • NetworkPolicy for egress rules
  • Egress gateways (Istio, Cilium)
  • NAT gateway for stable source IPs
  • Audit logging for compliance
Many enterprises run separate ingress controllers for external and internal traffic, often with different security policies. External ingress faces the internet and needs maximum protection. Internal ingress handles east-west traffic that crosses namespace or cluster boundaries. Egress control is equally important in regulated environments -- you need to know and control what your pods can reach outside the cluster.

Certificates for External Endpoints

Automating TLS with cert-manager

TLS certificates are a mandatory part of external access. Managing them manually is error-prone and does not scale. cert-manager is the de facto standard for automated certificate lifecycle management in Kubernetes.

TLS/SSL for Kubernetes Services

Every external endpoint must be encrypted in transit

Where TLS Terminates

  • Edge termination: At the Ingress controller / Gateway
  • Passthrough: Controller forwards encrypted traffic to the pod
  • Re-encrypt: Terminate at edge, re-encrypt to backend
  • Edge termination is most common

Certificate Challenges at Scale

  • Hundreds of services need unique certificates
  • Certificates expire -- renewals must be automated
  • Wildcard vs. per-service certificates
  • Private CA for internal services
  • Compliance: key storage and rotation policies
Walk through the three termination modes. Edge termination is simplest and most common -- the Ingress controller or Gateway handles TLS and backends receive plain HTTP. Passthrough is used when the application must handle its own TLS (e.g., for mTLS). Re-encrypt adds an extra hop of encryption for defense in depth. At scale, manual certificate management is untenable, which is why cert-manager exists.

cert-manager Overview

The standard Kubernetes add-on for automated certificate management

What cert-manager Does

  • Requests certificates from CAs automatically
  • Stores certificates as Kubernetes Secrets
  • Renews certificates before expiry
  • Supports ACME (Let's Encrypt), Vault, Venafi, private CAs
  • Integrates with Ingress and Gateway API

Architecture

  • Runs as a Deployment in the cluster
  • Watches Certificate and Issuer resources
  • Uses ACME HTTP-01 or DNS-01 challenges
  • Creates and manages TLS Secrets
  • Provides metrics and events for monitoring
cert-manager has become the de facto standard. It is maintained by the Jetstack team (now part of Venafi) and is a CNCF project. The key insight is that cert-manager turns certificate management into a declarative Kubernetes workflow: you define what you want, and cert-manager handles the rest. For enterprise environments, the Vault and Venafi issuers are particularly relevant since they integrate with existing PKI infrastructure.

🔶 Gateway API

The Evolution of Kubernetes Ingress

Gateway API is the successor to the Ingress resource, designed to address its many limitations. It's now GA for HTTP routing and is the recommended path forward.

📐 Ingress vs Gateway API

AspectIngressGateway API
ExtensibilityAnnotations (untyped strings)Typed, structured API fields
Role modelSingle resource, one ownerRole-oriented (infra, cluster, app)
ProtocolsHTTP/HTTPS onlyHTTP, TCP, UDP, gRPC, TLS
Traffic splittingNot nativeBuilt-in weight-based routing
Header matchingNot supportedNative header/query matching
PortabilityVendor-specific annotationsPortable across implementations
StatusStable, feature-frozenGA (HTTPRoute), evolving
Gateway API addresses every major pain point of Ingress. No more untyped annotations, explicit role separation, multi-protocol support, and native traffic management features. The Ingress API is feature-frozen — all innovation is happening in Gateway API.

🏗️ Gateway API Resource Model

🛠️ GatewayClass

Role: Infrastructure Provider

  • Defines the controller implementation
  • Similar to IngressClass
  • Cluster-scoped resource

🔶 Gateway

Role: Cluster Operator

  • Configures listeners (ports, protocols, TLS)
  • Provisions infrastructure (LB)
  • Namespace-scoped

📐 HTTPRoute

Role: Application Developer

  • Defines routing rules
  • Attaches to Gateway
  • Owned by app teams
Role Separation: Infrastructure teams manage GatewayClass and Gateway. Application teams create HTTPRoutes. This maps cleanly to Verisign's organizational model.
The three-tier resource model is Gateway API's biggest architectural advantage. Infrastructure providers define GatewayClasses. Platform teams create Gateways. Application developers attach routes. Each role has clear boundaries and RBAC can be applied at each level. This is much cleaner than Ingress where one resource mixes all concerns.

🔶 GatewayClass and Gateway

apiVersion: gateway.networking.k8s.io/v1
kind: GatewayClass
metadata:
  name: envoy-gateway
spec:
  controllerName: gateway.envoyproxy.io/gatewayclass-controller
---
apiVersion: gateway.networking.k8s.io/v1
kind: Gateway
metadata:
  name: verisign-gateway
  namespace: gateway-system
spec:
  gatewayClassName: envoy-gateway
  listeners:
  - name: http
    protocol: HTTP
    port: 80
    allowedRoutes:
      namespaces:
        from: All
  - name: https
    protocol: HTTPS
    port: 443
    tls:
      mode: Terminate
      certificateRefs:
      - kind: Secret
        name: wildcard-tls
        namespace: gateway-system
    allowedRoutes:
      namespaces:
        from: Selector
        selector:
          matchLabels:
            gateway-access: "true"
The Gateway resource defines listeners with protocols, ports, and TLS configuration. The allowedRoutes field controls which namespaces can attach routes — this is the RBAC model. Using a namespace selector with labels gives platform teams fine-grained control over which application namespaces can use the gateway.

📐 HTTPRoute Resource

apiVersion: gateway.networking.k8s.io/v1
kind: HTTPRoute
metadata:
  name: api-routes
  namespace: verisign-apps
spec:
  parentRefs:
  - name: verisign-gateway
    namespace: gateway-system
    sectionName: https
  hostnames:
  - "api.verisign.internal"
  rules:
  - matches:

        value: /v1/dns
      headers:
      - name: X-API-Version
        value: "v1"
    backendRefs:
    - name: dns-service-v1
      port: 8080
  - matches:
    - path:
        type: PathPrefix
        value: /v2/dns
    backendRefs:
    - name: dns-service-v2
      port: 8080
  - matches:
    - path:
        type: PathPrefix
        value: /health
    backendRefs:
    - name: health-service
      port: 8080
HTTPRoute is where application developers define their routing rules. Notice how matches can include both path and header conditions natively — no annotations needed. The parentRefs field attaches this route to a specific Gateway listener. Each rule can have multiple matches and multiple backends.

⚖️ Traffic Splitting

Canary Deployments with HTTPRoute Weights

One of Gateway API's killer features is native traffic splitting for canary deployments.

⚖️ Canary Deployments with HTTPRoute

apiVersion: gateway.networking.k8s.io/v1
kind: HTTPRoute
metadata:
  name: dns-service-canary
  namespace: verisign-apps
spec:
  parentRefs:
  - name: verisign-gateway
    namespace: gateway-system
  hostnames:
  - "dns.verisign.internal"
  rules:
  - matches:
    - path:
        type: PathPrefix
        value: /
    backendRefs:
    # Stable version gets 90% of traffic
    - name: dns-service-stable
      port: 8080
      weight: 90
    # Canary version gets 10% of traffic
    - name: dns-service-canary
      port: 8080
      weight: 10

Progressive Rollout Strategy

Start at 5% → Monitor metrics → 25% → Verify → 50% → Validate → 100%

Weight-based traffic splitting is built into the HTTPRoute spec — no custom annotations or service meshes needed. This enables canary deployments where you gradually shift traffic from the stable to the canary version. Combined with metrics monitoring, this is a powerful deployment strategy. You simply update the weights to roll forward or back.

📊 Header-Based Canary Routing

Route specific users to the canary using headers:

apiVersion: gateway.networking.k8s.io/v1
kind: HTTPRoute
metadata:
  name: dns-service-header-canary
  namespace: verisign-apps
spec:
  parentRefs:
  - name: verisign-gateway
    namespace: gateway-system
  hostnames:
  - "dns.verisign.internal"
  rules:
  # Internal testers get canary version
  - matches:
    - headers:
      - name: X-Canary
        value: "true"
    backendRefs:
    - name: dns-service-canary
      port: 8080
  # Everyone else gets stable
  - matches:
    - path:
        type: PathPrefix
        value: /
    backendRefs:
    - name: dns-service-stable
      port: 8080
Verisign Use Case: QA team sets X-Canary: true header to test new versions in production before enabling weight-based rollout for all traffic.
Header-based routing is another powerful pattern. Internal teams can test the canary version by adding a specific header, while all other traffic goes to stable. This lets you verify the canary in the real production environment before exposing it to a percentage of all users.

🔚 Egress Controls

Managing Outbound Traffic

Ingress handles traffic coming in. Egress controls manage what your applications can reach on the way out — equally important for security.

🔚 Why Egress Controls Matter

🛡️ Security Concerns

  • Compromised pods phoning home to C2 servers
  • Data exfiltration to unauthorized endpoints

Automated Certificate Lifecycle

From request to renewal -- fully automated

  1. Define: Create a Certificate resource specifying domains and issuer
  2. Request: cert-manager creates a CertificateRequest and initiates the ACME challenge
  3. Validate: CA validates domain ownership (HTTP-01 or DNS-01 challenge)
  4. Issue: CA issues the certificate; cert-manager stores it in a Secret
  5. Consume: Ingress controller or Gateway reads the Secret for TLS termination
  6. Renew: cert-manager renews automatically before renewBefore deadline

Ingress annotation shortcut: Add cert-manager.io/cluster-issuer: letsencrypt-prod to an Ingress, and cert-manager creates the Certificate resource automatically.

The annotation shortcut is the simplest path for teams already using Ingress. cert-manager's ingress-shim watches for Ingress resources with the cert-manager annotation and automatically creates Certificate resources. For Gateway API, cert-manager also supports direct integration. The renewal process is automatic and generates Kubernetes events that can be monitored. Failed renewals should trigger alerts in your monitoring system.

Enterprise Certificate Strategies

Internal Services

  • Private CA (Vault, step-ca)
  • Short-lived certificates (24h)
  • mTLS between services
  • No public trust needed

External Services

  • Public CA (Let's Encrypt, DigiCert)
  • Wildcard or SAN certificates
  • OCSP stapling
  • Certificate transparency logs

Compliance

  • Key material never leaves the cluster
  • Audit trail for all cert operations
  • Rotation policies enforced via CRDs
  • Integration with HSMs for key storage
Enterprise environments typically use a mix of internal and external certificate strategies. Internal services use a private CA for speed and cost -- cert-manager with Vault issuer is a common pattern. External services need publicly trusted certificates. Compliance requirements often dictate how keys are stored and rotated. For Verisign, PKI expertise is a core competency, and cert-manager can integrate with existing PKI infrastructure through custom issuers.
