🔒 Container Security Best Practices

Hardening images from the ground up

This first section covers the foundational techniques for building secure container images. Every production container should follow these principles. We will start with base image selection and work through the full hardening checklist.

📦 Minimal Base Images

Principle: The fewer components in your image, the smaller your attack surface. Every unnecessary binary, library, or shell is a potential vulnerability.

Alpine Linux

  • ~5 MB base image
  • Uses musl libc + BusyBox
  • Includes package manager (apk)
  • Good for most workloads
  • Shell available for debugging

Distroless (Google)

  • No shell, no package manager
  • Contains only app + runtime deps
  • Language-specific variants (Java, Python, Node, Go)
  • Dramatically reduces CVEs

Scratch

  • Completely empty base
  • Zero overhead
  • Ideal for statically compiled binaries (Go, Rust)
  • No debugging tools at all
Alpine is the most popular minimal base image — it has a package manager so you can install what you need. Distroless takes it further by stripping out everything except the language runtime and your app. Scratch is the absolute minimum — an empty filesystem. For Go and Rust apps that compile to static binaries, scratch is ideal. For Java or Python, distroless is the sweet spot.

📊 Base Image Comparison

Base Image Size Shell Package Mgr Typical CVEs Best For
ubuntu:22.04 ~77 MB Yes apt 50-200+ Dev/testing
alpine:3.19 ~5 MB Yes (sh) apk 0-10 General production
distroless ~2-20 MB No No 0-5 Production services
scratch 0 MB No No 0 Static Go/Rust binaries

Smaller images = faster pulls, fewer vulnerabilities, smaller blast radius.

This comparison highlights the trade-offs. Ubuntu is familiar and easy to debug but carries the most CVEs. Alpine strikes a good balance. Distroless and scratch are the most secure but harder to debug — you cannot exec into a scratch container because there is no shell. In production, consider using distroless for runtime and alpine for build stages.

🏗️ Multi-Stage Builds

Multi-stage builds let you use one image for building and a different, minimal image for running — keeping build tools out of the final image.

# Stage 1: Build
FROM golang:1.22-alpine AS builder
WORKDIR /app
COPY go.mod go.sum ./
RUN go mod download
COPY . .
RUN CGO_ENABLED=0 GOOS=linux go build -o /app/server .

# Stage 2: Runtime (minimal)
FROM gcr.io/distroless/static-debian12:nonroot
COPY --from=builder /app/server /server
EXPOSE 8080
ENTRYPOINT ["/server"]

Result: Build tools (Go compiler, git, etc.) stay in the builder stage. The final image contains only the compiled binary.

Multi-stage builds are the single most impactful Dockerfile technique for security. The builder stage can be a full-featured image with compilers, test frameworks, and debugging tools. The final stage only copies the artifact — the compiled binary or bundled assets. This can reduce image size by 90% or more and eliminates entire categories of vulnerabilities from the production image.

👥 Running as Non-Root

Running containers as root is one of the most common security misconfigurations. If an attacker escapes the container, they have root on the host.

In the Dockerfile

RUN addgroup -S appgroup && \
    adduser -S appuser -G appgroup

# Set ownership
COPY --chown=appuser:appgroup . /app

USER appuser

In the Pod Spec

securityContext:
  runAsNonRoot: true
  runAsUser: 1000
  runAsGroup: 1000
  allowPrivilegeEscalation: false

Defence in depth: enforce non-root in both the image AND the Pod spec.

Setting USER in the Dockerfile is the first line of defence, but it can be overridden at deploy time. Adding runAsNonRoot in the securityContext ensures the kubelet rejects any container that tries to run as UID 0. Always use both layers. The allowPrivilegeEscalation flag prevents setuid binaries from gaining elevated privileges.

🛡️ Read-Only Root Filesystem

Why Read-Only?

  • Prevents attackers from writing malware to disk
  • Blocks modification of application binaries
  • Stops reverse shells from persisting
  • Forces explicit declaration of writable paths
securityContext:
  readOnlyRootFilesystem: true
# Mount writable dirs as emptyDir
volumeMounts:
  - name: tmp
    mountPath: /tmp
  - name: cache
    mountPath: /var/cache

No Unnecessary Packages

  • Remove curl, wget, netcat from production images
  • These tools are used by attackers for data exfiltration and reverse shells
  • Use --no-install-recommends with apt
  • Clean up caches: rm -rf /var/cache/apk/*

Tip: If you need debugging tools, use ephemeral debug containers (kubectl debug) instead of baking them into the image.

A read-only root filesystem is an underutilized security control. Most applications only need to write to /tmp or a specific data directory. Mount those as emptyDir volumes and lock down everything else. For packages, the rule is simple: if the application does not need it at runtime, do not include it. Tools like curl and wget are commonly exploited in container breakout scenarios.

🔍 Image Scanning & Vulnerability Management

Scanning images for known CVEs is essential — integrate it into your CI/CD pipeline so vulnerabilities are caught before deployment.

Trivy

  • Open-source (Aqua Security)
  • OS packages + language deps
  • Scans images, filesystems, repos
  • trivy image myapp:v1.0

Grype / Syft

  • Anchore open-source tools
  • SBOM generation (Syft)
  • Vulnerability matching (Grype)
  • Supports multiple formats

Registry Scanning

  • Harbor (built-in Trivy)
  • ECR, ACR, GCR native scanning
  • Scan on push + scheduled rescans
  • Block deployment of critical CVEs
# CI pipeline example: fail build on CRITICAL/HIGH CVEs
trivy image --exit-code 1 --severity CRITICAL,HIGH myapp:${GIT_SHA}
Trivy is the most widely adopted open-source scanner and works well in CI pipelines. The key practice is shifting left — scan early in the pipeline and block builds with critical vulnerabilities. Syft generates SBOMs (Software Bill of Materials) which are increasingly required for compliance. Registry-level scanning provides a second gate before images reach production clusters.

📌 Pinning Versions & Signing Images

Never Use :latest in Production

  • :latest is mutable — it changes with every push
  • Breaks reproducibility and rollbacks
  • Impossible to audit which version is running

Pin by Digest (Strongest)

FROM node:20-alpine@sha256:a1b2c3d4...

Digest-pinning guarantees byte-for-byte reproducibility.

🔐 Image Signing (Cosign)

  • Sign images in CI after build + scan
  • Verify signatures before deployment
  • Part of the Sigstore project
# Sign
cosign sign --key cosign.key myregistry/app:v1.0

# Verify
cosign verify --key cosign.pub myregistry/app:v1.0

Supply chain: Combine scanning + signing + admission control to create a trusted image pipeline.

Tag mutability is a real production risk. If someone pushes a compromised image with the same tag, every new pod pull gets the malicious version. Digest pinning eliminates this entirely. Cosign, from the Sigstore project, lets you cryptographically sign images and verify them at admission time using a webhook or policy engine. This forms the foundation of supply chain security.

📝 Dockerfile Best Practices

Writing production-grade container definitions

Now that we understand the security principles, let us look at how to apply them in practice through well-written Dockerfiles. These patterns apply regardless of the language or framework you use.

🧱 Layer Optimization

Inefficient (Many Layers)

RUN apt-get update
RUN apt-get install -y curl
RUN apt-get install -y git
RUN rm -rf /var/lib/apt/lists/*

Each RUN creates a layer. Files deleted in later layers still exist in earlier ones.

Optimized (Single Layer)

RUN apt-get update && \
    apt-get install -y --no-install-recommends \
      curl git && \
    rm -rf /var/lib/apt/lists/*

Single layer — deleted files are truly gone. Smaller final image.

Layer Caching Strategy

  • Order instructions from least to most frequently changed
  • Copy dependency manifests (package.json, go.mod) before source code
  • This maximizes cache hits and minimizes rebuild time
Docker image layers are additive — each instruction creates a new layer on top of the previous ones. When you delete files in a later layer, the data still exists in the earlier layer, consuming space. Combining commands into a single RUN instruction ensures that temporary files are cleaned up within the same layer. For caching, order matters: put things that change rarely at the top of the Dockerfile.

🚫 .dockerignore & Health Checks

.dockerignore

Prevents sensitive and unnecessary files from entering the build context.

.git
.env
node_modules
*.md
Dockerfile
docker-compose*.yml
.github/
__pycache__/
*.pyc
.DS_Store

HEALTHCHECK Instruction

Docker-native health monitoring (complements Kubernetes probes).

HEALTHCHECK --interval=30s \
  --timeout=3s \
  --start-period=10s \
  --retries=3 \
  CMD wget --no-verbose --tries=1 \
    --spider http://localhost:8080/healthz \
    || exit 1

Note: In Kubernetes, use livenessProbe and readinessProbe instead. HEALTHCHECK is primarily useful for Docker Compose and standalone Docker.

The .dockerignore file is often overlooked but critically important. Without it, secrets like .env files, entire .git directories, and large node_modules folders get sent to the Docker daemon as part of the build context. This wastes time and can leak secrets into the image. For health checks, note the distinction: Docker HEALTHCHECK is useful for non-Kubernetes environments, but in Kubernetes you should use native probes for better integration with the scheduler.

⚡ Proper Signal Handling (PID 1)

The PID 1 problem: In a container, your process runs as PID 1. PID 1 does not receive default signal handlers — if your app does not handle SIGTERM, kubectl stop / rolling updates will hang until the grace period expires, then SIGKILL.

Shell Form (Broken)

# PID 1 = /bin/sh, your app is a child
CMD node server.js

Shell wraps the process — signals go to the shell, not your app.

Exec Form (Correct)

# PID 1 = node, receives signals directly
CMD ["node", "server.js"]

Process runs directly as PID 1 and receives SIGTERM for graceful shutdown.

Alternative: Use tini as an init process: ENTRYPOINT ["tini", "--"] — it properly forwards signals and reaps zombie processes.

This is a subtle but important issue. During rolling updates, Kubernetes sends SIGTERM to the container process. If the process does not handle it, Kubernetes waits for the terminationGracePeriodSeconds (default 30s) and then sends SIGKILL. This causes slow deployments and potential data loss. Always use the exec form for CMD and ENTRYPOINT. For languages that do not handle signals natively (like bash scripts), use tini as a lightweight init system.

🔧 Build Arguments vs Environment Variables

Feature ARG (Build-Time) ENV (Runtime)
Available at Build time only Build time + runtime
Persists in image No (but visible in history) Yes
Override --build-arg KEY=val -e KEY=val / Pod env
Use for Version pins, build config App configuration

Security warning: Never put secrets in ARG or ENV. Both are visible in docker history. Use Kubernetes Secrets or a vault at runtime instead.

# Good: ARG for build-time version pinning
ARG NODE_VERSION=20
FROM node:${NODE_VERSION}-alpine

# Good: ENV for runtime configuration
ENV NODE_ENV=production
ENV PORT=8080
This is a common source of confusion. ARG values are not available at runtime but can still be seen in the image layer history. ENV values persist into the running container. Neither should ever contain secrets — use Kubernetes Secrets mounted as files or environment variables injected by the kubelet, never baked into the image. ARG is useful for parameterizing builds, like selecting a base image version.

📄 Production Dockerfile Example

# Build stage
FROM node:20-alpine@sha256:abc123... AS builder
WORKDIR /app
COPY package.json package-lock.json ./
RUN npm ci --only=production
COPY src/ ./src/

# Runtime stage
FROM gcr.io/distroless/nodejs20-debian12:nonroot
WORKDIR /app
COPY --from=builder --chown=nonroot:nonroot /app /app

EXPOSE 3000
ENV NODE_ENV=production

CMD ["src/server.js"]

Checklist Applied

  • ✅ Digest-pinned base image
  • ✅ Multi-stage build (no build tools in runtime)
  • ✅ Non-root user (distroless nonroot variant)
  • ✅ Only production dependencies copied
  • ✅ Exec form CMD for proper signal handling
This example brings together all the best practices we have discussed. Walk through each line and connect it back to the security principle it enforces. Note the use of npm ci instead of npm install — ci is faster, deterministic, and respects the lock file exactly.

🛡️ Pod Security Admission (PSA)

Cluster-level enforcement of security standards

With secure images built, we now turn to what happens at deployment time. Pod Security Admission is the built-in Kubernetes mechanism for enforcing security standards on pods. It replaced PodSecurityPolicy in Kubernetes 1.25.

🔄 From PodSecurityPolicy to PSA

PodSecurityPolicy (PSP) was removed in Kubernetes 1.25. Pod Security Admission (PSA) is the built-in replacement.

PSP (Deprecated)

  • Complex RBAC binding required
  • Difficult to audit and troubleshoot
  • Mutation + validation combined
  • Ordering issues with multiple PSPs
  • Removed in K8s 1.25

PSA (Current)

  • Built into the API server
  • Simple namespace-level labels
  • Three predefined security profiles
  • Three enforcement modes
  • Stable since K8s 1.25

PSA is simpler by design — trade customization for ease of adoption.

PodSecurityPolicy was powerful but notoriously difficult to configure correctly. The RBAC binding model was confusing, and having multiple PSPs could lead to unpredictable behavior. PSA takes a different approach — it defines three fixed security profiles and applies them at the namespace level using labels. This makes it much easier to understand and roll out, though less flexible. For custom policies beyond what PSA provides, teams use OPA/Gatekeeper or Kyverno.

🔐 PSA Security Profiles

Privileged

  • Unrestricted policy
  • Allows all pod configurations
  • No security restrictions
  • For system-level workloads and infrastructure

Use only for cluster infrastructure (CNI, storage drivers, monitoring agents).

Baseline

  • Prevents known privilege escalations
  • Blocks hostNetwork, hostPID, hostIPC
  • Blocks privileged containers
  • Restricts dangerous capabilities
  • Allows most standard workloads

Sensible minimum for application namespaces.

Restricted

  • Hardened security posture
  • Requires non-root user
  • Drops ALL capabilities
  • Requires read-only root FS
  • Requires seccomp profile

Target for all production application workloads.

Think of these as escalating levels of lockdown. Privileged is a blank check — needed only for system components. Baseline blocks the most dangerous configurations but still allows running as root. Restricted is the gold standard — it requires non-root, read-only root filesystem, dropped capabilities, and a seccomp profile. Your goal should be to run all application workloads under the Restricted profile.

⚙️ PSA Enforcement Modes

Mode Behavior Use Case
enforce Rejects pods that violate the profile Production enforcement
audit Logs violations to the audit log, but allows the pod Discovery and migration
warn Returns a warning to the user, but allows the pod Developer feedback

Recommended Rollout Strategy

  1. Start with warn + audit at the target profile level
  2. Fix violations surfaced by warnings and audit logs
  3. Switch to enforce once workloads comply
The three modes allow a gradual rollout. Start with warn and audit to discover which workloads violate the policy without breaking anything. Developers will see warnings when they kubectl apply, and the audit log captures violations for review. Once all workloads are compliant, switch to enforce. You can combine all three modes at different levels on the same namespace.

🏷️ Namespace Labels for PSA

PSA is configured entirely through namespace labels. No CRDs, no RBAC bindings — just labels.

apiVersion: v1
kind: Namespace
metadata:
  name: production-apps
  labels:
    # Format: pod-security.kubernetes.io/<MODE>: <PROFILE>
    pod-security.kubernetes.io/enforce: restricted
    pod-security.kubernetes.io/enforce-version: v1.30
    pod-security.kubernetes.io/audit: restricted
    pod-security.kubernetes.io/audit-version: v1.30
    pod-security.kubernetes.io/warn: restricted
    pod-security.kubernetes.io/warn-version: v1.30

Version pinning: The -version labels lock the profile to a specific Kubernetes version's definition. Use latest or a specific version like v1.30.

# Apply PSA labels to an existing namespace
kubectl label namespace my-app \
  pod-security.kubernetes.io/enforce=baseline \
  pod-security.kubernetes.io/warn=restricted
This is where PSA shines in simplicity. Compare this to the PSP model which required creating a PSP object, a ClusterRole, and a ClusterRoleBinding. With PSA, you just label the namespace. A common pattern is to enforce baseline and warn on restricted — this blocks dangerous configurations while giving developers feedback about what they need to fix to reach the restricted level.

👨‍💻 PSA Impact on Developer Workflows

Common Failures Under "Restricted"

  • Container runs as root (no runAsNonRoot)
  • Missing seccompProfile in security context
  • Capabilities not dropped (drop: ["ALL"])
  • Writable root filesystem
  • Privilege escalation not disabled

Compliant Pod Template

securityContext:
  runAsNonRoot: true
  runAsUser: 1000
  seccompProfile:
    type: RuntimeDefault
containers:
  - name: app
    securityContext:
      allowPrivilegeEscalation: false
      readOnlyRootFilesystem: true
      capabilities:
        drop: ["ALL"]

Provide developers with compliant Pod templates and Helm chart defaults — do not expect them to memorize every field.

When you enable PSA enforcement, developers will encounter rejection errors. The most common issue is not setting seccompProfile — many teams overlook this. Provide a standard Pod template or Helm chart values file that includes all required fields. Teams that adopt the restricted profile typically create a shared library chart or Kustomize overlay with these defaults baked in.

📚 PSA in Practice: Profile Comparison

Blocked by Baseline

# This pod will be REJECTED under baseline enforcement
spec:
  hostNetwork: true        # Blocked
  containers:
    - name: app
      securityContext:
        privileged: true   # Blocked

Blocked by Restricted (but allowed by Baseline)

# This pod passes baseline but FAILS restricted
spec:
  containers:
    - name: app
      securityContext:
        runAsUser: 0                         # Blocked (must be non-root)
        # Missing seccompProfile             # Blocked (required)
        # Missing capabilities drop: ["ALL"] # Blocked (required)
Walk through these examples to show the concrete difference between profile levels. Under baseline, you can still run as root and skip seccomp profiles. Under restricted, these are all violations. This is why the recommended strategy is to enforce baseline as a floor and work toward restricted. Show how the warn mode surfaces these issues as actionable feedback.

🏛️ Governance & Compliance

Organizational controls for multi-tenant clusters

Beyond PSA, production clusters need additional governance controls. This section covers image policies, resource constraints, and policy engines that enforce organizational standards.

📥 Image Pull Policies

Policy Behavior When to Use
Always Pulls the image on every pod start Mutable tags, registry-scanned images
IfNotPresent Uses cached image if available Immutable tags, digest-pinned images
Never Only uses images already on the node Air-gapped / pre-loaded environments

Default Behavior

  • If tag is :latest → defaults to Always
  • If tag is anything else → defaults to IfNotPresent
  • Best practice: always set imagePullPolicy explicitly
Image pull policy determines whether the kubelet checks the registry for a newer version. Always is safest for mutable tags but adds latency to pod startup. IfNotPresent is better for performance when you use immutable tags or digests. Never is rare — typically used in air-gapped environments where images are pre-loaded onto nodes. Always be explicit about the policy rather than relying on defaults.

🔒 Allowed Registries & Resource Constraints

Restricting Image Sources

  • Allow only trusted, internal registries
  • Block pulls from Docker Hub, public repos
  • Enforce via admission webhooks or policy engines
# OPA/Gatekeeper constraint
apiVersion: constraints.gatekeeper.sh/v1beta1
kind: K8sAllowedRepos
metadata:
  name: allowed-repos
spec:
  match:
    kinds:
      - apiGroups: [""]
        kinds: ["Pod"]
  parameters:
    repos:
      - "registry.internal.company.com/"
      - "gcr.io/my-project/"

ResourceQuotas

apiVersion: v1
kind: ResourceQuota
metadata:
  name: team-quota
spec:
  hard:
    requests.cpu: "20"
    requests.memory: 40Gi
    limits.cpu: "40"
    limits.memory: 80Gi
    pods: "50"

LimitRanges

apiVersion: v1
kind: LimitRange
metadata:
  name: default-limits
spec:
  limits:
    - default:
        cpu: 500m
        memory: 256Mi
      defaultRequest:
        cpu: 100m
        memory: 128Mi
      type: Container
Allowed registries prevent supply chain attacks by ensuring only vetted images can run in the cluster. This is typically enforced via OPA/Gatekeeper or Kyverno rather than PSA, which does not cover image sources. ResourceQuotas cap the total resources a namespace can consume — essential for multi-tenant clusters. LimitRanges set per-container defaults and constraints, ensuring every container has resource requests and limits even if the developer forgets to set them.

📛 OPA / Gatekeeper Concepts

Open Policy Agent (OPA) Gatekeeper extends Kubernetes admission control with custom policies written in Rego. It fills the gaps that PSA does not cover.

What Gatekeeper Adds

  • Custom admission policies (any field, any resource)
  • Allowed registries enforcement
  • Required labels and annotations
  • Naming conventions
  • Ingress hostname uniqueness
  • Audit of existing violations

Architecture

  • ConstraintTemplate: Defines the policy logic (Rego)
  • Constraint: Applies the template with parameters
  • Runs as a ValidatingAdmissionWebhook
  • Audit controller checks existing resources

Alternative: Kyverno — Kubernetes-native policies in YAML (no Rego).

OPA Gatekeeper is the most widely deployed policy engine for Kubernetes. It uses ConstraintTemplates to define reusable policy logic in the Rego language, and Constraints to apply those templates with specific parameters. Common policies include: requiring resource limits, enforcing image registries, mandating labels, and restricting host paths. Kyverno is a popular alternative that defines policies in YAML rather than Rego, which has a lower learning curve. Both integrate as admission webhooks.

🗺️ Security & Governance Layers

Defence in depth: Each layer catches what the previous layer missed.

Layer Control What It Catches
Build Secure Dockerfile, image scanning Vulnerable packages, bloated images
Registry Signing, scanning, allowed repos Tampered or untrusted images
Admission PSA, OPA/Gatekeeper, Kyverno Misconfigured pods, policy violations
Runtime seccomp, AppArmor, read-only FS Container breakout, malicious behavior
Namespace ResourceQuotas, LimitRanges, NetworkPolicies Resource abuse, lateral movement
This table summarizes the full security stack. No single layer is sufficient on its own. Emphasize that container security is a pipeline concern that spans from Dockerfile authoring through CI scanning, registry policies, admission control, and runtime enforcement. Each layer should be independently effective so that a failure in one does not compromise the cluster.

Key Takeaways

  • Minimal base images (distroless, Alpine, scratch) dramatically reduce your vulnerability surface — prefer them over full OS images
  • Multi-stage builds keep build tools out of production images, reducing size by 90% or more
  • Non-root + read-only root filesystem should be enforced in both the Dockerfile and the Pod securityContext
  • Pod Security Admission replaces PSP with namespace-level labels — aim for the Restricted profile in production

Recap these six points. The most impactful quick wins are switching to minimal base images, adding multi-stage builds, and enabling PSA with at least baseline enforcement. The broader governance controls (OPA, resource quotas) are typically owned by platform teams but every developer should understand their impact.
