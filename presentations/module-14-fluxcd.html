🏗️ FluxCD Architecture

A Toolkit Approach to GitOps

Flux v2 was redesigned as a set of composable controllers rather than a monolithic operator. Each controller owns a specific domain, and they communicate through Kubernetes CRDs and status conditions.

⚙️ Flux Controllers Overview

Source

Fetches artifacts from Git, Helm, OCI, and S3-compatible sources

Kustomize

Reconciles Kustomization manifests and plain YAML to the cluster

Helm

Manages HelmRelease lifecycle — install, upgrade, test, rollback

Notification

Dispatches alerts to external systems and receives webhooks

Image Automation

Scans registries, applies policies, and commits image updates to Git

Each controller is independently deployable and can be installed separately via the Flux CLI or Terraform. The source controller is foundational — every other controller depends on it to fetch artifacts. The notification controller is optional but critical for production observability.

🔄 Reconciliation Loop

The Core GitOps Principle

Flux continuously compares desired state (in Git) with actual state (in cluster) and converges any drift.

  • Fetch — source controller pulls latest artifact revision
  • Detect — compute diff between desired and actual state
  • Apply — server-side apply changed resources
  • Validate — run health checks against applied resources
  • Report — update status conditions and emit events
Key: Reconciliation is idempotent and runs on a configurable interval. No manual kubectl apply needed.
The reconciliation loop is the heartbeat of Flux. Each controller runs its own loop at a configurable interval. The source controller fetches and caches artifacts, then downstream controllers react to new revisions. Server-side apply ensures proper field ownership and conflict detection.

🔎 GitOps Principles Recap

The operational model that underpins FluxCD

Before diving into FluxCD specifics, let us ground ourselves in the GitOps principles that Flux implements. This is not just theory; every design decision in Flux traces back to these principles.

📚 GitOps Core Principles

📌 Git as Single Source of Truth

  • All desired state is declared in Git
  • Git history = deployment history
  • Rollback = git revert

🔄 Pull-Based Reconciliation

  • Agents inside the cluster pull desired state
  • No external CI system needs cluster credentials
  • Continuous reconciliation, not one-shot apply

🔍 Drift Detection & Correction

  • Cluster state is continuously compared to Git
  • Manual changes are automatically reverted
  • Alerts fire when drift is detected

📝 Declarative Desired State

  • You describe what, not how
  • Controllers converge actual state to desired
  • Idempotent by design
These four principles are the foundation of GitOps. Pull-based reconciliation is the key differentiator from traditional CI/CD push models. In a push model, Jenkins or GitHub Actions applies changes to the cluster. In pull-based GitOps, the agent inside the cluster watches Git and pulls changes. This means your CI pipeline never needs kubectl access to the cluster, which is a massive security improvement.

🔨 Push vs Pull Delivery Models

Key Distinction: In push-based CI/CD, the pipeline needs cluster credentials and pushes changes outward. In pull-based GitOps, the cluster agent watches Git and pulls changes inward — no external access required.

Push Model (Traditional CI/CD)

CI Pipeline → kubectl apply → Cluster

  • Pipeline needs cluster credentials
  • One-shot apply, no drift correction
  • Cluster state can diverge from Git

Pull Model (GitOps with Flux)

Git Commit → Flux Agent (in-cluster) → Reconcile

  • Agent runs inside the cluster
  • Continuous reconciliation loop
  • Cluster always matches Git
This is a critical distinction for security. With push-based deployment, every CI runner that deploys to your cluster needs kubeconfig credentials. That is a broad attack surface. With pull-based GitOps, only the in-cluster agent needs access to Git, and it already has the Kubernetes API available locally. The Verisign platform uses this pull model exclusively through FluxCD.

⚙️ FluxCD Architecture Deep Dive

Four controllers, one reconciliation engine

Now let us look at how FluxCD actually implements these GitOps principles. Flux is not a single binary; it is a set of specialized controllers, each managing a specific concern. This separation of concerns makes Flux both flexible and composable.

🏗️ FluxCD Controller Architecture

Source Controller → Kustomize Controller / Helm Controller → Notification Controller

📦 Source Controller

Fetches artifacts from Git repos, Helm repos, OCI registries, and S3-compatible buckets. Produces versioned artifacts for other controllers to consume.

📑 Kustomize Controller

Reconciles Kubernetes manifests from sources. Runs kustomize build, applies results, prunes deleted resources, and performs health checks.

⛐️ Helm Controller

Manages Helm chart releases declaratively. Handles install, upgrade, test, rollback, and uninstall lifecycle operations via HelmRelease CRDs.

🔔 Notification Controller

Sends alerts to external systems (Slack, WebEx, GitHub). Receives webhooks from Git providers to trigger immediate reconciliation.

All four controllers run as Deployments in the flux-system namespace. They communicate through Kubernetes CRDs and the artifact storage managed by source-controller. The source controller is the entry point: it fetches artifacts, and the other controllers consume those artifacts. The notification controller is the exit point: it reports the results back to external systems.

📊 FluxCD Custom Resource Definitions

Controller CRDs Purpose

Image Automation ImageRepository, ImagePolicy, ImageUpdateAutomation Automated image tag updates in Git
This table gives a complete map of the FluxCD CRD landscape. The core four controllers cover the vast majority of use cases. Image automation is an optional add-on that watches container registries for new tags and automatically commits updated image references back to Git. We will focus primarily on Source, Kustomize, and Helm controllers since those are what you interact with most on the Verisign platform.

🛠️ Kustomize Controller

Reconciling Manifests to the Cluster

The kustomize controller is the workhorse of Flux. Despite the name, it handles both Kustomize overlays and plain YAML directories. It is responsible for applying manifests, pruning removed resources, and running health checks.

📑 Kustomization CRD

apiVersion: kustomize.toolkit.fluxcd.io/v1
kind: Kustomization
metadata:
  name: app-stack
  namespace: flux-system
spec:
  interval: 10m
  retryInterval: 2m
  timeout: 5m
  sourceRef:
    kind: GitRepository
    name: app-manifests
  path: ./clusters/production
  prune: true
  force: false
  healthChecks:
    - apiVersion: apps/v1
      kind: Deployment
      name: api-server
      namespace: production
  • prune: true — garbage-collect resources removed from Git
  • healthChecks — wait for rollout before marking as Ready
  • retryInterval — back-off interval on reconciliation failure
The Kustomization CRD is not the same as kustomize.config.k8s.io — it is a Flux-specific resource. The prune flag is critical for production: when you delete a manifest from Git, Flux will delete the corresponding cluster resource. The healthChecks field blocks the Ready condition until all specified resources are healthy, which is essential for dependency chains.

🔠 Variable Substitution (postBuild)

apiVersion: kustomize.toolkit.fluxcd.io/v1
kind: Kustomization
metadata:
  name: app-staging
  namespace: flux-system
spec:
  sourceRef:
    kind: GitRepository
    name: app-manifests
  path: ./apps/base
  postBuild:
    substitute:
      CLUSTER_ENV: staging
      REPLICAS: "3"
      DOMAIN: staging.verisign.com
    substituteFrom:
      - kind: ConfigMap
        name: cluster-vars
      - kind: Secret
        name: cluster-secrets
Pattern: Use ${VARIABLE} placeholders in your manifests. Flux replaces them at apply time — no templating engine required.
PostBuild substitution is one of the most powerful Flux features for multi-environment management. You define a single base manifest with variable placeholders, then each Kustomization provides environment-specific values. substituteFrom pulls values from existing ConfigMaps and Secrets, keeping sensitive data out of Git. This is simpler than Helm templating for many use cases.

🔗 Dependency Ordering

apiVersion: kustomize.toolkit.fluxcd.io/v1
kind: Kustomization
metadata:
  name: infrastructure
  namespace: flux-system
spec:
  sourceRef:
    kind: GitRepository
    name: app-manifests
  path: ./infrastructure
  prune: true
---
apiVersion: kustomize.toolkit.fluxcd.io/v1
kind: Kustomization
metadata:
  name: applications
  namespace: flux-system
spec:
  dependsOn:
    - name: infrastructure
  sourceRef:
    kind: GitRepository
    name: app-manifests
  path: ./applications
  prune: true
Ordering chain: CRDs → Infrastructure (cert-manager, ingress) → Platform (monitoring, logging) → Applications
DependsOn creates an explicit ordering between Kustomizations. The dependent Kustomization will not reconcile until all its dependencies report a Ready status. This is essential for bootstrapping — you need CRDs installed before controllers, and controllers running before application resources that use those CRDs. A typical production setup has 3-4 layers in the dependency chain.

⚖️ Helm Controller

GitOps-Driven Helm Release Management

The Helm controller bridges Flux's GitOps model with the Helm ecosystem. It manages HelmRelease resources declaratively, handling the full lifecycle of install, upgrade, test, and rollback without direct Helm CLI usage.

📜 HelmRelease CRD

apiVersion: helm.toolkit.fluxcd.io/v2
kind: HelmRelease
metadata:
  name: nginx-ingress
  namespace: infrastructure
spec:
  interval: 15m
  chart:
    spec:
      chart: ingress-nginx
      version: "4.x"     # Semver range
      sourceRef:
        kind: HelmRepository
        name: ingress-nginx
  values:
    controller:
      replicaCount: 3
      service:
        type: LoadBalancer
  upgrade:
    remediation:
      retries: 3
  rollback:
    cleanupOnFail: true
  • version — supports semver ranges for automatic minor/patch upgrades
  • upgrade.remediation — auto-retry or rollback on failure
  • rollback.cleanupOnFail — delete new resources on failed rollback
The HelmRelease CRD replaces imperative helm install/upgrade commands with a declarative spec. Semver ranges let you automatically pick up patch releases while staying within a major version. The remediation settings are critical for production — they define what happens when an upgrade fails, including automatic rollback after a configurable number of retries.

📑 Kustomize Controller

Reconciling manifests from source to cluster

The Kustomize Controller is the workhorse of Flux. It takes artifacts produced by the source controller and applies them to the cluster. Despite the name, it handles both plain YAML manifests and Kustomize overlays.

📑 The Kustomization CRD

Important: The Flux Kustomization CRD (API group kustomize.toolkit.fluxcd.io) is not the same as a kustomization.yaml file used by kubectl kustomize. They share the name but serve different purposes.


apiVersion: kustomize.toolkit.fluxcd.io/v1
kind: Kustomization
metadata:
  name: app-deployment
  namespace: flux-system
spec:
  interval: 10m             # Reconciliation interval
  sourceRef:
    kind: GitRepository
    name: app-manifests      # Reference to a source
  path: ./clusters/prod/app  # Path within the source
  prune: true                # Delete resources removed from Git
  timeout: 3m               # Max time for apply operation
  healthChecks:
    - apiVersion: apps/v1
      kind: Deployment
      name: my-app
      namespace: default
  dependsOn:
    - name: core-components  # Wait for this Kustomization first
This is the most important CRD to understand. The sourceRef ties this Kustomization to a source artifact. The path field tells Flux where within the repository to find the manifests. Prune is critical: when set to true, Flux will garbage-collect resources that were removed from Git. Without prune, deleting a manifest from Git would leave the resource running in the cluster. Health checks let Flux verify that applied resources are actually healthy before reporting success. The dependsOn field creates ordering between Kustomizations, which we will explore in the dependency management section.

🔧 Kustomization Key Configuration

Field Purpose Common Values
spec.interval How often to reconcile 5m, 10m, 30m
spec.sourceRef Which source to read from GitRepository, OCIRepository
spec.path Directory path within the source ./clusters/prod
spec.prune Delete resources removed from Git true (recommended)
spec.dependsOn Ordering dependencies List of Kustomization names
spec.healthChecks Resources to verify after apply Deployments, StatefulSets
spec.postBuild.substituteFrom Variable substitution ConfigMaps, Secrets
Walk through each field. The postBuild substituteFrom is particularly useful for injecting environment-specific values into manifests without maintaining separate files per environment. For example, you can substitute ${CLUSTER_NAME} or ${ENVIRONMENT} at apply time from a ConfigMap. This reduces duplication across environments significantly.

⛐️ Helm Controller

Declarative Helm release management

The Helm Controller brings the power of Helm charts into the GitOps model. Instead of running helm install or helm upgrade from a CI pipeline, you declare HelmRelease resources and Flux manages the entire lifecycle.

⛐️ HelmRelease CRD


apiVersion: helm.toolkit.fluxcd.io/v2
kind: HelmRelease
metadata:
  name: nginx-ingress
  namespace: networking
spec:
  interval: 30m
  chart:
    spec:
      chart: ingress-nginx
      version: "4.8.x"      # Semver range
      sourceRef:
        kind: HelmRepository
        name: ingress-nginx
        namespace: flux-system
  values:
    controller:
      replicaCount: 3
      resources:
        requests:
          cpu: 100m
          memory: 128Mi
  install:
    remediation:
      retries: 3
  upgrade:
    remediation:
      retries: 3
      remediationStrategy: rollback
The HelmRelease CRD wraps the entire Helm release lifecycle. The chart spec references a HelmRepository source and a chart name. The version field supports semver ranges, so Flux can automatically pick up patch releases. The values field is equivalent to a values.yaml override. The remediation section is critical for production: it tells Flux how many times to retry a failed install or upgrade, and whether to rollback on failure. The remediationStrategy of rollback means Flux will automatically revert to the previous working release if an upgrade fails.

🔄 HelmRelease Lifecycle

State Machine: Install → Upgrade → Test → Rollback

Phase Trigger Configuration
Install First reconciliation install.remediation.retries
Upgrade Values or chart version change upgrade.remediation.retries
Test After install/upgrade test.enable: true
Rollback Failed upgrade or test rollback.timeout, cleanupOnFail
Uninstall HelmRelease deleted Automatic via garbage collection
Tip: Set upgrade.remediation.remediateLastFailure: true to automatically retry the last failed release on the next reconciliation.
Understanding the lifecycle is critical for production operations. When Flux detects a change (new chart version, updated values, or interval expiry), it runs through the appropriate phase. If an upgrade fails, Flux can automatically rollback to the last successful release. Helm tests run after install or upgrade and can trigger rollback on failure. Each phase has its own timeout and remediation settings.

🚀 Advanced Flux

Image Automation, Notifications & Multi-Tenancy

These advanced features transform Flux from a deployment tool into a full platform for continuous delivery. Image automation closes the loop from container build to deployment, notifications provide observability, and multi-tenancy enables platform teams to safely share clusters.

📷 Image Automation

apiVersion: image.toolkit.fluxcd.io/v1beta2

  name: app-image

  image: ghcr.io/verisign/my-app

kind: ImagePolicy

  name: app-policy
spec:
  imageRepositoryRef:
    name: app-image
  policy:
    semver:
      range: ">=1.0.0 <2.0.0"
---
apiVersion: image.toolkit.fluxcd.io/v1beta2
kind: ImageUpdateAutomation
metadata:
  name: app-update
spec:
  sourceRef:
    kind: GitRepository
    name: app-manifests
  interval: 10m
  update:
    path: ./clusters/production
    strategy: Setters
Image automation is a three-step pipeline. ImageRepository scans a container registry for tags. ImagePolicy selects the latest tag matching a semver range, alphabetical, or numerical filter. ImageUpdateAutomation writes the selected tag back to Git using setter markers in your YAML. This closes the CI/CD loop — when your CI builds a new image, Flux detects it and updates the deployment manifest automatically.

    cleanupOnFail: true

    enable: true
    ignoreFailures: false

Remediation is what makes Flux-managed Helm releases production-safe. Without remediation, a bad chart version would leave the release in a failed state requiring manual intervention. If a test fails, the release is marked as failed and remediation kicks in. This gives you automated canary-style validation of every Helm upgrade.

Day-Two Management with Flux

⏸️ Suspend & Resume Reconciliation

# Suspend a specific Kustomization

# Suspend all HelmReleases in a namespace

# Resume reconciliation
flux resume kustomization app-stack

# Force an immediate reconciliation

Maintenance: Suspend before manual cluster changes to prevent Flux from reverting them
Emergency: Suspend, apply hotfix manually, then update Git and resume
  • Suspension persists across controller restarts (stored in CRD spec)
  • --with-source forces the source controller to re-fetch immediately
Suspend and resume are essential operational tools. During maintenance windows or incident response, you may need to stop Flux from reconciling while you make manual changes. Always remember to update Git to match your manual changes before resuming, otherwise Flux will revert them. The reconcile command with --with-source is useful when you've pushed a fix to Git and don't want to wait for the next interval.

🔍 Drift Detection & Correction

How Flux Detects Drift

Flux uses server-side apply with field ownership tracking. Any field managed by Flux that is modified externally triggers a drift correction on the next reconciliation.

# Check for drift manually
flux diff kustomization app-stack

# View which fields Flux manages
kubectl get deployment api-server -o yaml | grep "manager: flux"

# Force drift correction now
flux reconcile kustomization app-stack
Warning: With force: true in the Kustomization spec, Flux will replace resources instead of patching — use with caution as it recreates objects.
Drift detection is automatic and continuous. Flux relies on server-side apply's field management to know which fields it owns. If someone runs kubectl edit and changes a Flux-managed field, Flux will revert it on the next reconciliation. Fields not managed by Flux (like manually added annotations) are left untouched. The flux diff command is useful for previewing what changes Flux would make without actually applying them.

🔗 Dependency Management

Ordering reconciliation with dependsOn chains

In any real platform, resources have dependencies. CRDs must be installed before custom resources. Namespaces must exist before Deployments can be created in them. Flux handles this through dependency chains.

🔗 spec.dependsOn Chains

How it works: A Kustomization with dependsOn will not begin reconciliation until all dependencies report a Ready status. If a dependency fails, dependents are blocked until it recovers. 

# CRDs must be installed first
apiVersion: kustomize.toolkit.fluxcd.io/v1
kind: Kustomization
metadata:
  name: cert-manager-crds
spec:
  sourceRef: { kind: GitRepository, name: platform }
  path: ./core/cert-manager/crds
---
# Then install cert-manager itself
apiVersion: kustomize.toolkit.fluxcd.io/v1
kind: Kustomization
metadata:
  name: cert-manager
spec:
  dependsOn:
    - name: cert-manager-crds
  sourceRef: { kind: GitRepository, name: platform }
  path: ./core/cert-manager
---
# Then resources that need cert-manager
apiVersion: kustomize.toolkit.fluxcd.io/v1
kind: Kustomization
metadata:
  name: cluster-issuers
spec:
  dependsOn:
    - name: cert-manager
  sourceRef: { kind: GitRepository, name: platform }
  path: ./core/cert-manager/issuers
This three-stage dependency chain is a common pattern. CRDs are installed first because they define the API types. Then the controller that manages those CRDs is installed. Finally, the custom resources that use those CRDs are created. Without this ordering, Kubernetes would reject the custom resources because the CRD definitions would not exist yet. Flux evaluates the dependency graph at each reconciliation cycle and processes Kustomizations in the correct order.

🔍 Drift Detection & Reconciliation

How Flux keeps the cluster in sync with Git

This is one of the most important operational aspects of GitOps. Understanding how Flux detects and corrects drift is essential for working effectively with the platform.

🔍 What Happens When Someone Makes Manual Changes?

Scenario: A developer runs kubectl edit deployment my-app

  1. The deployment is modified in the cluster (e.g., replicas changed from 3 to 5)
  2. On the next reconciliation interval, Flux compares cluster state to Git
  3. Flux detects the drift and re-applies the desired state from Git
  4. Replicas revert from 5 back to 3 (the value in Git)
  5. If notifications are configured, an alert fires indicating drift was corrected

Key Takeaway: In a GitOps workflow, kubectl edit and kubectl apply are temporary. Git is the only source of truth. Always commit changes to Git rather than modifying the cluster directly.

This is the most common surprise for developers new to GitOps. Any manual change to a Flux-managed resource will be reverted on the next reconciliation cycle. This is by design. The maximum time a manual change can persist is the reconciliation interval. For a 5-minute interval, manual changes last at most 5 minutes. If you need to make an emergency change, you should either commit directly to the Git repo or use flux suspend to temporarily pause reconciliation while you work.

🗑️ Prune Behavior: Garbage Collection

spec.prune: true

  • Flux tracks all resources it creates via labels
  • When a manifest is removed from Git, Flux deletes the corresponding resource from the cluster
  • This is how git rm deployment.yaml translates to kubectl delete deployment

spec.prune: false

  • Flux only applies and updates resources
  • Removing a manifest from Git does NOT delete the resource
  • Resources become "orphaned" — still running but unmanaged
Recommendation: Always set spec.prune: true for application workloads. This ensures your cluster does not accumulate orphaned resources over time. Use prune: false only for shared infrastructure where accidental deletion would be catastrophic.
Prune behavior is often misunderstood. With prune enabled, Flux uses labels to track which resources belong to which Kustomization. When a resource disappears from Git, Flux removes the label-matched resource from the cluster. This is extremely powerful for keeping the cluster clean, but it also means you need to be careful when restructuring your Git repository. Moving manifests between directories without updating the Kustomization path could cause Flux to delete and recreate resources.

⚡ Force Reconciliation


# Force source to re-fetch from Git immediately
$ flux reconcile source git platform-config
✔ source reconciliation completed
✔ fetched revision main@sha1:d4e5f6

# Force Kustomization to re-apply
$ flux reconcile kustomization app-prod --with-source
✔ source reconciliation completed
✔ kustomization reconciliation completed

# Force HelmRelease to re-install
$ flux reconcile helmrelease nginx-ingress --with-source
✔ source reconciliation completed
✔ helm release reconciliation completed

Pro tip: The --with-source flag triggers a source fetch before reconciling the workload. This ensures you are applying the absolute latest commit, not a cached artifact.

Force reconciliation is the most common Flux CLI operation in daily use. After pushing a commit, developers want to see changes immediately rather than waiting for the interval. The with-source flag is important because without it, Flux reconciles against whatever artifact it already has cached. With the flag, it first pulls the latest from Git, then reconciles. This is equivalent to pressing the deploy button in a traditional CI/CD system.

🏆 Best Practices

Patterns for production-grade GitOps with FluxCD

Let us consolidate what we have learned into actionable best practices. These are patterns that have proven effective in production environments like the Verisign platform.

🏆 FluxCD Best Practices

Repository Organization

  • Use directory-based overlays for multi-env
  • Keep platform config separate from app config
  • Store Flux Kustomization CRDs in the same repo they manage
  • Use consistent naming conventions

Secret Management

  • Never store plain secrets in Git
  • Use SOPS for encrypted secrets in Git
  • Use External Secrets Operator with AWS Secrets Manager
  • Use Sealed Secrets for simpler setups

Health Checks & Reliability

  • Always define healthChecks in Kustomizations
  • Set timeout for apply operations
  • Use remediation in HelmReleases
  • Enable prune: true for clean garbage collection

Image Automation

  • ImageRepository watches a container registry
  • ImagePolicy defines which tags to track (semver, regex)
  • ImageUpdateAutomation commits tag updates back to Git
  • Closes the loop: push image → auto-deploy
These best practices reflect lessons learned from operating FluxCD at scale. The secret management question comes up frequently. On the Verisign platform, we use External Secrets Operator to pull secrets from AWS Secrets Manager, which means secrets never pass through Git at all. SOPS is the Flux-native alternative where secrets are encrypted in Git and decrypted at apply time. For health checks, always include the primary Deployment or StatefulSet so Flux knows whether the reconciliation actually succeeded, not just whether the YAML was valid. Image automation is an advanced topic that we will not cover in the lab, but it is worth knowing about as it enables fully automated deployments from container image push to cluster deployment.

🔧 Troubleshooting FluxCD

Diagnosing and resolving common issues

Troubleshooting is where theory meets practice. Let us look at the most common issues you will encounter and the commands to diagnose each one.

🔧 Common FluxCD Issues

Symptom Likely Cause Diagnostic Command
Source not Ready Auth failure, bad URL, network issue flux get sources git
flux events --for GitRepository/name
Dependency failed Upstream Kustomization not Ready flux get kustomizations
flux logs --kind=Kustomization
Helm remediation exhausted Chart values error, resource conflict flux get helmreleases
flux events --for HelmRelease/name
Kustomize build error Invalid YAML, missing base, bad patch flux logs --kind=Kustomization --name=X
Resources not pruned spec.prune: false or label mismatch kubectl get all -l kustomize.toolkit.fluxcd.io/name=X
This table covers the five most common issues. The troubleshooting workflow is always the same: first check the status with flux get, then look at events with flux events, and finally check logs with flux logs. For Helm issues specifically, you may also need to check helm history to see the release state. For kustomize build errors, you can reproduce them locally by running kustomize build on the path to verify the manifests are valid before pushing to Git. Always start with the source: if the source is not Ready, nothing downstream will work.

🔎 Troubleshooting Workflow


# Step 1: Get the big picture
$ flux get all

# Step 2: Check the specific resource
$ flux get kustomization app-prod
NAME       REVISION  SUSPENDED  READY  MESSAGE
app-prod             False      False  kustomize build failed: ...

# Step 3: Get detailed events
$ flux events --for Kustomization/app-prod

# Step 4: Check controller logs
$ flux logs --kind=Kustomization --name=app-prod --level=error

# Step 5: If source related, check source status
$ flux get sources git platform-config
$ kubectl describe gitrepository platform-config -n flux-system

# Step 6: After fixing, force reconciliation
$ flux reconcile kustomization app-prod --with-source

Rule of thumb: Always troubleshoot from the source outward. If the source is healthy, check the Kustomization or HelmRelease. If those are healthy, check the actual Kubernetes resources they manage.

This six-step workflow will resolve the vast majority of Flux issues. The key insight is to work from the source outward. A failed source blocks everything downstream. Once the source is confirmed healthy, move to the Kustomization or HelmRelease level. And if those report success but the application is not working, the issue is in the manifests themselves, not in Flux. Always force reconcile with the with-source flag after making a fix to verify the fix works immediately.
