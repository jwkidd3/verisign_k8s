⚠️ The Pod Identity Problem

Why node-level IAM is not enough

Before IRSA, every pod on an EKS node shared the same IAM role. Let's explore why that's a serious security concern and how the ecosystem evolved to solve it.

🚨 Why Node-Level IAM Is Dangerous

The Problem: Without IRSA, every pod on a node inherits the EC2 instance profile — the same IAM role for all workloads.
  • Over-privileged pods — a logging pod gets the same S3 access as your data pipeline
  • Blast radius — one compromised pod can access all AWS resources the node role permits
  • No audit granularity — CloudTrail shows the node role, not which pod made the call
  • Violates least privilege — impossible to scope permissions per workload
  • Instance Metadata Service (IMDS) — any pod can query 169.254.169.254 for node credentials
The node IAM role is typically quite broad because it needs to support all workloads. A compromised container can hit the IMDS endpoint and retrieve temporary credentials for the entire node role. This is one of the top EKS security risks.

🔄 Evolution of Pod Identity Solutions

Solution Approach Limitations
kube2iam iptables + IMDS proxy Race conditions, single point of failure
kiam Agent/server architecture Complex setup, TLS management overhead
IRSA OIDC + projected tokens Trust policy management at scale
EKS Pod Identity Native EKS agent EKS-only, newer (GA 2023)
Key Takeaway: IRSA is the current standard for EKS workload identity. EKS Pod Identity is the emerging successor with a simpler operational model.
kube2iam and kiam were community solutions that intercepted IMDS traffic. They worked but had operational challenges. IRSA was AWS's first-party answer, built on open standards (OIDC). EKS Pod Identity simplifies things further but IRSA remains widely deployed.

🏗️ IRSA Architecture

OIDC, projected tokens, and STS

Now let's dissect how IRSA actually works under the hood. Understanding the architecture is essential for troubleshooting and for designing secure trust policies.

🔑 OIDC Provider — The Foundation

What is the EKS OIDC Endpoint?

Every EKS cluster exposes an OpenID Connect (OIDC) discovery endpoint that AWS IAM can use to verify Kubernetes-issued tokens.

  • URL format: https://oidc.eks.{region}.amazonaws.com/id/{CLUSTER_ID}
  • Publishes a JWKS (JSON Web Key Set) for token signature verification
  • Must be registered as an IAM OIDC Identity Provider in your AWS account
  • Acts as the bridge between Kubernetes identity and AWS IAM
# Get your cluster's OIDC issuer URL
aws eks describe-cluster --name my-cluster \
  --query "cluster.identity.oidc.issuer" --output text

# Associate OIDC provider with IAM
eksctl utils associate-iam-oidc-provider \
  --cluster my-cluster --approve
The OIDC provider is what allows AWS to trust tokens issued by your Kubernetes API server. Without this registration, STS has no way to validate the JWT tokens that pods present. This is a one-time cluster setup step.

📄 JWT Token Projection into Pods

IRSA uses Kubernetes projected service account tokens — not the legacy static tokens.

Legacy Tokens

  • Never expire
  • Bound to secret objects
  • Same token for all audiences
  • No identity claims for AWS

Projected Tokens (IRSA)

  • Time-bound (default 24h)
  • Audience-scoped (sts.amazonaws.com)
  • Auto-rotated by kubelet
  • Contains SA name & namespace
Token path: /var/run/secrets/eks.amazonaws.com/serviceaccount/token
The mutating webhook in EKS automatically injects the projected token volume and AWS-related environment variables into pods that use an IRSA-annotated service account. The token is short-lived and continuously rotated by the kubelet.

🔀 STS AssumeRoleWithWebIdentity Flow

End-to-End IRSA Flow

  1. Pod starts — EKS webhook injects projected token + env vars
  2. AWS SDK detects AWS_ROLE_ARN and AWS_WEB_IDENTITY_TOKEN_FILE
  3. SDK calls STS AssumeRoleWithWebIdentity with the JWT
  4. STS validates the JWT signature via the OIDC provider's JWKS
  5. STS checks trust policy — issuer, audience, subject match?
  6. STS returns temporary AWS credentials to the SDK
Key Insight: The application code never changes. The AWS SDK credential chain handles IRSA transparently.
This is the critical flow to understand. The AWS SDK has a credential provider that checks for these environment variables. If present, it automatically uses the web identity token to assume the specified IAM role. No code changes needed in your application.

🔒 The Problem: Pods and AWS Access

How do containers running in Kubernetes access AWS services?

Before explaining IRSA, we need to understand the problem space. Pods frequently need to talk to AWS services — S3, DynamoDB, SQS, Route53, etc. The question is: how do they authenticate?

⚠️ Pre-IRSA Approaches

Node-Level IAM Roles

  • Every pod on the node gets the same permissions
  • Overly broad access
  • Violates least-privilege
  • One compromised pod = full node permissions

Hardcoded Credentials

  • AWS access keys stored in Secrets
  • Long-lived credentials
  • Rotation nightmare
  • Risk of leaking into logs or repos

kube2iam / kiam

  • Third-party, community tools
  • Added network latency (proxy-based)
  • Complex setup and failure modes
  • Race conditions on pod startup
Walk through each approach and its weaknesses. Node-level roles are the most common "easy" path and the most dangerous — a logging sidecar has the same AWS access as the main application. Hardcoded credentials are an anti-pattern in any cloud-native environment. kube2iam/kiam were clever community solutions but introduced their own operational burden.

🚫 Why These Approaches Fail

Node-Level Roles: If a pod running a web frontend is compromised, the attacker inherits permissions meant for backend services — including access to production databases and secrets.

Static Credentials: AWS access keys in Kubernetes Secrets are only base64-encoded (not encrypted by default). They never expire unless manually rotated. One leaked key = persistent unauthorized access.

kube2iam/kiam: These tools intercept the EC2 metadata endpoint. Under load, credential requests can time out or return stale tokens. AWS does not officially support these tools.

What we need: Per-pod, fine-grained IAM permissions with short-lived credentials and no stored secrets.
Reinforce the security implications. The node-level role scenario is especially important — in a multi-tenant cluster, this is a critical vulnerability. The key insight is that we need identity at the pod level, not the node level.

✅ IRSA: The Solution

Fine-grained, per-pod AWS IAM permissions

Now we introduce IRSA as the AWS-native, officially supported solution to this problem. It was introduced at re:Invent 2019 and has become the standard for EKS workloads.

🔑 What is IRSA?

IAM Roles for Service Accounts

IRSA associates an AWS IAM Role with a Kubernetes ServiceAccount, enabling pods to assume that role and receive temporary, scoped AWS credentials.

Key Properties

  • Per-pod granularity — each SA gets its own role
  • No stored credentials — tokens are projected and short-lived
  • AWS-native — uses STS AssumeRoleWithWebIdentity
  • Automatic — AWS SDKs handle token exchange transparently

Benefits

  • Least-privilege per workload
  • No credential rotation needed
  • Auditable via CloudTrail
  • Works with existing IAM policies
Emphasize that IRSA is built into EKS — no third-party components. The AWS SDK credential chain automatically picks up the projected token, so application code typically does not need any changes. This is a huge advantage over kube2iam where the application had to be aware of the metadata proxy.

🔗 How IRSA Works

The OIDC Trust Chain

This is the most important conceptual section. Understanding the trust chain is essential for debugging IRSA issues. Walk through it step by step.

🔄 The IRSA Trust Chain

EKS OIDC Provider → IAM Trust Policy → ServiceAccount Annotation → Pod Token Projection → STS Token Exchange

  1. EKS cluster has an OIDC provider — acts as an identity issuer for ServiceAccount tokens
  2. IAM Role has a trust policy — trusts tokens from this specific OIDC provider for a specific ServiceAccount
  3. ServiceAccount is annotated — with eks.amazonaws.com/role-arn pointing to the IAM Role
  4. Pod uses the ServiceAccount — EKS mutating webhook injects a projected token volume and AWS env vars
  5. AWS SDK exchanges the token — calls STS AssumeRoleWithWebIdentity to get temporary credentials
Draw this on a whiteboard if possible. The key insight is that there is a bidirectional link: the IAM Role trusts the ServiceAccount (via the trust policy condition), and the ServiceAccount references the IAM Role (via the annotation). Both must be correct for IRSA to work.

📐 Trust Chain Detail

Kubernetes Side

  • EKS OIDC provider issues signed JWT tokens for ServiceAccounts
  • Token projected into pod at a known path
  • Token contains: iss (OIDC URL), sub (system:serviceaccount:NS:SA)
  • Token is short-lived (default 24h, refreshed automatically)

AWS Side

  • IAM OIDC Identity Provider registered for the cluster
  • IAM Role trust policy validates the token issuer and subject
  • STS validates the JWT signature against the OIDC provider’s public keys
  • Returns temporary credentials (access key, secret key, session token)
The JWT token is the bridge between Kubernetes identity and AWS identity. STS never contacts the EKS cluster directly — it validates the token using the OIDC provider's public key endpoint. This is why the OIDC provider must be publicly accessible.

📝 Creating the IAM Policy

Define least-privilege permissions for your workload.

{
=======
                

                    Walk through every field. The Principal is the OIDC provider ARN — this is who is asserting the identity. The Condition block is critical: it restricts which ServiceAccount can assume this role. The aud condition verifies the token audience, and the sub condition verifies the specific namespace and ServiceAccount name. Miss either of these and IRSA will silently fail.

🔑 Condition Keys: Scoping Access

StringEquals (Recommended)

"Condition": {
  "StringEquals": {
    "...OIDC_ID:sub":
      "system:serviceaccount:prod:my-app"
  }
}
  • Exact match on namespace + SA
  • Most restrictive, most secure
  • Use for production workloads

StringLike (Wildcard)

"Condition": {
  "StringLike": {
    "...OIDC_ID:sub":
      "system:serviceaccount:dev:*"
  }
}
  • Wildcard match with *
  • Less restrictive — any SA in namespace
  • Use sparingly, only for dev/test

Best Practice: Always use StringEquals with the exact namespace and ServiceAccount name in production. Wildcards expand the blast radius of a compromised ServiceAccount.

This is a security-critical decision. StringLike with a wildcard on the ServiceAccount means any ServiceAccount in that namespace can assume the role. If an attacker can create a ServiceAccount in that namespace, they get the AWS permissions. Always prefer StringEquals in production.

⚙️ Terraform: IRSA Role

resource "aws_iam_role" "app_irsa_role" {
  name = "platform-lab-my-app-irsa"

  assume_role_policy = jsonencode({
    Version = "2012-10-17"
    Statement = [{
      Effect = "Allow"
      Principal = {
        Federated = aws_iam_openid_connect_provider.eks.arn
      }
      Action = "sts:AssumeRoleWithWebIdentity"
      Condition = {
        StringEquals = {
          "${replace(aws_iam_openid_connect_provider.eks.url, "https://", "")}:aud" = "sts.amazonaws.com"
          "${replace(aws_iam_openid_connect_provider.eks.url, "https://", "")}:sub" = "system:serviceaccount:my-namespace:my-app"
        }
      }
    }]
  })
}

resource "aws_iam_role_policy_attachment" "app_s3_policy" {
  role       = aws_iam_role.app_irsa_role.name
  policy_arn = aws_iam_policy.app_s3_read.arn
}
The replace function strips the https:// prefix from the OIDC URL because IAM condition keys use the hostname format, not the full URL. This is a common gotcha in Terraform. Note how the Condition uses the dynamic OIDC provider reference rather than hardcoded values — this makes the module portable across clusters.

📄 Annotating the ServiceAccount

Linking Kubernetes identity to AWS identity

The ServiceAccount annotation is the Kubernetes-side link. Without it, the pod will not receive the projected token or the AWS environment variables. This is the other half of the bidirectional trust.

📄 ServiceAccount with IRSA Annotation

Declarative YAML

apiVersion: v1
kind: ServiceAccount
metadata:
  name: my-app
  namespace: my-namespace
  annotations:
    eks.amazonaws.com/role-arn: |-
      arn:aws:iam::123456789012:role/
      platform-lab-my-app-irsa

Imperative kubectl

# Annotate an existing SA
kubectl annotate serviceaccount \
  my-app \
  -n my-namespace \
  eks.amazonaws.com/role-arn=\
arn:aws:iam::123456789012:role/\
platform-lab-my-app-irsa
Important: Pods must be restarted after annotating the ServiceAccount. Existing pods will not pick up the new annotation — the mutating webhook only runs at pod creation time.
The annotation key eks.amazonaws.com/role-arn is watched by the EKS pod identity mutating webhook. When a pod is created with a ServiceAccount that has this annotation, the webhook injects the projected token volume and the AWS environment variables. Emphasize: existing pods must be restarted.

🔎 What Happens Inside the Pod

Projected tokens and environment variables

Now let's look at what the mutating webhook actually injects into the pod. Understanding these internals is essential for debugging.

🔎 Inside an IRSA-Enabled Pod

The EKS mutating webhook automatically injects:

  • A projected service account token volume mounted at:
    /var/run/secrets/eks.amazonaws.com/serviceaccount/token
  • Environment variable: AWS_ROLE_ARN — the IAM Role ARN to assume
  • Environment variable: AWS_WEB_IDENTITY_TOKEN_FILE — path to the projected token

Verify from inside the pod:

# Check environment variables
kubectl exec -it my-pod -n my-namespace -- env | grep AWS

# AWS_ROLE_ARN=arn:aws:iam::123456789012:role/platform-lab-my-app-irsa
# AWS_WEB_IDENTITY_TOKEN_FILE=/var/run/secrets/eks.amazonaws.com/serviceaccount/token

# Verify the token file exists
kubectl exec -it my-pod -n my-namespace -- \
  cat /var/run/secrets/eks.amazonaws.com/serviceaccount/token

# Verify identity
kubectl exec -it my-pod -n my-namespace -- aws sts get-caller-identity
The AWS SDK credential chain automatically detects these environment variables and performs the STS AssumeRoleWithWebIdentity call transparently. Application code does not need to be modified — it just works with the standard SDK credential provider chain. The token is a signed JWT that gets refreshed automatically before expiration.

# "Arn": "arn:aws:sts::111122223333:assumed-role/my-app-irsa-role/..."

# 4. Decode the JWT (inspect claims)
kubectl exec -n my-namespace deploy/my-app -- \

Always verify IRSA is working before moving to production. The most common issue is a mismatch between the trust policy's :sub condition and the actual namespace:serviceaccount. Decoding the JWT lets you see exactly what claims the token contains.

🔬 Advanced IRSA Patterns

Cross-account, scoping, and Helm integration

With the fundamentals covered, let's look at advanced patterns that come up in enterprise environments: cross-account access, trust policy scoping strategies, and integrating IRSA with Helm-managed deployments.

🌐 Cross-Account IRSA

Access AWS resources in a different account from your EKS cluster.

Two-Step Role Chain

  1. Pod assumes Role A in the EKS account (via IRSA)
  2. Role A assumes Role B in the target account (via sts:AssumeRole)
// Role B trust policy (in target account 444455556666)
{
  "Effect": "Allow",
  "Principal": {
    "AWS": "arn:aws:iam::111122223333:role/my-app-irsa-role"
  },
  "Action": "sts:AssumeRole"
}

// Role A permissions (in EKS account 111122223333)
{
  "Effect": "Allow",
  "Action": "sts:AssumeRole",
  "Resource": "arn:aws:iam::444455556666:role/cross-account-role"
}
Cross-account IRSA is common in organizations with separate AWS accounts for different environments or teams. The pod first assumes its IRSA role, then chains to a role in the remote account. Some SDKs support this natively via role chaining configuration.

🎯 Scoping Trust Policies

Control exactly which pods can assume a role using condition keys.

Strict — Single SA

"Condition": {
  "StringEquals": {
    "...:sub":
      "system:serviceaccount:prod:app-sa"
  }
}

Only prod/app-sa can assume this role.

Flexible — Namespace Wildcard

"Condition": {
  "StringLike": {
    "...:sub":
      "system:serviceaccount:prod:*"
  }
}

Any SA in prod namespace.

Security tradeoff: StringLike with wildcards is convenient but broadens the blast radius. Prefer StringEquals with explicit SA names for production.
The :sub condition is the main scoping mechanism. StringEquals is the strictest and recommended for production. StringLike with wildcards can be useful for shared roles in development environments, but avoid it in production. You can also use multiple conditions for multi-SA access.

💫 EKS Pod Identity

The next generation of workload identity

AWS introduced EKS Pod Identity as a simpler alternative to IRSA. Let's understand what it offers and when you might consider migrating.

🤖 EKS Pod Identity Agent

How It Works

An EKS-managed DaemonSet runs on each node, intercepting credential requests and exchanging Kubernetes tokens for AWS credentials.

  • EKS Add-on: eks-pod-identity-agent — installed via EKS add-on mechanism
  • No OIDC setup: eliminates the IAM OIDC provider requirement
  • No trust policy edits: associations managed via EKS API
  • Simpler IAM: uses pods.eks.amazonaws.com as the trusted principal
  • Reusable roles: same role can be associated across clusters without trust policy changes
# Install the Pod Identity Agent add-on
aws eks create-addon --cluster-name my-cluster \
  --addon-name eks-pod-identity-agent

# Create a pod identity association
aws eks create-pod-identity-association \
  --cluster-name my-cluster \
  --namespace my-namespace \
  --service-account my-sa \
  --role-arn arn:aws:iam::111122223333:role/my-role
EKS Pod Identity was announced at re:Invent 2023 and is now GA. The agent runs as a DaemonSet and handles credential vending locally on each node. The big win is no more OIDC provider management and no more complex trust policies. Associations are managed centrally through the EKS API.

🛡️ Security Best Practices

Hardening IRSA configurations

IRSA is only as secure as its configuration. Let's cover the practices that make the difference between a secure setup and a false sense of security.

🛡️ IRSA Security Best Practices

  • Least privilege: Scope IAM policies to specific resources and actions
  • Exact conditions: Use StringEquals to lock roles to specific namespace:serviceaccount pairs
  • Avoid wildcard actions: Never use "Action": "*" or "Resource": "*" without justification
  • One role per workload: Do not share IRSA roles across different applications
  • Audit with CloudTrail: Monitor AssumeRoleWithWebIdentity events for unexpected access patterns
  • Verify identity: Use aws sts get-caller-identity from inside pods to confirm the assumed role
  • No credential rotation needed: Tokens are short-lived; rotate the role if compromised, not credentials
  • Namespace isolation: IRSA respects namespace boundaries — leverage this for multi-tenancy
Walk through each practice. Emphasize that IRSA eliminates the need for credential rotation (a huge operational win), but you still need to design policies carefully. CloudTrail is your friend for auditing — every AssumeRoleWithWebIdentity call is logged, including the ServiceAccount that assumed the role.

⚠️ Common Pitfalls

What goes wrong with IRSA and how to avoid it

IRSA configuration has several moving parts and multiple places where things can go wrong. Let's cover the most common issues students and teams encounter.

⚠️ Common IRSA Pitfalls

1. OIDC Provider Thumbprint Mismatch
If the OIDC provider's TLS certificate changes (e.g., AWS rotates it), the thumbprint in IAM may become stale. AWS now auto-manages this for EKS, but older clusters may still have this issue.

2. Missing ServiceAccount Annotation
The IAM Role exists and the trust policy is correct, but the ServiceAccount was never annotated with eks.amazonaws.com/role-arn. The pod starts without AWS credentials.

3. Trust Policy Condition Mismatch
Typo in namespace or ServiceAccount name in the trust policy condition. STS returns AccessDenied with a cryptic error.

These are the top three issues. The thumbprint issue is mostly historical now, but students should know about it. The annotation and condition mismatch issues are by far the most common — they are pure configuration errors that are easy to make and hard to spot without knowing what to look for.

⚠️ More IRSA Pitfalls

4. AWS SDK Version Too Old
The AWS SDK must support WebIdentityTokenCredentialProvider. Older SDKs (pre-2019) do not understand the AWS_WEB_IDENTITY_TOKEN_FILE environment variable and fall back to the instance metadata service.

5. Pod Not Restarted After SA Change
The mutating webhook only runs at pod creation. If you annotate an existing ServiceAccount, running pods must be deleted and recreated to pick up the projected token.

6. Multiple Containers in a Pod
All containers in the pod share the same ServiceAccount and IRSA role. Sidecars get the same AWS permissions as the main container — design accordingly.

The SDK version issue catches people who use custom-built containers with older dependencies. The pod restart issue is the most common "it does not work" complaint after configuring IRSA. The sidecar issue is subtle — if you have a logging sidecar that does not need AWS access, it still gets it.

🔎 Debugging IRSA

Systematic troubleshooting approach

When IRSA is not working, you need a systematic approach to identify where the chain is broken. Let's walk through the debugging steps.

🔎 Debugging IRSA: Step by Step

1. Check the pod for projected token volume

kubectl describe pod my-pod -n my-namespace | grep -A5 "Volumes" | grep "eks.amazonaws.com"
# Look for: aws-iam-token projected volume

2. Verify environment variables inside the pod

kubectl exec my-pod -n my-namespace -- env | grep AWS
# Must see: AWS_ROLE_ARN and AWS_WEB_IDENTITY_TOKEN_FILE

3. Test identity from inside the pod

kubectl exec my-pod -n my-namespace -- aws sts get-caller-identity
# Should show the IRSA role ARN, not the node role

4. Check CloudTrail for STS failures

# Filter CloudTrail for AssumeRoleWithWebIdentity errors
aws cloudtrail lookup-events \
  --lookup-attributes AttributeKey=EventName,AttributeValue=AssumeRoleWithWebIdentity \
  --max-results 10
Follow these steps in order. If the projected token volume is missing, the ServiceAccount annotation is wrong or missing. If the env vars are missing, the mutating webhook did not fire — possibly the pod was created before the annotation was added. If sts get-caller-identity fails, check the trust policy conditions in IAM. CloudTrail gives you the definitive answer on why STS rejected the token.

🛣️ Migrating from IRSA to Pod Identity

Good news: IRSA and Pod Identity can coexist on the same cluster. Migrate workloads incrementally.
  1. Install the eks-pod-identity-agent add-on
  2. Update the IAM role trust policy to add pods.eks.amazonaws.com as a principal (keep IRSA trust)
  3. Create pod identity association for the target SA
  4. Remove the IRSA annotation from the ServiceAccount
  5. Restart pods to pick up new credential source
  6. Validate with aws sts get-caller-identity
  7. Clean up old OIDC trust policy conditions once all workloads are migrated
The migration is non-disruptive because both mechanisms can coexist. Update the trust policy to accept both credential sources, create the pod identity association, then remove the IRSA annotation. Roll out changes gradually, validating each workload before moving to the next.

🏆 Best Practices

Production-grade IRSA patterns

Let's consolidate the key best practices for running IRSA in production. These patterns come from real-world experience managing IRSA at scale.

🛡️ Least Privilege Patterns

IAM Policy

  • Scope to specific resource ARNs
  • Use condition keys (e.g., s3:prefix)
  • Separate read and write roles

Trust Policy

  • Always use StringEquals
  • Specify both :aud and :sub
  • One role per workload

Kubernetes

  • Dedicated SA per deployment
  • Block IMDS access via NetworkPolicy
  • Audit SA annotations regularly
Defense in depth: Combine IRSA with NetworkPolicy to block IMDS (169.254.169.254) and prevent pods from falling back to the node role. 
# Block IMDS access
apiVersion: networking.k8s.io/v1
kind: NetworkPolicy
metadata:
  name: deny-imds
spec:
  podSelector: {}
  policyTypes: [Egress]
  egress:
  - to:
    - ipBlock:
        cidr: 0.0.0.0/0
        except: [169.254.169.254/32]
Blocking IMDS is a critical complementary control. Even with IRSA configured, if a pod can reach IMDS, a compromised container could still retrieve node-level credentials. The NetworkPolicy ensures pods can only use their IRSA credentials, never the node role.
