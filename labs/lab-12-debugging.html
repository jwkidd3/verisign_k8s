Lab 12: Debugging Applications in Kubernetes

Systematic Troubleshooting for Common Failures

Duration: 45 minutes

This lab presents four broken deployments that students must diagnose and fix. Each scenario represents a common production issue. Students can work individually or in pairs. Deploy all four broken manifests before starting.

📋 Lab Overview

Objectives

  • Diagnose CrashLoopBackOff, ImagePullBackOff, Pending, and probe failures
  • Use kubectl commands systematically to identify root causes
  • Fix broken manifests and verify successful recovery
  • Build a mental debugging decision tree
Format: 4 scenarios, individual or pairs. Each scenario is a deliberately broken deployment.
Approach: Observe symptoms → Gather data → Identify root cause → Apply fix → Verify
Each scenario takes roughly 10 minutes. Encourage students to follow the observe-gather-identify-fix-verify cycle. Resist the urge to give hints too early — the diagnostic process is the learning objective.

💥 Scenario A

CrashLoopBackOff

💥 Deploy the Broken Application


apiVersion: v1
kind: ConfigMap
metadata:
  name: scenario-a-config
  namespace: default
data:
  APP_MODE: "production"
  # Note: DATABASE_URL is intentionally missing
---
apiVersion: apps/v1
kind: Deployment
metadata:
  name: scenario-a-app
  namespace: default
spec:
  replicas: 1
  selector:
    matchLabels:
      app: scenario-a
  template:
    metadata:
      labels:
        app: scenario-a
    spec:
      containers:
      - name: app
        image: busybox:1.36
        command: ["/bin/sh", "-c"]
        args:
          - |
            if [ -z "$DATABASE_URL" ]; then
              echo "FATAL: DATABASE_URL environment variable is not set"
              echo "Application cannot start without database configuration"
              exit 1
            fi
            echo "Connected to $DATABASE_URL"
            echo "Application started successfully"
            sleep infinity
        envFrom:
          - configMapRef:
              name: scenario-a-config
                

kubectl apply -f scenario-a.yaml
The application requires DATABASE_URL which is missing from the ConfigMap. The container exits with code 1, causing CrashLoopBackOff. Let students discover this on their own.

🔍 Diagnose the Issue

Your task: The application is failing to start. Use kubectl to determine why and fix it.

Step-by-step debugging:


# 1. Check pod status — observe CrashLoopBackOff
kubectl get pods -l app=scenario-a

# 2. Describe the pod — check Events section
kubectl describe pod -l app=scenario-a

# 3. Check current container logs
kubectl logs -l app=scenario-a

# 4. Check logs from the previous crashed container
kubectl logs -l app=scenario-a --previous
Expected finding: Logs show FATAL: DATABASE_URL environment variable is not set
Walk through the debugging flow: get pods shows the status, describe shows events and restart count, logs (and --previous) reveal the error message. The root cause is a missing environment variable.

✅ Fix and Verify

Add the missing DATABASE_URL to the ConfigMap:


apiVersion: v1
kind: ConfigMap
metadata:
  name: scenario-a-config
  namespace: default
data:
  APP_MODE: "production"
  DATABASE_URL: "postgresql://db-svc:5432/myapp"   # Added!
                

# Apply the fixed ConfigMap
kubectl apply -f scenario-a-configmap-fixed.yaml

# Restart the deployment to pick up the new ConfigMap
kubectl rollout restart deployment scenario-a-app

# Verify the pod is now running
kubectl get pods -l app=scenario-a -w

# Confirm logs show successful startup
kubectl logs -l app=scenario-a
# Expected: "Connected to postgresql://db-svc:5432/myapp"
# Expected: "Application started successfully"
Key lesson: ConfigMap changes are not automatically picked up by running pods. You must restart the Deployment or use a mechanism like Reloader to trigger restarts on ConfigMap changes.
Emphasize that ConfigMap changes don't auto-restart pods. This is a very common gotcha. At Verisign, Reloader or a similar controller handles this automatically, but it's important to understand the underlying behavior.

📷 Scenario B

ImagePullBackOff

📷 Deploy with Wrong Image


apiVersion: apps/v1
kind: Deployment
metadata:
  name: scenario-b-app
  namespace: default
spec:
  replicas: 1
  selector:
    matchLabels:
      app: scenario-b
  template:
    metadata:
      labels:
        app: scenario-b
    spec:
      containers:
      - name: app
        # Deliberate typo: "ngingx" instead of "nginx"
        image: ngingx:1.25-alpine
        ports:
        - containerPort: 80
                

kubectl apply -f scenario-b.yaml
Your task: The pod is stuck and never starts. Identify the problem and fix it.
The image name has a typo: "ngingx" instead of "nginx". This is a very common issue in production, especially with private registries. Let students find it through kubectl describe.

🔍 Diagnose and Fix


# 1. Check pod status — observe ImagePullBackOff or ErrImagePull
kubectl get pods -l app=scenario-b

# 2. Describe the pod — look at Events section
kubectl describe pod -l app=scenario-b
# Events will show:
#   Failed to pull image "ngingx:1.25-alpine":
#   rpc error: manifest unknown

Fix the image reference:


# Option 1: Edit the deployment directly
kubectl set image deployment/scenario-b-app \
  app=nginx:1.25-alpine

# Option 2: Fix the YAML and reapply
# Change: image: ngingx:1.25-alpine
# To:     image: nginx:1.25-alpine
kubectl apply -f scenario-b-fixed.yaml

# Verify
kubectl get pods -l app=scenario-b -w
kubectl describe pod -l app=scenario-b | grep "Image:"
Common causes of ImagePullBackOff: Typos in image name/tag, image doesn't exist in registry, missing imagePullSecrets for private registries, network issues reaching the registry.
The Events section in kubectl describe is the key diagnostic tool here. It clearly states "Failed to pull image" with the reason. Remind students to always check Events first.

📈 Scenario C

Resource Constraints

📈 Part 1: Excessive Resource Requests


apiVersion: apps/v1
kind: Deployment
metadata:
  name: scenario-c-greedy
  namespace: default
spec:
  replicas: 1
  selector:
    matchLabels:
      app: scenario-c-greedy
  template:
    metadata:
      labels:
        app: scenario-c-greedy
    spec:
      containers:
      - name: app
        image: nginx:1.25-alpine
        resources:
          requests:
            cpu: "32"
            memory: 64Gi    # Far more than any node has
          limits:
            cpu: "32"
            memory: 64Gi
                

kubectl apply -f scenario-c-greedy.yaml
Your task: The pod stays in Pending state indefinitely. Why?
This pod requests 64Gi memory, which likely exceeds any single node's capacity. The scheduler cannot find a node to place it on. Students should see "Insufficient memory" in the Events.

🔍 Diagnose Pending State


# 1. Check pod status — observe Pending
kubectl get pods -l app=scenario-c-greedy

# 2. Describe the pod — check Events
kubectl describe pod -l app=scenario-c-greedy
# Events:
#   Warning  FailedScheduling  0/3 nodes are available:
#   3 Insufficient memory.

# 3. Check node capacity for comparison
kubectl describe nodes | grep -A 5 "Allocatable"
kubectl top nodes

Fix by setting reasonable resource requests:


        resources:
          requests:
            cpu: 100m
            memory: 128Mi
          limits:
            cpu: 500m
            memory: 256Mi
                    

kubectl apply -f scenario-c-greedy-fixed.yaml
kubectl get pods -l app=scenario-c-greedy -w
Show students how to check node capacity with kubectl describe nodes. The Allocatable section shows what's available. Resource requests must fit within what the scheduler can allocate.

💣 Part 2: OOMKilled


apiVersion: apps/v1
kind: Deployment
metadata:
  name: scenario-c-oom
  namespace: default
spec:
  replicas: 1
  selector:
    matchLabels:
      app: scenario-c-oom
  template:
    metadata:
      labels:
        app: scenario-c-oom
    spec:
      containers:
      - name: app
        image: busybox:1.36
        command: ["/bin/sh", "-c"]
        args:
          - |
            echo "Starting memory-hungry process..."
            # Allocate memory until killed
            dd if=/dev/zero of=/dev/null bs=1M &
            head -c 200m /dev/zero | tail -c +1 > /dev/null
            sleep infinity
        resources:
          requests:
            cpu: 50m
            memory: 32Mi
          limits:
            cpu: 100m
            memory: 64Mi    # Too low for the workload
                

kubectl apply -f scenario-c-oom.yaml
The container tries to use more memory than its limit allows. The kernel OOM killer terminates it. Students will see OOMKilled as the reason in kubectl describe.

🔍 Diagnose OOMKilled


# 1. Check pod status — CrashLoopBackOff with OOMKilled reason
kubectl get pods -l app=scenario-c-oom

# 2. Describe the pod — look for "OOMKilled" in Last State
kubectl describe pod -l app=scenario-c-oom
# Last State:  Terminated
#   Reason:    OOMKilled
#   Exit Code: 137

# 3. Check resource usage (if pod is running briefly)
kubectl top pod -l app=scenario-c-oom

Fix by increasing the memory limit:


        resources:
          requests:
            cpu: 50m
            memory: 128Mi
          limits:
            cpu: 100m
            memory: 256Mi    # Increased to accommodate workload
Exit code 137 = 128 + 9 (SIGKILL). This always indicates the container was killed by the system, usually OOM. Exit code 139 = 128 + 11 (SIGSEGV) indicates a segmentation fault.
The exit code math is a useful diagnostic trick: 128 + signal number. SIGKILL (9) = 137 means OOM or external kill. Teach students to check exit codes as part of their debugging toolkit.

🩹 Scenario D

Probe Failures

🩹 Part 1: Liveness Probe Failure


apiVersion: apps/v1
kind: Deployment
metadata:
  name: scenario-d-liveness
  namespace: default
spec:
  replicas: 1
  selector:
    matchLabels:
      app: scenario-d-liveness
  template:
    metadata:
      labels:
        app: scenario-d-liveness
    spec:
      containers:
      - name: app
        image: nginx:1.25-alpine
        ports:
        - containerPort: 80
        livenessProbe:
          httpGet:
            path: /healthz      # Wrong path! Nginx returns 404
            port: 8080          # Wrong port! Nginx listens on 80
          initialDelaySeconds: 5
          periodSeconds: 3
          failureThreshold: 3
                

kubectl apply -f scenario-d-liveness.yaml
Your task: The pod keeps restarting. Identify why and fix the probe configuration.
Two errors: wrong port (8080 instead of 80) and wrong path (/healthz doesn't exist). The liveness probe fails, kubelet kills the container, causing CrashLoopBackOff. Students should see "Liveness probe failed" in Events.

🔍 Diagnose Liveness Failure


# 1. Check pods — note high restart count
kubectl get pods -l app=scenario-d-liveness

# 2. Describe pod — look for probe failure events
kubectl describe pod -l app=scenario-d-liveness
# Events:
#   Warning  Unhealthy  Liveness probe failed:
#     HTTP probe failed with statuscode: ...
#   Normal   Killing    Container app failed liveness probe, will be restarted

# 3. Test what's actually available on the container
kubectl exec deploy/scenario-d-liveness -- wget -qO- http://localhost:80/

Fix the liveness probe:


        livenessProbe:
          httpGet:
            path: /          # Nginx default page returns 200
            port: 80         # Correct port
          initialDelaySeconds: 5
          periodSeconds: 10
          failureThreshold: 3
                    

kubectl apply -f scenario-d-liveness-fixed.yaml
kubectl get pods -l app=scenario-d-liveness -w
Point out how liveness probe failures cause restarts (unlike readiness failures). The exec command helps verify what endpoints are actually available inside the container.

🩹 Part 2: Readiness Probe Failure


apiVersion: apps/v1
kind: Deployment
metadata:
  name: scenario-d-readiness
  namespace: default
spec:
  replicas: 1
  selector:
    matchLabels:
      app: scenario-d-readiness
  template:
    metadata:
      labels:
        app: scenario-d-readiness
    spec:
      containers:
      - name: app
        image: nginx:1.25-alpine
        ports:
        - containerPort: 80
        readinessProbe:
          httpGet:
            path: /ready     # This path doesn't exist
            port: 80
          initialDelaySeconds: 2
          periodSeconds: 5
                

kubectl apply -f scenario-d-readiness.yaml
                

# Observe: READY shows 0/1 but NO restarts (key difference)
kubectl get pods -l app=scenario-d-readiness
# NAME                         READY   STATUS    RESTARTS
# scenario-d-readiness-xxx     0/1     Running   0

# The pod is Running but not Ready — it won't receive traffic
kubectl describe pod -l app=scenario-d-readiness
Liveness vs Readiness: Liveness probe failure → container restarts. Readiness probe failure → pod removed from Service endpoints (no traffic), but no restarts.
This is a critical distinction. Readiness failures are subtler — the pod looks healthy but doesn't receive traffic. Show how the Service endpoints are empty when readiness fails.

🧰 Debugging Toolkit

Essential Commands Reference

🧰 Essential Debugging Commands

Command Purpose When to Use
kubectl get pods Pod status overview Always start here
kubectl describe pod Events, conditions, config Pending, ImagePull, probe issues
kubectl logs [--previous] Container stdout/stderr CrashLoopBackOff, app errors
kubectl exec -it -- sh Shell into container Network, filesystem, env issues
kubectl port-forward Local access to pod/service Testing endpoints
kubectl top pod CPU/memory usage OOM, resource issues
kubectl get events --sort-by=.lastTimestamp Cluster-wide events Broad investigation
kubectl debug Ephemeral debug container Distroless/minimal images
This is the essential toolkit every developer should memorize. kubectl debug is especially useful for distroless images where you can't exec into the container because there's no shell.

🌳 Debugging Decision Tree

Pod Not Starting?

  • Pending → Check events: scheduling, resources, node selectors, taints
  • ImagePullBackOff → Wrong image name/tag, missing pull secrets, registry unreachable
  • CrashLoopBackOff → Check logs and --previous logs for app errors
  • CreateContainerError → Missing ConfigMap, Secret, or volume

Pod Running But Broken?

  • 0/1 READY (no restarts) → Readiness probe failing, check probe config
  • High restart count → Liveness probe or OOMKilled, check describe
  • Not accessible → Check Service selector matches pod labels, check endpoints
  • Slow/degraded → Check resource usage with top, check metrics
Always follow this order: get pods → describe pod (Events) → logs / logs --previous → exec for deeper investigation
This decision tree should become second nature. The status tells you which branch to follow, and the commands narrow down the root cause. Encourage students to bookmark or print this reference.

📚 Summary: Common Failure States

Status Cause Key Diagnostic Fix
CrashLoopBackOff App error / missing config kubectl logs --previous Fix app config or code
ImagePullBackOff Wrong image / no pull secret kubectl describe Events Fix image ref or add secret
Pending Insufficient resources kubectl describe Events Reduce requests or add nodes
OOMKilled (137) Memory limit too low describe Last State Increase memory limit
0/1 Ready Readiness probe failing describe + exec test Fix probe path/port
This table is a quick reference for the most common issues. In production, 90% of pod issues fall into one of these categories. The remaining 10% are usually network policy or RBAC related.
