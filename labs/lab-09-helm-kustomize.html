This 45-60 minute lab covers both Helm and Kustomize. Students will use Helm for package management and Kustomize for environment-specific overlays. Both tools are used in the Verisign stack with FluxCD.

📋 Lab Overview

Objectives

  • Install and customize Helm charts from public repositories
  • Manage releases with upgrades, rollbacks, and history
  • Create a custom Helm chart from scratch
  • Build Kustomize bases and environment-specific overlays
  • Understand how Helm and Kustomize integrate with FluxCD

⏱️ Duration

45 – 60 minutes

⚙️ Prerequisites

  • Helm CLI installed (brew install helm)
  • EKS cluster access with kubectl
  • kubectl v1.14+ (built-in Kustomize support)
Verify helm version and kubectl version before starting. Kustomize is built into kubectl since v1.14 via the -k flag, so no separate install is needed.

Step 1: Install a Chart from a Public Repository

Start with a familiar workflow: installing a third-party chart. This is how most teams first encounter Helm.

📥 Add a Helm Repository

# Create a namespace for this lab
kubectl create namespace lab09

# Add the Bitnami repository
helm repo add bitnami https://charts.bitnami.com/bitnami

# Update repository index
helm repo update

# Search for nginx charts
helm search repo bitnami/nginx

💡 Helm Repositories

Helm repositories are HTTP servers hosting packaged charts (.tgz files) and an index. Bitnami maintains one of the most popular public chart repositories.

helm repo update fetches the latest chart index. Without it, you may install outdated chart versions. In production, Verisign uses private OCI-based chart repositories in ECR.

🚀 Install the Chart

# Install nginx with default values
helm install my-nginx bitnami/nginx \
  --namespace lab09 \
  --set service.type=ClusterIP

# Verify the release
helm list --namespace lab09

# Check the deployed resources
kubectl get all -n lab09

✅ Expected Resources

  • Deployment: my-nginx
  • Service: my-nginx (ClusterIP)
  • Pod: my-nginx-xxx (Running)
# View the values Helm used for this release
helm get values my-nginx --namespace lab09

# View ALL computed values (defaults + overrides)
helm get values my-nginx --namespace lab09 --all
helm get values shows what was explicitly set. --all shows the full computed values including defaults from values.yaml. This is invaluable for debugging.

Step 2: Customize with Values File

1

Values files are the primary way to customize Helm charts. This is the recommended approach over --set flags for anything beyond trivial changes.

📝 Create a Custom Values File

Create custom-values.yaml:

# custom-values.yaml
replicaCount: 3

service:
  type: ClusterIP
  port: 80

resources:
  requests:
    memory: "64Mi"
    cpu: "50m"
  limits:
    memory: "128Mi"
    cpu: "100m"

podLabels:
  team: platform-engineering
  environment: lab

autoscaling:
  enabled: false

metrics:
  enabled: true
  service:
    port: 9113
Values files map directly to the chart's values.yaml schema. Use helm show values bitnami/nginx to see all available options.

⬆️ Upgrade the Release

# Upgrade with the custom values
helm upgrade my-nginx bitnami/nginx \
  --namespace lab09 \
  -f custom-values.yaml

# Verify the upgrade
helm list --namespace lab09
# Note the REVISION column should now be 2

# Check that we have 3 replicas
kubectl get pods -n lab09 -l app.kubernetes.io/name=nginx

# Verify resource limits are applied
kubectl get pod -n lab09 -l app.kubernetes.io/name=nginx \
  -o jsonpath='{.items[0].spec.containers[0].resources}' | jq .

✅ Result: The release is upgraded to revision 2 with 3 replicas, resource limits, and custom labels applied.

helm upgrade is idempotent - you can run it multiple times safely. Each upgrade creates a new revision. Point out that the revision number incremented.

Step 3: Rollback a Release

1
2
3
4
5

10
Rollback is one of Helm's most valuable features. It gives you a safety net for any upgrade.

⏪ Rollback to Previous Revision

# View release history
helm history my-nginx --namespace lab09

Expected Output

REVISION  STATUS      DESCRIPTION
1         superseded  Install complete
2         deployed    Upgrade complete
# Rollback to revision 1
helm rollback my-nginx 1 --namespace lab09

# Verify rollback
helm history my-nginx --namespace lab09
# Revision 3 should appear as "Rollback to 1"

# Confirm we're back to 1 replica
kubectl get pods -n lab09 -l app.kubernetes.io/name=nginx

💡 Note: A rollback creates a new revision (3), it doesn't delete history. This gives you a full audit trail of every change.

Helm stores release metadata as Kubernetes Secrets in the release namespace. This is how it maintains history. Each revision is a separate Secret.

Step 4: Create a Custom Helm Chart

1

Creating custom charts is an essential skill for packaging your own applications.

🏗️ Scaffold a New Chart

# Create a new chart
helm create myapp

# Explore the generated structure
find myapp -type f

📁 Chart Structure

myapp/
├── Chart.yaml          # Chart metadata (name, version, description)
├── values.yaml         # Default configuration values
├── charts/             # Dependency charts
├── templates/          # Kubernetes manifest templates
│   ├── deployment.yaml
│   ├── service.yaml
│   ├── serviceaccount.yaml
│   ├── hpa.yaml
│   ├── ingress.yaml
│   ├── NOTES.txt       # Post-install message template
│   ├── _helpers.tpl    # Template helper functions
│   └── tests/
│       └── test-connection.yaml
└── .helmignore
Walk through each file's purpose. _helpers.tpl is especially important - it contains reusable template snippets like labels and names.

✏️ Customize Chart.yaml

Edit myapp/Chart.yaml:

apiVersion: v2
name: myapp
description: A sample web application for Verisign K8s training
type: application
version: 0.1.0
appVersion: "1.0.0"
maintainers:
- name: Platform Engineering
  email: platform-team@verisign.com
Chart.yaml is the chart's identity. version is the chart version (packaging), appVersion is the application version. They're independent.

✏️ Customize values.yaml

Edit myapp/values.yaml:

replicaCount: 1

image:
  repository: hashicorp/http-echo
  pullPolicy: IfNotPresent
  tag: "0.2.3"

args:
  - "-text=Hello from Helm chart myapp!"
  - "-listen=:8080"

service:
  type: ClusterIP
  port: 8080

resources:
  requests:
    cpu: 50m
    memory: 32Mi
  limits:
    cpu: 100m
    memory: 64Mi

serviceAccount:
  create: true
  name: ""

ingress:
  enabled: false

autoscaling:
  enabled: false
We use http-echo for simplicity. In production, this would point to the team's application image in ECR.

✏️ Customize the Deployment Template

Edit myapp/templates/deployment.yaml to add the args:

      containers:
        - name: {{ .Chart.Name }}
          securityContext:
            {{- toYaml .Values.securityContext | nindent 12 }}
          image: "{{ .Values.image.repository }}:{{ .Values.image.tag }}"
          imagePullPolicy: {{ .Values.image.pullPolicy }}
          {{- if .Values.args }}
          args:
            {{- toYaml .Values.args | nindent 12 }}
          {{- end }}
          ports:
            - name: http
              containerPort: {{ .Values.service.port }}
              protocol: TCP
# Install the custom chart
helm install myapp ./myapp --namespace lab09

# Verify
kubectl get all -n lab09 -l app.kubernetes.io/name=myapp

# Test the application
kubectl exec -n lab09 deploy/myapp -- wget -qO- http://localhost:8080
The key addition is the args block with the conditional. The default helm create template doesn't include args, so we need to add it for http-echo.

📄 Render Templates Locally

Now create prod-values.yaml for production overrides:

# prod-values.yaml
replicaCount: 3

    cpu: 200m
    memory: 128Mi
  limits:
    cpu: 500m
    memory: 256Mi
# Render with production values

# Compare outputs - notice the differences
helm template myapp ./myapp --namespace lab09 | grep replicas
helm template myapp ./myapp --namespace lab09 -f prod-values.yaml | grep replicas

💡 Use Case: helm template is invaluable for CI/CD validation, code review, and understanding what a chart will actually deploy before applying it.

helm template renders locally without contacting the cluster. This is perfect for CI validation - you can diff the output against expected manifests.

Step 6: Kustomize Basics

1

10
Transition from Helm to Kustomize. Kustomize takes a fundamentally different approach: no templates, just patches on plain YAML.

📁 Create the Base Directory

# Create the directory structure
mkdir -p kustomize-demo/base
mkdir -p kustomize-demo/overlays/dev
mkdir -p kustomize-demo/overlays/prod

Create kustomize-demo/base/deployment.yaml:

apiVersion: apps/v1
kind: Deployment
metadata:
  name: webapp
  labels:
    app: webapp
spec:
  replicas: 1
  selector:
    matchLabels:
      app: webapp
  template:
    metadata:
      labels:
        app: webapp
    spec:
      containers:
      - name: webapp
        image: hashicorp/http-echo:0.2.3
        args:
        - "-text=Hello from Kustomize base!"
        - "-listen=:8080"
        ports:
        - containerPort: 8080
        resources:
          requests:
            cpu: 50m
            memory: 32Mi
          limits:
            cpu: 100m
            memory: 64Mi
The base contains plain, valid Kubernetes YAML. No templating syntax, no special markers. This is the key difference from Helm.

📁 Create Base Service and Kustomization

Create kustomize-demo/base/service.yaml:

apiVersion: v1
kind: Service
metadata:
  name: webapp
spec:
  selector:
    app: webapp
  ports:
  - port: 8080
    targetPort: 8080
  type: ClusterIP

Create kustomize-demo/base/kustomization.yaml:

apiVersion: kustomize.config.k8s.io/v1beta1
kind: Kustomization

resources:
- deployment.yaml
- service.yaml

commonLabels:
  managed-by: kustomize
# Preview what the base generates
kubectl kustomize kustomize-demo/base/
The kustomization.yaml is the only Kustomize-specific file. It references the resources and declares any transformations. commonLabels adds labels to all resources.

Step 7: Kustomize Overlays

1

Overlays are how Kustomize handles environment-specific configuration. Each overlay patches the base.

🔧 Dev Overlay

Create kustomize-demo/overlays/dev/kustomization.yaml:

apiVersion: kustomize.config.k8s.io/v1beta1
kind: Kustomization

namespace: lab09-dev

resources:
- ../../base

namePrefix: dev-

commonLabels:
  environment: dev

patches:
- target:
    kind: Deployment
    name: webapp
  patch: |-
    - op: replace
      path: /spec/replicas
      value: 1
    - op: replace
      path: /spec/template/spec/containers/0/args
      value:
        - "-text=Hello from DEV environment!"
        - "-listen=:8080"
The overlay references the base with a relative path. namePrefix adds 'dev-' to all resource names. The JSON patch format allows precise modifications.

🏭 Prod Overlay

Create kustomize-demo/overlays/prod/kustomization.yaml:

apiVersion: kustomize.config.k8s.io/v1beta1
kind: Kustomization

namespace: lab09-prod

resources:
- ../../base

namePrefix: prod-

commonLabels:
  environment: prod

patches:
- target:
    kind: Deployment
    name: webapp
  patch: |-
    - op: replace
      path: /spec/replicas
      value: 3
    - op: replace
      path: /spec/template/spec/containers/0/args
      value:
        - "-text=Hello from PRODUCTION environment!"
        - "-listen=:8080"
    - op: replace
      path: /spec/template/spec/containers/0/resources
      value:
        requests:
          cpu: 200m
          memory: 128Mi
        limits:
          cpu: 500m
          memory: 256Mi
Prod overlay has more replicas and higher resource limits. The same base manifests produce different outputs for each environment.

🔍 Compare Overlays

# Preview dev overlay output
kubectl kustomize kustomize-demo/overlays/dev/

# Preview prod overlay output
kubectl kustomize kustomize-demo/overlays/prod/

🟢 Dev

  • Namespace: lab09-dev
  • Name prefix: dev-
  • Replicas: 1
  • CPU limit: 100m
  • Label: environment: dev

🔴 Prod

  • Namespace: lab09-prod
  • Name prefix: prod-
  • Replicas: 3
  • CPU limit: 500m
  • Label: environment: prod
Have students diff the two outputs to see exactly what changes. The base YAML stays identical - only the overlay patches differ.

Now deploy the dev overlay and verify the resources match our expectations.

🚀 Deploy the Dev Overlay

# Create the dev namespace
kubectl create namespace lab09-dev

# Deploy using the dev overlay
kubectl apply -k kustomize-demo/overlays/dev/

# Verify the deployment
kubectl get all -n lab09-dev

# Check the pod response
kubectl exec -n lab09-dev deploy/dev-webapp -- \
  wget -qO- http://localhost:8080
# Expected: "Hello from DEV environment!"

✅ Verification Points

  • Resource names start with dev-
  • Resources are in namespace lab09-dev
  • Labels include environment: dev
  • Only 1 replica running
The -k flag tells kubectl to process the directory as a Kustomize overlay. This is equivalent to running kubectl kustomize and piping to kubectl apply.

🔍 Inspect Labels and Metadata

# Show labels on all resources
kubectl get all -n lab09-dev --show-labels

# Verify the environment label
kubectl get deploy dev-webapp -n lab09-dev \
  -o jsonpath='{.metadata.labels}' | jq .

# Compare: what would prod look like?
kubectl kustomize kustomize-demo/overlays/prod/ | head -30

💡 Key Insight: The same base manifests produce different resources for each environment. No duplication, no template syntax — just patches.

Highlight that with Kustomize, you can kubectl apply the base directory directly for local development, or use overlays for deployed environments.

In practice, Helm and Kustomize complement each other. FluxCD uses both together in Verisign's GitOps pipeline.

🔄 The Helm Template + Kustomize Pattern

Helm Chart
→
helm template
→
Plain YAML
→
Kustomize Overlay
→
kubectl apply
# Example: render Helm chart, then kustomize the output
helm template my-nginx bitnami/nginx \
  -f custom-values.yaml \
  --namespace lab09 > kustomize-demo/base/helm-rendered.yaml

# Then apply Kustomize patches on top

💡 Why Combine?

  • Helm excels at packaging and distributing reusable charts
  • Kustomize excels at environment-specific patches without templates
  • Together: use Helm for third-party charts, Kustomize for env customization
This pattern is very common in GitOps. The rendered Helm output becomes the Kustomize base, and each environment gets its own overlay.

🏛️ FluxCD Integration (Verisign Stack)

In the Verisign GitOps pipeline, FluxCD manages both Helm and Kustomize natively:

# FluxCD HelmRelease example
apiVersion: helm.toolkit.fluxcd.io/v2beta2
kind: HelmRelease
metadata:
  name: envoy-gateway
  namespace: gateway-system
spec:
  interval: 10m
  chart:
    spec:
      chart: gateway-helm
      version: "1.0.x"
      sourceRef:
        kind: HelmRepository
        name: envoy-gateway
  values:
    config:
      envoyGateway:
        logging:
          level: info
FluxCD reconciles HelmReleases automatically from Git. When a values change is merged to main, Flux detects it and performs the upgrade. This is how Verisign manages Envoy Gateway, Kyverno, and other infrastructure components.

📂 FluxCD Kustomization Example

# FluxCD Kustomization example
apiVersion: kustomize.toolkit.fluxcd.io/v1
kind: Kustomization
metadata:
  name: app-team-alpha
  namespace: flux-system
spec:
  interval: 5m
  path: ./overlays/prod
  prune: true
  sourceRef:
    kind: GitRepository
    name: app-team-alpha
  targetNamespace: team-alpha
  postBuild:
    substituteFrom:
    - kind: ConfigMap
      name: cluster-vars

🔄 GitOps Flow

Git commit → FluxCD detects change → Kustomize builds manifests → kubectl apply → Cluster updated

No manual kubectl apply needed. The Git repository is the single source of truth.

FluxCD Kustomization (capital K) is different from kustomize (lowercase k). The FluxCD resource orchestrates the application of Kustomize overlays from Git. postBuild.substituteFrom allows injecting cluster-specific variables.

📝 Summary — Helm vs Kustomize

FeatureHelmKustomize
Approach Templating (Go templates) Patching (overlays on plain YAML)
Package management Yes (charts, repos) No
Release management Yes (history, rollback) No
Environment overlays Values files Native overlay system
Learning curve Steeper (Go templates) Gentler (plain YAML)
Built into kubectl No (separate CLI) Yes (-k flag)
Best for Distributing reusable apps Environment-specific config
Neither tool replaces the other. They solve different problems and work best together. Helm for packaging, Kustomize for environment customization.

🎯 When to Use Each

⚙️ Use Helm When...

  • Installing third-party applications (Prometheus, cert-manager, Kyverno)
  • Packaging your app for distribution across teams
  • You need release history and rollback capability
  • Complex conditional logic in manifests
  • Managing chart dependencies

📁 Use Kustomize When...

  • Managing environment-specific differences (dev/staging/prod)
  • Simple patches on existing YAML
  • You want to avoid templating complexity
  • GitOps workflows with FluxCD/ArgoCD
  • Team-specific overlays on a shared base

🏛️ Verisign Pattern

Infrastructure components (Envoy Gateway, Kyverno, Prometheus) are managed as FluxCD HelmReleases. Application deployments use Kustomize overlays per environment, with FluxCD Kustomizations reconciling from Git.

This is the standard pattern across the Verisign platform. Platform engineering manages Helm charts for infrastructure, application teams use Kustomize for their services.
