🔬 Lab 1: Exploring Your Kubernetes Environment

Intermediate Kubernetes — Hands-On Lab

Duration: ~30 minutes

This is the first hands-on lab. Make sure all participants have kubectl configured and can reach the cluster before starting. Walk the room to verify access during Step 1.

📋 Lab Overview

Objectives

  • Verify cluster connectivity and examine node topology
  • Navigate Kubernetes namespaces and system components
  • Deploy a sample application with production-ready configuration
  • Expose the application via a Service and test connectivity
  • Observe pod lifecycle events, logs, and resource consumption

⏱ Duration

~30 minutes

✅ Prerequisites

  • kubectl installed and configured
  • Access to the EKS cluster
  • Terminal with internet connectivity
Review the objectives with the class. Emphasize this lab builds a foundation — every subsequent lab assumes comfort with these commands. If someone is blocked on prerequisites, pair them with a neighbor while you troubleshoot.

▶ Step 1

Verify Cluster Access

🔐 Verify Cluster Access

Confirm your kubectl context is pointing to the correct cluster.

Check cluster information


kubectl cluster-info

Inspect the nodes


kubectl get nodes -o wide

Verify client and server versions


kubectl version
💡 Key Point: The -o wide flag reveals additional columns including internal IPs, OS images, and container runtimes — essential for debugging node-level issues.
Walk around the room during this step. Common issues: wrong kubeconfig context, expired AWS credentials, VPN not connected. Have participants run `kubectl config current-context` if they see errors.

▶ Step 2

Explore Namespaces

📁 Explore Namespaces

List all namespaces


kubectl get namespaces

Common System Namespaces on EKS

  • kube-system — Core components (CoreDNS, kube-proxy, Calico, metrics-server)
  • kube-public — Publicly readable resources
  • kube-node-lease — Node heartbeat leases
  • monitoring — Prometheus & Grafana stack
  • logging — Splunk forwarders & log aggregation
  • flux-system — FluxCD GitOps controllers
Point out which namespaces are Kubernetes defaults vs. platform-specific. This is a good opportunity to discuss namespace isolation and RBAC scoping. Ask: "Which namespaces do you recognize?"

🔍 Inspect System Components

Examine pods in kube-system


kubectl get pods -n kube-system

Explore the monitoring namespace


kubectl get pods -n monitoring

Examine FluxCD controllers


kubectl get pods -n flux-system
💡 Key Point: Understanding what runs in system namespaces helps you differentiate platform issues from application issues during troubleshooting.
Ask participants to identify Calico, Prometheus, and FluxCD pods. This reinforces the platform architecture covered in the presentation.

▶ Step 3

Deploy a Sample Application

📦 Deployment Manifest

Create a file named lab01-deployment.yaml with the following content:


apiVersion: apps/v1
kind: Deployment
metadata:
  name: lab01-nginx
  labels:
    app: lab01-nginx
spec:
  replicas: 2
  selector:
    matchLabels:
      app: lab01-nginx
  template:
    metadata:
      labels:
        app: lab01-nginx
    spec:
      containers:
      - name: nginx
        image: nginx:1.25-alpine
        ports:
        - containerPort: 80
        resources:
          requests:
            cpu: 100m
            memory: 64Mi
          limits:
            cpu: 250m
            memory: 128Mi
        readinessProbe:
          httpGet:
            path: /
            port: 80
          initialDelaySeconds: 5
          periodSeconds: 10
        livenessProbe:
          httpGet:
            path: /
            port: 80
          initialDelaySeconds: 10
          periodSeconds: 15
Walk through each section: the resource requests/limits ensure scheduling fairness, and the probes give Kubernetes signals for rolling updates and traffic routing. Ask: "Why do we set requests lower than limits?"

🚀 Apply and Verify

Apply the deployment


kubectl apply -f lab01-deployment.yaml

Check the deployment status


kubectl get deployment lab01-nginx

Watch pods come up


kubectl get pods -l app=lab01-nginx
✅ Expected Output: You should see 2 pods in Running state with 1/1 containers ready.
If pods stay in Pending, check resource quotas or node capacity. If CrashLoopBackOff, check image pull policies. Common issue: corporate proxy blocking image pulls.

▶ Step 4

Expose with a Service

🌐 Service Manifest

Create a file named lab01-service.yaml:


apiVersion: v1
kind: Service
metadata:
  name: lab01-nginx-svc
  labels:
    app: lab01-nginx
spec:
  type: ClusterIP
  selector:
    app: lab01-nginx
  ports:
  - port: 80
    targetPort: 80
    protocol: TCP
    name: http

Apply and verify


kubectl apply -f lab01-service.yaml
kubectl get svc lab01-nginx-svc
Explain ClusterIP is internal-only. In the Verisign platform, external traffic arrives through Envoy Gateway — we'll cover that later. For now, we test with port-forward.

🔗 Test Connectivity

Port-forward to your local machine


kubectl port-forward svc/lab01-nginx-svc 8080:80

In another terminal, test the connection


curl http://localhost:8080
✅ Expected Output: You should see the default nginx welcome page HTML.
💡 Key Point: port-forward creates a tunnel from your workstation to the cluster. It is useful for development and debugging — not for production traffic routing.
Remind participants to press Ctrl+C to stop port-forward when done. If port 8080 is taken, they can use any available local port.

▶ Step 5

Observe Pod Lifecycle

👁 Watch Pod Events

Watch pods in real time


kubectl get pods -l app=lab01-nginx -w

Describe a pod for detailed info


# Replace with your actual pod name
kubectl describe pod lab01-nginx-<pod-hash>

📜 What to look for in describe output

  • Events — Scheduling, pulling, creating, starting events
  • Conditions — PodScheduled, Initialized, ContainersReady, Ready
  • QoS Class — Burstable (because requests ≠ limits)
This is a key debugging skill. Point out the Events section at the bottom of describe output — it shows the chronological lifecycle. Ask: "What QoS class would this pod get if requests equaled limits?"

📋 Logs and Resource Usage

View container logs


# View logs from one pod
kubectl logs -l app=lab01-nginx --tail=20

# Stream logs in real time
kubectl logs -l app=lab01-nginx -f

Check resource consumption


kubectl top pods -l app=lab01-nginx
⚠️ Note: kubectl top requires a working metrics-server. If you see an error, verify the metrics-server is running in kube-system.
Mention that in the Verisign platform, logs also flow to Splunk via the logging namespace. kubectl logs is great for real-time debugging but Splunk is the long-term store.

▶ Step 6

Clean Up

🗑️ Clean Up Resources

Delete the deployment and service


kubectl delete -f lab01-deployment.yaml
kubectl delete -f lab01-service.yaml

Verify everything is removed


kubectl get all -l app=lab01-nginx
✅ Expected: No resources found. Clean clusters are happy clusters.
Emphasize the importance of cleaning up lab resources. In a shared cluster, leftover workloads consume resources and cause confusion. Mention that Kyverno policies may enforce cleanup in production namespaces.

📚 Summary

What You Accomplished

  • Verified cluster access and inspected node topology
  • Explored system namespaces and platform components
  • Deployed a production-style application with resource management and health probes
  • Created a ClusterIP Service and tested connectivity via port-forward

Recap these points quickly. Ask if anyone encountered anything unexpected — those are great learning moments.

📑 Key Commands Reference

Command Purpose
kubectl cluster-info Display cluster endpoint and DNS
kubectl get nodes -o wide List nodes with extended details
kubectl get namespaces List all namespaces
kubectl apply -f <file> Declaratively create/update resources
kubectl get pods -w Watch pods in real time
kubectl describe pod <name> Detailed pod info and events
kubectl logs -l app=<label> View logs by label selector
kubectl top pods Resource consumption (requires metrics-server)
kubectl port-forward Tunnel traffic to a Service or Pod
kubectl delete -f <file> Remove resources defined in a manifest
Suggest participants bookmark this slide or keep a cheat sheet. These commands form the foundation for every subsequent lab.
