⚙️ Step 1

Examine a Jenkins Pipeline

📝 Jenkinsfile Overview

Review the complete CI/CD pipeline definition:


pipeline {
    agent {
        kubernetes {
            yaml """
apiVersion: v1
kind: Pod
spec:
  containers:
  - name: docker
    image: docker:24-dind
    securityContext:
      privileged: true
  - name: kubectl
    image: bitnami/kubectl:1.28
    command: ['sleep', 'infinity']
"""
        }
    }

    environment {
        REGISTRY = 'registry.verisign.com'
        IMAGE    = "${REGISTRY}/apps/my-api"
        TAG      = "${env.BUILD_NUMBER}-${env.GIT_COMMIT.take(7)}"
    }

    stages {
        stage('Checkout') {
            steps {
                checkout scm
            }
        }

        stage('Build Image') {
            steps {
                container('docker') {
                    sh "docker build -t ${IMAGE}:${TAG} ."
                }
            }
        }

        stage('Scan Image') {
            steps {
                container('docker') {
                    sh """
                        trivy image --exit-code 1 \
                          --severity CRITICAL,HIGH \
                          ${IMAGE}:${TAG}
                    """
                }
            }
        }

        stage('Push to Registry') {
            steps {
                container('docker') {
                    withCredentials([
                        usernamePassword(
                            credentialsId: 'registry-creds',
                            usernameVariable: 'REG_USER',
                            passwordVariable: 'REG_PASS'
                        )
                    ]) {
                        sh """
                            docker login ${REGISTRY} \
                              -u ${REG_USER} -p ${REG_PASS}
                            docker push ${IMAGE}:${TAG}
                        """
                    }
                }
            }
        }

        stage('Deploy to K8s') {
            steps {
                container('kubectl') {
                    sh """
                        kubectl set image deployment/my-api \
                          my-api=${IMAGE}:${TAG} \
                          -n production
                        kubectl rollout status \
                          deployment/my-api \
                          -n production --timeout=120s
                    """
                }
            }
        }

        stage('Verify Deployment') {
            steps {
                container('kubectl') {
                    sh """
                        kubectl get pods -n production \
                          -l app=my-api
                        kubectl run verify --rm -it \
                          --image=curlimages/curl \
                          --restart=Never -- \
                          curl -sf http://my-api.production/health
                    """
                }
            }
        }
    }

    post {
        failure {
            slackSend channel: '#deploys',
                color: 'danger',
                message: "Build ${env.BUILD_NUMBER} FAILED"
        }
        success {
            slackSend channel: '#deploys',
                color: 'good',
                message: "Build ${env.BUILD_NUMBER} deployed"
        }
    }
}
Walk through each stage. Point out the Kubernetes agent pod definition, the Trivy scan stage that gates on CRITICAL/HIGH vulnerabilities, and the post block for notifications. This is a push-based model where Jenkins directly modifies the cluster.

🔍 Pipeline Stage Breakdown

Build & Security Stages

  • Checkout — Pulls source code from Git
  • Build Image — Creates container image with Docker-in-Docker
  • Scan Image — Trivy scans for CRITICAL/HIGH CVEs; fails the build if found

Delivery Stages

  • Push to Registry — Authenticates and pushes to internal registry
  • Deploy to K8s — kubectl set image updates the Deployment
  • Verify — Confirms pods are running and health check passes
Emphasize that image scanning is a gate — if Trivy finds critical vulnerabilities, the pipeline stops. This is a key security control at Verisign.

🚀 Step 2

Understanding the Pipeline Flow

🛠️ Push-Based CI/CD Flow

➔

➔

➔

➔

➔

Key characteristic: Jenkins pushes changes into the cluster. It holds cluster credentials and directly executes kubectl commands.
Limitations of push-based: External system needs cluster credentials, no continuous reconciliation, cluster state can drift from desired state.
This is the traditional CI/CD approach. It works, but has drawbacks around credential management and drift. Use this as a bridge to introduce GitOps as an improvement for the CD portion.

📚 Step 3

GitOps with FluxCD — Concepts

⚖️ Pull-Based vs Push-Based

📤 Push-Based (Jenkins)

  • CI tool pushes to cluster
  • Cluster credentials stored externally
  • No continuous reconciliation
  • Drift goes undetected

📥 Pull-Based (FluxCD)

  • Agent inside cluster pulls from Git
  • No external cluster credentials needed
  • Continuous reconciliation loop
  • Drift is automatically corrected
GitOps Principles: Git is the single source of truth. All changes go through Git. The cluster state is continuously reconciled to match Git. Drift is detected and corrected automatically.
GitOps shifts the deployment responsibility from an external CI system to an in-cluster agent. This is more secure and provides continuous enforcement of desired state.

🏗️ Step 4

FluxCD Architecture

🧩 FluxCD Components

source-controller

Fetches manifests from Git repos, Helm repos, S3 buckets

kustomize-controller

Applies Kustomize overlays and plain YAML manifests

helm-controller

Manages Helm chart installations and upgrades

notification-controller

Sends alerts to Slack, Teams, webhooks on events

Check all FluxCD components:


# View all Flux resources and their status
flux get all

# Check Flux system pods
kubectl get pods -n flux-system

# View Flux version
flux version
Each controller has a specific responsibility. The source-controller handles fetching, while kustomize and helm controllers handle applying resources. The notification-controller integrates with external alerting systems.

📝 Step 5

Create a FluxCD Kustomization

📄 Flux Kustomization YAML


apiVersion: kustomize.toolkit.fluxcd.io/v1
kind: Kustomization
metadata:
  name: my-api
  namespace: flux-system
spec:
  # Reconciliation interval - how often Flux checks for changes
  interval: 5m

  # Path within the Git repository
  path: ./clusters/production/my-api

  # Prune resources removed from Git
  prune: true

  # Reference to the GitRepository source
  sourceRef:
    kind: GitRepository
    name: verisign-apps

  # Health checks before marking as ready
  healthChecks:
    - apiVersion: apps/v1
      kind: Deployment
      name: my-api
      namespace: production

  # Timeout for health checks
  timeout: 2m
spec.interval: How often Flux polls for changes (5m). spec.prune: Deletes resources removed from Git. spec.sourceRef: Links to a GitRepository object.
Walk through each field. The interval determines how quickly changes are picked up. Prune is important — it means removing a file from Git will remove the resource from the cluster. Health checks ensure the deployment is actually healthy before marking reconciliation as successful.

⚗️ Flux HelmRelease YAML


apiVersion: helm.toolkit.fluxcd.io/v2beta2
kind: HelmRelease
metadata:
  name: my-api
  namespace: production
spec:
  interval: 10m
  chart:
    spec:
      chart: my-api
      version: "1.2.x"    # Semver range
      sourceRef:
        kind: HelmRepository
        name: verisign-charts
        namespace: flux-system
  values:
    replicaCount: 3
    image:
      repository: registry.verisign.com/apps/my-api
      tag: "v1.5.0"
    resources:
      requests:
        cpu: 100m
        memory: 128Mi
      limits:
        cpu: 500m
        memory: 256Mi
  upgrade:
    remediation:
      retries: 3
Apply both resources: 

kubectl apply -f flux-kustomization.yaml
kubectl apply -f flux-helmrelease.yaml
flux get kustomizations
flux get helmreleases -n production
HelmReleases let Flux manage Helm charts declaratively. The version field supports semver ranges. The upgrade.remediation section tells Flux to retry failed upgrades up to 3 times before rolling back.

🚀 Step 6

Simulate a GitOps Deployment

🔁 Update Image Tag via Git

Make a change to the deployment manifest in your Git repository:


# Clone the app repository
git clone https://git.verisign.com/apps/my-api-manifests.git
cd my-api-manifests/clusters/production/my-api

# Edit the deployment to update the image tag

Change the image tag in deployment.yaml:


# Before:
    spec:
      containers:
      - name: my-api
        image: registry.verisign.com/apps/my-api:v1.5.0

# After:
    spec:
      containers:
      - name: my-api
        image: registry.verisign.com/apps/my-api:v1.6.0
In a real workflow, this change might be made by a CI pipeline that updates the image tag after building, or by an image automation controller. Have students make the change manually to understand the flow.

👀 Commit, Push, and Watch


# Commit and push the change
git add deployment.yaml
git commit -m "Update my-api image to v1.6.0"
git push origin main

# Watch Flux reconcile (this will update in real-time)
flux get kustomizations -w

# Expected output after reconciliation:
# NAME    REVISION        READY   MESSAGE
# my-api  main@sha1:abc   True    Applied revision: main@sha1:abc123

Verify the deployment was updated:


# Check the deployment image
kubectl get deployment my-api -n production \
  -o jsonpath='{.spec.template.spec.containers[0].image}'

# Watch the rollout
kubectl rollout status deployment/my-api -n production

# Verify pods are running with new image
kubectl get pods -n production -l app=my-api -o wide
The -w flag on flux get shows real-time updates. Depending on the interval, students may need to wait up to 5 minutes. You can trigger an immediate reconciliation with: flux reconcile kustomization my-api

↺️ Step 7

GitOps Rollback

🔄 Rollback via Git Revert

In GitOps, rollback means reverting the Git commit:


# Revert the last commit (the image update)
git revert HEAD --no-edit

# Push the revert commit
git push origin main

# Watch Flux reconcile the rollback
flux get kustomizations -w

# Force immediate reconciliation (optional)
flux reconcile kustomization my-api

Verify the rollback:


# Confirm the image is back to v1.5.0
kubectl get deployment my-api -n production \
  -o jsonpath='{.spec.template.spec.containers[0].image}'

# Check rollout status
kubectl rollout status deployment/my-api -n production
GitOps rollback vs imperative rollback: git revert creates an auditable trail. kubectl rollout undo modifies cluster state directly without updating Git, causing drift.
Stress the difference: git revert keeps Git and the cluster in sync. kubectl rollout undo creates drift because Git still says v1.6.0 but the cluster has v1.5.0. Flux would eventually reconcile back to v1.6.0, undoing the imperative rollback.

🔭 Step 8

Drift Detection

🚫 Simulate and Detect Drift

Manually modify the cluster state (creating drift):


# Manually scale the deployment (drift from Git)
kubectl scale deployment my-api -n production --replicas=1

# Verify the manual change
kubectl get deployment my-api -n production
# NAME     READY   UP-TO-DATE   AVAILABLE
# my-api   1/1     1            1

Watch Flux detect and correct the drift:


# Wait for the next reconciliation cycle (or force it)
flux reconcile kustomization my-api

# Watch the replica count get corrected
kubectl get deployment my-api -n production -w
# NAME     READY   UP-TO-DATE   AVAILABLE
# my-api   1/3     3            1
# my-api   2/3     3            2
# my-api   3/3     3            3
Reconciliation loop: Flux continuously compares the cluster state with the desired state in Git. Any manual changes (drift) are automatically reverted to match the Git manifests.
This is one of the most powerful features of GitOps. No more "someone changed something in production and we don't know what." Every change must go through Git to persist. Demonstrate this live if possible.

🏭 Step 9

Verisign CI/CD Context

🛠️ End-to-End Verisign Pipeline

Jenkins (CI) + FluxCD (CD)

Developer Push
➔
Jenkins Build
➔
Trivy Scan
➔
Push Image
➔
Update Git Manifests
➔
FluxCD Reconcile
➔
Cluster Updated

Jenkins Responsibilities

  • Build container images
  • Run security scans (Trivy)
  • Run tests
  • Push images to registry
  • Update image tags in Git manifests

FluxCD Responsibilities

  • Watch Git for manifest changes
  • Apply changes to EKS clusters
  • Enforce desired state
  • Detect and correct drift
  • Send notifications on events
At Verisign, Jenkins handles CI (build, test, scan, push) and then updates a manifests repository. FluxCD watches that repository and handles CD (deploying to EKS). This separation provides clear boundaries and better security since FluxCD runs inside the cluster.

🧹 Step 10

Clean Up

🗑️ Clean Up Resources


# Remove the Flux Kustomization (stops reconciliation)
kubectl delete kustomization my-api -n flux-system

# Remove the HelmRelease
kubectl delete helmrelease my-api -n production

# Verify resources are removed
flux get kustomizations
flux get helmreleases --all-namespaces

# Clean up the local git clone
cd ..
rm -rf my-api-manifests
Note: If prune was enabled, deleting the Kustomization will also delete the managed resources (Deployments, Services, etc.) from the cluster.
Remind students that prune: true means managed resources get garbage collected when the Kustomization is deleted. This is intentional — it keeps the cluster clean.

📚 Summary

Aspect Push-Based (Jenkins) Pull-Based (FluxCD)
Trigger External CI system Git commit / polling interval
Credentials Cluster creds in CI Git creds in cluster
Reconciliation One-time apply Continuous loop
Drift Handling Not detected Auto-corrected
Rollback kubectl rollout undo git revert
Audit Trail CI logs Git history
