📜 Part A

kubectl Logs

📦 Deploy a Sample Application

Create an application that generates log output:


apiVersion: apps/v1
kind: Deployment
metadata:
  name: log-generator
  namespace: default
spec:
  replicas: 2
  selector:
    matchLabels:
      app: log-generator
  template:
    metadata:
      labels:
        app: log-generator
    spec:
      containers:
      - name: logger
        image: busybox:1.36
        command: ["/bin/sh", "-c"]
        args:
          - |
            i=0
            while true; do
              echo "$(date -Iseconds) level=info msg=\"Request processed\" request_id=$i status=200 duration_ms=$((RANDOM % 500))"
              if [ $((i % 10)) -eq 0 ]; then
                echo "$(date -Iseconds) level=error msg=\"Database connection timeout\" request_id=$i error_code=504"
              fi
              i=$((i+1))
              sleep 2
            done
      - name: sidecar
        image: busybox:1.36
        command: ["/bin/sh", "-c"]
        args:
          - |
            while true; do
              echo "$(date -Iseconds) sidecar health-check OK"
              sleep 10
            done
                

kubectl apply -f log-generator.yaml
kubectl get pods -l app=log-generator
This deployment has two containers per pod: a main logger that generates simulated request logs and occasional errors, and a sidecar that emits health checks. This is useful for demonstrating multi-container log access.

🔎 Essential Log Commands


# Basic log output (default container)
kubectl logs deploy/log-generator

# Follow logs in real-time (Ctrl+C to stop)
kubectl logs deploy/log-generator -f

# View logs from a crashed/restarted container
kubectl logs <pod-name> --previous

# Filter by label selector (all pods matching label)
kubectl logs -l app=log-generator

# Time-based filtering
kubectl logs deploy/log-generator --since=5m
kubectl logs deploy/log-generator --since-time="2024-01-15T10:00:00Z"

# Specific container in a multi-container pod
kubectl logs <pod-name> -c logger
kubectl logs <pod-name> -c sidecar

# Tail last N lines
kubectl logs deploy/log-generator --tail=20

# All containers in a pod
kubectl logs <pod-name> --all-containers=true
Try each command. Notice how -c lets you target a specific container when pods have sidecars. The --previous flag is essential for debugging containers that have already crashed and restarted.
Have students try each command. The --previous flag is critical for CrashLoopBackOff debugging. Emphasize that logs are ephemeral — when a pod is deleted, its logs are gone unless shipped to a log aggregator like Splunk.

📊 Part B

Structured Logging

📄 Deploy a JSON Logger


apiVersion: apps/v1
kind: Deployment
metadata:
  name: json-logger
  namespace: default
spec:
  replicas: 1
  selector:
    matchLabels:
      app: json-logger
  template:
    metadata:
      labels:
        app: json-logger
    spec:
      containers:
      - name: app
        image: busybox:1.36
        command: ["/bin/sh", "-c"]
        args:
          - |
            i=0
            while true; do
              status=$((200 + (RANDOM % 3) * 100))
              dur=$((RANDOM % 800))
              echo "{\"timestamp\":\"$(date -Iseconds)\",\"level\":\"info\",\"service\":\"api\",\"method\":\"GET\",\"path\":\"/api/users\",\"status\":$status,\"duration_ms\":$dur,\"request_id\":\"req-$i\"}"
              i=$((i+1))
              sleep 1
            done
                

kubectl apply -f json-logger.yaml
Structured JSON logs are the standard at Verisign. They enable field extraction, filtering, and aggregation in Splunk without custom parsing rules.

🔍 Comparing Log Formats

Unstructured (text)


2024-01-15T10:30:00 level=info
  msg="Request processed"
  request_id=42 status=200
  duration_ms=127

Requires regex parsing

Structured (JSON)


{
  "timestamp": "2024-01-15T10:30:00",
  "level": "info",
  "service": "api",
  "method": "GET",
  "path": "/api/users",
  "status": 200,
  "duration_ms": 127,
  "request_id": "req-42"
}

Machine-parseable fields

Filter and extract fields with jq:


# Parse JSON logs and extract specific fields
kubectl logs deploy/json-logger | jq '.'

# Filter for errors only
kubectl logs deploy/json-logger | jq 'select(.status >= 400)'

# Extract specific fields
kubectl logs deploy/json-logger | jq '{path, status, duration_ms}'

# Calculate average duration
kubectl logs deploy/json-logger --tail=100 | jq -s 'map(.duration_ms) | add / length'
jq is an essential tool for working with JSON logs locally. In production, Splunk handles this automatically. The key point is that structured logs enable programmatic analysis.

🔴 RED Method Panels

Rate

Requests per second


rate(
  http_requests_total[5m]
)

How much traffic is your service handling?

Errors

Error rate percentage


rate(
  http_requests_total
    {code=~"5.."}[5m]
) / rate(
  http_requests_total[5m]
)

What percentage of requests fail?

Duration

Request latency (p99)


histogram_quantile(0.99,
  rate(
    http_request_duration_seconds_bucket[5m]
  )
)

How long do requests take?

RED Method: Rate, Errors, and Duration are the three golden signals for monitoring request-driven services. Every service dashboard should include these panels.
The RED method is the industry standard for microservice monitoring. Rate tells you traffic volume, Errors tell you reliability, Duration tells you performance. These three signals cover most alerting needs.

⚙️ Explore Dashboard Features

Practice These Actions

  1. Time range: Click the clock icon (top right) and set to "Last 30 minutes"
  2. Auto-refresh: Set refresh interval to 10s for live updates
  3. Template variables: Use the namespace/pod dropdowns to filter panels
  4. Panel inspect: Click a panel title → "Inspect" to see raw PromQL queries
  5. Explore: Click "Explore" in the sidebar to run ad-hoc PromQL queries
Try this query in Explore: 

sum(rate(container_cpu_usage_seconds_total{namespace="default"}[5m])) by (pod)
Let students spend a few minutes clicking around. The Inspect feature is particularly useful — it shows the exact PromQL query powering each panel, which helps when creating custom dashboards.

🚨 Part D

Prometheus Alerts

📝 Create a PrometheusRule


apiVersion: monitoring.coreos.com/v1
kind: PrometheusRule
metadata:
  name: api-alerts
  namespace: monitoring
  labels:
    # Must match the Prometheus ruleSelector
    release: kube-prometheus-stack
spec:
  groups:
  - name: api.rules
    rules:
    - alert: HighErrorRate
      expr: |
        rate(http_requests_total{code=~"5.."}[5m])
        / rate(http_requests_total[5m]) > 0.1
      for: 2m
      labels:
        severity: critical
        team: platform
      annotations:
        summary: "High 5xx error rate on {{ $labels.service }}"
        description: >
          Error rate is {{ $value | humanizePercentage }}
          for {{ $labels.service }} in {{ $labels.namespace }}.
        runbook_url: "https://wiki.verisign.com/runbooks/high-error-rate"

    - alert: HighLatency
      expr: |
        histogram_quantile(0.99,
          rate(http_request_duration_seconds_bucket[5m])
        ) > 1.0
      for: 5m
      labels:
        severity: warning
        team: platform
      annotations:
        summary: "P99 latency above 1s on {{ $labels.service }}"
                

kubectl apply -f api-alerts.yaml
Walk through the PrometheusRule structure: groups contain rules, each rule has an expr (PromQL), for (duration before firing), labels (routing), and annotations (human-readable info). The release label must match the Prometheus operator's ruleSelector.

⏰ Alert Lifecycle

Pending

Condition is true but for duration not yet elapsed

➔

Firing

Condition sustained for the for duration; alert sent

➔

Resolved

Condition no longer true; resolution notification sent

Verify the alert was created:


# List PrometheusRules
kubectl get prometheusrule -n monitoring

# Describe the rule
kubectl describe prometheusrule api-alerts -n monitoring

# Check in Prometheus UI (port-forward first)
kubectl port-forward svc/kube-prometheus-stack-prometheus \
  9090:9090 -n monitoring
# Navigate to http://localhost:9090/alerts
The "for" duration prevents flapping — a brief spike won't trigger an alert. At Verisign, critical alerts typically use "for: 2m" and warnings use "for: 5m". Show the alerts page in the Prometheus UI if possible.

🏭 Verisign Observability Stack

Production Architecture

🛠️ Logs: Cluster to Splunk

Log Pipeline Architecture


Pods (stdout/stderr)
       |
  Logging Operator (Fluentd/Fluentbit DaemonSet)
       |
  ClusterFlow (filtering, routing rules)
       |
  ClusterOutput (Splunk HEC endpoint)
       |
  Splunk (indexing, search, dashboards)

Example ClusterOutput for Splunk:


apiVersion: logging.banzaicloud.io/v1beta1
kind: ClusterOutput
metadata:
  name: splunk-output
spec:
  splunkHec:
    hec_host: splunk-hec.verisign.com
    hec_port: 8088
    hec_token:
      valueFrom:
        secretKeyRef:
          name: splunk-hec-token
          key: token
    index: kubernetes_logs
    source: kubernetes
    insecure_ssl: false
At Verisign, the Logging Operator runs as a DaemonSet collecting logs from all pods. ClusterFlows define filtering and routing rules. ClusterOutputs define where logs go — in this case, Splunk via HTTP Event Collector (HEC).

🚨 Alerts: Prometheus to Netcool

Alert Routing Architecture


PrometheusRule (alert definition)
       |
  Prometheus (evaluates rules)
       |
  Alertmanager (routing, grouping, silencing)
       |
  +--> Netcool (critical alerts, on-call)
  +--> Email (warning alerts, team inbox)
  +--> WebEx (notifications, team channels)
Routing by severity: critical alerts go to Netcool for immediate on-call response. warning alerts go to email and WebEx for team awareness.
Alertmanager handles the routing logic. At Verisign, the alert severity label determines which receiver gets notified. Critical alerts page the on-call engineer via Netcool, while warnings are informational.
