⚠️ The Insecure Dockerfile

Create Dockerfile.insecure:

FROM ubuntu:latest

RUN apt-get update && apt-get install -y \
    python3 \
    python3-pip \
    curl \
    vim \
    net-tools \
    ssh

COPY . .

ENV DATABASE_PASSWORD=supersecret123
ENV API_KEY=ak_live_abcdef123456

RUN pip3 install flask requests

EXPOSE 5000

CMD ["python3", "app.py"]
This Dockerfile has at least 8 security issues. Have students identify them before revealing the checklist on the next slide.

🔍 Security Issue Checklist

# Issue Severity
1 FROM ubuntu:latest — unpinned, mutable tag HIGH
2 Full OS image — large attack surface HIGH
3 Unnecessary packages: vim, net-tools, ssh MEDIUM
4 Runs as root (no USER directive) CRITICAL
5 COPY . . — no .dockerignore, copies everything MEDIUM
6 Secrets in ENV (DATABASE_PASSWORD, API_KEY) CRITICAL
7 No HEALTHCHECK instruction LOW
8 Single-stage build — includes build tools in final image MEDIUM
Walk through each issue. The critical ones are running as root and embedding secrets. These are the most common mistakes in production.

Step 2: Harden the Dockerfile

1

Now we rewrite the Dockerfile applying every best practice. This is the reference pattern students should follow.

📝 Create .dockerignore

First, create a .dockerignore file:

.git
.gitignore
.env
*.md
__pycache__
*.pyc
.pytest_cache
docker-compose*.yml
Dockerfile*
.dockerignore
node_modules
.vscode
.idea
secrets/
credentials/

💡 Why: Without a .dockerignore, COPY . . sends your entire build context to the Docker daemon — including .git history, secrets files, and IDE configs.

The .dockerignore works exactly like .gitignore. It prevents files from being included in the build context. This is both a security and performance optimization.

✅ The Hardened Dockerfile

Create Dockerfile.secure:

# Stage 1: Build
FROM python:3.12-slim AS builder

WORKDIR /build

COPY requirements.txt .
RUN pip install --no-cache-dir --prefix=/install -r requirements.txt

# Stage 2: Production
FROM python:3.12-alpine3.19

LABEL maintainer="platform-team@verisign.com"
LABEL org.opencontainers.image.source="https://github.com/verisign/myapp"

# Create non-root user
RUN addgroup -g 1001 -S appgroup && \
    adduser -u 1001 -S appuser -G appgroup

WORKDIR /app

# Copy only installed dependencies from builder
COPY --from=builder /install /usr/local

# Copy application code with correct ownership
COPY --chown=appuser:appgroup app.py .
COPY --chown=appuser:appgroup templates/ ./templates/

# No secrets in the image - use Vault or K8s secrets at runtime
# ENV DATABASE_PASSWORD  ← NEVER do this

USER appuser:appgroup

EXPOSE 5000

HEALTHCHECK --interval=30s --timeout=3s --start-period=5s --retries=3 \
  CMD wget --no-verbose --tries=1 --spider http://localhost:5000/health || exit 1

CMD ["python3", "app.py"]
Walk through each improvement: multi-stage build, alpine base, non-root user, COPY --chown, no secrets, HEALTHCHECK. The builder stage is discarded - only the runtime image ships.

📊 Before vs. After Comparison

⚠️ Insecure Image

  • ~850 MB image size
  • Runs as root
  • Secrets embedded
  • 200+ CVEs (ubuntu:latest)
  • Unnecessary tools installed
→

✅ Secure Image

  • ~65 MB image size
  • Non-root user (UID 1001)
  • No secrets in image
  • Minimal CVEs (alpine)
  • Only runtime dependencies
The size reduction alone is dramatic - over 90% smaller. But the real win is the reduced attack surface and no root access.

Step 4: Pod Security Admission — Setup

1
2
3
4
5
6
7
Pod Security Admission replaced PodSecurityPolicy in Kubernetes 1.25. It's built-in and requires no additional controllers.

🔒 Pod Security Admission Levels

🟢 Privileged

Unrestricted. No security restrictions applied. For system-level workloads only.

🟡 Baseline

Minimally restrictive. Prevents known privilege escalations. Good default for most workloads.

🔴 Restricted

Heavily restricted. Current best practices for pod hardening. Required for security-sensitive workloads.

💡 PSA Modes

  • enforce — Rejects pods that violate the policy
  • warn — Allows but shows a warning to the user
  • audit — Allows but records in the audit log
The recommended approach is to start with warn/audit on restricted, and enforce on baseline. Then gradually tighten to enforce restricted.

⚙️ Configure PSA on a Namespace

# Create lab namespace
kubectl create namespace lab08

# Apply PSA labels: enforce baseline, warn+audit restricted
kubectl label namespace lab08 \
  pod-security.kubernetes.io/enforce=baseline \
  pod-security.kubernetes.io/warn=restricted \
  pod-security.kubernetes.io/audit=restricted

# Verify labels
kubectl get namespace lab08 --show-labels

💡 What This Means

  • Pods violating baseline are rejected
  • Pods violating restricted show warnings and are audit logged
  • This lets teams see what would break before enforcing restricted
This graduated approach is key for production adoption. Teams can see the warnings first and fix their manifests before enforcement kicks in.

Step 5: Test PSA Enforcement

1
2
3
4
5
6
7
Now students see PSA in action. Three tests: privileged (rejected), hostNetwork (rejected), and compliant (accepted with warnings).

🚫 Test 1: Privileged Pod (Rejected)

# privileged-pod.yaml
apiVersion: v1
kind: Pod
metadata:
  name: privileged-test
  namespace: lab08
spec:
  containers:
  - name: test
    image: nginx:1.25-alpine
    securityContext:
      privileged: true
kubectl apply -f privileged-pod.yaml

🚨 Expected: REJECTED

Error from server (Forbidden): error when creating "privileged-pod.yaml":
pods "privileged-test" is forbidden: violates PodSecurity "baseline:latest":
privileged (container "test" must not set securityContext.privileged=true)
The error message is very descriptive - it tells you exactly which field violates which policy level. This is helpful for developers fixing their manifests.

🚫 Test 2: hostNetwork Pod (Rejected)

# hostnetwork-pod.yaml
apiVersion: v1
kind: Pod
metadata:
  name: hostnetwork-test
  namespace: lab08
spec:
  hostNetwork: true
  containers:
  - name: test
    image: nginx:1.25-alpine
kubectl apply -f hostnetwork-pod.yaml

🚨 Expected: REJECTED

Error from server (Forbidden): error when creating "hostnetwork-pod.yaml":
pods "hostnetwork-test" is forbidden: violates PodSecurity "baseline:latest":
host namespaces (hostNetwork=true)
hostNetwork gives the pod direct access to the node's network interfaces - a significant security risk. Baseline correctly blocks this.

✅ Test 3: Compliant Pod (Accepted with Warnings)

# compliant-pod.yaml
apiVersion: v1
kind: Pod
metadata:
  name: compliant-test
  namespace: lab08
spec:
  containers:
  - name: test
    image: nginx:1.25-alpine
    ports:
    - containerPort: 80
kubectl apply -f compliant-pod.yaml

⚠️ Expected: CREATED with Warnings

Warning: would violate PodSecurity "restricted:latest":
  allowPrivilegeEscalation != false
  unrestricted capabilities
  runAsNonRoot != true
  seccompProfile (pod or container must set)
pod/compliant-test created

💡 Key Insight: The pod was created (passes baseline) but shows warnings about what would fail under restricted enforcement.

This is the graduated enforcement in action. The pod deploys but the warnings tell developers exactly what to fix for restricted compliance.

Step 6: PSA in Restricted Mode

1
2
3
4
5

Now we create a namespace that enforces the restricted level and deploy a fully compliant pod.

🔒 Create Restricted Namespace

# Create a namespace with restricted enforcement
kubectl create namespace lab08-restricted

kubectl label namespace lab08-restricted \
  pod-security.kubernetes.io/enforce=restricted \
  pod-security.kubernetes.io/warn=restricted \
  pod-security.kubernetes.io/audit=restricted
Full restricted enforcement is the target state for production application namespaces at Verisign.

✅ Deploy a Fully Restricted-Compliant Pod

# restricted-pod.yaml
apiVersion: v1
kind: Pod
metadata:
  name: secure-app
  namespace: lab08-restricted
spec:
  securityContext:
    runAsNonRoot: true
    runAsUser: 1001
    runAsGroup: 1001
    fsGroup: 1001
    seccompProfile:
      type: RuntimeDefault
  containers:
  - name: app
    image: nginx:1.25-alpine
    ports:
    - containerPort: 8080
    securityContext:
      allowPrivilegeEscalation: false
      readOnlyRootFilesystem: true
      capabilities:
        drop:
        - ALL
    resources:
      requests:
        memory: "64Mi"
        cpu: "50m"
      limits:
        memory: "128Mi"
        cpu: "100m"
    volumeMounts:
    - name: tmp
      mountPath: /tmp
    - name: cache
      mountPath: /var/cache/nginx
    - name: run
      mountPath: /var/run
  volumes:
  - name: tmp
    emptyDir: {}
  - name: cache
    emptyDir: {}
  - name: run
    emptyDir: {}
Every field here is intentional: runAsNonRoot, seccompProfile, drop ALL capabilities, readOnlyRootFilesystem. The emptyDir volumes provide writable directories nginx needs without making the entire filesystem writable.

🧪 Deploy and Verify

# Deploy the restricted pod
kubectl apply -f restricted-pod.yaml
# Expected: pod/secure-app created (NO warnings)

# Verify it's running
kubectl get pod secure-app -n lab08-restricted

# Inspect the security context
kubectl get pod secure-app -n lab08-restricted -o jsonpath='{.spec.securityContext}' | jq .
kubectl get pod secure-app -n lab08-restricted -o jsonpath='{.spec.containers[0].securityContext}' | jq .

✅ Security Context Summary

  • runAsNonRoot: true — container cannot run as UID 0
  • allowPrivilegeEscalation: false — no setuid/setgid
  • capabilities.drop: [ALL] — no Linux capabilities
  • readOnlyRootFilesystem: true — immutable container filesystem
  • seccompProfile: RuntimeDefault — syscall filtering
This pod template is the gold standard for production workloads. Students should save this as a reference pattern.

Kyverno provides fine-grained policy enforcement beyond what PSA offers. It's a key part of the Verisign governance model.

🏛️ Kyverno: Image Registry Restriction

Only allow images from approved registries:

# restrict-registries.yaml
apiVersion: kyverno.io/v1
kind: ClusterPolicy
metadata:
  name: restrict-image-registries
  annotations:
    policies.kyverno.io/title: Restrict Image Registries
    policies.kyverno.io/description: >-
      Ensures all container images come from approved registries.
spec:
  validationFailureAction: Enforce
  background: true
  rules:
  - name: validate-registries
    match:
      any:
      - resources:
          kinds:
          - Pod
    validate:
      message: >-
        Images must come from an approved registry.
        Allowed: ECR (*.dkr.ecr.*.amazonaws.com), Docker Hub official.
      pattern:
        spec:
          containers:
          - image: "*.dkr.ecr.*.amazonaws.com/*"
In practice, Verisign restricts images to their private ECR registries. This prevents developers from accidentally deploying untrusted public images.

🏷️ Kyverno: Require Labels

# require-labels.yaml
apiVersion: kyverno.io/v1
kind: ClusterPolicy
metadata:
  name: require-labels
  annotations:
    policies.kyverno.io/title: Require Standard Labels
spec:
  validationFailureAction: Enforce
  background: true
  rules:
  - name: check-required-labels
    match:
      any:
      - resources:
          kinds:
          - Deployment
          - StatefulSet
    validate:
      message: >-
        Deployments and StatefulSets must have labels:
        app.kubernetes.io/name, app.kubernetes.io/version,
        app.kubernetes.io/managed-by
      pattern:
        metadata:
          labels:
            app.kubernetes.io/name: "?*"
            app.kubernetes.io/version: "?*"
            app.kubernetes.io/managed-by: "?*"
Standard labels are critical for observability. Prometheus, Grafana dashboards, and Splunk queries all depend on consistent labeling.

🚫 Kyverno: Block Latest Tag

# block-latest-tag.yaml
apiVersion: kyverno.io/v1
kind: ClusterPolicy
metadata:
  name: block-latest-tag
  annotations:
    policies.kyverno.io/title: Block Latest Tag
spec:
  validationFailureAction: Enforce
  background: true
  rules:
  - name: validate-image-tag
    match:
      any:
      - resources:
          kinds:
          - Pod
    validate:
      message: >-
        Using ':latest' or untagged images is not allowed.
        Pin images to a specific version tag or SHA digest.
      pattern:
        spec:
          containers:
          - image: "!*:latest"
# Apply the policies (use Audit mode for testing)
kubectl apply -f restrict-registries.yaml
kubectl apply -f require-labels.yaml
kubectl apply -f block-latest-tag.yaml

# Verify policies are installed
kubectl get clusterpolicies

🏛️ Verisign Governance Model

These Kyverno policies complement PSA by enforcing organization-specific standards that go beyond generic pod security: approved registries, mandatory labels, and image pinning.

In production, these policies are managed via FluxCD GitOps. Platform engineering maintains the policies in a central repo and they're automatically applied across all clusters.

📝 Summary — Container Security Checklist

📦 Image Build — Multi-stage builds, minimal base images, pinned versions, non-root user, no secrets
🔎 Scanning — Trivy in CI/CD, block images with CRITICAL CVEs, scan on schedule for new CVEs
🔒 Runtime — PSA restricted mode, readOnlyRootFilesystem, drop ALL capabilities, seccomp profiles
🏛️ Governance — Kyverno policies for registry, labels, and tag enforcement
🔐 Secrets — Never in images. Use Vault, K8s Secrets, or cert-manager for runtime injection
Recap the full security lifecycle: build, scan, deploy, enforce. Each layer adds defense in depth.
