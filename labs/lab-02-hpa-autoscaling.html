📈 Lab 2: Horizontal Pod Autoscaling

Intermediate Kubernetes — Hands-On Lab

Duration: ~45 minutes

This lab is where participants see the HPA in action. It requires a working metrics-server and some patience for the scale-down phase. Consider having two terminal windows ready — one for the load generator and one for watching the HPA.

📋 Lab Overview

Objectives

  • Verify the metrics-server is operational
  • Deploy a CPU-intensive target application
  • Create an HPA using both imperative and declarative approaches
  • Generate load and observe automatic scale-up
  • Remove load and observe the stabilization window during scale-down

⏱ Duration

~45 minutes

✅ Prerequisites

  • kubectl configured with cluster access
  • Metrics-server running in the cluster
  • Two terminal windows available
Remind participants this lab takes longer because we need to wait for metrics collection (15-30 second intervals) and the scale-down stabilization window (5 minutes by default). Use the wait time for discussion.

▶ Step 1

Verify Metrics Server

📊 Verify Metrics Server

The HPA relies on metrics from the metrics-server to make scaling decisions.

Check node-level metrics


kubectl top nodes

Check pod-level metrics across all namespaces


kubectl top pods -A

Verify the metrics-server deployment


kubectl get deployment metrics-server -n kube-system
✅ Expected: All commands return data without errors. If kubectl top returns "metrics not available," wait 60 seconds and retry — the metrics pipeline may need time to initialize.
If metrics-server is not deployed, you can install it with `kubectl apply -f https://github.com/kubernetes-sigs/metrics-server/releases/latest/download/components.yaml`. On EKS, it should already be present.

▶ Step 2

Deploy the Target Application

📦 Target Application Manifest

Create lab02-php-apache.yaml — a CPU-intensive app for load testing:


apiVersion: apps/v1
kind: Deployment
metadata:
  name: php-apache
  labels:
    app: php-apache
spec:
  replicas: 1
  selector:
    matchLabels:
      app: php-apache
  template:
    metadata:
      labels:
        app: php-apache
    spec:
      containers:
      - name: php-apache
        image: registry.k8s.io/hpa-example
        ports:
        - containerPort: 80
        resources:
          requests:
            cpu: 200m
            memory: 128Mi
          limits:
            cpu: 500m
            memory: 256Mi
💡 Key Point: Resource requests are critical for HPA. The autoscaler calculates CPU utilization as a percentage of the requested CPU, not the limit.
The hpa-example image runs a simple PHP page that performs sqrt() in a loop on each request, making it CPU-intensive. The 200m CPU request gives us a clear baseline for percentage calculations.

🚀 Deploy the Application


kubectl apply -f lab02-php-apache.yaml

Verify the pod is running


kubectl get pods -l app=php-apache

Confirm metrics are being collected


# Wait ~30 seconds after pod starts, then:
kubectl top pod -l app=php-apache
✅ Expected: Pod is Running and kubectl top shows a small CPU value (1-3m at idle).
If kubectl top shows "metrics not yet available," the pod just started. Metrics are scraped every 15 seconds and need at least one collection cycle.

▶ Step 3

Expose the Application

🌐 Service for the Target App

Create lab02-service.yaml:


apiVersion: v1
kind: Service
metadata:
  name: php-apache
  labels:
    app: php-apache
spec:
  type: ClusterIP
  selector:
    app: php-apache
  ports:
  - port: 80
    targetPort: 80
    protocol: TCP
                

kubectl apply -f lab02-service.yaml
kubectl get svc php-apache
💡 Key Point: The load generator will target this Service DNS name: php-apache.default.svc.cluster.local. The Service distributes traffic across all pods managed by the HPA.
The Service is essential — our load generator will send HTTP requests to this ClusterIP. As the HPA adds replicas, the Service automatically includes them in its endpoint list.

▶ Step 4

Create an HPA

⚙️ Imperative Approach

The quickest way to create an HPA for testing:


kubectl autoscale deployment php-apache \
  --cpu-percent=50 \
  --min=1 \
  --max=10

Verify the HPA


kubectl get hpa php-apache
⚠️ Note: If you see <unknown>/50% in the TARGETS column, wait 15-30 seconds for the first metrics scrape. It should resolve to something like 0%/50%.
The imperative approach is fast but not GitOps-friendly. We'll show the declarative YAML next, which is what you would commit to a FluxCD repository in the Verisign platform.

📄 Declarative Approach (HPA YAML)

For production, use a declarative manifest. Delete the imperative HPA first:


kubectl delete hpa php-apache

Create lab02-hpa.yaml:


apiVersion: autoscaling/v2
kind: HorizontalPodAutoscaler
metadata:
  name: php-apache
spec:
  scaleTargetRef:
    apiVersion: apps/v1
    kind: Deployment
    name: php-apache
  minReplicas: 1
  maxReplicas: 10
  metrics:
  - type: Resource
    resource:
      name: cpu
      target:
        type: Utilization
        averageUtilization: 50
                

kubectl apply -f lab02-hpa.yaml
kubectl get hpa php-apache
Highlight the use of autoscaling/v2 — this is the current stable API version that supports multiple metrics and custom metrics. The v1 API only supported CPU percentage.

▶ Step 5

Generate Load

🔥 Generate Load

You will need two terminal windows

  • Terminal 1: Run the load generator
  • Terminal 2: Watch the HPA scale

Terminal 1 — Start the load generator


kubectl run load-generator \
  --image=busybox:1.36 \
  --restart=Never \
  -- /bin/sh -c "while sleep 0.01; do \
     wget -q -O- http://php-apache; done"

Terminal 2 — Watch the HPA


kubectl get hpa php-apache -w
💡 Key Point: The load generator runs inside the cluster, so it can reach the ClusterIP Service directly. Each iteration sends one HTTP request every 10ms.
Give this 1-2 minutes. You should see CPU climb past 50% and the replica count increase. Walk through what participants are observing — this is the "aha" moment of the lab.

▶ Step 6

Observe Scale-Up

📈 Watch Replicas Increase

Check the pod count


kubectl get pods -l app=php-apache

Get detailed HPA status


kubectl describe hpa php-apache

🧮 The HPA Algorithm

The controller computes desired replicas using:


desiredReplicas = ceil[ currentReplicas × (currentMetricValue / desiredMetricValue) ]

Example: 1 replica at 250% CPU (target 50%) → ceil(1 × 250/50) = 5 replicas

Walk through the algorithm with real numbers from the HPA output. Ask participants to calculate the expected replica count themselves using the formula. Point out the Events section in `describe hpa` which logs each scaling decision.

🔍 Inspect the Scale Events

Look at the Events section of kubectl describe hpa:

Sample Events Output


Type    Reason             Age   Message
----    ------             ----  -------
Normal  SuccessfulRescale  2m    New size: 4; reason: cpu resource
                                 utilization (percentage of request)
                                 above target
Normal  SuccessfulRescale  1m    New size: 7; reason: cpu resource
                                 utilization (percentage of request)
                                 above target
💡 Key Point: The HPA evaluates metrics every 15 seconds by default (controlled by --horizontal-pod-autoscaler-sync-period on the controller manager).
Point out that the HPA may overshoot initially, then converge. The controller uses a tolerance of 0.1 (10%) — if the ratio is within 10% of the target, it won't scale. This prevents thrashing.

▶ Step 7

Observe Scale-Down

🕗 Remove Load and Watch Scale-Down

Stop the load generator


kubectl delete pod load-generator

Continue watching the HPA


kubectl get hpa php-apache -w
⚠️ Be patient: Scale-down does not happen immediately. The default stabilization window is 5 minutes (--horizontal-pod-autoscaler-downscale-stabilization). This prevents flapping during intermittent traffic spikes.

🕒 Scale-Down Stabilization

  • The HPA remembers the highest recommended replica count from the past 5 minutes
  • It will not scale below that number until the window expires
  • This is asymmetric by design — scale-up is fast, scale-down is conservative
Use the 5-minute wait time to discuss real-world scaling patterns. Ask: "Why would you want asymmetric scaling behavior?" Answer: sudden load spikes are common, and scaling up quickly is more important than saving a few minutes of extra capacity.

▶ Step 8

Clean Up

🗑️ Clean Up Resources


kubectl delete -f lab02-hpa.yaml
kubectl delete -f lab02-service.yaml
kubectl delete -f lab02-php-apache.yaml

# Verify clean up
kubectl get all -l app=php-apache
kubectl get hpa
✅ Expected: No resources found for the php-apache label.
Make sure everyone also deleted the load-generator pod from Step 7. Quick check: `kubectl get pod load-generator` should return not found.

🏆 Optional Challenge

Extend your HPA skills

💪 Challenge: Advanced HPA Configuration

Try one or both of these stretch goals:

Challenge A: Memory-Based Scaling

Add a memory metric to your HPA:


metrics:
- type: Resource
  resource:
    name: memory
    target:
      type: Utilization
      averageUtilization: 70

Hint: With multiple metrics, the HPA scales to the highest recommendation.

Challenge B: Custom Scale-Down Behavior

Add a behavior block to control scaling speed:


behavior:
  scaleDown:
    stabilizationWindowSeconds: 120
    policies:
    - type: Percent
      value: 25
      periodSeconds: 60
  scaleUp:
    stabilizationWindowSeconds: 0
    policies:
    - type: Percent
      value: 100
      periodSeconds: 15

This limits scale-down to 25% of pods per minute.

These challenges are for participants who finish early. Challenge B is particularly relevant for production — the behavior API gives fine-grained control over scaling velocity. Discuss how Verisign might tune these for their workloads.
