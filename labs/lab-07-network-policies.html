Welcome to Lab 7. This is a 45-60 minute hands-on lab where students will implement network policies step-by-step. Make sure all students have Calico CNI confirmed before starting.

📋 Lab Overview

Objectives

  • Deploy a multi-tier application with proper labeling
  • Observe default allow-all behavior in Kubernetes networking
  • Implement a default-deny-all network policy
  • Selectively allow traffic between tiers using NetworkPolicy resources
  • Configure DNS egress rules and namespace-based isolation
  • Verify a least-privilege network security model

⏱️ Duration

45 – 60 minutes

⚙️ Prerequisites

  • EKS cluster access with kubectl
  • Calico CNI installed and operational
  • Permissions to create namespaces and NetworkPolicies
Confirm Calico is running: kubectl get pods -n calico-system. If students don't see Calico pods, they need to resolve that first. Network policies without a CNI that supports them will have no effect.

📦 Create the Namespace

First, create a dedicated namespace for this lab:

kubectl create namespace lab07
kubectl config set-context --current --namespace=lab07
Setting the namespace context avoids having to add -n lab07 to every command.

🌐 Deploy the Frontend (nginx)

Create frontend.yaml:

apiVersion: apps/v1
kind: Deployment
metadata:
  name: frontend
  namespace: lab07
  labels:
    app: frontend
    tier: frontend
spec:
  replicas: 1
  selector:
    matchLabels:
      app: frontend
      tier: frontend
  template:
    metadata:
      labels:
        app: frontend
        tier: frontend
    spec:
      containers:
      - name: nginx
        image: nginx:1.25-alpine
        ports:
        - containerPort: 80
---
apiVersion: v1
kind: Service
metadata:
  name: frontend
  namespace: lab07
spec:
  selector:
    app: frontend
    tier: frontend
  ports:
  - port: 80
    targetPort: 80
  type: ClusterIP
kubectl apply -f frontend.yaml
Note the dual labels: app and tier. Both will be useful for network policy selectors. We use alpine-based images for smaller footprint.

🔍 Test Default Connectivity

Without any network policies, all pods can communicate freely:

# Frontend → Backend (should succeed)
kubectl exec -n lab07 deploy/frontend -- \
  curl -s --max-time 5 http://backend:8080

# Backend → Database (should succeed)
kubectl exec -n lab07 deploy/backend -- \
  sh -c "nc -z -w3 database 6379 && echo 'Connection successful' || echo 'Connection failed'"

# Frontend → Database directly (should succeed)
kubectl exec -n lab07 deploy/frontend -- \
  sh -c "nc -z -w3 database 6379 && echo 'Connection successful' || echo 'Connection failed'"
All three tests should succeed. Use the nc (netcat) command for Redis since it's not HTTP. The -z flag is for scanning, -w3 sets a 3 second timeout.

📊 Connectivity Matrix — Before Policies

Source Destination Port Result
Frontend Backend 8080 ✅ ALLOWED
Backend Database 6379 ✅ ALLOWED
Frontend Database 6379 ✅ ALLOWED

⚠️ Security Concern: The frontend can reach the database directly, bypassing the backend. In a real application, this violates the principle of least privilege.

Draw attention to the frontend-to-database path. This is exactly what we want to block with network policies.

Step 3: Apply Default Deny All

1

This is the most important step conceptually. Default deny is the foundation of a zero-trust network model.

🚫 Default Deny Network Policy

Create default-deny-all.yaml:

apiVersion: networking.k8s.io/v1
kind: NetworkPolicy
metadata:
  name: default-deny-all
  namespace: lab07
spec:
  podSelector: {}
  policyTypes:
  - Ingress
  - Egress
kubectl apply -f default-deny-all.yaml

💡 How It Works

  • podSelector: {} — matches all pods in the namespace
  • Both Ingress and Egress in policyTypes with no rules — blocks everything
The empty podSelector is the key. It selects all pods in the namespace. Since we specify both Ingress and Egress in policyTypes but provide no rules, all traffic is denied.

🚨 Test — All Connections Blocked

# Frontend → Backend (should FAIL)
kubectl exec -n lab07 deploy/frontend -- \
  curl -s --max-time 5 http://backend:8080
# Expected: connection timeout

# Backend → Database (should FAIL)
kubectl exec -n lab07 deploy/backend -- \
  sh -c "nc -z -w3 database 6379 && echo 'Connection successful' || echo 'Connection failed'"
# Expected: Connection failed

# Frontend → Database (should FAIL)
kubectl exec -n lab07 deploy/frontend -- \
  sh -c "nc -z -w3 database 6379 && echo 'Connection successful' || echo 'Connection failed'"
# Expected: Connection failed

🔒 Result: All connections are now blocked. We have achieved a zero-trust baseline.

All tests should timeout or fail. If any succeed, Calico may not be properly installed. DNS resolution will also fail at this point, so curl may show a DNS error rather than a timeout.

Now we selectively open traffic. This is the allow-list approach: deny everything, then permit only what's needed.

✅ Allow Frontend to Backend Policy

Create allow-frontend-to-backend.yaml:

apiVersion: networking.k8s.io/v1
kind: NetworkPolicy
metadata:
  name: allow-frontend-to-backend
  namespace: lab07
spec:
  podSelector:
    matchLabels:
      tier: backend
  policyTypes:
  - Ingress
  ingress:
  - from:
    - podSelector:
        matchLabels:
          tier: frontend
    ports:
    - protocol: TCP
      port: 8080
kubectl apply -f allow-frontend-to-backend.yaml

💡 Policy Breakdown

  • Target: pods with tier: backend
  • Allowed ingress: from pods with tier: frontend on port 8080
Note that this policy only opens ingress to the backend. The frontend also needs egress permission, but we'll handle that with the DNS policy in Step 6. For now, this alone won't make curl work because DNS is still blocked.

🧪 Test Frontend → Backend

⚠️ Note: This test may still fail because DNS egress is blocked. We will fix that in Step 6. For now, test using the backend pod IP directly:

# Get the backend pod IP
BACKEND_IP=$(kubectl get pod -n lab07 -l tier=backend \
  -o jsonpath='{.items[0].status.podIP}')

# Test using IP (bypasses DNS)
kubectl exec -n lab07 deploy/frontend -- \
  curl -s --max-time 5 http://${BACKEND_IP}:8080
# Expected: "Hello from backend"

# Frontend → Database should still FAIL
kubectl exec -n lab07 deploy/frontend -- \
  sh -c "nc -z -w3 database 6379 && echo 'Connection successful' || echo 'Connection failed'"
If testing with IP also fails, ensure the frontend has egress allowed. With a strict default-deny on both ingress and egress, you need both sides. We'll add a broader egress allowance in Step 6 with DNS.

DNS is often overlooked when implementing network policies. Without DNS egress, pods can't resolve service names.

🌐 Why DNS Matters

⚠️ The DNS Problem

With default deny on egress, pods cannot reach kube-dns in the kube-system namespace. This means service name resolution (e.g., backend.lab07.svc.cluster.local) fails entirely.

Create allow-dns.yaml:

apiVersion: networking.k8s.io/v1
kind: NetworkPolicy
metadata:
  name: allow-dns
  namespace: lab07
spec:
  podSelector: {}
  policyTypes:
  - Egress
  egress:
  - to:
    - namespaceSelector: {}
    ports:
    - protocol: UDP
      port: 53
    - protocol: TCP
      port: 53
kubectl apply -f allow-dns.yaml
We allow egress to port 53 (DNS) in any namespace. This is a common pattern since kube-dns lives in kube-system. Both UDP and TCP on port 53 are needed - UDP for standard lookups, TCP for large responses.

🔍 Also Allow Egress for Application Traffic

We also need egress policies so pods can initiate connections. Create allow-egress-frontend.yaml:

apiVersion: networking.k8s.io/v1
kind: NetworkPolicy
metadata:
  name: allow-egress-frontend
  namespace: lab07
spec:
  podSelector:
    matchLabels:
      tier: frontend
  policyTypes:
  - Egress
  egress:
  - to:
    - podSelector:
        matchLabels:
          tier: backend
    ports:
    - protocol: TCP
      port: 8080
---
apiVersion: networking.k8s.io/v1
kind: NetworkPolicy
metadata:
  name: allow-egress-backend
  namespace: lab07
spec:
  podSelector:
    matchLabels:
      tier: backend
  policyTypes:
  - Egress
  egress:
  - to:
    - podSelector:
        matchLabels:
          tier: database
    ports:
    - protocol: TCP
      port: 6379
kubectl apply -f allow-egress-frontend.yaml
Egress policies are the counterpart to ingress. Both must be allowed for a connection to succeed. This gives us tight control over who can initiate which connections.

✅ Test with DNS Resolution

# Frontend → Backend via service name (should now work)
kubectl exec -n lab07 deploy/frontend -- \
  curl -s --max-time 5 http://backend:8080
# Expected: "Hello from backend"

# Backend → Database via service name (should now work)
kubectl exec -n lab07 deploy/backend -- \
  sh -c "nc -z -w3 database 6379 && echo 'Connection successful' || echo 'Connection failed'"
# Expected: Connection successful

# Frontend → Database (should STILL fail)
kubectl exec -n lab07 deploy/frontend -- \
  sh -c "nc -z -w3 database 6379 && echo 'Connection successful' || echo 'Connection failed'"
# Expected: Connection failed

✅ Success: Service-name-based communication now works for allowed paths, while unauthorized paths remain blocked.

This is the "aha" moment for most students - seeing DNS resolution work alongside tight network policies. Emphasize that both ingress and egress must be allowed.

Namespace selectors are essential for cross-namespace communication like Prometheus scraping.

📡 Create Monitoring Namespace & Test Pod

# Create monitoring namespace with a label
kubectl create namespace monitoring
kubectl label namespace monitoring purpose=monitoring

# Deploy a test pod in monitoring namespace
kubectl run monitor-probe --namespace=monitoring \
  --image=busybox:1.36 --restart=Never \
  -- sleep 3600

# Wait for pod to be ready
kubectl wait --for=condition=Ready pod/monitor-probe \
  -n monitoring --timeout=60s

💡 Key Concept: The namespace label purpose=monitoring is what our network policy will select on. This mirrors how Prometheus accesses application pods in the Verisign stack.

In Verisign's environment, Prometheus runs in a monitoring namespace. It needs to scrape metrics from application pods across namespaces. This pattern demonstrates how to allow that.

🔒 Allow Ingress from Monitoring Namespace

Create allow-monitoring-ingress.yaml:

apiVersion: networking.k8s.io/v1
kind: NetworkPolicy
metadata:
  name: allow-monitoring-ingress
  namespace: lab07
spec:
  podSelector: {}
  policyTypes:
  - Ingress
  ingress:
  - from:
    - namespaceSelector:
        matchLabels:
          purpose: monitoring
    ports:
    - protocol: TCP
      port: 8080
    - protocol: TCP
      port: 80
    - protocol: TCP
      port: 6379
kubectl apply -f allow-monitoring-ingress.yaml
The namespaceSelector selects namespaces by label, not by name. This is more flexible and survives namespace recreation. We open all service ports to the monitoring namespace.

🧪 Verify Monitoring Access

# Monitoring → Backend (should succeed)
kubectl exec -n monitoring monitor-probe -- \
  wget -qO- --timeout=5 http://backend.lab07.svc.cluster.local:8080
# Expected: "Hello from backend"

# Monitoring → Frontend (should succeed)
kubectl exec -n monitoring monitor-probe -- \
  wget -qO- --timeout=5 http://frontend.lab07.svc.cluster.local:80
# Expected: nginx welcome page HTML

✅ Result: Pods in the monitoring namespace can reach our application pods, while pods without the correct namespace label remain blocked.

Note the use of fully qualified service names since we're crossing namespace boundaries. The monitoring pod uses wget since busybox doesn't include curl.

📝 Summary — Network Policy Patterns

🔒 Default Deny

Empty podSelector with no rules blocks all traffic. Always start here.

✅ Allow by Label

Use podSelector to target pods and allow specific sources on specific ports.

🌍 Namespace Selector

Use namespaceSelector for cross-namespace access like monitoring and logging.

🐅 Calico-Specific Features (Beyond Standard K8s)

  • GlobalNetworkPolicy — cluster-wide policies without namespace scope
  • NetworkSet — define CIDR-based IP ranges for external traffic control
  • Tier-based ordering — hierarchical policy evaluation (security → platform → application)
  • DNS-based policies — allow/deny traffic based on domain names
Mention that in production, Verisign uses Calico's GlobalNetworkPolicies for cluster-wide baselines, and application teams layer on namespace-specific policies.

🔑 Key Concepts

1. Default Deny is Foundational — Always start with a deny-all policy, then selectively open paths.
2. Policies are Additive — Multiple NetworkPolicy resources are OR'd together. If any policy allows the traffic, it flows.
3. Both Ingress and Egress Matter — A connection requires both egress from the source and ingress to the destination.
4. DNS is Critical Infrastructure — Always allow egress to port 53 (UDP/TCP) for service discovery.
5. Labels Drive Everything — Consistent labeling on pods and namespaces is essential for effective policies.
These five takeaways are the core of network policy implementation. Reinforce that labels are the selectors for everything.
