Lab 13: Capstone — Multi-Tier Application Deployment

Integrating All Course Concepts

Duration: 60-90 minutes (Capstone Workshop)

This is the capstone lab integrating every concept from the course. Give students ample time — this is meant to be challenging. Circulate and assist as needed, but encourage students to refer back to previous lab materials before asking for help.

📋 Lab Overview

Capstone Objectives

  • Deploy a complete 3-tier application (Database, API, Frontend)
  • Configure persistent storage, ConfigMaps, and Secrets
  • Implement Network Policies for zero-trust segmentation
  • Configure Horizontal Pod Autoscaling
  • Add Prometheus monitoring and alerting
  • Set up Ingress with TLS and security hardening
This lab integrates Modules 1-12. Refer to previous labs for detailed explanations of individual concepts.
Set expectations: this is the most comprehensive lab. Students should work through it systematically, deploying each tier and verifying before moving to the next. Partial completion is fine — the goal is integration practice.

🏗️ Architecture Overview

Internet | +--------------+ | Ingress | | (TLS / L7) | +--------------+ / \ / (/) (/api) \ +-----------+ +-----------+ | Frontend | | API | | (Nginx) |----->| (Node) | | 2 repls | | 2 repls | +-----------+ +-----------+ | +-----------+ | Database | | (Postgres)| | StatefulSet| +-----------+ | +-----------+ | PVC | | (5Gi) | +-----------+
Three tiers: Frontend (Nginx) serves static content and proxies /api requests. API tier handles business logic. PostgreSQL provides persistent data storage via a StatefulSet with PVC.
Walk through the architecture diagram. Traffic enters through Ingress, routes to Frontend for / and to API for /api. Only the API tier talks to the database. Network policies will enforce these boundaries.

🗃️ PostgreSQL StatefulSet


# database/statefulset.yaml
apiVersion: apps/v1
kind: StatefulSet
metadata:
  name: postgres
  namespace: capstone
spec:
  serviceName: postgres-headless
  replicas: 1
  selector:
    matchLabels:
      app: postgres
      tier: database
  template:
    metadata:
      labels:
        app: postgres
        tier: database
    spec:
      containers:
      - name: postgres
        image: postgres:16-alpine
        ports:
        - containerPort: 5432
          name: postgresql
        envFrom:
        - secretRef:
            name: postgres-credentials
        volumeMounts:
        - name: postgres-data
          mountPath: /var/lib/postgresql/data
          subPath: pgdata
        - name: init-scripts
          mountPath: /docker-entrypoint-initdb.d
        resources:
          requests:
            cpu: 100m
            memory: 256Mi
          limits:
            cpu: 500m
            memory: 512Mi
        readinessProbe:
          exec:
            command:
              - pg_isready
              - -U
              - appuser
              - -d
              - capstone_db
          initialDelaySeconds: 10
          periodSeconds: 5
        livenessProbe:
          exec:
            command:
              - pg_isready
              - -U
              - appuser
              - -d
              - capstone_db
          initialDelaySeconds: 30
          periodSeconds: 10
      volumes:
      - name: init-scripts
        configMap:
          name: postgres-init
  volumeClaimTemplates:
  - metadata:
      name: postgres-data
    spec:
      accessModes: ["ReadWriteOnce"]
      resources:
        requests:
          storage: 5Gi
Key points: StatefulSet provides stable network identity and persistent storage. The subPath prevents PostgreSQL from complaining about a non-empty directory. pg_isready is the standard health check for PostgreSQL.

  name: api-config
  namespace: capstone
data:
  DB_HOST: "postgres.capstone.svc.cluster.local"

kubectl apply -f capstone/api/configmap.yaml
The DB_HOST uses the fully qualified DNS name. In practice, you can also use the short form "postgres" since both are in the same namespace. The explicit FQDN is clearer and works across namespaces.

⚙️ API Deployment


# api/deployment.yaml
apiVersion: apps/v1
kind: Deployment
metadata:
  name: api
  namespace: capstone
  labels:
    app: api
    tier: backend
spec:
  replicas: 2
  selector:
    matchLabels:
      app: api
      tier: backend
  template:
    metadata:
      labels:
        app: api
        tier: backend
    spec:
      containers:
      - name: api
        image: hashicorp/http-echo:0.2.3
        args:
          - "-listen=:3000"
          - '-text={"status":"ok","service":"capstone-api","version":"1.0.0"}'
        ports:
        - containerPort: 3000
          name: http
        envFrom:
        - configMapRef:
            name: api-config
        env:
        - name: DB_USER
          valueFrom:
            secretKeyRef:
              name: postgres-credentials
              key: POSTGRES_USER
        - name: DB_PASSWORD
          valueFrom:
            secretKeyRef:
              name: postgres-credentials
              key: POSTGRES_PASSWORD
        resources:
          requests:
            cpu: 100m
            memory: 128Mi
          limits:
            cpu: 500m
            memory: 256Mi
        readinessProbe:
          httpGet:
            path: /
            port: 3000
          initialDelaySeconds: 5
          periodSeconds: 10
        livenessProbe:
          httpGet:
            path: /
            port: 3000
          initialDelaySeconds: 10
          periodSeconds: 15
      securityContext:
        runAsNonRoot: true
        runAsUser: 1000
We use http-echo as a lightweight API stand-in. It responds on port 3000 with JSON. Environment variables come from both ConfigMap (non-sensitive) and Secret (credentials). Note the pod-level securityContext.

🎨 Step 4

Deploy the Frontend Tier

📄 Frontend Nginx Config


# frontend/configmap.yaml
apiVersion: v1
kind: ConfigMap
metadata:
  name: frontend-nginx-config
  namespace: capstone
data:
  default.conf: |
    server {
        listen 80;
        server_name _;

        location / {
            root /usr/share/nginx/html;
            index index.html;
            try_files $uri $uri/ /index.html;
        }

        location /api/ {
            proxy_pass http://api.capstone.svc.cluster.local:3000/;
            proxy_set_header Host $host;
            proxy_set_header X-Real-IP $remote_addr;
            proxy_set_header X-Forwarded-For $proxy_add_x_forwarded_for;
            proxy_set_header X-Forwarded-Proto $scheme;
        }

        location /health {
            access_log off;
            return 200 '{"status":"ok"}';
            add_header Content-Type application/json;
        }
    }
  index.html: |
    
    
    Capstone App
    
      
Capstone Multi-Tier Application

      
Frontend served by Nginx

      
Loading API status...
The nginx config proxies /api/ requests to the API service. This is a common frontend pattern — the frontend serves static assets and proxies API calls. The /health endpoint is for probes.

🎨 Frontend Deployment & Service


# frontend/deployment.yaml
apiVersion: apps/v1
kind: Deployment
metadata:
  name: frontend
  namespace: capstone
  labels:
    app: frontend
    tier: frontend
spec:
  replicas: 2
  selector:
    matchLabels:
      app: frontend
      tier: frontend
  template:
    metadata:
      labels:
        app: frontend
        tier: frontend
    spec:
      containers:
      - name: nginx
        image: nginx:1.25-alpine
        ports:
        - containerPort: 80
          name: http
        volumeMounts:
        - name: nginx-config
          mountPath: /etc/nginx/conf.d
        - name: html-content
          mountPath: /usr/share/nginx/html
        resources:
          requests:
            cpu: 50m
            memory: 64Mi
          limits:
            cpu: 200m
            memory: 128Mi
        readinessProbe:
          httpGet:
            path: /health
            port: 80
          initialDelaySeconds: 3
          periodSeconds: 5
        livenessProbe:
          httpGet:
            path: /health
            port: 80
          initialDelaySeconds: 5
          periodSeconds: 10
      volumes:
      - name: nginx-config
        configMap:
          name: frontend-nginx-config
          items:
          - key: default.conf
            path: default.conf
      - name: html-content
        configMap:
          name: frontend-nginx-config
          items:
          - key: index.html
            path: index.html
Two volume mounts from the same ConfigMap: one for the nginx config and one for the HTML content. In production, the HTML would come from a built container image.

🔒 Step 5

Configure Network Policies

🔒 Default Deny & Tier Policies


# network/network-policies.yaml
# Default deny all ingress and egress in the namespace
apiVersion: networking.k8s.io/v1
kind: NetworkPolicy
metadata:
  name: default-deny-all
  namespace: capstone
spec:
  podSelector: {}
  policyTypes:
  - Ingress
  - Egress
---
# Allow DNS egress for all pods
apiVersion: networking.k8s.io/v1
kind: NetworkPolicy
metadata:
  name: allow-dns
  namespace: capstone
spec:
  podSelector: {}
  policyTypes:
  - Egress
  egress:
  - to:
    - namespaceSelector: {}
    ports:
    - protocol: UDP
      port: 53
    - protocol: TCP
      port: 53
Start with deny-all, then add specific allows. DNS egress must be permitted or nothing works — pods can't resolve service names without it. This is the zero-trust approach.

🔒 Tier-Specific Policies


---
# Allow ingress to frontend from ingress controller
apiVersion: networking.k8s.io/v1
kind: NetworkPolicy
metadata:
  name: allow-frontend-ingress
  namespace: capstone
spec:
  podSelector:
    matchLabels:
      tier: frontend
  policyTypes:
  - Ingress
  ingress:
  - from:
    - namespaceSelector:
        matchLabels:
          kubernetes.io/metadata.name: ingress-nginx
    ports:
    - port: 80
---
# Allow frontend to reach API
apiVersion: networking.k8s.io/v1
kind: NetworkPolicy
metadata:
  name: allow-frontend-to-api
  namespace: capstone
spec:
  podSelector:
    matchLabels:
      tier: frontend
  policyTypes:
  - Egress
  egress:
  - to:
    - podSelector:
        matchLabels:
          tier: backend
    ports:
    - port: 3000
---
# Allow API ingress from frontend
apiVersion: networking.k8s.io/v1
kind: NetworkPolicy
metadata:
  name: allow-api-ingress
  namespace: capstone
spec:
  podSelector:
    matchLabels:
      tier: backend
  policyTypes:
  - Ingress
  ingress:
  - from:
    - podSelector:
        matchLabels:
          tier: frontend
    ports:
    - port: 3000
---
# Allow API to reach database
apiVersion: networking.k8s.io/v1
kind: NetworkPolicy
metadata:
  name: allow-api-to-db
  namespace: capstone
spec:
  podSelector:
    matchLabels:
      tier: backend
  policyTypes:
  - Egress
  egress:
  - to:
    - podSelector:
        matchLabels:
          tier: database
    ports:
    - port: 5432
---
# Allow database ingress from API only
apiVersion: networking.k8s.io/v1
kind: NetworkPolicy
metadata:
  name: allow-db-ingress
  namespace: capstone
spec:
  podSelector:
    matchLabels:
      tier: database
  policyTypes:
  - Ingress
  ingress:
  - from:
    - podSelector:
        matchLabels:
          tier: backend
    ports:
    - port: 5432
Each policy explicitly allows only the necessary communication. Frontend can only reach API. API can only reach Database. Database accepts connections only from API. This is microsegmentation with Calico.

✅ Apply and Verify Network Policies


kubectl apply -f capstone/network/network-policies.yaml

# List all network policies
kubectl get networkpolicy -n capstone

# Verify: frontend can reach API
kubectl exec -n capstone deploy/frontend -- \
  wget -qO- --timeout=5 http://api:3000/

# Verify: frontend CANNOT reach database (should timeout)
kubectl exec -n capstone deploy/frontend -- \
  wget -qO- --timeout=5 http://postgres:5432/ 2>&1 || echo "BLOCKED (expected)"

# Verify: API can reach database
kubectl exec -n capstone deploy/api -- \
  wget -qO- --timeout=5 http://postgres:5432/ 2>&1 || echo "Connection refused (expected - not HTTP)"
Zero-trust networking: Default deny everything, then allow only the specific traffic flows required. This prevents lateral movement in case of a compromised pod.
Test all the policies. The frontend-to-database test should fail (timeout or connection refused). This validates that the network policies are enforcing tier boundaries correctly.

📈 Step 7

Add Observability

📈 ServiceMonitor & PrometheusRule


# monitoring/servicemonitor.yaml
apiVersion: monitoring.coreos.com/v1
kind: ServiceMonitor
metadata:
  name: api-metrics
  namespace: capstone
  labels:
    release: kube-prometheus-stack
spec:
  selector:
    matchLabels:
      app: api
      tier: backend
  endpoints:
  - port: http
    interval: 15s
    path: /metrics
---
# monitoring/prometheusrule.yaml
apiVersion: monitoring.coreos.com/v1
kind: PrometheusRule
metadata:
  name: capstone-alerts
  namespace: monitoring
  labels:
    release: kube-prometheus-stack
spec:
  groups:
  - name: capstone.rules
    rules:
    - alert: CapstoneAPIHighErrorRate
      expr: |
        sum(rate(http_requests_total{namespace="capstone",
          code=~"5.."}[5m]))
        / sum(rate(http_requests_total{namespace="capstone"}[5m]))
        > 0.05
      for: 2m
      labels:
        severity: critical
        team: platform
        namespace: capstone
      annotations:
        summary: "Capstone API error rate above 5%"
        description: >
          The Capstone API in namespace {{ $labels.namespace }}
          has an error rate of {{ $value | humanizePercentage }}.

    - alert: CapstoneAPIHighLatency
      expr: |
        histogram_quantile(0.95,
          sum(rate(http_request_duration_seconds_bucket{
            namespace="capstone"}[5m])) by (le)
        ) > 2.0
      for: 5m
      labels:
        severity: warning
        team: platform
      annotations:
        summary: "Capstone API P95 latency above 2s"
                

kubectl apply -f capstone/monitoring/servicemonitor.yaml
kubectl apply -f capstone/monitoring/prometheusrule.yaml

# Verify
kubectl get servicemonitor -n capstone
kubectl get prometheusrule -n monitoring | grep capstone
The ServiceMonitor tells Prometheus to scrape the API pods. The PrometheusRule defines alerting rules. Both need the correct labels to be picked up by the Prometheus Operator.

🌐 Step 8

Configure External Access

🌐 Ingress with TLS


# ingress/ingress.yaml
apiVersion: networking.k8s.io/v1
kind: Ingress
metadata:
  name: capstone-ingress
  namespace: capstone
  annotations:
    cert-manager.io/cluster-issuer: "letsencrypt-prod"
    nginx.ingress.kubernetes.io/ssl-redirect: "true"
    nginx.ingress.kubernetes.io/proxy-body-size: "10m"
spec:
  ingressClassName: nginx
  tls:
  - hosts:
    - capstone.lab.verisign.com
    secretName: capstone-tls
  rules:
  - host: capstone.lab.verisign.com
    http:
      paths:
      - path: /api
        pathType: Prefix
        backend:
          service:
            name: api
            port:
              number: 3000
      - path: /
        pathType: Prefix
        backend:
          service:
            name: frontend
            port:
              number: 80
                

kubectl apply -f capstone/ingress/ingress.yaml

# Check ingress
kubectl get ingress -n capstone
kubectl describe ingress capstone-ingress -n capstone
cert-manager automatically provisions the TLS certificate via the letsencrypt-prod ClusterIssuer. The capstone-tls Secret is created and managed automatically.
The Ingress routes / to the frontend and /api to the API service. TLS is handled by cert-manager. In the Verisign environment, Envoy Gateway may be used instead of nginx-ingress, with HTTPRoute resources instead of Ingress.

🛡️ Step 9

Security Hardening

🛡️ Pod Security & Namespace Labels


# Apply Pod Security Admission labels to the namespace
kubectl label namespace capstone \
  pod-security.kubernetes.io/enforce=restricted \
  pod-security.kubernetes.io/warn=restricted \
  pod-security.kubernetes.io/audit=restricted

# Verify labels
kubectl get namespace capstone --show-labels

Ensure all pods have proper security contexts:


# Required securityContext for restricted PSA
spec:
  securityContext:
    runAsNonRoot: true
    runAsUser: 1000
    fsGroup: 1000
    seccompProfile:
      type: RuntimeDefault
  containers:
  - name: app
    securityContext:
      allowPrivilegeEscalation: false
      readOnlyRootFilesystem: true
      capabilities:
        drop:
          - ALL
Note: Some images (like PostgreSQL) need writable directories. Use emptyDir volumes for temp/data paths when readOnlyRootFilesystem: true is set.
The restricted PSA profile enforces the tightest security. Students may need to update their deployments to add proper security contexts. Nginx needs writable /tmp and /var/cache/nginx directories via emptyDir.

Clean Up

🗑️ Clean Up All Resources


# Delete the entire namespace (removes all resources within it)
kubectl delete namespace capstone

# Remove PrometheusRule from monitoring namespace
kubectl delete prometheusrule capstone-alerts -n monitoring

# Verify cleanup
kubectl get all -n capstone 2>&1
# Expected: "No resources found in capstone namespace"
# or "namespaces "capstone" not found"

# Clean up local files
rm -rf capstone/
Note: Deleting the namespace removes all resources including PVCs. In production, PVC reclaim policies determine whether the underlying volume is retained or deleted.
Deleting the namespace is the cleanest way to remove everything. Remind students that PVCs with a Retain reclaim policy would preserve the underlying volume even after namespace deletion.
