🔐 Lab 5

ConfigMaps and Secrets

Intermediate Kubernetes — Hands-On Lab

This lab covers the full lifecycle of ConfigMaps and Secrets: creation, consumption as environment variables and volume mounts, hot reload behavior, and immutability. Allow 45-60 minutes.

📋 Lab Overview

Duration: 45–60 minutes

Objectives

  • Create ConfigMaps from literals and files
  • Inject configuration as environment variables and volume mounts
  • Create and consume Kubernetes Secrets
  • Observe ConfigMap hot reload behavior
  • Understand immutable ConfigMaps and Secrets
  • Learn best practices for secret management in production

Prerequisites: Cluster access verified, kubectl configured, familiarity with Pods and Deployments.

ConfigMaps and Secrets are fundamental to externalizing configuration from container images. This lab covers both the mechanics and the operational considerations. Remind students that Secrets are only base64-encoded, not encrypted, by default.

📝 Step 1

Create ConfigMaps from Literals

The simplest way to create a ConfigMap is from literal key-value pairs on the command line.

📝 Step 1: Setup and Create ConfigMap

Create the lab namespace and a ConfigMap from literal values:

# Create and switch to the lab namespace
kubectl create namespace lab05
kubectl config set-context --current --namespace=lab05
# Create a ConfigMap from literal key-value pairs
kubectl create configmap app-config \
  --from-literal=DB_HOST=postgres \
  --from-literal=DB_PORT=5432 \
  --from-literal=LOG_LEVEL=info \
  --from-literal=CACHE_TTL=300
# Inspect the ConfigMap
kubectl get configmap app-config -o yaml

Verify: You should see all four key-value pairs under the data field.

Show the YAML output and point out the data section. Each literal becomes a key-value pair. The keys become the environment variable names when injected into pods.

📄 Step 2

Create ConfigMap from File

ConfigMaps can also be created from entire files. The filename becomes the key and the file content becomes the value.

📄 Step 2: Create a Properties File

First, create a local configuration file:

# Create an application.properties file
cat <<'EOF' > application.properties
# Application Configuration
spring.datasource.url=jdbc:postgresql://postgres:5432/mydb
spring.datasource.driver-class-name=org.postgresql.Driver
spring.jpa.hibernate.ddl-auto=validate
server.port=8080
logging.level.root=INFO
logging.level.com.verisign=DEBUG
management.endpoints.web.exposure.include=health,metrics,prometheus
EOF
# Create ConfigMap from the file
kubectl create configmap app-file-config \
  --from-file=application.properties

# Verify
kubectl get configmap app-file-config -o yaml

Notice: The filename application.properties becomes the key, and the entire file content is the value.

Point out the difference from --from-literal: the key is the filename, and the value is multi-line content. You can override the key name with --from-file=custom-key=application.properties.

⚙️ Step 3

ConfigMap as Environment Variables

There are two patterns for injecting ConfigMap data as environment variables: envFrom (all keys) and valueFrom (individual keys).

⚙️ Step 3: Pod with envFrom

Create env-pod.yaml that loads all ConfigMap keys as env vars:

apiVersion: v1
kind: Pod
metadata:
  name: env-demo
  namespace: lab05
spec:
  containers:
  - name: app
    image: busybox:1.36
    command: ["sleep", "3600"]
    envFrom:
    - configMapRef:
        name: app-config
    env:
    - name: APP_NAME
      valueFrom:
        configMapKeyRef:
          name: app-config
          key: DB_HOST
          # Remap DB_HOST to APP_NAME
kubectl apply -f env-pod.yaml
envFrom injects ALL keys from the ConfigMap. The env section with valueFrom lets you cherry-pick individual keys and optionally rename them. Both patterns are common in practice.

⚙️ Step 3: Verify Environment Variables

Exec into the pod and check the environment:

# Check all config values
kubectl exec env-demo -- env | grep -E \
  'DB_HOST|DB_PORT|LOG_LEVEL|CACHE_TTL|APP_NAME'

Expected output:

DB_HOST=postgres
DB_PORT=5432
LOG_LEVEL=info
CACHE_TTL=300
APP_NAME=postgres
Key insight: envFrom loads all keys. valueFrom.configMapKeyRef lets you select individual keys and rename them.
Both DB_HOST and APP_NAME contain "postgres" because APP_NAME was mapped from the DB_HOST key. This demonstrates how you can adapt ConfigMap keys to match what your application expects.

📁 Step 4

ConfigMap as Volume Mount

Volume-mounted ConfigMaps project each key as a file in a directory. This is ideal for configuration files that applications read from disk.

📁 Step 4: Volume Mount YAML

Create volume-pod.yaml:

apiVersion: v1
kind: Pod
metadata:
  name: volume-demo
  namespace: lab05
spec:
  containers:
  - name: app
    image: busybox:1.36
    command: ["sleep", "3600"]
    volumeMounts:
    - name: config-volume
      mountPath: /etc/config
      readOnly: true
  volumes:
  - name: config-volume
    configMap:
      name: app-file-config
kubectl apply -f volume-pod.yaml
The ConfigMap is mounted as a volume. Each key in the ConfigMap becomes a file in /etc/config. The file name is the key and the file content is the value.

📁 Step 4: Verify Mounted Files

Check that the configuration file exists in the container:

# List files in the mount path
kubectl exec volume-demo -- ls -la /etc/config/

# Read the configuration file
kubectl exec volume-demo -- cat \
  /etc/config/application.properties

Expected: The full contents of application.properties should appear, matching what you created earlier.

Volume Mount Behavior

  • Each ConfigMap key becomes a file in the mount directory
  • The mount directory is managed by Kubernetes (existing files are hidden)
  • Use subPath to mount a single key without hiding other files
Important subtlety: mounting a ConfigMap to a directory replaces the directory contents entirely. If the container has existing files at /etc/config, they will be hidden. Use subPath to mount individual files alongside existing content.

🔒 Step 5

Create Secrets

Secrets are structurally similar to ConfigMaps but intended for sensitive data. Values are base64-encoded in the API but not encrypted by default.

🔒 Step 5: Create a Generic Secret

# Create a secret from literal values
kubectl create secret generic db-creds \
  --from-literal=username=admin \
  --from-literal=password='s3cure!P@ss'
# View the secret (values are base64-encoded)
kubectl get secret db-creds -o yaml

Notice: The values are base64-encoded, not encrypted. Anyone with API access can decode them.

# Decode values using jsonpath
kubectl get secret db-creds \
  -o jsonpath='{.data.username}' | base64 -d; echo

kubectl get secret db-creds \
  -o jsonpath='{.data.password}' | base64 -d; echo
Emphasize that base64 is encoding, not encryption. In production, RBAC restricts who can read secrets, and encryption at rest should be enabled on the etcd datastore. Verisign uses Vault via the External Secrets Operator for production secrets.

🔓 Step 6

Consume Secrets in Pods

Secrets can be consumed the same two ways as ConfigMaps: environment variables and volume mounts.

🔓 Step 6: Secret Pod YAML

Create secret-pod.yaml using both env vars and volume mount:

apiVersion: v1
kind: Pod
metadata:
  name: secret-demo
  namespace: lab05
spec:
  containers:
  - name: app
    image: busybox:1.36
    command: ["sleep", "3600"]
    env:
    - name: DB_USERNAME
      valueFrom:
        secretKeyRef:
          name: db-creds
          key: username
    - name: DB_PASSWORD
      valueFrom:
        secretKeyRef:
          name: db-creds
          key: password
    volumeMounts:
    - name: secret-volume
      mountPath: /etc/secrets
      readOnly: true
  volumes:
  - name: secret-volume
    secret:
      secretName: db-creds
kubectl apply -f secret-pod.yaml
This pod demonstrates both consumption patterns simultaneously. In practice, volume mounts are slightly more secure because environment variables can leak through process listings and crash dumps.

🔓 Step 6: Verify Secret Injection

Verify both environment variables and mounted files:

# Check environment variables (values are decoded automatically)
kubectl exec secret-demo -- env | grep DB_

# Check mounted secret files
kubectl exec secret-demo -- ls /etc/secrets/
kubectl exec secret-demo -- cat /etc/secrets/username
kubectl exec secret-demo -- cat /etc/secrets/password

Key point: Kubernetes automatically base64-decodes the values when injecting them into pods, whether as env vars or files.

Security note: Volume-mounted secrets use tmpfs (in-memory filesystem) and are never written to disk on the node.
The automatic decoding means your application never needs to deal with base64. The tmpfs mount is a security feature: secret data exists only in memory on the node, not on disk. Point out that env vars, however, may appear in /proc filesystem.

🔄 Step 7

ConfigMap Hot Reload

One of the most practical features of volume-mounted ConfigMaps is that changes propagate automatically without restarting the pod. This step demonstrates that behavior.

🔄 Step 7: Update a ConfigMap Value

Modify the ConfigMap and watch the volume update:

# Check current value
kubectl exec volume-demo -- cat \
  /etc/config/application.properties | head -3
# Update the ConfigMap
kubectl create configmap app-file-config \
  --from-literal=application.properties="\
# UPDATED Configuration
spring.datasource.url=jdbc:postgresql://postgres-ha:5432/mydb
logging.level.root=DEBUG
server.port=8080" \
  --dry-run=client -o yaml | kubectl apply -f -
# Wait ~60 seconds, then check the mounted file
kubectl exec volume-demo -- cat \
  /etc/config/application.properties

The file content updates automatically! Kubernetes syncs volume-mounted ConfigMaps approximately every 60 seconds.

The kubelet periodically checks for ConfigMap updates and refreshes the mounted files. The default sync period is about 60 seconds but can vary. If the file hasn't updated yet, wait a bit longer. Note: subPath mounts do NOT receive updates.

🔄 Step 7: Env Vars Do NOT Hot Reload

✅ Volume Mounts

  • Auto-update ~60 seconds
  • No pod restart needed
  • App must re-read the file
  • Exception: subPath mounts do not update

❌ Environment Variables

  • Set at pod startup only
  • Require pod restart to update
  • Use kubectl rollout restart
# To update env vars, restart the deployment
kubectl rollout restart deployment <name>
Design decision: Use volume mounts for configuration that may change. Use env vars for values that are stable across the pod lifetime.
This is a critical operational distinction. Many teams have been caught by this: they update a ConfigMap expecting environment variables to change, but nothing happens until the pod restarts. Design your applications to read from files if you need hot reload.

🔒 Step 8

Immutable ConfigMaps and Secrets

Kubernetes supports marking ConfigMaps and Secrets as immutable, which provides both performance and safety benefits.

🔒 Step 8: Create an Immutable ConfigMap

Create immutable-config.yaml:

apiVersion: v1
kind: ConfigMap
metadata:
  name: app-config-v2
  namespace: lab05
data:
  DB_HOST: "postgres-ha"
  LOG_LEVEL: "warn"
  CACHE_TTL: "600"
immutable: true
kubectl apply -f immutable-config.yaml
# Try to update it (this will FAIL)
kubectl patch configmap app-config-v2 \
  -p '{"data":{"LOG_LEVEL":"debug"}}'

Error: Invalid value: "Immutable": field is immutable

Immutable ConfigMaps cannot be modified after creation. To change values, you must create a new ConfigMap with a different name (e.g., app-config-v3) and update the pod spec. This is a safety mechanism that prevents accidental changes and improves API server performance by eliminating watch overhead.

🔒 Step 8: Benefits of Immutability

Why Use Immutable ConfigMaps and Secrets?

  • Performance: The API server skips watch events for immutable objects, reducing load at scale
  • Safety: Prevents accidental updates that could cause widespread outages
  • Versioning: Forces a versioned naming strategy (e.g., app-config-v1, app-config-v2)
  • Rollback: Old versions remain available for quick rollback
Pattern: Name immutable configs with a version suffix or content hash. Update the Deployment to reference the new version, triggering a controlled rollout.
This pattern works well with GitOps (FluxCD in the Verisign stack). The ConfigMap version is in the Git repo, and changes trigger a new Deployment rollout. Helm and Kustomize can automate the hash-based naming.

💡 Step 9

Best Practices

This section covers production-grade patterns for managing configuration and secrets, particularly in the Verisign environment.

💡 Step 9: Production Secret Management

🛠️ RBAC for Secrets

  • Restrict get/list on secrets
  • Separate roles for read vs. write
  • Audit secret access with logging
  • Use kubectl auth can-i to verify

🛡️ Encryption at Rest

  • EKS encrypts etcd with AWS KMS
  • Enable envelope encryption
  • Secrets stored encrypted on disk
  • Transparent to applications

Verisign Stack: In production, Verisign uses the External Secrets Operator (ESO) with HashiCorp Vault to sync secrets into Kubernetes. This keeps the source of truth outside the cluster and provides audit trails, rotation, and dynamic secrets.

ESO creates Kubernetes Secrets from external providers like Vault. Developers define ExternalSecret resources that reference Vault paths. The operator handles syncing, rotation, and lifecycle management. This avoids storing secrets in Git entirely.

💡 Step 9: External Secrets Operator Pattern

How ESO + Vault Works at Verisign

# ExternalSecret resource (conceptual)
apiVersion: external-secrets.io/v1beta1
kind: ExternalSecret
metadata:
  name: db-credentials
spec:
  refreshInterval: 1h
  secretStoreRef:
    name: vault-backend
    kind: ClusterSecretStore
  target:
    name: db-creds         # Creates this K8s Secret
    creationPolicy: Owner
  data:
  - secretKey: username
    remoteRef:
      key: secret/data/myapp/db
      property: username
  - secretKey: password
    remoteRef:
      key: secret/data/myapp/db
      property: password
Result: ESO creates and manages a standard Kubernetes Secret. Your pods consume it exactly as you practiced in Step 6 — no application changes needed.
This is a conceptual example showing the ESO pattern. The ExternalSecret references a Vault path, and ESO creates a regular Kubernetes Secret that pods consume normally. The refreshInterval handles automatic rotation. This is how production secrets are managed at Verisign.

📚 Summary

ConfigMap vs. Secret Comparison

Feature ConfigMap Secret
Purpose Non-sensitive configuration Sensitive data (creds, tokens, keys)
Storage Plain text in etcd Base64-encoded (encrypted at rest on EKS)
Size limit 1 MiB 1 MiB
Env var injection configMapRef / configMapKeyRef secretRef / secretKeyRef
Volume mount Regular filesystem tmpfs (in-memory)
Hot reload (volume) Yes (~60s) Yes (~60s)
Hot reload (env) No (pod restart) No (pod restart)
Walk through the comparison table. The key takeaway is that ConfigMaps and Secrets have nearly identical mechanics, but Secrets have additional protections: base64 encoding, tmpfs mounts, RBAC restrictions, and encryption at rest.
