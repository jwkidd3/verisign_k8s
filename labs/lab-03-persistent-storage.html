💾 Lab 3: Working with Persistent Storage

Intermediate Kubernetes — Hands-On Lab

Duration: ~40 minutes

This lab covers the full PV lifecycle on EKS with gp2/gp3 EBS volumes. Emphasize that storage is where stateless and stateful workloads diverge — this is essential knowledge for running databases, caches, and file-based applications.

📋 Lab Overview

Objectives

  • Explore StorageClasses available on EKS
  • Create a PersistentVolumeClaim and understand the binding lifecycle
  • Mount persistent storage into a running application
  • Verify data persists across pod restarts
  • Compare persistent volumes with ephemeral emptyDir volumes

⏱ Duration

~40 minutes

✅ Prerequisites

  • kubectl configured with cluster access
  • EBS CSI driver installed on EKS
  • At least one StorageClass available
On EKS, the Amazon EBS CSI driver must be installed as an addon. Verify with `kubectl get pods -n kube-system -l app.kubernetes.io/name=aws-ebs-csi-driver`. If not present, the PVC will stay Pending indefinitely.

▶ Step 1

Explore Storage Classes

🗃️ Explore Storage Classes

List all available StorageClasses


kubectl get storageclasses

Describe the default StorageClass


kubectl describe storageclass gp2

📜 Key StorageClass Fields

  • provisioner: kubernetes.io/aws-ebs or ebs.csi.aws.com — the driver that creates volumes
  • reclaimPolicy: Delete (default) or Retain — what happens to the PV when the PVC is deleted
  • volumeBindingMode: WaitForFirstConsumer — delays binding until a pod is scheduled (AZ-aware)
  • parameters.type: gp2 or gp3 — the EBS volume type
Explain WaitForFirstConsumer — this is critical on EKS because EBS volumes are AZ-specific. If the PV binds to us-east-1a but the pod lands in us-east-1b, it cannot mount. WaitForFirstConsumer solves this by deferring PV creation until scheduling.

▶ Step 2

Create a PersistentVolumeClaim

📄 PVC Manifest

Create lab03-pvc.yaml:


apiVersion: v1
kind: PersistentVolumeClaim
metadata:
  name: lab03-data-pvc
  labels:
    app: lab03-storage
spec:
  accessModes:
    - ReadWriteOnce
  storageClassName: gp2
  resources:
    requests:
      storage: 5Gi
                

kubectl apply -f lab03-pvc.yaml
kubectl get pvc lab03-data-pvc
⚠️ Note: With WaitForFirstConsumer binding mode, the PVC will show Pending until a pod that references it is scheduled. This is expected behavior — not an error.
Walk through access modes: ReadWriteOnce (single node), ReadOnlyMany (multiple nodes read-only), ReadWriteMany (multiple nodes read-write — not supported by EBS). EBS only supports RWO because it's a block device attached to one EC2 instance.

🔄 PVC Lifecycle

PersistentVolumeClaim States

  • Pending — Waiting for a PV to be provisioned or bound
  • Bound — Successfully attached to a PV
  • Lost — The underlying PV was deleted (rare)

PVC Created
    │
    ▼
 Pending ──────► Bound
 (waiting for     (PV provisioned
  first consumer)  and attached)
    │
    ▼
  Lost (if PV
  is deleted)
On EKS with dynamic provisioning, you rarely see the Lost state. It can happen if an admin manually deletes the PV or the underlying EBS volume. Mention that Vault and cert-manager on the Verisign platform also use PVCs for storing secrets and certificates.

▶ Step 3

Deploy a Stateful Application

📦 Stateful Deployment Manifest

Create lab03-deployment.yaml:


apiVersion: apps/v1
kind: Deployment
metadata:
  name: lab03-storage-app
  labels:
    app: lab03-storage
spec:
  replicas: 1
  selector:
    matchLabels:
      app: lab03-storage
  template:
    metadata:
      labels:
        app: lab03-storage
    spec:
      containers:
      - name: app
        image: nginx:1.25-alpine
        ports:
        - containerPort: 80
        volumeMounts:
        - name: persistent-data
          mountPath: /usr/share/nginx/html
        resources:
          requests:
            cpu: 100m
            memory: 64Mi
          limits:
            cpu: 250m
            memory: 128Mi
      volumes:
      - name: persistent-data
        persistentVolumeClaim:
          claimName: lab03-data-pvc
Point out the volume/volumeMount relationship: volumes are defined at the pod level, volumeMounts are defined at the container level. The claimName must match the PVC name exactly.

🚀 Apply and Verify


kubectl apply -f lab03-deployment.yaml

Check the PVC status — it should now be Bound


kubectl get pvc lab03-data-pvc

Verify the pod is running with the volume mounted


kubectl get pods -l app=lab03-storage
kubectl describe pod -l app=lab03-storage | grep -A5 "Volumes:"
✅ Expected: PVC is Bound, pod is Running, and the Volumes section shows the PVC is mounted.
If the pod stays in Pending, check `kubectl describe pod` for scheduling errors. Common issue: no nodes in the same AZ as the PV. With WaitForFirstConsumer this is rare, but can happen in constrained clusters.

▶ Step 4

Write Data to Persistent Storage

✍️ Write Data to the Volume

Exec into the pod and create files


# Get the pod name
POD_NAME=$(kubectl get pod -l app=lab03-storage \
  -o jsonpath='{.items[0].metadata.name}')

# Create an HTML file on the persistent volume
kubectl exec $POD_NAME -- sh -c \
  'echo "<h1>Hello from persistent storage!</h1>" \
   > /usr/share/nginx/html/index.html'

# Create additional test data
kubectl exec $POD_NAME -- sh -c \
  'echo "Lab 3 data written at $(date)" \
   > /usr/share/nginx/html/data.txt'

Verify the data exists


kubectl exec $POD_NAME -- cat /usr/share/nginx/html/index.html
kubectl exec $POD_NAME -- cat /usr/share/nginx/html/data.txt
kubectl exec $POD_NAME -- ls -la /usr/share/nginx/html/
✅ Expected: Both files display their content correctly. The volume is mounted at /usr/share/nginx/html and writable.
This is straightforward but sets up the critical next step. Make sure everyone verifies the data exists before moving on to the persistence test.

▶ Step 5

Test Data Persistence

💥 Delete the Pod (Not the Deployment)

Delete the running pod


kubectl delete pod $POD_NAME

Watch the Deployment create a replacement


kubectl get pods -l app=lab03-storage -w
💡 Key Point: The Deployment controller detects the missing pod and immediately creates a replacement. The new pod gets the same PVC mounted — the data survives.
This is the "proof" moment. The pod is ephemeral but the data is not. Emphasize that this is why we separate compute from storage in Kubernetes. Ask: "What would happen if we deleted the Deployment instead?"

✅ Verify Data Survived

Get the new pod name and check the data


# Get the new pod name
NEW_POD=$(kubectl get pod -l app=lab03-storage \
  -o jsonpath='{.items[0].metadata.name}')

# Verify data persisted
kubectl exec $NEW_POD -- cat /usr/share/nginx/html/index.html
kubectl exec $NEW_POD -- cat /usr/share/nginx/html/data.txt
✅ Expected: Both files are present with their original content. The data survived the pod deletion because it lives on the EBS volume, not in the container filesystem.

Optional: Test via port-forward


kubectl port-forward $NEW_POD 8080:80
# In another terminal: curl http://localhost:8080
The curl should return the custom HTML page. This confirms nginx is serving content from the persistent volume. Great visual proof of persistence.

▶ Step 6

Ephemeral Volumes — emptyDir

💧 emptyDir Example

Create lab03-emptydir.yaml to compare with persistent storage:


apiVersion: v1
kind: Pod
metadata:
  name: lab03-emptydir
  labels:
    app: lab03-emptydir
spec:
  containers:
  - name: writer
    image: busybox:1.36
    command: ['sh', '-c',
      'echo "Written at $(date)" > /data/test.txt
       && cat /data/test.txt
       && sleep 3600']
    volumeMounts:
    - name: scratch
      mountPath: /data
  volumes:
  - name: scratch
    emptyDir: {}
                

kubectl apply -f lab03-emptydir.yaml
emptyDir is created when the pod is assigned to a node and destroyed when the pod is removed. It's useful for scratch space, caching, and inter-container communication within a pod.

💣 Demonstrate Data Loss

Verify data exists


kubectl exec lab03-emptydir -- cat /data/test.txt

Delete and recreate the pod


kubectl delete pod lab03-emptydir
kubectl apply -f lab03-emptydir.yaml

# Wait for the new pod to be running
kubectl wait --for=condition=Ready pod/lab03-emptydir --timeout=60s

Check — the original data is gone


kubectl exec lab03-emptydir -- cat /data/test.txt
❌ Result: The file contains a new timestamp. The original data was lost when the pod was deleted. This is the fundamental difference between emptyDir and PersistentVolumeClaim.
This contrast is powerful — the PVC data survived pod deletion, the emptyDir data did not. Ask: "When would you intentionally choose emptyDir over a PVC?" Answers: temp files, build caches, sidecar communication.

📋 Volume Type Comparison

Feature emptyDir PVC (EBS)
Lifecycle Tied to pod Independent of pod
Data persistence Lost on pod deletion Survives pod deletion
Provisioning Automatic (node disk) Dynamic via StorageClass
Capacity Shared with node disk Dedicated EBS volume
Access modes Pod-local only ReadWriteOnce (EBS)
Use cases Scratch space, caching Databases, file stores
Cost Free (node disk) EBS pricing applies
This table is a great reference. Mention that there are other volume types too — configMap, secret, hostPath, CSI-based network file systems (EFS for ReadWriteMany). We focus on EBS because it's the most common on the Verisign platform.

▶ Step 7

Clean Up

🗑️ Clean Up Resources

Delete all lab resources


# Delete the stateful deployment
kubectl delete -f lab03-deployment.yaml

# Delete the emptyDir pod
kubectl delete -f lab03-emptydir.yaml

# Delete the PVC (this triggers PV deletion with reclaimPolicy: Delete)
kubectl delete -f lab03-pvc.yaml

Verify PV cleanup


# PVC should be gone
kubectl get pvc lab03-data-pvc

# The PV should also be deleted (reclaimPolicy: Delete)
kubectl get pv | grep lab03
💡 reclaimPolicy behavior:
  • Delete (default) — PV and underlying EBS volume are automatically deleted
  • Retain — PV becomes Released and the EBS volume persists (manual cleanup required)
Emphasize that reclaimPolicy: Delete means the EBS volume is gone forever. For production databases, you might want Retain so you can recover data. Kyverno policies on the Verisign platform may enforce specific reclaim policies for certain namespaces.
