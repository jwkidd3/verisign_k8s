🌐 Lab 6

Configuring External Access

Intermediate Kubernetes — Hands-On Lab

This lab covers Ingress resources, the Gateway API, TLS termination, and traffic splitting. Students will configure path-based and host-based routing, set up TLS, and explore the Gateway API as the next-generation approach to external access. Allow 45-60 minutes.

📋 Lab Overview

Duration: 45–60 minutes

Objectives

  • Deploy sample applications with services for routing
  • Configure path-based and host-based Ingress routing
  • Set up TLS termination with self-signed certificates
  • Explore useful Ingress annotations
  • Understand the Gateway API and HTTPRoute resources
  • Configure traffic splitting for canary deployments

Prerequisites: Cluster access verified, kubectl configured, familiarity with Services (Lab 4).

This lab builds directly on the Services lab. Students should understand ClusterIP services before working with Ingress. If the NGINX Ingress Controller is not pre-installed, Step 2 covers installation.

🚀 Step 1

Deploy Sample Applications

We deploy two distinct applications so we can demonstrate routing traffic to different backends based on path or hostname.

🚀 Step 1: Create Namespace and Deployments

Create lab06-apps.yaml:

apiVersion: v1
kind: Namespace
metadata:
  name: lab06
---
apiVersion: apps/v1
kind: Deployment
metadata:
  name: nginx-blue
  namespace: lab06
spec:
  replicas: 2
  selector:
    matchLabels:
      app: nginx-blue
  template:
    metadata:
      labels:
        app: nginx-blue
    spec:
      containers:
      - name: nginx
        image: hashicorp/http-echo:0.2.3
        args: ["-text=BLUE environment", "-listen=:8080"]
        ports:
        - containerPort: 8080
---
apiVersion: apps/v1
kind: Deployment
metadata:
  name: nginx-green
  namespace: lab06
spec:
  replicas: 2
  selector:
    matchLabels:
      app: nginx-green
  template:
    metadata:
      labels:
        app: nginx-green
    spec:
      containers:
      - name: nginx
        image: hashicorp/http-echo:0.2.3
        args: ["-text=GREEN environment", "-listen=:8080"]
        ports:
        - containerPort: 8080
kubectl apply -f lab06-apps.yaml
kubectl config set-context --current --namespace=lab06
We use http-echo with different text responses so students can easily verify which backend is serving each request. Two replicas each demonstrate that Ingress routes to the service, which load balances across pods.

🚀 Step 1: Create Services

Create lab06-services.yaml:

apiVersion: v1
kind: Service
metadata:
  name: blue-svc
  namespace: lab06
spec:
  selector:
    app: nginx-blue
  ports:
  - port: 80
    targetPort: 8080
---
apiVersion: v1
kind: Service
metadata:
  name: green-svc
  namespace: lab06
spec:
  selector:
    app: nginx-green
  ports:
  - port: 80
    targetPort: 8080
kubectl apply -f lab06-services.yaml

Verify: kubectl get pods,svc -n lab06 — you should see 4 pods and 2 services.

Both services are ClusterIP (default type). Ingress will route external traffic to these internal services. Verify pods are Running before proceeding.

🛠️ Step 2

Deploy NGINX Ingress Controller

An Ingress resource requires an Ingress Controller to function. NGINX Ingress Controller is the most widely used implementation.

🛠️ Step 2: Install via Helm

Install the NGINX Ingress Controller (skip if pre-installed):

# Add the ingress-nginx Helm repository
helm repo add ingress-nginx \
  https://kubernetes.github.io/ingress-nginx
helm repo update
# Install the controller
helm install ingress-nginx ingress-nginx/ingress-nginx \
  --namespace ingress-nginx \
  --create-namespace \
  --set controller.service.type=LoadBalancer
# Verify the controller is running
kubectl get pods -n ingress-nginx
kubectl get svc -n ingress-nginx

Wait for the EXTERNAL-IP of the ingress-nginx-controller service to populate. This is the entry point for all Ingress traffic.

# Store the Ingress controller's external address
INGRESS_HOST=$(kubectl get svc ingress-nginx-controller \
  -n ingress-nginx \
  -o jsonpath='{.status.loadBalancer.ingress[0].hostname}')
If the controller is already installed in the lab environment, students can skip the helm install and just verify the pods and service. The EXTERNAL-IP may take 2-3 minutes to provision on EKS. The controller creates an NLB that handles all Ingress traffic.

🔌 Step 3

Create Path-Based Ingress

Path-based routing directs traffic to different backends based on the URL path. This is the most common Ingress pattern.

🔌 Step 3: Path-Based Ingress YAML

Create path-ingress.yaml:

apiVersion: networking.k8s.io/v1
kind: Ingress
metadata:
  name: path-based-ingress
  namespace: lab06
  annotations:
    nginx.ingress.kubernetes.io/rewrite-target: /
spec:
  ingressClassName: nginx
  rules:
  - http:
      paths:
      - path: /blue
        pathType: Prefix
        backend:
          service:
            name: blue-svc
            port:
              number: 80
      - path: /green
        pathType: Prefix
        backend:
          service:
            name: green-svc
            port:
              number: 80
kubectl apply -f path-ingress.yaml
The rewrite-target annotation strips the path prefix before forwarding to the backend. Without it, the backend would receive /blue or /green in the request path. ingressClassName: nginx binds this Ingress to the NGINX controller.

🔌 Step 3: Test Path-Based Routing

# Test the /blue path
curl -s http://$INGRESS_HOST/blue

# Test the /green path
curl -s http://$INGRESS_HOST/green

Expected:

/blue  → "BLUE environment"
/green → "GREEN environment"
Single entry point, multiple backends. One LoadBalancer serves all Ingress routes, unlike the one-LB-per-service model from Lab 4.
This is a key advantage of Ingress over LoadBalancer services: cost savings by sharing a single LB. On EKS, each LoadBalancer service creates a separate NLB, which costs money. Ingress consolidates all routes behind one LB.

🏠 Step 4

Host-Based Routing

Host-based routing uses the HTTP Host header to route traffic to different backends. This is how virtual hosting works.

🏠 Step 4: Host-Based Ingress YAML

Create host-ingress.yaml:

apiVersion: networking.k8s.io/v1
kind: Ingress
metadata:
  name: host-based-ingress
  namespace: lab06
spec:
  ingressClassName: nginx
  rules:
  - host: blue.lab.local
    http:
      paths:
      - path: /
        pathType: Prefix
        backend:
          service:
            name: blue-svc
            port:
              number: 80
  - host: green.lab.local
    http:
      paths:
      - path: /
        pathType: Prefix
        backend:
          service:
            name: green-svc
            port:
              number: 80
kubectl apply -f host-ingress.yaml
Each rule has a host field that matches the HTTP Host header. This allows multiple domains to share the same IP address and Ingress controller.

🏠 Step 4: Test Host-Based Routing

Use the Host header to simulate DNS:

# Route via blue.lab.local
curl -s -H "Host: blue.lab.local" http://$INGRESS_HOST/

# Route via green.lab.local
curl -s -H "Host: green.lab.local" http://$INGRESS_HOST/

Expected:

Host: blue.lab.local  → "BLUE environment"
Host: green.lab.local → "GREEN environment"

In production

DNS records (via External DNS or manual entries) point blue.example.com and green.example.com to the Ingress controller's external IP. The -H "Host:" flag simulates this without DNS changes.

In real environments, you would configure DNS A or CNAME records pointing to the Ingress controller. External DNS (used in the Verisign stack) can automate this by watching Ingress resources and creating DNS records automatically.

🔒 Step 5

TLS Termination

TLS termination at the Ingress controller encrypts traffic between clients and the cluster. The Ingress controller handles the certificate and forwards decrypted traffic to backends.

🔒 Step 5: Generate a Self-Signed Certificate

# Generate a self-signed certificate
openssl req -x509 -nodes -days 365 -newkey rsa:2048 \
  -keyout tls.key -out tls.crt \
  -subj "/CN=blue.lab.local/O=Lab06"
# Create a TLS secret from the certificate
kubectl create secret tls blue-tls \
  --cert=tls.crt --key=tls.key \
  -n lab06
# Verify
kubectl get secret blue-tls -n lab06

Note: In production, cert-manager automates certificate issuance and renewal from Let's Encrypt or internal CAs. Self-signed certs are for lab purposes only.

Walk through the openssl command: -x509 creates a self-signed cert, -nodes means no passphrase on the key, -days 365 sets the validity. The TLS secret type is special in Kubernetes and requires both tls.crt and tls.key.

🔒 Step 5: TLS Ingress YAML

Create tls-ingress.yaml:

apiVersion: networking.k8s.io/v1
kind: Ingress
metadata:
  name: tls-ingress
  namespace: lab06
  annotations:
    nginx.ingress.kubernetes.io/ssl-redirect: "true"
spec:
  ingressClassName: nginx
  tls:
  - hosts:
    - blue.lab.local
    secretName: blue-tls
  rules:
  - host: blue.lab.local
    http:
      paths:
      - path: /
        pathType: Prefix
        backend:
          service:
            name: blue-svc
            port:
              number: 80
kubectl apply -f tls-ingress.yaml
The tls section references the secret and specifies which hosts it covers. ssl-redirect forces HTTP to HTTPS. Multiple hosts can share the same certificate (SAN certs) or have separate tls entries.

🔒 Step 5: Test TLS Access

# Test HTTPS access (-k skips certificate verification)
curl -sk -H "Host: blue.lab.local" \
  https://$INGRESS_HOST/

Expected: BLUE environment served over HTTPS.

# Verify the certificate details
openssl s_client -connect $INGRESS_HOST:443 \
  -servername blue.lab.local </dev/null 2>/dev/null \
  | openssl x509 -noout -subject -dates
TLS termination flow: Client → HTTPS → Ingress Controller (decrypts) → HTTP → Backend Service → Pod
The -k flag is needed because the certificate is self-signed. The openssl s_client command shows the certificate subject and validity dates. In production with cert-manager, certificates are trusted and -k is not needed.

🔧 Step 6

Ingress Annotations

Annotations extend Ingress behavior beyond what the spec provides. Different Ingress controllers support different annotations.

🔧 Step 6: Useful NGINX Ingress Annotations

🛡️ Security

# Force HTTPS redirect
nginx.ingress.kubernetes.io/
  ssl-redirect: "true"

# Enable CORS
nginx.ingress.kubernetes.io/
  enable-cors: "true"
nginx.ingress.kubernetes.io/
  cors-allow-origin: "https://app.example.com"

# Rate limiting
nginx.ingress.kubernetes.io/
  limit-rps: "10"
nginx.ingress.kubernetes.io/
  limit-burst-multiplier: "5"

⚙️ Traffic Control

# Max request body size
nginx.ingress.kubernetes.io/
  proxy-body-size: "50m"

# Timeouts
nginx.ingress.kubernetes.io/
  proxy-read-timeout: "60"
nginx.ingress.kubernetes.io/
  proxy-send-timeout: "60"

# Custom headers
nginx.ingress.kubernetes.io/
  configuration-snippet: |
    add_header X-Frame-Options
      "SAMEORIGIN";

Important: Annotations are controller-specific. NGINX, Traefik, and AWS ALB Ingress each have their own annotation sets. Always check your controller's documentation.

These are the most commonly used annotations. Rate limiting is critical for public-facing services. proxy-body-size is important for file upload endpoints. The configuration-snippet allows arbitrary NGINX config but should be used carefully.

🔮 Step 7

Gateway API

The Gateway API is the next generation of Kubernetes traffic management, designed to replace Ingress with a more expressive, role-oriented model.

🔮 Step 7: Gateway API Architecture

Three-Layer Model

  • GatewayClass: Defines the controller implementation (managed by platform team)
  • Gateway: Configures the listener (ports, protocols, TLS) — shared infrastructure
  • HTTPRoute: Defines routing rules — owned by application teams
Key advantage: Clear separation of concerns. Platform engineers manage GatewayClass and Gateway. Developers manage HTTPRoutes for their own applications.
This is a conceptual section. The Gateway API may or may not be fully installed in the lab cluster. Walk through the YAML to explain the model even if students can't apply it. The role-based model is why Verisign is adopting Gateway API with Envoy Gateway.

🔮 Step 7: GatewayClass and Gateway

Platform team resources (conceptual):

apiVersion: gateway.networking.k8s.io/v1
kind: GatewayClass
metadata:
  name: envoy-gateway
spec:
  controllerName: gateway.envoyproxy.io/gatewayclass-controller
---
apiVersion: gateway.networking.k8s.io/v1
kind: Gateway
metadata:
  name: main-gateway
  namespace: gateway-system
spec:
  gatewayClassName: envoy-gateway
  listeners:
  - name: http
    protocol: HTTP
    port: 80
    allowedRoutes:
      namespaces:
        from: All
  - name: https
    protocol: HTTPS
    port: 443
    tls:
      mode: Terminate
      certificateRefs:
      - name: wildcard-tls
    allowedRoutes:
      namespaces:
        from: All
The GatewayClass defines which controller handles the gateway (Envoy Gateway at Verisign). The Gateway defines listeners with ports, protocols, and which namespaces can attach routes. This is infrastructure that the platform team manages once.

🔮 Step 7: HTTPRoute

Application team resource (conceptual):

apiVersion: gateway.networking.k8s.io/v1
kind: HTTPRoute
metadata:
  name: color-routes
  namespace: lab06
spec:
  parentRefs:
  - name: main-gateway
    namespace: gateway-system
  hostnames:
  - "colors.lab.local"
  rules:
  - matches:
    - path:
        type: PathPrefix
        value: /blue
    backendRefs:
    - name: blue-svc
      port: 80
  - matches:
    - path:
        type: PathPrefix
        value: /green
    backendRefs:
    - name: green-svc
      port: 80

Notice: The HTTPRoute references the Gateway via parentRefs. Developers only need to know the Gateway name, not its implementation details.

The HTTPRoute is what application developers create. It references the shared Gateway and defines routing rules. This is more portable than Ingress because the routing rules are not tied to controller-specific annotations.

📐 Step 8

Traffic Splitting with Gateway API

Traffic splitting is a first-class feature of the Gateway API. This enables canary deployments and A/B testing at the routing layer.

📐 Step 8: Weighted Traffic Split

HTTPRoute with weighted backends (conceptual):

apiVersion: gateway.networking.k8s.io/v1
kind: HTTPRoute
metadata:
  name: canary-route
  namespace: lab06
spec:
  parentRefs:
  - name: main-gateway
    namespace: gateway-system
  hostnames:
  - "app.lab.local"
  rules:
  - matches:
    - path:
        type: PathPrefix
        value: /
    backendRefs:
    - name: blue-svc
      port: 80
      weight: 80
    - name: green-svc
      port: 80
      weight: 20
The weight field distributes traffic: 80% to blue (stable) and 20% to green (canary). This is built into the API spec, no annotations required. You can gradually shift weights as confidence in the canary grows.

📐 Step 8: Canary Deployment Pattern

Progressive Delivery with Traffic Splitting

Phase 1: Canary

  • blue-svc (stable): 80%
  • green-svc (canary): 20%

Monitor error rates, latency, and logs

Phase 2: Promote or Rollback

  • Promote: Shift to 0/100
  • Rollback: Shift to 100/0

Update the HTTPRoute weights

Gateway API advantage: Traffic splitting is declarative and version-controlled. Combined with FluxCD, weight changes can be automated based on metrics from Prometheus.
This is the pattern Verisign uses with Envoy Gateway and FluxCD. Argo Rollouts or Flagger can automate the weight progression based on Prometheus metrics. The key insight is that this is infrastructure-as-code for traffic management.

🏙️ Step 9

Verisign Context

This section connects what students learned to the actual Verisign platform stack.

🏙️ Step 9: The Verisign External Access Stack

How Envoy Gateway + External DNS + cert-manager Work Together

🔮 Envoy Gateway

  • Implements Gateway API
  • High-performance Envoy proxy
  • Advanced routing, retries, rate limiting
  • Observability via Prometheus metrics

🌐 External DNS

  • Watches Gateway/HTTPRoute resources
  • Auto-creates DNS records in Route 53
  • Syncs on resource create/update/delete
  • Eliminates manual DNS management

🔒 cert-manager

  • Auto-issues TLS certificates
  • Integrates with internal CA or Let's Encrypt
  • Handles renewal before expiry
  • Stores certs as K8s TLS Secrets
Developer workflow: Create a Gateway + HTTPRoute → External DNS creates the DNS record → cert-manager issues the TLS certificate → traffic flows automatically. No tickets, no manual steps.
This is the power of the Verisign platform. Developers declare their intent through Kubernetes resources, and the platform handles DNS, TLS, and routing automatically. This self-service model is why Kubernetes is a developer productivity platform, not just infrastructure.

📚 Summary

Ingress vs. Gateway API

Feature Ingress Gateway API
API maturity Stable (v1) Stable (v1, graduated 2023)
Role separation Single resource, one owner GatewayClass / Gateway / Route
Traffic splitting Annotations (controller-specific) Built-in (weight field)
Header matching Limited or via annotations Native (headers, query params)
Portability Low (annotation-dependent) High (standardized spec)
Verisign direction Legacy, being phased out Standard via Envoy Gateway
Gateway API is the future of Kubernetes traffic management. While Ingress is stable and widely deployed, the Gateway API solves its fundamental limitations around portability, role separation, and advanced routing. Verisign is standardizing on Envoy Gateway for all new services.

📚 Key Annotations Reference

  • ✅ ssl-redirect: Force HTTP to HTTPS — always enable in production
  • ✅ proxy-body-size: Set max upload size — default is often 1m
  • ✅ limit-rps: Rate limit per client IP — protect against abuse
  • ✅ rewrite-target: Strip or rewrite URL paths before forwarding
  • ✅ enable-cors: Allow cross-origin requests for APIs
These are the annotations students will use most often. Remind them that annotations are NGINX-specific. When migrating to Gateway API, these become part of the HTTPRoute or policy resources.
